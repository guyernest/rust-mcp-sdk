<objective>
Implement composable `[[stage]]` blocks in TOML config and a stage-driven scheduler in the engine that ramps VU count up and down linearly over configured durations, with per-VU selective cancellation for graceful ramp-down.

Purpose: Without load shaping, all VUs start simultaneously, which does not reflect real-world traffic patterns and makes it impossible to find the VU count where a server begins to degrade. This plan enables developers to define multi-stage load profiles (ramp-up, hold, ramp-down) that reveal how their MCP server behaves as load gradually increases.

Output: Extended `LoadTestConfig` with `Stage` struct, refactored `LoadTestEngine::run()` with stage scheduler loop, per-VU `CancellationToken::child_token()` wiring in `vu_loop`, stage-aware `DisplayState` for live terminal rendering.
</objective>

**Engine refactor (engine.rs):**

1. Define `DisplayState` struct (in engine.rs or a shared location):
```rust
#[derive(Debug, Clone)]
pub struct DisplayState {
    pub snapshot: MetricsSnapshot,
    pub stage_label: Option<String>,  // e.g., "stage 2/3"
}
```

2. Refactor `LoadTestEngine::run()` to branch on `config.has_stages()`:
   - **Stages path:** Replace the flat spawn-all-then-wait controller with a stage scheduler loop.
   - **No stages path:** Preserve existing behavior exactly (flat load, all VUs at once, optional ramp_up).

3. Stage scheduler loop (when stages are present):
   - Track `vu_tokens: Vec<CancellationToken>` for per-VU selective shutdown.
   - For each stage in `config.stage`:
     a. Compute `current_vus = active_vus.get()` and `target = stage.target_vus`.
     b. Update stage label via display state channel: `format!("stage {}/{}", i + 1, total_stages)`.
     c. If `target > current_vus` (ramp up): spawn `(target - current_vus)` VUs with linear stagger (`stage_duration / vus_to_spawn`). Each VU gets `global_cancel.child_token()` pushed to `vu_tokens`. Use `tokio::select!` with cancel check during sleep between spawns.
     d. If `target < current_vus` (ramp down): cancel `(current_vus - target)` VU tokens in LIFO order via `vu_tokens.pop().map(|t| t.cancel())`. VUs finish current iteration then exit.
     e. If `target == current_vus` (hold): just wait for remaining stage duration.
     f. Wait for remaining stage time (total stage duration minus time spent ramping). Use `tokio::select!` with cancel and ctrl+c.
   - After all stages complete, cancel remaining VUs and drain.

4. Safety timeout: `tokio::select!` with `effective_duration_secs() + 30` seconds as absolute ceiling (prevents runaway if stage timing drifts).

5. Use `watch::channel(DisplayState)` instead of `watch::channel(MetricsSnapshot)` for the display channel. The metrics aggregator publishes `DisplayState { snapshot, stage_label: current_stage_label.clone() }`.

6. The `ramp_up_end` for dual-recorder (report recorder excluding ramp-up) should be set to the `Instant` when the first non-ramp stage begins (i.e., after the first stage completes if it ramps from 0). For simplicity, set `ramp_up_end = test_start` when using stages (all stage data goes into report recorder since ramp data is part of the defined test shape).
