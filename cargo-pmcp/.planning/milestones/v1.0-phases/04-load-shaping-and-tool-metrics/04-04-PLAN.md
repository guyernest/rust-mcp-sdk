# Plan 04-04: Fix Breaking Point VU Count Wiring (Gap Closure)

## Gap Being Closed

**Verification gap from 04-VERIFICATION.md:**
`BreakingPointDetector.observe()` is called with hardcoded `0` for `active_vus` in both `metrics_aggregator` (line ~523) and `metrics_aggregator_with_label` (line ~610) in `engine.rs`. The `ActiveVuCounter` exists and is updated by VU tasks but is not passed into the aggregator functions. As a result, `BreakingPoint.vus` is always `0` in real test runs.

**Root cause:** Neither aggregator function accepts an `ActiveVuCounter` parameter. The counter is only cloned for the display loop.

**Impact:** Terminal warning reads "Breaking point detected at 0 VUs" and JSON report has `"vus": 0` instead of the actual VU count.

## Context

- `ActiveVuCounter` is defined in `src/loadtest/vu.rs` as a newtype around `Arc<AtomicU32>` with `.get()` method
- It is already imported in `engine.rs` (line 18: `use crate::loadtest::vu::{vu_loop, ActiveVuCounter}`)
- It is already cloned in both `run_flat()` (line 140) and `run_staged()` (line 282) as `active_vus`
- The `display_loop` already receives `active_vus.clone()` -- the aggregator functions simply need the same treatment
- Both aggregator functions are private async functions within `engine.rs`
