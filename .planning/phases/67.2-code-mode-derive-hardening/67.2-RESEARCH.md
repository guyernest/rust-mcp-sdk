# Phase 67.2: Code Mode Derive Hardening - Research

**Researched:** 2026-04-11
**Domain:** Rust proc-macro (derive), Code Mode validation pipeline, policy evaluation, JS runtime performance
**Confidence:** HIGH

## Summary

Phase 67.2 addresses 14 success criteria spanning four categories: critical derive macro fixes (policy evaluator wiring, context_from attribute, language attribute), derive warnings (HmacTokenGenerator::new Result, compile-fail tests, shared Arc pipeline), performance improvements (HashMap clone elimination, double-parse removal, async fallback reuse), and code quality cleanups (json_to_string unification, LoopContinue/LoopBreak refactoring, ValidationResponse wrapping).

The derive macro in `crates/pmcp-code-mode-derive/src/lib.rs` is 345 lines and currently generates two handler structs (`ValidateCodeHandler`, `ExecuteCodeHandler`) inside a hidden module. The critical defects are: (1) the generated `ValidateCodeHandler::handle` never calls `policy_evaluator.evaluate_operation()` -- it only calls the sync `validate_graphql_query()` which skips policy entirely, (2) the `ValidationContext` uses hardcoded placeholder strings ("anonymous", "session", "schema", "perms"), and (3) the language is hardcoded to "graphql" in both `CodeModeToolBuilder::new("graphql")` calls. These three issues make the derive macro unusable for production deployment.

**Primary recommendation:** Wave 1 tackles the three critical derive issues together since they all modify the same generated handler code. The handler must become async (calling `validate_graphql_query_async`), accept a policy evaluator Arc, call a user-specified context method, and parameterize the language string.

### Pattern 1: Darling Struct-Level Attributes for CodeModeOpts
**What:** darling's `FromDeriveInput` parses `#[code_mode(...)]` attributes at the struct level and `FromField` parses field-level attributes. Currently `CodeModeOpts` has no struct-level attributes beyond `ident` and `data`. [VERIFIED: lib.rs:89-94]
**How to extend for D-02 and SC-3:**

```rust
// Source: crates/pmcp-code-mode-derive/src/lib.rs (current) + darling docs
#[derive(Debug, FromDeriveInput)]
#[darling(attributes(code_mode))]
struct CodeModeOpts {
    ident: syn::Ident,
    data: darling::ast::Data<(), CodeModeField>,
    // NEW: struct-level attributes
    #[darling(default)]
    context_from: Option<String>,    // D-02: method name for ValidationContext
    #[darling(default)]
    language: Option<String>,        // SC-3: "graphql"|"javascript"|"sql"
}
```

darling parses `#[code_mode(context_from = "get_context", language = "graphql")]` into these Option fields automatically. If absent, they default to `None`. [VERIFIED: darling 0.23 uses `FromMeta` for named values, same pattern as pmcp-macros `McpToolArgs` at pmcp-macros/src/mcp_tool.rs:34-47]

### Pattern 2: Generated Handler Must Be Async for Policy Evaluation
**What:** The current `ValidateCodeHandler::handle` calls `self.pipeline.validate_graphql_query()` (sync). To call `policy_evaluator.evaluate_operation()`, the handler must call `validate_graphql_query_async()` instead. The `pmcp::ToolHandler::handle` method IS already async. [VERIFIED: src/server/mod.rs:203]
**Key insight:** The generated handler already uses `#[pmcp_code_mode::async_trait]` and returns a Future. Switching from the sync to async validation path requires changing one method call and wiring the policy evaluator into the handler struct.

### Pattern 3: Policy Evaluator Arc in Generated Handler
**What:** The parent struct has `policy_evaluator: Arc<dyn PolicyEvaluator>` (or `Arc<P>` where P: PolicyEvaluator). The generated `ValidateCodeHandler` needs access to call `evaluate_operation`. Two approaches:
- (a) Store `Arc<dyn PolicyEvaluator>` in the handler struct
- (b) Create the `ValidationPipeline` with the policy evaluator already set

Approach (b) is cleaner: use `ValidationPipeline::with_policy_evaluator()` [VERIFIED: validation.rs:132-147] during `register_code_mode_tools()`, then the handler's `validate_graphql_query_async()` call naturally routes through the policy evaluator. The pipeline's existing `policy_evaluator: Option<Box<dyn PolicyEvaluator>>` field handles the plumbing.

**Problem with approach (b):** `ValidationPipeline::with_policy_evaluator` takes `Box<dyn PolicyEvaluator>`, but the struct has `Arc<dyn PolicyEvaluator>`. You cannot convert `Arc<dyn T>` to `Box<dyn T>` without cloning. Two solutions:
1. Add `ValidationPipeline::with_arc_policy_evaluator(config, secret, Arc<dyn PolicyEvaluator>)` that wraps the Arc in a newtype implementing PolicyEvaluator
2. Change `ValidationPipeline.policy_evaluator` to `Option<Arc<dyn PolicyEvaluator>>` (breaking internal API but not public -- the field is private)

**Recommended:** Option 2 -- change the internal field to `Arc<dyn PolicyEvaluator>`. This is the cleanest and matches how the struct stores the evaluator. The `set_policy_evaluator` method would accept `Arc<dyn PolicyEvaluator>` instead of `Box<dyn PolicyEvaluator>`. Since `ValidationPipeline` is not `Clone`, sharing via `Arc<ValidationPipeline>` is the right pattern anyway. [ASSUMED -- no breaking public API since the field is private]

### Pattern 4: Shared Arc<ValidationPipeline> (SC-6)
**What:** Currently `register_code_mode_tools()` creates TWO separate `ValidationPipeline` instances (lib.rs:317-324). Both validate and execute handlers need token verification, so both need a pipeline. But they should share one instance via `Arc`.

```rust
// CURRENT (wasteful):
let pipeline_validate = ValidationPipeline::from_token_secret(config.clone(), &secret);
let pipeline_execute = ValidationPipeline::from_token_secret(config.clone(), &secret);

// FIXED:
let pipeline = Arc::new(ValidationPipeline::from_token_secret(config.clone(), &secret));
// Both handlers get Arc::clone(&pipeline)
```

The `ValidateCodeHandler` and `ExecuteCodeHandler` structs change from `pipeline: ValidationPipeline` to `pipeline: Arc<ValidationPipeline>`. Since `ValidationPipeline` methods take `&self`, this is transparent. [VERIFIED: validation.rs methods all take `&self`]

### Anti-Patterns to Avoid
- **Cloning ValidationPipeline instead of sharing:** Would duplicate the HMAC key material (security concern) and waste memory. Use `Arc`.
- **Making the handler sync when policy evaluation is async:** The `PolicyEvaluator::evaluate_operation()` is an async method. [VERIFIED: policy/mod.rs:40]
- **Panic in HmacTokenGenerator::new:** Currently panics on short secrets (token.rs:202-208). Must become `Result`.

## Common Pitfalls

### Pitfall 1: Breaking ToolHandler Trait by Changing Handler Signature
**What goes wrong:** If the generated `ValidateCodeHandler::handle` signature diverges from `pmcp::ToolHandler::handle`, compilation fails.
**Why it happens:** `ToolHandler::handle` takes `(args: Value, extra: RequestHandlerExtra)`. The current generated code ignores `_extra`. The new code needs `extra` for `context_from`.
**How to avoid:** Rename `_extra` to `extra` and pass it to the context method. The method name comes from the attribute.
**Warning signs:** `_extra` prefix currently suppresses unused-variable warning.

### Pitfall 2: Box vs Arc PolicyEvaluator Type Mismatch
**What goes wrong:** `ValidationPipeline::with_policy_evaluator` accepts `Box<dyn PolicyEvaluator>` but user structs typically store `Arc<dyn PolicyEvaluator>`.
**Why it happens:** The original pipeline was designed for single-owner usage; the derive macro needs shared ownership.
**How to avoid:** Add an `Arc`-accepting constructor or change the internal field type. The cleanest path is changing the internal `Option<Box<dyn PolicyEvaluator>>` to `Option<Arc<dyn PolicyEvaluator>>` in validation.rs:89.
**Warning signs:** Cannot move out of `Arc`, cannot convert `Arc<dyn T>` to `Box<dyn T>`.

### Pitfall 3: Trybuild Stderr Snapshot Brittleness
**What goes wrong:** trybuild `.stderr` files contain exact compiler output. Changing the macro error messages or upgrading Rust versions breaks the snapshots.
**Why it happens:** Rust nightly/stable version differences in error rendering.
**How to avoid:** Use `TRYBUILD=overwrite cargo test` to regenerate snapshots. Keep error messages stable. Test on the same Rust version as CI.
**Warning signs:** CI failures on trybuild tests after toolchain update.

### Pitfall 4: HashMap Clone Optimization Scope
**What goes wrong:** Changing the scope-chain pattern in eval.rs affects all array methods across both sync and async executors.
**Why it happens:** Both `PlanExecutor` (executor.rs) and the eval module's `evaluate_array_method_with_scope` share the same pattern.
**How to avoid:** The change is localized to `evaluate_array_method_with_scope` in eval.rs since it's the shared function. `PlanExecutor::execute_step` for BoundedLoop (executor.rs:2887-2888) does NOT clone -- it inserts directly into `self.variables`. Only the eval.rs array method closures clone.
**Warning signs:** Test failures in array method tests if scope is not properly restored after each element.

### Pitfall 5: LoopContinue/LoopBreak Removal Ripple Effects
**What goes wrong:** Removing `ExecutionError::LoopContinue` and `ExecutionError::LoopBreak` from the public enum is a breaking API change for anyone pattern-matching on `ExecutionError`.
**Why it happens:** These variants are `pub` in the `ExecutionError` enum.
**How to avoid:** The phase replaces them with a separate internal `StepOutcome` enum. Since `ExecutionError` is re-exported from `pmcp-code-mode::types`, this is a semver break. However, since `pmcp-code-mode` is v0.1.0, breaking changes are expected.
**Warning signs:** Any external code that matches `ExecutionError::LoopContinue` or `ExecutionError::LoopBreak`.

The current generated handler (lib.rs:188-245) calls `self.pipeline.validate_graphql_query()` (sync, no policy eval). The fix:

pub(super) struct ValidateCodeHandler {

    .map_err(|e| pmcp::Error::Internal(format!("Validation error: {}", e)))?;

### Critical Fix 2: context_from Attribute Code Generation

```rust
// User writes:
#[derive(CodeMode)]
#[code_mode(context_from = "get_context")]
struct MyServer {
    code_mode_config: CodeModeConfig,
    token_secret: TokenSecret,
    policy_evaluator: Arc<NoopPolicyEvaluator>,
    code_executor: Arc<MyExecutor>,
}

impl MyServer {
    fn get_context(&self, extra: &pmcp::RequestHandlerExtra) -> pmcp_code_mode::ValidationContext {
        pmcp_code_mode::ValidationContext::new(
            extra.auth_info.as_ref().map(|a| a.subject.as_str()).unwrap_or("anonymous"),
            extra.session_id.as_deref().unwrap_or("no-session"),
            "schema-hash-v1",
            "perms-hash-v1",
        )
    }
}
```

The generated handler needs access to the parent struct to call the method. The pattern: store an `Arc<ParentStruct>` in the handler, call `self.parent.get_context(&extra)`.

But wait -- `register_code_mode_tools` takes `&self`. To get `Arc<Self>`, the struct must already be wrapped in an Arc, or we clone the method's output. Looking at the current code (lib.rs:313-339), `register_code_mode_tools(&self, builder)` creates handler structs with cloned fields. It does NOT hold a reference to `self`.

**Key design decision:** The handler cannot hold `&self` (lifetime issues). Options:
1. Store a closure `Arc<dyn Fn(&RequestHandlerExtra) -> ValidationContext>` captured from `self` at registration time
2. Make `register_code_mode_tools` take `self: &Arc<Self>` (requires user to Arc-wrap)
3. Store the context method's output as a factory closure that captures only what it needs

**Recommended approach:** The handler stores a `context_factory: Arc<dyn Fn(&pmcp::RequestHandlerExtra) -> ValidationContext + Send + Sync>`. In `register_code_mode_tools(&self)`, the generated code creates this closure by calling `self.get_context_factory()` or by capturing relevant state. But this adds complexity.

**Simpler approach that matches the decision D-02:** Change `register_code_mode_tools` signature from `(&self, builder)` to `(self: Arc<Self>, builder)`. Then the handler holds `Arc<Self>` and calls `self.parent.get_context(&extra)` at request time.

**Problem:** This changes the public API. Current trybuild test (valid_code_mode.rs:31) calls `server.register_code_mode_tools(builder)` with a non-Arc reference.

**Pragmatic approach:** Store a clone of the context-producing closure. The generated code captures what it needs at registration time:

```rust
// Generated code when context_from = "get_context":
// At registration time, create a closure that captures self's state
let parent_arc: Arc<#struct_name> = ... // need Arc
let context_fn = {
    let parent = Arc::clone(&parent_arc);
    move |extra: &pmcp::RequestHandlerExtra| -> pmcp_code_mode::ValidationContext {
        parent.get_context(extra)
    }
};
```

**Cleanest solution:** Accept `self: &Arc<Self>` for the registration method. This is the pattern used by `pmcp-macros`'s `#[mcp_server]` which stores `Arc<ServerType>` in generated handlers. [VERIFIED: pmcp-macros/src/mcp_server.rs generates handlers with `Arc<T>` of the impl block's Self type]

For backward compatibility, provide both:
```rust
// New: context-aware registration (requires Arc)
pub fn register_code_mode_tools_arc(self: &Arc<Self>, builder: ServerBuilder) -> ServerBuilder

// Old: deprecated placeholder context (takes &self)
#[deprecated(note = "Use register_code_mode_tools_arc for production. This uses placeholder ValidationContext.")]
pub fn register_code_mode_tools(&self, builder: ServerBuilder) -> ServerBuilder
```

Actually, looking more carefully at the generate pattern -- the simplest approach that D-02 supports:

When `context_from` is NOT specified: keep current `&self` signature with placeholder context (emit deprecation warning).

When `context_from` IS specified: change signature to take `self: &Arc<Self>`, generate handler that holds `Arc<Self>` and calls `self.parent.method_name(&extra)`. This is type-safe -- if the method doesn't exist or has wrong signature, the user gets a standard Rust compile error pointing at the method call.

### Performance Fix P-01: Scope-Chain for Array Methods

The pattern at eval.rs:487-598 clones `local_vars` HashMap for every array element in `.map()`, `.filter()`, `.find()`, `.some()`, `.every()`, `.flatMap()`, `.reduce()`, `.sort()`.

```rust
// CURRENT (eval.rs:489-494, .map example):
for item in arr {
    let mut merged = local_vars.clone();  // CLONE per element!
    merged.insert(item_var.clone(), item);
    let value = evaluate_with_scope(body, global_vars, &merged)?;
    results.push(value);
}

// FIXED (scope-chain push/pop):
for item in arr {
    // Temporarily insert binding, evaluate, then remove
    let old = local_vars.insert(item_var.clone(), item);
    let value = evaluate_with_scope(body, global_vars, local_vars)?;
    // Restore previous binding (or remove if new)
    match old {
        Some(prev) => { local_vars.insert(item_var.clone(), prev); },
        None => { local_vars.remove(item_var); },
    }
    results.push(value);
}
```

**Problem:** `local_vars` is `&HashMap<String, JsonValue>` (immutable reference). The function signature at eval.rs:464 takes `local_vars: &HashMap<String, JsonValue>`. Changing to `&mut HashMap` would require changing the function signature and all callers.

**Better approach:** Change `evaluate_array_method_with_scope` to take `local_vars: &mut HashMap<String, JsonValue>`, which avoids the clone. The parent function `evaluate_with_scope` (eval.rs:59) already owns a stack of merged variables -- the change is localized.

Actually, looking more carefully at the call chain: `evaluate_with_scope` takes `local_vars: &HashMap` (immutable). It creates new HashMaps by cloning when needed. The cleanest fix is to change the array method function to accept `&mut HashMap` and use push/pop. The caller at `evaluate_with_scope` would need to clone once (to get a mutable copy), then pass `&mut` to the array method. This is still one clone per array method call instead of N clones per element.

## Open Questions

1. **register_code_mode_tools signature for context_from**
   - What we know: D-02 says the method signature is `fn get_context(&self, extra: &RequestHandlerExtra) -> ValidationContext`. The handler needs `Arc<Self>` to call this at request time.
   - What's unclear: Should the registration method signature change to `self: &Arc<Self>`, or should there be two methods?
   - Recommendation: Generate `register_code_mode_tools(self: &Arc<Self>, builder)` when `context_from` is specified, and keep the old `(&self, builder)` when not specified. The trybuild valid test should be updated to use `Arc<MyServer>`.

2. **Box-to-Arc policy evaluator migration**
   - What we know: `ValidationPipeline::with_policy_evaluator` takes `Box<dyn PolicyEvaluator>`. The derive macro user has `Arc<dyn PolicyEvaluator>`.
   - What's unclear: Whether to add a new constructor or change the existing one.
   - Recommendation: Change internal field to `Arc`, update `with_policy_evaluator` to take `Arc<dyn PolicyEvaluator>`, add backward-compat `with_boxed_policy_evaluator` that wraps `Box` in `Arc`. This is v0.1.0 so the break is acceptable.

3. **eval.rs mutability for scope-chain optimization**
   - What we know: `evaluate_array_method_with_scope` takes `&HashMap` (immutable).
   - What's unclear: How many callers need updating to pass `&mut HashMap`.
   - Recommendation: The function is only called from `evaluate_with_scope` at one site. Change the array method function to take `&mut HashMap`, have the caller clone once and pass `&mut`. Net effect: 1 clone per array method call instead of N clones per element.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | Yes (via context_from) | `RequestHandlerExtra.auth_info` / `auth_context` for user identity |
| V3 Session Management | Yes (via context_from) | `RequestHandlerExtra.session_id` for session binding |
| V4 Access Control | Yes (policy evaluator) | `PolicyEvaluator::evaluate_operation()` via Cedar or AVP |
| V5 Input Validation | Yes | GraphQL parser + `CodeModeConfig` limits (max_depth, max_query_length) |
| V6 Cryptography | Yes (HMAC tokens) | `HmacTokenGenerator` with SHA-256, 16-byte minimum secret, `secrecy`/`zeroize` |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Token replay (current placeholder context) | Spoofing | SC-2: context_from binds tokens to real user/session/schema |
| Short HMAC secret (trivially forgeable tokens) | Tampering | SC-4: HmacTokenGenerator::new returns error instead of panicking |
| Missing policy evaluation (bypass authz) | Elevation | SC-1: Generated handler calls policy evaluator |
| Cross-session token reuse | Spoofing | context_hash includes session_id from ValidationContext |
