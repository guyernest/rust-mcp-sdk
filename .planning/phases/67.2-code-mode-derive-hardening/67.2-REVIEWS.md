# Cross-AI Plan Review: Phase 67.2 Code Mode Derive Hardening

This review evaluates the 6-plan execution strategy for Phase 67.2, focusing on the hardening of the `#[derive(CodeMode)]` macro and runtime performance/quality improvements.

## 1. Summary
The plan is exceptionally well-structured, following a logical dependency-driven wave approach that prioritizes critical security and functionality fixes (Waves 1-2) before moving to hardening (Wave 3), performance (Wave 4), and quality (Wave 5). It successfully addresses the "broken security contract" identified by the `pmcp.run` team by wiring the policy evaluator into the async validation path and providing a type-safe mechanism for binding session context to tokens via the `context_from` attribute. The inclusion of `trybuild` compile-fail tests and the refactoring of "error-as-control-flow" patterns demonstrate a high level of engineering maturity.

## 2. Strengths
*   **Security-First Wave Ordering:** Addressing the policy evaluator bypass in Wave 1 ensures that the core security premise of Code Mode is functional before any other changes are applied.
*   **Type-Safe Context Binding:** The `context_from` attribute design (D-02) leveraging `self: &Arc<Self>` registration correctly solves the `'static` lifetime requirements for async handlers while providing real user/session data to the validation pipeline.
*   **Performance Wins:** P-01 (scope-chain push/pop) and P-03 (double GraphQL parse elimination) are high-impact optimizations that significantly reduce CPU and memory overhead during high-volume tool execution.
*   **Robust Error Handling:** Converting the `HmacTokenGenerator` and `ValidationPipeline` constructors to return `Result` (SC-4) prevents runtime panics on misconfigured secrets, which is vital for production stability.
*   **Comprehensive Testing:** The plan explicitly includes `trybuild` tests for both success and failure cases (missing fields), ensuring the macro's error messages remain helpful.

## 3. Concerns
*   **Breaking API Change (Registration Signature):** [MEDIUM] Changing the `register_code_mode_tools` signature to return `Result<ServerBuilder, TokenError>` (Plan 03) will break existing builder chains (e.g., `builder.tool(...).register_code_mode_tools(builder)` will no longer compile). 
    *   *Mitigation:* The plan correctly notes this is acceptable for v0.1.0, but users should be warned in a migration guide.
*   **Arc-Only Registration Requirement:** [LOW] Requiring `self: &Arc<Self>` when `context_from` is used (Plan 02) might be a friction point for users not yet using `Arc` for their server state.
    *   *Mitigation:* The plan preserves the old `&self` signature (with a deprecation warning) for the placeholder context path, maintaining backward compatibility.
*   **Internal vs. External Field Access:** [LOW] Plan 01 proposes changing `Box` to `Arc` in `ValidationPipeline`. If any user code was manually constructing or inspecting this (rare, as it's private), they would break.
    *   *Mitigation:* The plan confirms the field is private, minimizing impact.

## 4. Suggestions
*   **Macro Error Clarity:** In Plan 03 (SC-5), ensure the error messages for missing `token_secret` or `code_executor` explicitly mention that these fields are *required* by the `CodeMode` derive macro, not just missing from the struct.
*   **Clippy Baseline:** In Plan 06, perform a baseline `cargo clippy` run before making changes to ensure you aren't fighting unrelated warnings. The goal is to reduce suppressions, so having a clean "before" state is useful.
*   **Benchmark Validation:** For Plan 04 (P-01), consider adding a simple micro-benchmark (even just a `#[bench]` or a large array test) to verify the performance gain of the push/pop pattern over cloning.

## 5. Risk Assessment: LOW
The overall risk is **LOW**. The changes are surgically targeted at specific crates (`pmcp-code-mode` and `pmcp-code-mode-derive`), and the use of `trybuild` snapshots provides a high degree of confidence that macro behavior won't regress. The most complex logic (async validation with policy evaluation) is already implemented in the runtime; the plans merely wire it correctly into the code generation layer.
