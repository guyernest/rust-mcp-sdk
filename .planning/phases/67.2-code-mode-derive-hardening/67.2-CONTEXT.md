# Phase 67.2: Context Decisions

## D-01: Wave Scope — Derive-First (Revised by Planner)

**Decision:** Waves reorganized from the original 4-wave grouping for technical dependency reasons:
- Wave 1: SC-1 + SC-6 — policy evaluator wiring + shared Arc<ValidationPipeline> (foundation for all other derive changes)
- Wave 2: SC-2 + SC-3 — context_from + language attributes (depends on Wave 1's Arc changes)
- Wave 3: SC-4 + SC-5 — HmacTokenGenerator Result + trybuild tests
- Wave 4: SC-7 + SC-8 — performance (scope-chain P-01, GraphQL double-parse P-03)
- Wave 5: SC-9 through SC-13 — quality + cleanup

**Rationale:** SC-2/SC-3 (context_from attribute) needs the `Arc<Self>` pattern introduced by SC-1's policy wiring, so they can't share Wave 1. The planner split critical derive work across Waves 1-2 to respect this dependency while still unblocking pmcp.run migration earliest.

**Original proposal:** Wave 1: SC-1/SC-2/SC-3, Wave 2: SC-4/SC-5/SC-6, Wave 3: perf, Wave 4: quality. Revised because SC-6 (shared pipeline) is a prerequisite for SC-1, and SC-2/SC-3 depend on SC-1.

## D-04: P-02 (Double SWC Parse) Deferred

**Decision:** P-02 is deferred to a future phase. Requires a new `ValidatedCode { ast: Module, validation: ValidationResult }` type threading the parsed SWC AST from `javascript.rs` through to `executor.rs`. This is a cross-module interface change that is too complex for a hardening phase.

**SC-8 removed from ROADMAP.md.** P-01 and P-03 remain in scope.

## D-02: Context API — Attribute-Based Method Reference

**Decision:** `#[code_mode(context_from = "method_name")]` attribute. The struct must have a method like `fn get_context(&self, extra: &RequestHandlerExtra) -> ValidationContext`. The derive macro calls it in the generated handler.

**Rationale:** Clean, type-safe, no additional fields needed on the struct. The method has access to both `&self` (for server-side state like schema hash) and `&RequestHandlerExtra` (for request-level context like client_id, request_id).

**Default behavior (no attribute):** Keep current placeholder behavior but emit a compile-time warning via `#[deprecated]` or a doc comment. This preserves backward compatibility for testing.

## D-03: Performance Scope — Feature-Gated Only

**Decision:** P-01 (HashMap clone) and P-02 (double SWC parse) are scoped to `js-runtime` / `openapi-code-mode` feature gates only. P-03 (double GraphQL parse) is included as it affects the default feature set.

**Rationale:** No risk to default GraphQL-only users from P-01/P-02. P-03 is included because it affects every async validation call without a policy evaluator.
