<objective>
Add `#[code_mode(context_from = "method_name")]` and `#[code_mode(language = "...")]` struct-level attributes to the derive macro via darling, replacing hardcoded placeholder context and hardcoded "graphql" language string.

Purpose: The current derive macro uses placeholder ValidationContext values ("anonymous", "session", "schema", "perms"), making tokens replayable across users and sessions (SC-2). It also hardcodes "graphql" as the language, so SQL and OpenAPI servers get wrong tool schemas (SC-3). Per D-02, the fix uses attribute-based method reference.

Output: Updated derive macro with darling attribute parsing, two new trybuild compile-pass tests, updated existing compile-pass test.

Depends on Plan 01 which already changed the generated code to use `Arc<ValidationPipeline>`, `validate_graphql_query_async`, and `from_token_secret_with_policy`.
</objective>

**1. Extend CodeModeOpts with darling attributes:**

```rust
#[derive(Debug, FromDeriveInput)]
#[darling(attributes(code_mode))]
struct CodeModeOpts {
    ident: syn::Ident,
    data: darling::ast::Data<(), CodeModeField>,
    /// Optional method name for extracting ValidationContext from the struct.
    /// When specified, the generated registration method requires `self: &Arc<Self>`.
    #[darling(default)]
    context_from: Option<String>,
    /// Code language for tool metadata. Defaults to "graphql".
    #[darling(default)]
    language: Option<String>,
}
```

**2. Extract the language string** in `expand_code_mode()`, after parsing opts:

```rust
let language = opts.language.as_deref().unwrap_or("graphql");
let language_lit = syn::LitStr::new(language, struct_name.span());
```

**3. Replace hardcoded "graphql"** in both `metadata()` methods in the generated code. Change from:
```rust
let tools = pmcp_code_mode::CodeModeToolBuilder::new("graphql").build_tools();
```
To (using quote interpolation):
```rust
let tools = pmcp_code_mode::CodeModeToolBuilder::new(#language_lit).build_tools();
```

**4. Generate two different code paths based on context_from:**

Use `if let Some(ref method_name) = opts.context_from { ... } else { ... }` in the `expand_code_mode` function to produce different `proc_macro2::TokenStream` blocks.

**When `context_from` is `None` (backward compat):**
- Keep `register_code_mode_tools(&self, builder)` signature (from Plan 01)
- Keep placeholder `ValidationContext::new("anonymous", "session", "schema", "perms")`
- Add `#[deprecated(note = "Use #[code_mode(context_from = \"method_name\")] for production. This uses placeholder ValidationContext.")]` on the generated method
- Keep `_extra` parameter name in handler to suppress unused warning
- ValidateCodeHandler struct: same as Plan 01 (no parent field)

**IMPORTANT (addresses Gemini review concern):** The `&self` signature in the `None` path is the backward compatibility guarantee. Users who do NOT use `context_from` continue to call `server.register_code_mode_tools(builder)` without needing `Arc<Self>`. The `#[deprecated]` warning guides them toward the `context_from` path for production use. This dual path ensures the `Arc<Self>` requirement is ONLY imposed when `context_from` is explicitly opted into.

**When `context_from` is `Some(method_name)`:**
- Parse: `let method_ident = syn::Ident::new(method_name, struct_name.span());`
- ValidateCodeHandler gains a `parent` field:
  ```rust
  pub(super) struct ValidateCodeHandler {
      pub(super) pipeline: Arc<pmcp_code_mode::ValidationPipeline>,
      pub(super) config: pmcp_code_mode::CodeModeConfig,
      pub(super) parent: Arc<#struct_name>,
  }
  ```
- In handler `handle()`, rename `_extra` to `extra` and generate:
  ```rust
  let context = self.parent.#method_ident(&extra);
  ```
  instead of the placeholder.
- Registration method signature changes to:
  ```rust
  pub fn register_code_mode_tools(
      self: &std::sync::Arc<Self>,
      builder: pmcp::ServerBuilder,
  ) -> pmcp::ServerBuilder
  ```
- Handler construction adds `parent: std::sync::Arc::clone(self)` for validate handler

**5. Implementation approach:** Build the `expanded` TokenStream in two separate code paths. Keep the `quote! { ... }` block for the `None` case nearly identical to what Plan 01 produced. For the `Some` case, generate a modified version with the `parent` field and `self: &Arc<Self>` method. This avoids conditional logic inside `quote!` which is hard to read.

The `expand_code_mode` function structure becomes:
```rust
let expanded = if opts.context_from.is_some() {
    let method_ident = syn::Ident::new(opts.context_from.as_ref().unwrap(), struct_name.span());
    quote! {
        // Send+Sync assertion ...
        mod #mod_name {
            // Handler with parent: Arc<#struct_name>
            // handle() calls self.parent.#method_ident(&extra)
            // metadata() uses #language_lit
        }
        impl #struct_name {
            pub fn register_code_mode_tools(self: &std::sync::Arc<Self>, builder: ...) -> ... { ... }
        }
    }
} else {
    quote! {
        // Send+Sync assertion ...
        mod #mod_name {
            // Handler without parent
            // handle() uses placeholder context
            // metadata() uses #language_lit
        }
        impl #struct_name {
            #[deprecated(note = "Use #[code_mode(context_from = \"method_name\")] for production. This uses placeholder ValidationContext.")]
            pub fn register_code_mode_tools(&self, builder: ...) -> ... { ... }
        }
    }
};
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| RequestHandlerExtra -> context_from method | MCP session/auth data flows into ValidationContext |
| User-specified method -> ValidationContext | The method returns context that binds tokens to real identity |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-67.2-04 | Spoofing | ValidationContext | mitigate | SC-2: context_from attribute binds tokens to real user/session/schema from RequestHandlerExtra, preventing cross-session replay |
| T-67.2-05 | Information Disclosure | context_from method | accept | The user-defined method controls what context is exposed; derive macro trusts the method's return type (ValidationContext is a fixed struct) |
| T-67.2-06 | Elevation | language attribute | accept | Language string only affects tool schema metadata (format field), not security decisions; incorrect language doesn't bypass validation |
</threat_model>
