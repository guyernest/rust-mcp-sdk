<objective>
Close RESEARCH Gap #2: `register_code_mode_tools` (Phase 83) is shape-preserving — it builds + validates a `ValidationPipeline` (so R9/secret errors fire) but registers ZERO tools (Plan 08 was deferred). SC-3/SC-4 require `validate_code`/`execute_code` to EXIST and enforce policy. This is the core security-parity surface.

The toolkit's `SqlConnector` exposes a SINGLE `execute(sql, params)` method; the production `SqlCodeModeHandler` mirror is over a different 2-method `DatabaseConnector` trait (`execute_query`/`execute_statement`). The adapter must be RE-DERIVED for the single-method trait — re-validate the SQL (defense-in-depth) then call `connector.execute(...)` regardless of query type, since the toolkit trait has one entry point.

This plan ALSO adds the `--schema` file → code-mode prompt seam (`assemble_code_mode_prompt_with_schema`). The existing `assemble_code_mode_prompt(connector, config)` reads `connector.schema_text()` live — for non-SQLite backends that hits the network (breaking SC-1) and ignores the admin-redacted `--schema` file (D-04/D-05). All three changes live in `code_mode.rs` + the new builder_ext API, so they are co-located here (after Plan 01's `${VAR}` expansion to the same file).

**REVIEW FIX (Codex HIGH #4 — LOCK the connector-aware code-mode registration API; no alternatives).** The prior draft listed two A/B paths and a "planner decides" choice — that ambiguity is now RESOLVED. The single committed shape:
- **`try_code_mode_from_config_with_connector(self, cfg, connector: Arc<dyn SqlConnector>) -> Result<Self>`** — the binary's path. Builds the `SqlCodeExecutor` from the connector, then registers BOTH `validate_code` and `execute_code` with static `[code_mode]` policy. NO-op when `[code_mode]` is absent.
- **`try_code_mode_from_config(self, cfg) -> Result<Self>`** — KEPT, but explicitly documented as the connectorless **validation-only / no-tool** path (it builds + validates the pipeline so R9/secret errors fire, but registers no tools because there is no executor). Its existing callers compile unchanged.
- A free helper **`code_mode_tools_from_executor(builder, cfg, executor: Arc<SqlCodeExecutor>) -> Result<builder>`** in `code_mode.rs` does the actual two-tool registration; `try_code_mode_from_config_with_connector` constructs the executor and delegates to it (the Phase 83-06 R1 split precedent).
Plan 05 calls `try_code_mode_from_config_with_connector` EXPLICITLY — no A/B decision deferred to execution.

Purpose: Without registered code-mode tools, the 8 `validate_code` scenarios + `execute_code` scenario in the parity harness (Plan 06) cannot pass. Static policy enforcement IS the SC-3 claim (D-13). The file-based prompt seam is the D-05 redaction guarantee + SC-1 prerequisite.
Output: A `SqlCodeExecutor` adapter + the LOCKED `try_code_mode_from_config_with_connector` API (+ `code_mode_tools_from_executor` helper) + the file-based prompt helper + a registration/policy-enforcement test.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM-generated SQL (validate_code/execute_code) → SQL backend | untrusted code crosses into the connector; policy must gate it |
| approval token (validate_code → execute_code) | HMAC token binds code hash; replay with modified SQL must be caught |
| --schema file → public code-mode prompt | admin-redacted file is the source of truth; no live introspection bypass |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-85-02-01 | Tampering | SqlCodeExecutor::execute | mitigate | Defense-in-depth re-validation of the SQL against `[code_mode]` policy BEFORE connector.execute — guards token replay with a different statement (mirrors production SqlCodeModeHandler) |
| T-85-02-02 | Elevation of privilege | static `[code_mode]` policy | mitigate | allow_writes/allow_deletes/allow_ddl enforced statically from config (D-13); DELETE/DDL on read-only config rejected by validation pipeline — a config-driven server CANNOT bypass guards (SC-3 claim) |
| T-85-02-03 | Spoofing | HMAC approval token | mitigate | Token bound to code hash via `HmacTokenGenerator` (never hand-rolled); secret resolved via `resolve_token_secret` (>=16 bytes, T-85-01-01) |
| T-85-02-04 | Information disclosure | ConnectorError → ExecutionError mapping | mitigate | Backend error messages sanitized before surfacing (V7, T-84-04-02 precedent) — no raw internals echoed to the client |
| T-85-02-05 | Information disclosure | --schema → prompt body | mitigate | The file-based assembler (no connector.schema_text) means only the admin-redacted file content reaches the prompt; live DDL never leaks (D-04/D-05) |
</threat_model>
