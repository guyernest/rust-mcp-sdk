Close VERIFICATION Gap 2 (HIGH, test validity): the parity test masks failed
rejection scenarios. The `failure`-asserting validate scenarios in
`crates/pmcp-sql-server/tests/fixtures/generated.yaml` carry
`continue_on_failure: true`, and mcp-tester computes `ScenarioResult.success`
(`crates/mcp-tester/src/scenario_executor.rs:111-118`) by EXCLUDING any
continue_on_failure step. So a genuinely-failing rejection scenario (e.g. the
no-LIMIT `SELECT * FROM Artist` before Gap 1 was fixed) is silently dropped and
`tests/parity_chinook.rs` still passes on `assert!(result.success)`. The
negative-path parity proof is non-gating — it would stay green even if every
policy rejection regressed.

This plan strengthens `parity_chinook.rs` so the policy-rejection scenarios are
INDIVIDUALLY gating: assert every `result.step_results[i].success` is true
(regardless of `continue_on_failure`), and assert the named rejection scenarios
are present so they cannot silently disappear. The fix lives in the test, NOT in
the fixtures — fixtures are not weakened.

Depends on 85-07 (require_limit enforcement) so the no-LIMIT rejection genuinely
passes its `failure` assertion, and on 85-09 (prompt synthesis) so the
start_code_mode scenario still resolves. The acceptance proof is two-sided: with
Gap 1's fix in place the strengthened test passes; reverting Gap 1 makes the
strengthened test FAIL (demonstrating it now catches the masked regression).

Purpose: make the SC-3/SC-4 negative-path parity proof actually gating.
