**REVIEW FIX (Codex HIGH #1 — DDL-only fixture has no data; parity would FAIL).** The prior draft vendored ONLY the DDL (`sqlite3 .schema`), which creates EMPTY tables. But `generated.yaml` asserts on REAL data values — VERIFIED: `"Rock"` (line 51), `"Angus Young"` (line 62), `"AC/DC"` (lines 65/141), `"Bohemian"` (line 76), `"For Those About To Rock (We Salute You)"` (line 107). With empty tables, `search_tracks`/`list_artists`/`get_album_tracks`/`execute_code` return NO rows and the replay FAILS. **FIX:** vendor the real **`chinook.db`** (~1 MB, VERIFIED present at `reference/chinook.db` = 984 KB) into `tests/fixtures/chinook.db` so the connector returns real rows. To keep the published crate lean, add `exclude = ["tests/fixtures/chinook.db"]` (or a glob) to the crate's `Cargo.toml` so the blob is NOT shipped in the published crate. The separate `--schema` DDL file STAYS (it is the code-mode prompt/resource input, D-05/D-06) — but it is DISTINCT from the connector's database file: the `[database] file_path` points at the populated `chinook.db`, while `--schema` provides the DDL text for the prompt + schema resource.

Purpose: The parity harness can't run in CI against a fixture that lives in another repo, and a schema-only DB can't satisfy data-value assertions. Committing the populated `chinook.db` (excluded from publish) + the DDL text + the scenarios makes the test self-contained AND able to return real rows.
Output: New crate skeleton in the workspace + FOUR committed fixtures (chinook.db + chinook.ddl + generated.yaml + reference-config.toml) + a fixture-shape sanity test + a publish-exclude entry.
