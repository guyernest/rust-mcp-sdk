<objective>
Close VERIFICATION Gap 3 (MEDIUM): the assembled `start_code_mode` prompt drops
the code-mode instructions + policy content. The reference prompt's
`include_resources` lists 5 URIs (`docs://chinook/schema`, `code-mode://instructions`,
`code-mode://policies`, `docs://chinook/examples`, `code-mode://learnings`) but only
3 are declared `[[resources]]`. `StaticPromptHandler::resolve_body`
(`crates/pmcp-server-toolkit/src/prompts.rs:186-206`) warn-logs and SKIPS the two
undeclared URIs, so the served prompt omits the instructions + policy text that
production injects at startup (the reference config even documents this:
"The template system generates code-mode://instructions ... and code-mode://policies
from the [code_mode] config above").

This plan synthesizes those two resources during Shape A assembly from the
`[code_mode]` config + the dialect, merges them into the resource set the prompt
resolves against, and adds a prompt-body content assertion.

Purpose: the served `start_code_mode` prompt becomes behaviorally faithful to
production — the LLM receives the code-mode instructions and the active policy,
not a prompt missing two of its five sections.

Output: two synthesized resources (`code-mode://instructions`, `code-mode://policies`)
wired into assembly + a test asserting their text appears in the prompt body.
</objective>
