<objective>
Close VERIFICATION Gap 1 (HIGH, correctness): `[code_mode] require_limit` is parsed
but never enforced. `build_cm_config` discards it (`let _require_limit_gap = section.require_limit;`),
`pmcp-code-mode`'s `CodeModeConfig` has no require-limit concept, and the only
backstop (`estimated_rows > sql_max_rows`) never fires for a bare `SELECT` because
the unbounded estimate (1000) equals `sql_max_rows` (1000) under strict `>`, and
`UnboundedQuery` is not `is_critical()`. Empirically `validate_code("SELECT * FROM Artist")`
returns `valid:true, auto_approved:true` while production REJECTS it.

This plan wires `require_limit` through to actual enforcement (the WR-01 option-(a)
recommendation): add an additive `sql_require_limit` flag to `pmcp-code-mode`'s
`CodeModeConfig`, enforce it in the single shared SQL authorization path
(`check_sql_config_authorization`, used by BOTH `validate_sql_query` and
`validate_sql_query_async`), and map the toolkit's `section.require_limit` onto it.

Purpose: make the operator-facing `require_limit` config field deliver the safety
guarantee its doc comment promises, independent of `max_limit`. This is also the
correctness foundation Plan 85-08 (Gap 2) asserts against.

Output: a `sql_require_limit` config field + enforcement branch + toolkit mapping,
covered by unit tests in both crates.
</objective>
