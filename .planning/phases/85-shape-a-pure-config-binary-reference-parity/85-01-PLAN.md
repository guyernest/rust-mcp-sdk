<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| committed config.toml → toolkit parser | admin-authored TOML crosses into ServerConfig; malformed input must error, never panic |
| `${VAR}`/`env:VAR` → process env → HMAC secret | the resolved secret signs code-mode approval tokens; a weak/empty/missing secret is a forgery risk |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-85-01-01 | Information disclosure | `resolve_token_secret` `${VAR}` branch | mitigate | Missing/unset var returns `Err(ToolkitError::CodeMode)` — never falls back to a weak/empty secret; min length enforced downstream by `HmacTokenGenerator::MIN_SECRET_LEN` (16 bytes) |
| T-85-01-02 | Tampering | `ServerConfig` `deny_unknown_fields` | mitigate | Additive fields only; deny_unknown_fields preserved so a renamed/typo'd field is rejected at parse time (negative test `renames_rejected`) |
| T-85-01-03 | Denial of service | TOML parse of malformed config | accept | The existing `pmcp_server_toolkit_config_parser` fuzz target (Phase 84) already asserts no-panic on malformed input; the new additive fields are covered by extending its corpus (Plan 06) |
</threat_model>
