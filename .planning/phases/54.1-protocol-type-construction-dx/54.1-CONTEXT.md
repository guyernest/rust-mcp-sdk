# Phase 54.1: Protocol Type Construction DX - Context

**Gathered:** 2026-03-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix inconsistent construction patterns across all protocol types so downstream users don't break on every SDK upgrade. Add #[non_exhaustive] + Default + ::new() constructors + .with_*() fluent methods to every protocol struct. Add static construction helpers to Content enum variants. Normalize existing types (ToolInfo, CallToolResult) to follow the same pattern. Update SDK tests to dogfood constructors. Update MIGRATION.md with new construction patterns.

</domain>

<decisions>
## Implementation Decisions

### Default + constructor coverage
- **D-01:** Every protocol struct gets both `derive(Default)` AND a `::new()` constructor
- **D-02:** Constructors take required fields as parameters; all Optional fields default to None
- **D-03:** Constructors use `impl Into<String>` for string parameters (no .to_string() noise)
- **D-04:** Every protocol struct that downstream users might construct gets a constructor — systematic coverage (~15-20 types), not just the reported painful ones

### Content enum helpers
- **D-05:** Static methods on Content for all variants: `Content::text()`, `Content::image()`, `Content::resource()`, `Content::audio()`, `Content::resource_link()`
- **D-06:** Each helper takes required fields as parameters, Optional fields (annotations, meta) default to None
- **D-07:** Helpers use `impl Into<String>` consistently

### Non-exhaustive strategy
- **D-08:** Apply `#[non_exhaustive]` to ALL protocol structs uniformly — no selective judgment calls
- **D-09:** This forces constructor usage from outside the crate — future field additions won't break downstream
- **D-10:** SDK tests must also migrate from struct literals to constructors — dogfood the public API
- **D-11:** Tests become living documentation of intended construction patterns and validate constructor completeness

### Fluent API depth
- **D-12:** Every Optional field gets a `.with_*()` fluent method: `::new(required).with_description("...").with_annotations(ann)`
- **D-13:** `.with_*()` methods use `impl Into<String>` for string parameters, matching constructor ergonomics
- **D-14:** Normalize existing types (ToolInfo, CallToolResult, etc.) to follow the same pattern — take advantage of v2.0 breaking change window
- **D-15:** Pattern is: `#[non_exhaustive]` + `derive(Default)` + `::new(required_fields)` + `.with_*(optional_field)` for every protocol struct

### Claude's Discretion
- Exact list of which types qualify as "protocol structs" (vs internal/utility types)
- Method naming for .with_*() when field names are ambiguous
- Whether to keep existing ToolInfo convenience methods (.with_ui(), .with_widget_meta()) alongside normalized pattern
- Order of implementation (which modules first)
- How to handle types where Default has empty required fields (uri, name) — these exist for ..Default::default() convenience, documented as not standalone-valid

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- ToolInfo fluent pattern: the `.with_*()` + `impl Into<String>` pattern is proven and can be replicated systematically
- Capabilities static factories (minimal(), full()): good pattern for types with common configurations
- ListResourcesResult::new() + .with_next_cursor(): clean constructor + single fluent method pattern

### Established Patterns
- `#[derive(Debug, Clone, Serialize, Deserialize)]` on all protocol types
- `#[serde(rename_all = "camelCase")]` for JSON field mapping
- `Option<T>` with `#[serde(skip_serializing_if = "Option::is_none")]` for optional fields
- `#[non_exhaustive]` already on ~8 types — expanding to all is consistent

### Integration Points
- Every type module in src/types/ needs modification
- All tests within type modules need struct literal → constructor migration
- Examples in examples/ that construct protocol types directly need updates
- MIGRATION.md needs a section on new construction patterns
- Workspace crates (mcp-tester, mcp-preview, cargo-pmcp) may need updates if they construct protocol types

</code_context>

<specifics>
## Specific Ideas

- The uniform pattern `#[non_exhaustive] + Default + ::new() + .with_*()` should be documented once in MIGRATION.md and referenced from rustdoc on every type
- Content::text("hello") is the most common construction — make it the simplest possible
- ResourceInfo::new("file://x", "test.txt").with_description("A test file") should be the canonical example in docs
- Run through all examples/ files to ensure they demonstrate the new construction patterns
- The dev team that reported the issue should be able to drop all their .map_err() and struct literal workarounds after this phase

</specifics>

## Deferred Ideas

- Spec-schema codegen that auto-generates constructors from JSON Schema — deferred per Phase 54 decision
- Content::from("hello") → Content::Text convenience From impl — could add later but static methods are primary API
