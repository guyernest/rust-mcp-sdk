<objective>
Close Blocker 2 from 92-VERIFICATION.md: three fail-open paths undermine the
phase's "fails closed on any validation gap" claim (WBSV-06). Each is a small,
targeted hardening:

- WR-02: the override accept arm (input.rs:145-149) has no role filter — a
  Role::Output or Role::Formula key is accepted, seeded, and echoed in
  accepted_overrides. After 92-06 made the executor seed-preserving, a seeded
  output value now WINS over the IR formula, turning this into a live
  output-forging vector (a caller could pin a served output under a valid
  provenance stamp). This is why this plan depends on 92-06.
- WR-04: project_outputs (handler.rs:78-80) does `let Some(value) = ... else {
  continue }` — a declared output missing from the run result silently vanishes
  from the success payload, contradicting the advertised outputSchema (WBSV-07).
- WR-07: verify_stamp_binding (bundle_loader.rs:218) uses unwrap_or("") — an
  absent layout anchor + empty lock.workbook_hash passes the security gate that
  claims to fail closed.

Purpose: WBSV-06 (every domain failure returns a fail-closed isError envelope;
validation is fail-closed). The isError envelope shape, error codes, and
provenance stamps are already correct — these three gaps are the only fail-open
holes left in the validation and integrity layer.

Output: a role-filtered override gate, a fail-closed output projector, an
absent-anchor-rejecting stamp gate, and a regression test for each.
</objective>
