<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → MCP server (bearer) | MCP requests must carry a valid bearer; unauthenticated requests are rejected |
| client → tasks/* (owner) | A client may only read/cancel its OWN tasks (owner-scoped) |
| browser → IdP routes | Authorization redirect + token exchange terminate at the bundled IdP |
| concurrent task creates → updater | The background updater must complete the CORRECT task; a discovery race could complete another client's task (MEDIUM-6) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-103-auth | Spoofing | StreamableHttpServer auth_provider | mitigate | Server validates Authorization: Bearer via the chosen TokenValidator (streamable:758-770); demo_server_requires_bearer test asserts unauthenticated MCP requests are rejected |
| T-103-IDOR | Elevation of Privilege | tasks/* owner scoping | mitigate | Owner derived from AuthContext.subject (NOT client params, task_dispatch.rs:168-186); cross-owner isolation pattern mirrored from lifecycle_http test:289-325 |
| T-103-RACE | Tampering | background updater task discovery | mitigate | The updater correlates the minted task by a UNIQUE marker (task variable / tool-arg id) so concurrent clicks cannot complete the wrong task (MEDIUM-6). If the create-path/task-variable surface cannot carry a marker this phase, the demo is constrained to single-user/no-concurrency and that limitation is documented here + in the plan-05 README |
| T-103-OPENREDIR | Tampering | GET /oauth2/authorize redirect URI | mitigate | InMemoryOAuthProvider.validate_authorization (oauth2.rs:473) validates the redirect_uri against the registered client; do not echo an arbitrary client-supplied redirect |
| T-103-PKCE-SRV | Information Disclosure | token exchange | mitigate | exchange_code runs verify_pkce (oauth2.rs:431-447) binding the code to the verifier (S256) — the auth code alone cannot be redeemed |
| T-103-CORS | Information Disclosure | merged IdP+MCP origin | mitigate | Per 103-SPIKE, single-origin router-merge via the named public mechanism; if two-port fallback, constrain the CORS allow-list to the demo origin |
| T-103-SC | Tampering | crate installs | accept | Server crate native deps mirror s46 (already vetted in-tree); no new external packages |
</threat_model>
