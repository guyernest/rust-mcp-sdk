<objective>
FIX the existing, broken `WasmHttpTransport::{send,receive}` so the high-level `Client` (and all
four typed task helpers) work over browser Fetch (D-08). This is the central SDK fix and the second
reusable public-API piece in the new release â€” it is NOT a greenfield transport.

Purpose: Today `send()` POSTs and DISCARDS the response, and `receive()` returns a hard error
("HTTP transport requires send() before receive()"). The high-level `Client::send_request` calls
`send(msg)` once then LOOPS on `receive()` to correlate the response â€” so `Client.initialize()` over
Fetch fails immediately. Buffering the one-shot POST response in `send()` and popping it in
`receive()` makes the whole typed `Client` API ("call MCP from a web app") work over Fetch. The
one-slot buffer logic is target-agnostic plumbing, so it is extracted into a small host-testable
`PendingSlot` type â€” that is what makes the WEBCH-02 correlation proof a GREEN `cargo test` rather
than a wasm-only compile check. To prevent a silently-lost response, `put` returns `Result` and
errors when the slot is already occupied (MEDIUM-4). And to prove the actual transport contract
(send stores `do_request` output, receive returns it) rather than only the buffer plumbing, the
`do_request` call is factored behind a mockable seam exercised by a host test (MEDIUM-5).
Output: a repaired `Transport` impl backed by a host-testable `PendingSlot` one-slot buffer (with an
occupied-slot error path) + a mockable `do_request` seam + host `cargo test`s that prove BOTH the
buffer contract AND the sendâ†’storeâ†’receive transport contract on the HOST target.
</objective>
