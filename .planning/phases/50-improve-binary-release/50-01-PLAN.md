<objective>
Fix the broken binary release pipeline and add ARM64 targets for both platforms.

Purpose: Binary releases are broken because `on: release: types: [published]` does not fire for GITHUB_TOKEN-created releases. Recent releases (v1.17.0, v1.18.0) have zero binary assets. Additionally, the macOS build mislabels ARM64 binaries as x86_64 due to runner changes, and ARM64 targets for both macOS and Linux are missing entirely.

Output: Three workflow files rewritten so that pushing a v* tag produces 10 binaries (5 per tool) plus 10 .sha256 checksums, all uploaded to the GitHub release.
</objective>

<task type="auto">
  <name>Task 1: Convert release-tester.yml to reusable workflow with 5-target matrix</name>
  <files>.github/workflows/release-tester.yml</files>
  <action>
Rewrite `.github/workflows/release-tester.yml` to be a reusable workflow callable via `workflow_call` while also supporting `workflow_dispatch` for manual runs.

**Trigger section:** Replace `on: release: types: [published]` with:
```yaml
on:
  workflow_call:
    inputs:
      tag_name:
        required: true
        type: string
  workflow_dispatch:
    inputs:
      tag_name:
        description: 'Release tag (leave empty for latest)'
        required: false
        type: string
```

**Matrix:** Replace the 3-target matrix with 5 targets:
```yaml
strategy:
  fail-fast: false
  matrix:
    include:
      - os: ubuntu-latest
        target: x86_64-unknown-linux-gnu
        asset_name: mcp-tester-x86_64-unknown-linux-gnu
      - os: ubuntu-24.04-arm
        target: aarch64-unknown-linux-gnu
        asset_name: mcp-tester-aarch64-unknown-linux-gnu
      - os: macos-15-intel
        target: x86_64-apple-darwin
        asset_name: mcp-tester-x86_64-apple-darwin
      - os: macos-14
        target: aarch64-apple-darwin
        asset_name: mcp-tester-aarch64-apple-darwin
      - os: windows-latest
        target: x86_64-pc-windows-msvc
        asset_name: mcp-tester-x86_64-pc-windows-msvc.exe
```

**Tag resolution step:** Resolve the tag from either `workflow_call` input, `workflow_dispatch` input, or fall back to latest release:
```yaml
- name: Resolve tag
  id: tag
  shell: bash
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  run: |
    TAG="${{ inputs.tag_name }}"
    if [ -z "$TAG" ]; then
      TAG=$(gh release list --limit 1 --json tagName -q '.[0].tagName')
    fi
    echo "TAG_NAME=$TAG" >> $GITHUB_OUTPUT
```

**Build step:** Keep `cargo build --release --package mcp-tester --target ${{ matrix.target }}`. No `--target` override needed since runners match their native target, but keep it explicit for clarity.

**Prepare artifact step:** After build, rename binary to asset_name. Handle Windows (.exe) vs Unix.

**Checksum step:** After preparing the artifact, generate a per-binary SHA256 checksum:
```yaml
- name: Generate checksum
  shell: bash
  run: |
    cd target/${{ matrix.target }}/release
    if command -v sha256sum &>/dev/null; then
      sha256sum "${{ matrix.asset_name }}" > "${{ matrix.asset_name }}.sha256"
    else
      shasum -a 256 "${{ matrix.asset_name }}" > "${{ matrix.asset_name }}.sha256"
    fi
```

**Upload step:** Upload both the binary and its .sha256 file to the release:
```yaml
- name: Upload binary and checksum
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  run: |
    gh release upload "${{ steps.tag.outputs.TAG_NAME }}" \
      "target/${{ matrix.target }}/release/${{ matrix.asset_name }}" \
      "target/${{ matrix.target }}/release/${{ matrix.asset_name }}.sha256" \
      --clobber
```

**Keep:** `actions/checkout@v6`, `dtolnay/rust-toolchain@stable`, `actions/cache@v5` for cargo.

**Remove:** The old dev release creation logic (no longer needed since tag is always provided by caller or resolved from latest). Remove the outdated comment about ring crate / ARM64 runners.

Add `fail-fast: false` to the matrix strategy so a failure on one platform does not cancel other builds.

  <name>Task 3: Wire binary workflows into release.yml as downstream jobs</name>
  <files>.github/workflows/release.yml</files>
  <action>
Modify `.github/workflows/release.yml` to call the binary build workflows after `create-release`.

**Step 1:** Add `outputs` to the `create-release` job so downstream jobs can read the tag:
```yaml
  create-release:
    name: Create Release
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.get_version.outputs.VERSION }}
    steps:
      # ... existing steps unchanged ...
```

The existing `get_version` step already writes `VERSION` to `$GITHUB_OUTPUT`, so this just surfaces it as a job output.

**Step 2:** Add two new jobs after the `create-release` job (and parallel to `publish-crates`):

```yaml
  build-tester:
    name: Build Tester Binaries
    needs: create-release
    uses: ./.github/workflows/release-tester.yml
    with:
      tag_name: ${{ needs.create-release.outputs.version }}
    secrets: inherit

  build-preview:
    name: Build Preview Binaries
    needs: create-release
    uses: ./.github/workflows/release-preview.yml
    with:
      tag_name: ${{ needs.create-release.outputs.version }}
    secrets: inherit
```

These run in parallel with `publish-crates` since they only depend on `create-release` (the release must exist before uploading assets).

**Do NOT modify:** The existing `create-release`, `publish-crates`, or `publish-mcp` jobs beyond adding the `outputs` field to `create-release`. The binary build jobs are additive.

**Resulting job graph:**
```
create-release
  ├── publish-crates -> publish-mcp
  ├── build-tester (parallel)
  └── build-preview (parallel)
```
