  <name>Task 1: Create install.sh for Linux and macOS</name>
  <files>install/install.sh</files>
  <action>
Create `install/install.sh` -- a POSIX-compatible shell script that downloads and installs mcp-tester or mcp-preview.

**Script structure:**

1. **Shebang and options:** `#!/bin/sh` with `set -eu`

2. **Constants:**
```sh
REPO="paiml/rust-mcp-sdk"
DEFAULT_TOOL="mcp-tester"
DEFAULT_INSTALL_DIR="$HOME/.local/bin"
```

3. **Usage function:** Print help showing:
```
Usage: install.sh [OPTIONS]
  --tool TOOL        Tool to install: mcp-tester or mcp-preview (default: mcp-tester)
  --version VERSION  Version to install (default: latest)
  --dir DIR          Install directory (default: ~/.local/bin)
  --help             Show this help
```

4. **Argument parsing:** Parse `--tool`, `--version`, `--dir`, `--help` using a `while` loop with `case`.

5. **OS detection function:**
```sh
detect_os() {
  case "$(uname -s)" in
    Linux*)  echo "unknown-linux-gnu" ;;
    Darwin*) echo "apple-darwin" ;;
    *)       printf "Error: Unsupported OS: %s\n" "$(uname -s)" >&2; exit 1 ;;
  esac
}
```

6. **Architecture detection function:**
```sh
detect_arch() {
  case "$(uname -m)" in
    x86_64|amd64)   echo "x86_64" ;;
    aarch64|arm64)   echo "aarch64" ;;
    *)               printf "Error: Unsupported architecture: %s\n" "$(uname -m)" >&2; exit 1 ;;
  esac
}
```

7. **Checksum verification function:** Detect available tool and verify:
```sh
verify_checksum() {
  local file="$1" expected_file="$2"
  if command -v sha256sum >/dev/null 2>&1; then
    sha256sum --check "$expected_file" --quiet
  elif command -v shasum >/dev/null 2>&1; then
    shasum -a 256 --check "$expected_file" --quiet
  else
    printf "Warning: No sha256sum or shasum found, skipping checksum verification\n" >&2
    return 0
  fi
}
```
Note: Use POSIX-compatible function syntax (no `local` if targeting strict POSIX -- but `local` is widely supported in sh on Linux/macOS).

8. **Main logic:**
   - Resolve ARCH and OS using detection functions
   - Build TARGET as `${ARCH}-${OS}` (e.g., `x86_64-apple-darwin`)
   - Build ASSET_NAME as `${TOOL}-${TARGET}`
   - If version is "latest" (default), use URL: `https://github.com/${REPO}/releases/latest/download/${ASSET_NAME}`
   - Otherwise use: `https://github.com/${REPO}/releases/download/${VERSION}/${ASSET_NAME}`
   - Download the binary: `curl -fsSL -o "${TMPDIR}/${ASSET_NAME}" "${DOWNLOAD_URL}"`
   - Download the checksum: `curl -fsSL -o "${TMPDIR}/${ASSET_NAME}.sha256" "${DOWNLOAD_URL}.sha256"`
   - Verify checksum (cd to tmpdir, run verify_checksum)
   - Create install dir if needed: `mkdir -p "${INSTALL_DIR}"`
   - Move binary: `mv "${TMPDIR}/${ASSET_NAME}" "${INSTALL_DIR}/${TOOL}"`
   - Make executable: `chmod +x "${INSTALL_DIR}/${TOOL}"`
   - Print success message with version and path
   - Check if install dir is in PATH, warn if not

9. **Temp directory:** Use `mktemp -d` for secure temporary directory, clean up with trap:
```sh
TMPDIR=$(mktemp -d)
trap 'rm -rf "$TMPDIR"' EXIT
```

10. **Error handling:** `set -eu` catches errors. Use `curl -fsSL` (-f = fail on HTTP error, -sS = silent but show errors, -L = follow redirects).

Make the file executable (chmod +x in the action or note for executor).

<task type="auto">
  <name>Task 2: Create install.ps1 for Windows</name>
  <files>install/install.ps1</files>
  <action>
Create `install/install.ps1` -- a PowerShell script for installing mcp-tester or mcp-preview on Windows.

**Script structure:**

1. **Parameters block:**
```powershell
param(
    [string]$Tool = "mcp-tester",
    [string]$Version = "latest",
    [string]$InstallDir = "$env:USERPROFILE\.local\bin"
)
```

2. **Strict mode:**
```powershell
$ErrorActionPreference = "Stop"
```

3. **Architecture detection:**
```powershell
$arch = [System.Runtime.InteropServices.RuntimeInformation]::OSArchitecture
switch ($arch) {
    "X64"   { $target = "x86_64-pc-windows-msvc" }
    default { Write-Error "Unsupported architecture: $arch"; exit 1 }
}
```
Note: No ARM64 Windows target is built (no runner available). Future-proof the detection but error on non-x86_64.

4. **Build download URL:**
```powershell
$assetName = "$Tool-$target.exe"
if ($Version -eq "latest") {
    $url = "https://github.com/paiml/rust-mcp-sdk/releases/latest/download/$assetName"
} else {
    $url = "https://github.com/paiml/rust-mcp-sdk/releases/download/$Version/$assetName"
}
$checksumUrl = "$url.sha256"
```

5. **Download to temp:**
```powershell
$tempDir = New-Item -ItemType Directory -Path ([System.IO.Path]::Combine([System.IO.Path]::GetTempPath(), [System.IO.Path]::GetRandomFileName()))
$binaryPath = Join-Path $tempDir $assetName
$checksumPath = Join-Path $tempDir "$assetName.sha256"

Write-Host "Downloading $Tool..."
Invoke-WebRequest -Uri $url -OutFile $binaryPath -UseBasicParsing
Invoke-WebRequest -Uri $checksumUrl -OutFile $checksumPath -UseBasicParsing
```

6. **Checksum verification:**
```powershell
$expectedHash = (Get-Content $checksumPath).Split(" ")[0]
$actualHash = (Get-FileHash -Path $binaryPath -Algorithm SHA256).Hash.ToLower()
if ($actualHash -ne $expectedHash) {
    Write-Error "Checksum mismatch! Expected: $expectedHash, Got: $actualHash"
    exit 1
}
Write-Host "Checksum verified."
```

7. **Install:**
```powershell
if (-not (Test-Path $InstallDir)) {
    New-Item -ItemType Directory -Path $InstallDir -Force | Out-Null
}
$destPath = Join-Path $InstallDir "$Tool.exe"
Move-Item -Path $binaryPath -Destination $destPath -Force
Write-Host "$Tool installed to $destPath"
```

8. **PATH check:** Check if $InstallDir is in the user's PATH. If not, print instructions:
```powershell
$currentPath = [Environment]::GetEnvironmentVariable("PATH", "User")
if ($currentPath -notlike "*$InstallDir*") {
    Write-Host ""
    Write-Host "Add $InstallDir to your PATH:"
    Write-Host "  [Environment]::SetEnvironmentVariable('PATH', `"$InstallDir;`$env:PATH`", 'User')"
}
```

9. **Cleanup:** Remove temp directory in a finally block.

**For crates/mcp-preview/Cargo.toml**, add after the `[dev-dependencies]` section:

```toml
[package.metadata.binstall]
pkg-url = "https://github.com/paiml/rust-mcp-sdk/releases/download/v{ version }/{ name }-{ target }{ binary-ext }"
pkg-fmt = "bin"
```

**Key decisions:**
- Use explicit `https://github.com/paiml/rust-mcp-sdk/...` URL (not `{ repo }` template) to avoid the potential redirect issue from `paiml/pmcp` noted in RESEARCH.md pitfall 6.
- `pkg-fmt = "bin"` per user decision -- bare binaries, not archived.
- `{ name }` resolves to the crate name (mcp-tester or mcp-preview)
- `{ target }` resolves to the Rust target triple (e.g., x86_64-unknown-linux-gnu)
- `{ binary-ext }` resolves to `.exe` on Windows, empty string elsewhere
- `{ version }` resolves to the crate version on crates.io

**Important:** The version in the URL uses `v{ version }` (with v prefix) because GitHub release tags use `v1.18.0` format but crates.io versions are `1.18.0`. The `v` prefix must be hardcoded in the template.

**Note:** The binstall version references the crate's own version. Since mcp-tester is at 0.3.2 while the SDK release is v1.18.x, the tag `v0.3.2` would not exist. This is a known issue -- binstall will work correctly only when the release tag matches the crate version. Since this project uses a single tag for all crates, we need to verify the tag convention. Looking at the existing release.yml, tags are `v*` on push. The README says "One tag per release" using the SDK version. This means binstall needs the SDK version tag.

**Corrected approach:** Since the workspace uses a single release tag (the pmcp SDK version, e.g., v1.18.0), and mcp-tester has its own version (0.3.2), the binstall `{ version }` template will NOT match the release tag. To handle this, we need to use a different approach:

Actually, reviewing the release workflow: it triggers on `v*` tags. The crate versions are independent. When cargo-binstall resolves `{ version }`, it uses the crate version from the registry. Since the release tag is e.g. `v1.18.0` but mcp-tester is `v0.3.2`, the URL would be `releases/download/v0.3.2/...` which does not exist.

**Solution:** Use the `bin-dir` field and point to the latest release instead. OR, the simpler approach: since all crates are published in the same release, we can set the binstall `pkg-url` to use a fixed "latest" URL pattern that cargo-binstall supports. However, binstall does not support "latest" -- it always uses version.

**Practical solution:** Accept that cargo-binstall will only work if the user specifies `--version` matching the release tag, OR update the workflow to also create tool-specific tags. The simplest practical approach: document that `cargo binstall` requires `--version` matching the release tag, and use a pkg-url that maps version to the release tag. Since each release publishes all crates, the mapping is: find the release that contains this crate version.

**Simplest correct approach:** Use `pkg-url` with the crate's own version, and ensure release tags match. Since the project already uses a single v* tag for all crates, and cargo-binstall tries to download `v{crate_version}`, this will 404 unless crate versions align with tags.

Given this constraint, the most pragmatic approach is:
1. Add the binstall metadata pointing to the repo with version template
2. Note in the SUMMARY that binstall will work when crate versions align with release tags, or when tool-specific tags are added in the future
3. The metadata is still correct and useful -- it just requires version awareness

Add the metadata as described above. This is the standard binstall configuration.
