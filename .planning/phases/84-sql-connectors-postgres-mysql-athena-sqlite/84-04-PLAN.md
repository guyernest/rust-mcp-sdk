<objective>
Promote the canonical SQLite driver from spike 005's `sqlite_backend` module into a first-class public `SqliteConnector` type under the existing `sqlite` feature on `pmcp-server-toolkit`. This satisfies CONN-08 (SQLite as feature flag, no separate crate) + provides the concrete connector that Plan 03's `synthesizer_structured_content.rs` integration test needs.

Critical design choice (per RESEARCH Open Question #3 + PATTERNS row + RESEARCH Landmine #10): the existing `pub(crate) MockSqlConnector` is NOT deleted. It is a test-only fixture for `schema_text` testing across dialects (it can declare any `Dialect` for unit-testing dialect-aware prompt assembly). `SqliteConnector` is a separate real-driver impl that always declares `Dialect::Sqlite`. The two coexist.

Internal architecture (per RESEARCH §1.4 + Claude's Discretion):
- `Arc<Mutex<rusqlite::Connection>>` where `Mutex` is `std::sync::Mutex` (NOT `tokio::sync::Mutex`)
- `async fn execute()` wraps sync rusqlite work via `tokio::task::spawn_blocking`
- Mutex held ONLY inside `spawn_blocking` so the async runtime is never blocked
- `rusqlite` already declared `optional = true` in `Cargo.toml` line 51 with `bundled` feature — no Cargo changes needed for the dep itself

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Toolkit consumer → `SqliteConnector::open(path)` | Path string crosses into filesystem I/O; rusqlite's bundled SQLite handles path safety |
| User SQL via `execute()` → rusqlite parameter binding | All user values bound parametrically via `raw_bind_parameter` — never concatenated |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-84-04-01 | Tampering | SQL injection via unbound `:placeholder` | mitigate | `bind_params` iterates `ordered_params` from `translate_placeholders` + calls `raw_bind_parameter` with `json_to_sql(val)`. Values NEVER concatenated into the SQL string. Unit test verifies bound parameter actually used. |
| T-84-04-02 | Information Disclosure | `ConnectorError::Connection(e.to_string())` leaks file path on `open()` failure | mitigate | rusqlite error messages are bounded ("unable to open database file"); they don't include arbitrary user data. Acceptable for SQLite's local-only threat model. |
| T-84-04-03 | Denial of Service | Mutex poisoning blocks all subsequent execute calls | mitigate | `conn.lock().map_err(|_| ConnectorError::Driver("mutex poisoned".into()))` translates poisoning to a recoverable error variant. Server can reconstruct the connector if needed. |
| T-84-04-04 | Denial of Service | spawn_blocking spawns unbounded blocking tasks under high concurrency | accept | Tokio's default `block_on` thread pool is bounded (512 threads default). Per CLAUDE.md `--test-threads=1` and single-thread tokio runtime in toolkit + tests, this is non-issue in practice. |
</threat_model>
