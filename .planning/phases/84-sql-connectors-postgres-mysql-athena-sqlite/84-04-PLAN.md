Promote the canonical SQLite driver from spike 005's `sqlite_backend` module into a first-class public `SqliteConnector` type under the existing `sqlite` feature on `pmcp-server-toolkit`. This satisfies CONN-08 (SQLite as feature flag, no separate crate) + provides the concrete connector that Plan 03's `synthesizer_structured_content.rs` integration test needs.

**REVIEWS dispositions folded into this plan:**

- **H1 (Plan 03↔Plan 04 build-order mismatch):** The `tests/synthesizer_structured_content.rs` integration test is OWNED BY THIS PLAN (Plan 04), not Plan 03. Plan 04 ships `SqliteConnector` AND the test that consumes it together, eliminating the same-wave cross-plan compile dependency.
- **H2 (missing depends_on Plan 02):** Plan 04 now declares `depends_on: [00, 01, 02, 03]` (was `[00, 01]`). Plan 02 ships `translate_placeholders` (used by `SqliteConnector::execute` for named-binding); Plan 03 ships `synthesize_from_config_with_connector` + `synthesize_inner` widget_meta flip (used by the integration test).
- **H3 (example calls wrong synthesize fn):** The example body calls `synthesize_from_config_with_connector(&cfg, conn)` (NOT `synthesize_from_config(&cfg, conn)` — that doesn't exist; Plan 03 preserved the original 1-arg signature and ADDED the connector-threaded variant alongside).
- **H4 (`?` placeholder incompatibility):** The integration test seed SQL exclusively uses `:name` style placeholders (`:p1`, `:p2`), NOT `?`. `translate_placeholders(_, Dialect::Sqlite)` is identity but only recognises `:name`; bare `?` would pass through to rusqlite as a positional placeholder which the connector binds against an empty `ordered_params` (mismatch → driver error).

Critical design choice (per RESEARCH Open Question #3 + PATTERNS row + RESEARCH Landmine #10): the existing `pub(crate) MockSqlConnector` is NOT deleted. It is a test-only fixture for `schema_text` testing across dialects (it can declare any `Dialect` for unit-testing dialect-aware prompt assembly). `SqliteConnector` is a separate real-driver impl that always declares `Dialect::Sqlite`. The two coexist.

Internal architecture (per RESEARCH §1.4 + Claude's Discretion):
- `Arc<Mutex<rusqlite::Connection>>` where `Mutex` is `std::sync::Mutex` (NOT `tokio::sync::Mutex`)
- `async fn execute()` wraps sync rusqlite work via `tokio::task::spawn_blocking`
- Mutex held ONLY inside `spawn_blocking` so the async runtime is never blocked
- `rusqlite` already declared `optional = true` in `Cargo.toml` line 51 with `bundled` feature — no Cargo changes needed for the dep itself
