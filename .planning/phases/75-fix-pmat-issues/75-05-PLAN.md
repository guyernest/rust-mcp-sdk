## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PR author → ci.yml `gate` job | PRs are evaluated against the gate; org-required-check enforces merge block |
| GitHub Actions runner → crates.io | `cargo install pmat --version =3.15.0 --locked` fetches a pinned binary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-75-05-01 | Repudiation | A misconfigured CI gate that runs but doesn't fail (false sense of security) | mitigate | Task 75-05-02 explicitly opens a regression PR with a deliberately-complex function and verifies the `gate` job FAILS — empirical fail-closed test |
| T-75-05-02 | Tampering | A future contributor disables or weakens the gate by editing ci.yml | accept | Detected by code review on the workflow file change; CODEOWNERS for `.github/` (if configured) gates changes |
| T-75-05-03 | Denial of Service | PMAT step adds CI minutes to every PR | accept | One-time `cargo install pmat --version =3.15.0 --locked` per cache; subsequent PRs hit the cache. ~30s overhead per PR is acceptable for the gate it provides. |

**Refactor scope note:** Wave 5 is CI-process change. Adding the gate is itself a security/quality-process change with this STRIDE consideration: a CI gate that passes when it should fail (T-75-05-01) is more dangerous than no gate. The fail-closed empirical test (Task 5-02) is the explicit mitigation.
</threat_model>
