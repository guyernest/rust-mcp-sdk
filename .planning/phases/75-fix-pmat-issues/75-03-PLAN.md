**POST-REVIEW REVISION (2026-04-23 — Codex MEDIUM concern, helper-context allowance):**
Pure helper extraction in evaluate_with_scope and evaluate_array_method_with_scope can produce
6-arg function spam (each helper needs `expr`, `global_vars`, `local_vars`, evaluator config,
error sink, scope chain, etc.). The planner EXPLICITLY ALLOWS introducing a small `EvalContext`
(or similar context struct) to thread shared state through helpers:

```rust
struct EvalContext<'a, V: VariableProvider> {
    global_vars: &'a V,
    local_vars: &'a HashMap<String, JsonValue>,
    // optional: scope_chain, evaluator_config, error_sink, etc. as needed
}

fn evaluate_property_access(
    object: &ValueExpr,
    property: &str,
    ctx: &EvalContext<impl VariableProvider>,
) -> Result<JsonValue, ExecutionError> {
    let obj = evaluate_with_scope(object, ctx.global_vars, ctx.local_vars)?;
    /* ... */
}
```

If introduced, the new struct MUST be documented in `75-03-SUMMARY.md` with:
- The fields it threads (and why each is shared)
- The cognitive impact on helpers (helpers should still be ≤25)
- Whether the dispatcher itself benefits (often yes — fewer args at the call site)

This is an explicit license to introduce the struct WITHOUT it being scope creep. The
goal is readable refactor, not minimum-edit refactor.
