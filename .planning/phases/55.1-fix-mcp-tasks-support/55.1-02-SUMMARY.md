# Phase 55.1 Plan 02: Core Task Detection Summary

**Wired task detection into ServerCore handle_call_tool so tools declaring taskSupport return CreateTaskResult with _meta related-task instead of wrapping Task JSON as CallToolResult text**

## Performance

- **Duration:** 5 min

- **Completed:** 2026-03-23T05:05:01Z
- **Tasks:** 2
- **Files modified:** 2

## Accomplishments
- handle_call_tool now returns ToolCallOutcome enum enabling bifurcated dispatch between CallToolResult and CreateTaskResult
- Tools declaring execution.taskSupport = Required|Optional that return Task-shaped JSON (taskId + status) produce proper CreateTaskResult wire format
- _meta with io.modelcontextprotocol/related-task taskId reference added to CreateTaskResult responses per D-08/D-09
- Three unit tests verify all paths: Required (returns CreateTaskResult), absent (returns CallToolResult), Forbidden (returns CallToolResult)
- All 18 core_tests pass (15 existing + 3 new) with zero regression

## Files Created/Modified
- `src/server/core.rs` - Added ToolCallOutcome enum, task detection in handle_call_tool, CreateTaskResult dispatch with _meta in ClientRequest::CallTool handler
- `src/server/core_tests.rs` - Added 3 tests: test_call_tool_with_task_support_returns_create_task_result, test_call_tool_without_task_support_returns_call_tool_result, test_call_tool_with_forbidden_task_support_returns_call_tool_result

## Decisions Made
- ToolCallOutcome enum defined at module scope (not inside impl block) -- Rust does not allow type definitions inside impl blocks
- Task detection requires both task_store configured AND tool declaring TaskSupport::Required|Optional -- shape-matching alone is insufficient per D-06 security requirement
- TaskCreated variant and match arm are cfg-gated for wasm32 since task_store is native-only
- CreateTaskResult JSON built manually with serde_json::json! macro rather than serializing CreateTaskResult struct, to allow direct _meta injection
