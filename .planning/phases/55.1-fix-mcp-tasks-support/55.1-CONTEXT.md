# Phase 55.1: Fix MCP Tasks support - Context

**Gathered:** 2026-03-22
**Status:** Ready for planning
**Source:** Dev team investigation report (satellite-view/claude-instance/INVESTIGATION_REPORT.md)

<domain>
## Phase Boundary

Fix three SDK-side gaps that prevent the standard `task_store` path from returning proper `CreateTaskResult` wire format. Currently, tools that create tasks via `task_store.create()` have their results wrapped as `CallToolResult` text — clients see plain text instead of a protocol-level task signal and cannot auto-poll `tasks/get`.

</domain>

<decisions>
## Implementation Decisions

### Gap 1: TypedTool execution metadata
- **D-01:** Add `execution: Option<ToolExecution>` field to all TypedTool variants (TypedTool, TypedToolWithOutput, TypedSyncTool, WasmTypedTool)
- **D-02:** Add `.with_execution(execution: ToolExecution)` builder method to all TypedTool variants
- **D-03:** Update `metadata()` to include the `execution` field instead of hardcoded `None`

### Gap 2: SDK task result detection on standard path
- **D-04:** In `ServerCore::handle_call_tool` (core.rs), after the tool handler returns a `Value`, detect if the result is Task-shaped (has `taskId` + `status` fields) AND the tool declares `taskSupport` in its `ToolInfo.execution`
- **D-05:** When detected, return `CreateTaskResult { task: serde_json::from_value(value) }` instead of wrapping as `CallToolResult { content: [text(pretty_json)] }`
- **D-06:** Detection is based on the tool's declared `execution.taskSupport` (not heuristic shape-matching alone) — if `taskSupport` is `required` or `optional`, attempt the conversion; if `forbidden` or absent, use the normal `CallToolResult` path
- **D-07:** The existing `TaskRouter` (legacy pmcp-tasks) path remains unchanged — it already returns `CreateTaskResult` correctly

### Gap 3: RelatedTaskMetadata wiring
- **D-08:** When returning `CreateTaskResult`, also add `_meta` with `io.modelcontextprotocol/related-task` key containing the `taskId` for backwards compatibility
- **D-09:** Use the existing `RELATED_TASK_META_KEY` and `RelatedTaskMetadata` types from `src/types/tasks.rs` — they're already defined but unused

### Claude's Discretion
- Whether to add `with_execution()` to the `#[mcp_tool]` proc macro as well (via annotation attribute)
- Error handling when Task-shaped value fails `serde_json::from_value::<Task>()` deserialization
- Whether to log a warning when a tool returns Task-shaped JSON but doesn't declare `taskSupport`

</decisions>

<specifics>
## Specific Ideas

- The wire format must change from `{"content": [{"type": "text", "text": "{taskId: ...}"}]}` to `{"task": {"taskId": "abc", "status": "working", "pollInterval": 5000, ...}}`
- The dev team recommends: Change 1 + Change 3 first (TypedTool + tool declarations), then Change 2 (core.rs detection)
- Detection approach: use the tool's declared `execution.taskSupport` metadata from the cached `ToolInfo`, not heuristic JSON shape matching

</specifics>
