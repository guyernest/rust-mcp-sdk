### Summary

Phase 55.1 fully achieves its goal. All three gaps are closed:

**Gap 1 closed:** TypedTool, TypedSyncTool, TypedToolWithOutput, and WasmTypedTool now carry `execution: Option<ToolExecution>` and expose `.with_execution()` builders. The `metadata()`/`info()` methods return the stored value instead of `None`. WasmSimpleTool was also fixed to include `title` and `icons` fields in its ToolInfo literal (pre-existing WASM compilation bug).

**Gap 2 closed:** `ServerCore::handle_call_tool` now returns a `ToolCallOutcome` enum. When a tool declares `TaskSupport::Required` or `TaskSupport::Optional` AND the `task_store` is configured AND the returned Value has `taskId`+`status` fields, the dispatch site returns `CreateTaskResult` JSON instead of a `CallToolResult` text wrapper. The existing `TaskRouter` legacy path is completely untouched.

**Gap 3 closed:** `RelatedTaskMetadata` and `RELATED_TASK_META_KEY` — previously defined but unused in `src/types/tasks.rs` — are now imported into `core.rs` and used to populate `_meta` on every `CreateTaskResult` response. Clients receive `{"task": {...}, "_meta": {"io.modelcontextprotocol/related-task": {"taskId": "..."}}}` and can begin polling `tasks/get`.
