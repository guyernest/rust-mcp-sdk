# Cross-AI Plan Review — Phase 72.1

## Codex Review

### Summary

`72.1-01-PLAN.md` is generally strong: it is tightly scoped to CR-03 rev-2, preserves the `MCP_SERVER_NAME` boundary, orders the major dependencies correctly (`Task 1` before `Task 2`, `Task 4` last), and maps most acceptance criteria into explicit checks. The main weaknesses are around runtime confidence rather than design: `Task 1` claims `'error'` on unexpected shape without actually validating the JSON shape, `Task 2` uses full-file rewrites that could cause avoidable churn/regressions, and `Task 5` still leaves the most important runtime behaviors largely in manual verification rather than automation.

### Strengths

- Strong traceability. `must_haves`, `grep_guardrails`, and `Task 5` all map back to CR-03 ACs with very little ambiguity.
- Dependency ordering is explicit and correct.
  - `Task 1` establishes the hook contract before any consumer rewrites.
  - `Task 4` correctly delays the version bump until the end.
- Scope control is good.
  - `Task 2` repeatedly calls out the `MCP_SERVER_NAME` boundary.
  - `G2` and `G5` are useful protections against over-editing the template.
- The plan respects the `next.config.js` / `output: 'export'` constraint.
  - `Task 2c` converts only `connect/page.tsx` to a client component.
  - `Task 5` explicitly verifies offline/error rendering instead of sneaking in SSR.
- Verification breadth is good for a no-harness template.
  - `make quality-gate`
  - `cargo doc --no-deps`
  - template `tsc --noEmit`
  - `pnpm build` with `NEXT_PUBLIC_*` unset
  - G1..G6 grep checks
- The manual AC-11 gate is honest about what is and is not automated.

### Concerns

- **MEDIUM**: `Task 1` says `'error'` covers "unexpected shape", but the implementation does not validate shape at all. `setCfg(c as LandingConfig)` accepts any 200 JSON payload. A malformed `{ signup: "oops" }` or missing `mcpEndpoint` would be treated as success, which contradicts both the hook comments and the plan text.

- **MEDIUM**: `Task 2` uses full-file replacements where the CR only needs targeted behavior changes. `Task 2c` and `Task 2d` are especially prone to accidental churn: copy changes, className changes, or loss of existing connect-page-local branding/structure could slip in even though they are not required by CR-03.

- **MEDIUM**: AC-1 is only inferred, not exercised. The plan assumes "template source changed" means "`pmcp new --template landing` scaffolds correctly," but `Task 5` never actually runs the scaffold command. If the scaffold pipeline has an explicit file manifest/embed/index step, this plan would miss it.

- **LOW**: The performance note is directionally right but overstated. The plan/context talks about "4 concurrent fetches on first paint," but with the current route structure the real maximum is closer to 2 on a given page (`Header` + page component). Not a blocker, but the reasoning should be precise.

- **LOW**: `Task 1` repeatedly calls the cleanup behavior "abort-on-unmount," but it is only a cancel flag. That prevents stale state updates, but it does not abort the network request. Fine for this endpoint, but the wording is inaccurate.

- **LOW**: Some verification checks are weaker than they look. Example: `Task 2`'s `grep -c 'cfg.mcpEndpoint\|endpoint'` does not really prove all snippets use the runtime endpoint, only that the file mentions `endpoint` enough times.

- **LOW**: Runtime behavior still depends heavily on manual verification. Given the explicit "no test harness" constraint, this is understandable, but it means redirect behavior and disabled-state rendering are not protected by CI.

### Suggestions

- Add lightweight runtime shape validation in `Task 1`, or explicitly downgrade the claim. If CR-03 must be followed literally, keep the code but change the plan text from "unexpected shape -> error" to "non-OK/network failure -> error." If minor hardening is allowed, validate `mcpEndpoint` as a string and `signup` as `null | object` before `setCfg`.

- Tighten `Task 2` to require preservation of unrelated markup/copy. A line like "preserve existing styling/copy unless required by CR-03" would reduce regression risk from the full replacements.

- Add a real scaffold smoke check for AC-1 in `Task 5`. Even one command that generates a temp landing project and verifies the new hook file exists would close the biggest coverage gap.

- Strengthen AC-8 verification. Replace the loose `grep -c 'cfg.mcpEndpoint\|endpoint'` with checks that the four snippet props actually include `endpoint`/`cfg.mcpEndpoint`, plus the existing negative grep for `.us-east.true-mcp.com`.

- Clarify the fetch fan-out note. Call it "duplicate hook fetches across consumers, max ~2 per page with current route layout" rather than "4 concurrent fetches on first paint."

- Either rename "abort-on-unmount" to "ignore-after-unmount" or explicitly permit `AbortController` if the team wants the wording to match behavior.

### Risk Assessment

**Overall risk: MEDIUM**

The plan is well scoped and mostly complete, so implementation risk is not high. The remaining risk comes from verification quality, not architecture: the hook's malformed-response behavior is underspecified vs. implemented, scaffold correctness is not directly exercised, and key runtime behaviors still rely on manual checks. If those points are tightened, this drops close to LOW risk.

Single-reviewer only — Codex flagged (priority order by severity):

1. **MEDIUM — `Task 1` hook shape validation gap.** The hook types the `'error'` state as covering "unexpected shape" but `setCfg(c as LandingConfig)` accepts any 200 payload. Either (a) relax the plan claim to "non-OK/network failure → error" to match the CR-03 verbatim code, or (b) add a minimal shape check (`typeof mcpEndpoint === 'string'` and `signup === null || typeof signup === 'object'`) inside the hook before `setCfg`.

2. **MEDIUM — `Task 2` full-file rewrites risk regression churn.** The CR-03 inline code blocks are behavior-complete but may inadvertently lose copy/className/local branding in the four consumer files during wholesale replacement. 3. **MEDIUM — AC-1 not directly exercised.** `Task 5` never runs `pmcp new --template landing` against the updated template. If the scaffold pipeline has an explicit file manifest/embed/index step, a missing update there would go undetected. Add a minimal scaffold smoke check (generate temp landing project, assert `lib/useLandingConfig.ts` is present in the output).

4. **LOW — Fan-out note overstated.** The "4 concurrent fetches" framing is inaccurate; per-page fan-out is closer to 2 (`Header` + current page component). Update the wording in CONTEXT.md / PLAN.md.

5. **LOW — "abort-on-unmount" wording is imprecise.** The cancel flag only prevents state updates, it does not abort the fetch. Either rename to "ignore-after-unmount" or introduce `AbortController`.

6. **LOW — AC-8 grep is loose.** `grep -c 'cfg.mcpEndpoint\|endpoint'` is a weak proxy; tighten to count `cfg.mcpEndpoint` occurrences specifically alongside the existing negative grep for `.us-east.true-mcp.com`.

7. **LOW — No CI coverage for runtime redirect / disabled-state behavior.** Acknowledged tradeoff given "no test harness" constraint; noted for awareness, not action.
