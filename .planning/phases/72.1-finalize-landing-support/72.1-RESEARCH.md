## Phase Summary

Phase 72.1 ships a thin, PRD-complete consolidation of CR-03 rev-2: a new required React hook (`lib/useLandingConfig.ts`) plus surgical rewrites of four consumer files in the landing Next.js template, three rustdoc-only edits in `cargo-pmcp/src/landing/config.rs`, and a patch bump from 0.8.0 → 0.8.1. The driver is Phase 71.1 platform behavior: `deploy-landing` now actively strips the four `NEXT_PUBLIC_COGNITO_*` / `NEXT_PUBLIC_LANDING_CLIENT_ID` / `NEXT_PUBLIC_SIGNUP_REDIRECT_AFTER` env vars on every deploy, so any build-time read returns `undefined`. The template has been permanently broken for signup since the strip landed; this phase closes the gap by moving config discovery from `next build` time to browser runtime via a same-origin `GET /landing-config` fetch.

The change surface is small (~100 LOC of TypeScript + 3 rustdoc edits + 1 version bump) but tightly coupled — all four template files must shift atomically (or in an ordering that keeps the template type-checkable at every step), and the `connect/page.tsx` is the only page that flips from server to client component. The contract shape is fully locked in CR-03; research adds no new decisions. The remaining risk surface is (a) getting the `'use client'` flip on `connect/page.tsx` right (module-scope `mcpClientConfig` must move inside the component), (b) not overreaching into `MCP_SERVER_NAME` reads (scope boundary), and (c) validating AC-11 (offline `next build && next start`) which has no automation path — it's a manual gate by design.

Primary recommendation: single-wave, single PLAN.md with ~5 tasks. Hook creation, 4 consumer rewrites (bundle or split — planner's call), rustdoc edits, version bump, final `make quality-gate`. All grep guardrails from CONTEXT.md lift into `must_haves` verbatim.

## Technical Approach

### Hook design invariants (`lib/useLandingConfig.ts`)

The planner's tasks MUST enforce these invariants — they are the contract the four consumers depend on:

1. **File path:** `cargo-pmcp/templates/landing/nextjs/lib/useLandingConfig.ts` (NEW). The `lib/` directory already exists (`lib/config.ts` is there today).
2. **Directive:** `'use client'` at line 1.
3. **Exports:** `export type LandingConfig`, `export type LandingConfigState = LandingConfig | null | 'error'`, `export function useLandingConfig(): LandingConfigState`. Types must be exported so consumers narrow via the same symbols.
4. **Abort-on-unmount:** `useEffect` must set a `cancelled` flag in cleanup and guard both `setCfg` calls with it. Prevents React warning "Can't perform a state update on an unmounted component" on route churn.
5. **Fetch:** `fetch('/landing-config', { cache: 'no-store' })`. The literal string `'/landing-config'` is the only matching occurrence allowed in the template (AC-4 guardrail).
6. **Error contract:** anything non-OK (network failure, 404, non-JSON, wrong shape) → `setCfg('error')`. No retry, no backoff, no telemetry.
7. **State narrowing:** consumers distinguish via `cfg === null` (loading), `cfg === 'error'` (failed), else loaded — then branch on `cfg.signup`.

### Server→client conversion — `connect/page.tsx` landmine

Today the file has no `'use client'` directive — it's a React Server Component. The module-scope consts `serverName` (line 6) and `mcpClientConfig` (lines 8–15) read `process.env.MCP_SERVER_NAME` at SSR time, which is fine for a server component but **will not work inside a client component** the same way (Next.js inlines `process.env.MCP_SERVER_NAME` at build via `next.config.js`'s `env:` block — so it technically *would* still resolve, but the moral semantics change). More importantly, `mcpClientConfig` references `cfg.mcpEndpoint` once the fetch lands — and `cfg` is only available per-render, not at module scope. So the constant **must move inside the component body**.

Concrete order of operations in the rewrite:

1. Add `'use client'` as line 1.
2. Replace `import Header from '../components/Header'` / `import ConnectSnippet from '../components/ConnectSnippet'` imports left as-is.
3. Delete module-scope `serverName` and `mcpClientConfig`.
4. Inside `export default function Connect()`: call `useLandingConfig()`, handle `null` / `'error'` with JSX early returns, then compute `const endpoint = cfg.mcpEndpoint` and build per-render `mcpClientConfig` if needed.
5. Interpolate `endpoint` (or `cfg.mcpEndpoint`) into each of the 4 snippets. Do NOT reassemble a URL from any `serverName` variable.
6. Existing `<Header />` usage stays. Header is independently converted to a client component in the Header rewrite — no SSR regression because no ancestor is currently server-rendered except this page.

## Risks & Landmines

1. **`output: 'export'` incompatibility with server-side fetch.** `next.config.js` locks `output: 'export'`. Any `getServerSideProps` or Route Handler that calls `fetch('/landing-config')` at build time would fail or bake a stale value. Client-side fetch inside `useEffect` is the only path. Planner should not consider alternatives.

2. **Four concurrent fetches on first paint.** Four hook callers (`Header` + current-page among `signup` / `callback` / `connect`) each fire independent fetches on hydration. The `/landing-config` response is <200B and same-origin; HTTP caching is explicitly off (`no-store`). CR-03 §"Why bundle" + CONTEXT §"Hook design" both accept this. Planner should NOT add module-level caching. Note in plan; don't fix.

3. **`cargo doc` regenerates from source.** Any regression to the rustdoc text (e.g., an autofix that touches the file) would silently hide the fix. Validation must include both `grep -n 'NEXT_PUBLIC_SIGNUP_REDIRECT_AFTER' cargo-pmcp/src/landing/config.rs` (expect 0) AND `cargo doc --no-deps -p cargo-pmcp` (expect success). `cargo doc` is not currently part of `make quality-gate`, so the plan task MUST invoke it explicitly.

4. **No template test harness.** `package.json` has no jest/vitest/playwright. Verification is: grep guardrails (cheap, deterministic), `tsc --noEmit` (catches hook/consumer type drift), `next build` (catches static-export-breaking errors), manual `next start` check (AC-11). Planner should NOT propose adding a test framework — explicitly deferred.

5. **`cargo clippy --features full` pedantic lints.** The rustdoc edit is pure comment text; clippy's `missing_docs_in_private_items` / `doc_markdown` lints *could* fire if the new text contains bare identifiers rendered as `CamelCase` items (e.g., `ServerOAuthConfig.signupRedirectAfter`). Mitigation: wrap bare identifiers in backticks (`` `ServerOAuthConfig` ``, `` `signupRedirectAfter` ``) OR test with `make quality-gate` and fix only if it fires. Flag for planner as a verification-step concern, not a blocking design issue.

6. **Scope boundary drift.** A naive "remove env vars from landing template" search-and-replace will delete `MCP_SERVER_NAME` reads and churn branding in 6 files. This is the exact failure mode CR-03 §0 was written to prevent. Guardrail: `grep -r 'MCP_SERVER_NAME' cargo-pmcp/templates/landing/nextjs/` → **≥1** (today's baseline: 7 files matching; after this phase still 7 or 6 if `connect/page.tsx` branding is relocated — never 0).

7. **Version-bump ordering.** Bumping `Cargo.toml` to 0.8.1 before quality-gate passes means a failed build still mutates the version. Must be last task in the plan.
