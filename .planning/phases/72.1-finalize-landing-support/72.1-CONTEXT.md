## Phase Boundary

Ship the SDK-side template + rustdoc changes that make the `pmcp new --template landing` scaffold functional again after the pmcp.run platform (Phase 71.1) actively strips `NEXT_PUBLIC_*` env vars on every deploy.

Replace build-time env-var reads with a single runtime fetch of `/landing-config` (served same-origin by Amplify 200-rewrite → API GW → Lambda → DynamoDB; always HTTP 200) via a new required shared React hook `useLandingConfig`.

Target release: `cargo-pmcp 0.8.1` (patch bump — additive, no breaking API changes).

**In scope (SDK):**
- New shared hook `lib/useLandingConfig.ts`
- Rewrite 4 consumer files to use the hook
- Delete all `NEXT_PUBLIC_COGNITO_*` / `NEXT_PUBLIC_LANDING_CLIENT_ID` / `NEXT_PUBLIC_SIGNUP_REDIRECT_AFTER` reads in the template
- Fix 3 rustdoc references in `cargo-pmcp/src/landing/config.rs`
- Bump `cargo-pmcp` 0.8.0 → 0.8.1

**Out of scope:** sign-in flow; subscription/account/settings/try-live pages; A/B testing; custom-domain template for non-pmcp.run hosts; SSR variant of the template; tests for the template itself (template has no test harness today — see decision "Template testing" below).

## Implementation Decisions

### Contract (locked by CR-03, no SDK discretion)

- **Endpoint:** `GET {origin}/landing-config` (same-origin via Amplify rewrite — no CORS, no auth, `Cache-Control: no-store`).
- **Response shape (always HTTP 200):**
  ```typescript
  type LandingConfig = {
    mcpEndpoint: string;             // e.g. "https://cost-coach.us-east.true-mcp.com/mcp"
    signup: SignupConfig | null;     // null when OAuth disabled / pool not provisioned
  };
  type SignupConfig = {
    cognitoDomain: string;           // PREFIX only, e.g. "pmcp-cost-coach-mncj4och"
    cognitoRegion: string;           // FULL region, e.g. "us-east-1" (never shortname)
    landingClientId: string;         // public client ID (confidential secret stays server-side)
    signupRedirectAfter: string;     // relative path, default "/connect"
  };
  ```
- **404 handling:** template treats 404 the same as `'error'` (disabled state); no fallback URL assembly.
- **No `NEXT_PUBLIC_*` fallback window:** env-var reads are deleted outright (platform actively strips them on every deploy — keeping the reads creates silent confusion).

### Scope boundary — `MCP_SERVER_NAME` STAYS

The env-var `MCP_SERVER_NAME` is injected at build time (`dev.rs:89`, `deploy.rs:417`) and drives branding in 7 template files. It must continue to work. Only the **endpoint-URL assembly** in `connect/page.tsx` (currently `` `https://${serverName}.us-east.true-mcp.com` ``) is replaced by `cfg.mcpEndpoint`. Header / title / hero / install / config.ts / next.config.js / connect-page-branding reads of `MCP_SERVER_NAME` are untouched.

Grep guardrails:
- `grep -rE 'NEXT_PUBLIC_COGNITO|NEXT_PUBLIC_LANDING_CLIENT_ID|NEXT_PUBLIC_SIGNUP_REDIRECT_AFTER' cargo-pmcp/templates/landing/nextjs/` → **0 matches**.
- `grep -r 'MCP_SERVER_NAME' cargo-pmcp/templates/landing/nextjs/` → **≥1 match** (6 existing files must still match — enforcement is the ≥1 bound from AC-3).
- `grep -rE "fetch\('/landing-config'" cargo-pmcp/templates/landing/nextjs/` → **exactly 1 match** (the one inside the hook).
- `grep -n 'NEXT_PUBLIC_SIGNUP_REDIRECT_AFTER' cargo-pmcp/src/landing/config.rs` → **0 matches**.

### Hook design

- `'use client'` directive (hook is browser-only).
- Uses `useEffect` with an abort flag to avoid state updates after unmount.
- `fetch('/landing-config', { cache: 'no-store' })`.
- Promise chain: `r.ok ? r.json() : Promise.reject(r.status)` → `setCfg(c as LandingConfig)` on success; `setCfg('error')` on any failure (network, 404, bad shape).
- Three public states for consumers to narrow: `null` (loading), `'error'` (failed), `LandingConfig` (loaded — then branch on `cfg.signup`).
- Exported type: `export type LandingConfig = …; export type LandingConfigState = LandingConfig | null | 'error';`.
- No retry, no backoff — if fetch fails, consumers render the disabled state; platform deploys are idempotent and the user will reload.
- No caching across hook instances — each consumer calls `useLandingConfig()` independently. That's ~4 concurrent fetches on first paint. Acceptable because the endpoint is same-origin, cached HTTP-wise is off by design, and the response is small (<200B). A shared context / SWR-style cache is a future optimisation, not needed for 0.8.1.

### Template testing

The landing template has **no existing test harness** (no jest/vitest/playwright in `package.json`). This phase does not introduce one — it's a template, not an app. Verification relies on:
- Grep assertions (AC-2, AC-3, AC-4, AC-12) — cheap and deterministic.
- `pnpm --dir cargo-pmcp/templates/landing/nextjs tsc --noEmit` (or `npx -y typescript tsc --noEmit`) to catch type errors in the hook and its consumers.
- `pnpm --dir cargo-pmcp/templates/landing/nextjs build` (runs `next build` with `output: 'export'`) to validate AC-10.
- Rust: `cargo doc --no-deps -p cargo-pmcp` succeeds and the regenerated docs for `SignupConfig` / `redirect_after` describe the runtime `/landing-config` mechanism (AC-12).
- Rust: `cargo clippy --workspace --all-targets --features full -- -D warnings` passes (existing quality gate).
- Rust: `cargo fmt --all -- --check` passes.
- Project quality gate: `make quality-gate` passes (matches CI exactly).

**No runtime regression harness for the Next template** — AC-11 ("next build && next start without network access to /landing-config") is validated manually once during this phase; automating it would require a Playwright rig the template doesn't have and the SDK team has said explicitly is out of scope.

### Ordering / dependencies

The hook file MUST exist before any consumer rewrites land (otherwise the consumers won't type-check). Valid execution orderings:

1. Hook → signup → callback → connect → Header → rustdoc → Cargo.toml bump, **or**
2. All consumer rewrites + hook creation in one atomic commit (preferred — smaller review surface, no intermediate broken build states in git history).

The Rust rustdoc fix is independent of the template changes (separate crate, separate file). It can land in the same commit or a separate one.

The Cargo.toml version bump lands last in the phase, after all quality gates pass — bumping before the work is done means a failed build produces an already-bumped tag.

### Claude's discretion

- **Snippet ergonomics in `connect/page.tsx`** — CR-03 says "Exact snippet text, client-specific config shape, and copy-to-clipboard interactivity are SDK team discretion." The existing `ConnectSnippet` component API can be reused or extended as needed. Minimum: 4 snippets (Claude Desktop, Claude Code, ChatGPT, Generic MCP CLI), each referencing `cfg.mcpEndpoint`.
- **Loading-state copy** — "Loading connection instructions…" and "Connection instructions unavailable. Please try again." are reasonable defaults; writer may adjust.
- **Wave grouping** — single-wave vs. multi-wave is at planner's discretion. Given the tight coupling (hook + 4 consumers) and small surface area (~100 LOC of TypeScript + 3 rustdoc edits + 1 version bump), single wave, one PLAN.md is sensible.
- **Task granularity inside PLAN.md** — splitting by file is fine; bundling the 4 TypeScript consumer rewrites into one task is also fine. Executor needs each task to have read_first and acceptance_criteria.

<specifics>
## Specific Implementation Notes

### Hook file — copy verbatim from CR-03 §1, then extend with a minor clarity tweak
The CR prescribes the hook body. That's the baseline. The implementer may add inline comments that reference this CONTEXT.md rather than invent new behavior. Do NOT:
- Add retry logic (out of scope).
- Add a module-level cache (deferred — see decision above).
- Change the return-value encoding from `LandingConfig | null | 'error'` to anything else — consumers are locked to this discriminated shape.

### `connect/page.tsx` — the only page that flips server→client
Currently imports at the top of the file (`Header` and `ConnectSnippet`) are fine for a client component, but the file needs a `'use client'` directive added at line 1. The `mcpClientConfig` constant defined at module scope must move inside the component body (it references `cfg.mcpEndpoint` which is only available per-render). Keep the SEO/branding header — it's rendered via `<Header />` which is now itself a client component (Header was already effectively client-only because of the conditional button work, so no SSR regression).

### Quality-gate integration
- `cargo fmt --all -- --check` and `cargo clippy --workspace --all-targets --features full -- -D warnings` must pass. The rustdoc edits should not trigger clippy (pure comment changes); the Cargo.toml bump does not affect clippy.
- `cargo doc --no-deps -p cargo-pmcp` is not currently part of `make quality-gate`, but the acceptance criterion requires it succeed. Executor should run it explicitly in the verification step.
- `next build` requires Node 18+; check the dev environment. If Node/pnpm isn't available, AC-10 is flagged as a manual gate and the plan task marked `autonomous: false` for that verification step.

### Release
- cargo-pmcp is published from this workspace. 0.8.1 is a patch bump, so semver rules allow it without a PR from downstream crates (cargo-pmcp has no in-workspace reverse dependencies — `pmcp`, `mcp-tester`, `mcp-preview` don't depend on `cargo-pmcp`).
- Do NOT tag / push / publish in this phase. Release is a separate step (`/gsd-ship` or manual `git tag vX.Y.Z` per CLAUDE.md release workflow) taken after the phase merges to main.

</specifics>
