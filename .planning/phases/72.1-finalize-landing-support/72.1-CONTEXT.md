## Implementation Decisions

### Contract (locked by CR-03, no SDK discretion)

- **Endpoint:** `GET {origin}/landing-config` (same-origin via Amplify rewrite — no CORS, no auth, `Cache-Control: no-store`).
- **Response shape (always HTTP 200):**
  ```typescript
  type LandingConfig = {
    mcpEndpoint: string;             // e.g. "https://cost-coach.us-east.true-mcp.com/mcp"
    signup: SignupConfig | null;     // null when OAuth disabled / pool not provisioned
  };
  type SignupConfig = {
    cognitoDomain: string;           // PREFIX only, e.g. "pmcp-cost-coach-mncj4och"
    cognitoRegion: string;           // FULL region, e.g. "us-east-1" (never shortname)
    landingClientId: string;         // public client ID (confidential secret stays server-side)
    signupRedirectAfter: string;     // relative path, default "/connect"
  };
  ```
- **404 handling:** template treats 404 the same as `'error'` (disabled state); no fallback URL assembly.
- **No `NEXT_PUBLIC_*` fallback window:** env-var reads are deleted outright (platform actively strips them on every deploy — keeping the reads creates silent confusion).

### Hook design

- `'use client'` directive (hook is browser-only).
- Uses `useEffect` with a **cancel flag** (`let cancelled = false`) to ignore state updates after unmount. The flag does NOT abort the network request — there is no `AbortController` by design (CR-03 §1 verbatim). "Ignore-after-unmount" is the accurate characterization; earlier drafts called this "abort-on-unmount", which is imprecise and has been corrected (Codex L2).
- `fetch('/landing-config', { cache: 'no-store' })`.
- Promise chain: `r.ok ? r.json() : Promise.reject(r.status)` → `setCfg(c as LandingConfig)` on success; `setCfg('error')` on any failure (non-OK HTTP response, network error, or non-JSON parse failure).
- **No JSON shape validation.** The hook casts any 200 JSON body to `LandingConfig` via `c as LandingConfig` — it does NOT validate that `mcpEndpoint` is a string or that `signup` is `null | object`. The `'error'` state therefore covers transport / HTTP / parse failures ONLY, not shape mismatches. Earlier drafts claimed `'error'` covers "unexpected shape", which is inaccurate and has been corrected (Codex M1, disposition "a" — relax the plan text to match the verbatim CR-03 hook rather than introduce shape validation the platform side did not agree to).
- Three public states for consumers to narrow: `null` (loading), `'error'` (failed fetch/parse), `LandingConfig` (loaded — then branch on `cfg.signup`).
- Exported type: `export type LandingConfig = …; export type LandingConfigState = LandingConfig | null | 'error';`.
- No retry, no backoff — if fetch fails, consumers render the disabled state; platform deploys are idempotent and the user will reload.
- No caching across hook instances — each consumer calls `useLandingConfig()` independently. Realistic fan-out per page is **duplicate hook fetches across consumers, max ~2 per page under the current route layout** (`Header` always mounts plus the current-page component calling its own `useLandingConfig`). Earlier drafts said "~4 concurrent fetches on first paint", which overstates the actual per-page fan-out (Codex L1 — corrected). Acceptable because the endpoint is same-origin, HTTP caching is off by design, and the response is small (<200B). A shared context / SWR-style cache is a future optimisation, not needed for 0.8.1.

### Ordering / dependencies

The hook file MUST exist before any consumer rewrites land (otherwise the consumers won't type-check). Valid execution orderings:

1. Hook → signup → callback → connect → Header → rustdoc → Cargo.toml bump, **or**
2. All consumer rewrites + hook creation in one atomic commit (preferred — smaller review surface, no intermediate broken build states in git history).

The Rust rustdoc fix is independent of the template changes (separate crate, separate file). It can land in the same commit or a separate one.

The Cargo.toml version bump lands last in the phase, after all quality gates pass — bumping before the work is done means a failed build produces an already-bumped tag.

## Specific Implementation Notes

### Hook file — copy verbatim from CR-03 §1
The CR prescribes the hook body. That's the baseline. The implementer may add inline comments that reference this CONTEXT.md rather than invent new behavior. The hook:
- Uses a cancel flag (`let cancelled = false`) in the effect cleanup to ignore state updates after unmount. This is NOT `AbortController` — the fetch itself continues; only the `setCfg` calls are guarded. Characterize this as "ignore-after-unmount" (not "abort-on-unmount") in any narrative text (Codex L2).
- Does NOT validate the JSON shape of a 200 response. `setCfg(c as LandingConfig)` accepts any 200 JSON payload as-is. The `'error'` state therefore covers **non-OK HTTP response / network failure / non-JSON parse failure** — NOT "unexpected shape". Any shape-mismatch concerns are the platform's responsibility to enforce server-side; the template trusts the contract (Codex M1, disposition "a").

Do NOT:
- Add retry logic (out of scope).
- Add a module-level cache (deferred — see decision above).
- Add JSON shape validation (`typeof mcpEndpoint === 'string'`, etc.) — the CR-03 hook is verbatim; introducing shape checks would invent a contract the platform did not agree to.
- Introduce `AbortController` — the cancel flag is sufficient for this endpoint.
- Change the return-value encoding from `LandingConfig | null | 'error'` to anything else — consumers are locked to this discriminated shape.

### `connect/page.tsx` — the only page that flips server→client
Currently imports at the top of the file (`Header` and `ConnectSnippet`) are fine for a client component, but the file needs a `'use client'` directive added at line 1. The `mcpClientConfig` constant defined at module scope must move inside the component body (it references `cfg.mcpEndpoint` which is only available per-render). Keep the SEO/branding header — it's rendered via `<Header />` which is now itself a client component (Header was already effectively client-only because of the conditional button work, so no SSR regression).

### Consumer rewrite preservation rule (Codex M2)
CR-03 §2–§5 inline code blocks are a **specification of the new behavior**, not an instruction to replace whole files. When the CR-03 snippet contains only a subset of what's in the current file (e.g., Header's existing logo/nav), merge that subset into the existing file rather than overwriting. Preserve all existing styling (Tailwind classNames), copy, imports, component structure, and comments that are not required to change by CR-03. LOC-added budget across the 4 consumer files: ≤ 100 lines (rough regression guard enforced via `git diff --numstat` in Task 2's verify block).

### Quality-gate integration
- `cargo fmt --all -- --check` and `cargo clippy --workspace --all-targets --features full -- -D warnings` must pass. The rustdoc edits should not trigger clippy (pure comment changes); the Cargo.toml bump does not affect clippy.
- `cargo doc --no-deps -p cargo-pmcp` is not currently part of `make quality-gate`, but the acceptance criterion requires it succeed. Executor should run it explicitly in the verification step.
- `next build` requires Node 18+; check the dev environment. If Node/pnpm isn't available, AC-10 is flagged as a manual gate and the plan task marked `autonomous: false` for that verification step.

### Scaffold command (G7 — Codex M3)
The real repo command is `cargo pmcp landing init --template nextjs --output <dir> --server-name <name>` (confirmed via `cargo-pmcp/src/commands/landing/init.rs` — signature: `init_landing_page(project_root, template_name, output, server_name)`). CR-03 AC-1 uses the loose phrasing `pmcp new --template landing`; there is no `pmcp new` subcommand in this repo. The scaffold smoke test (G7) uses the real command; see PLAN.md Task 5 Step 5.4b.

- cargo-pmcp is published from this workspace. 0.8.1 is a patch bump, so semver rules allow it without a PR from downstream crates (cargo-pmcp has no in-workspace reverse dependencies — `pmcp`, `mcp-tester`, `mcp-preview` don't depend on `cargo-pmcp`).
