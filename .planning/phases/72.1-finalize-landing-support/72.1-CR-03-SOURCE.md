## Summary

Replace the four `NEXT_PUBLIC_*` build-time env vars introduced in CR-02 with a single runtime fetch of `/landing-config` on the landing origin. The response body unifies sign-up config and MCP endpoint discovery into one always-200 JSON object, eliminating the Next.js `output: 'export'` / build-time-baking trap that caused the UAT Test 7 hang.

Affected files:

**Template:**
- `app/signup/page.tsx` — uses `useLandingConfig` then redirects (was: read env vars then redirect)
- `app/signup/callback/page.tsx` — uses `useLandingConfig` then redirects (was: read env var then redirect)
- `app/connect/page.tsx` — uses `useLandingConfig`; snippets interpolate `cfg.mcpEndpoint` (was: interpolate `MCP_SERVER_NAME` with hand-assembled URL)
- `components/Header.tsx` — uses `useLandingConfig`; renders "Sign up" only when `cfg.signup !== null`
- `lib/useLandingConfig.ts` — **new**, required shared hook
- `pmcp-landing.toml.template` — no changes beyond CR-02

**Rust crate:**
- `src/landing/config.rs` — rustdoc fix only (lines 29, 180, 445): rewrite the three `NEXT_PUBLIC_SIGNUP_REDIRECT_AFTER` references to describe the new runtime-fetch contract so the doc comments stop lying about behavior.

**Not changed:** No new TOML fields. No CLI changes. No signature changes on `LandingConfig` / `SignupConfig` structs. `MCP_SERVER_NAME` env injection in `dev.rs:89` and `deploy.rs:417` is unchanged (see §0).

## 0. Scope boundary — what does NOT change

"Remove `NEXT_PUBLIC_*`" is precise: it means the four `NEXT_PUBLIC_COGNITO_*` / `NEXT_PUBLIC_LANDING_CLIENT_ID` / `NEXT_PUBLIC_SIGNUP_REDIRECT_AFTER` env vars the platform was injecting and now strips. It does **not** mean "purge all build-time env vars from the template."

**`MCP_SERVER_NAME` stays.** It is a different kind of signal — it brands the static, SEO-sensitive surfaces of the site (title tags, marketing copy, snippet headers). It's safe to bake at build time because renaming a server is a rare, intentional redeploy event, not a runtime state change like "admin enabled OAuth yesterday." It ships in the template via:

- `dev.rs:89` — local dev server env
- `deploy.rs:417` — `npm run build` env on Amplify

and is consumed in seven files:

- `app/layout.tsx` — `<title>` / metadata
- `components/Header.tsx` — logo label
- `components/Hero.tsx` — hero copy
- `components/Installation.tsx` — installation section text
- `lib/config.ts` — shared constant
- `next.config.js` — build-time passthrough
- `app/connect/page.tsx` — connect-page header text (NOT the endpoint assembly)

Of these, only the last one is touched by this CR — and even there, the change is narrow: the **endpoint-URL assembly** in `connect/page.tsx` (currently something like `` `https://${serverName}.us-east.true-mcp.com` ``) is replaced by `cfg.mcpEndpoint`. Any header / label / copy usage of `process.env.MCP_SERVER_NAME` in that same file stays.

**Why state this explicitly?** Past CRs that said "remove env-var X" have been overinterpreted as "remove all env-var reads that look similar." The resulting diffs churn branding copy and force marketing rewrites. The boundary here is: kill the four `NEXT_PUBLIC_*` reads (platform actively strips them, they're always `undefined`); keep `MCP_SERVER_NAME` (platform actively injects it, it's always correct).

## Problem

CR-02 shipped the `/signup` surface reading four `NEXT_PUBLIC_*` env vars baked at `next build` time by the pmcp.run `deploy-landing` Lambda. Three problems surfaced in Phase 71 UAT and were resolved at the **platform** layer in Phase 71.1:

1. **Static-export trap.** The default landing template uses `output: 'export'` + `next build` runs at Amplify build time. Any `NEXT_PUBLIC_*` value read from `process.env` at build time is frozen into the JS bundle. When the operator later runs `cargo pmcp landing deploy` the build runs again, but `deploy-landing` had no way to mutate the already-uploaded static bundle without rebuild. If any of the four values changed after the initial deploy (e.g., Enable OAuth after the landing existed), the browser kept the stale value.
2. **"Re-deploy to pick up Cognito changes" guidance** was toxic for a platform promising "just `cargo pmcp deploy` and it works".
3. **Env-var stripping** — as of Phase 71.1, `deploy-landing` *actively removes* the four `NEXT_PUBLIC_*` keys from every Amplify app on every deploy. The template's current behavior (read stale env vars or `undefined`) is now permanently broken on any platform-side redeploy.

Phase 71.1 shipped the runtime alternative on the platform:

- Per-server Amplify **native custom domain** at `{server}-landing.{region}.{domain}` (pretty URL, verified live).
- **Amplify 200-rewrite** from `{landing-origin}/landing-config` to an API Gateway–backed Lambda that reads from DynamoDB at request time. Same-origin to the browser → no CORS preflight.
- **Always-200 contract** — unified sign-up + MCP endpoint shape. See §"Contract" below.

Ever since the platform strip landed, the template has been non-functional for signup. This CR closes the gap.

## Contract

### Endpoint

`GET {origin}/landing-config`

Served same-origin by Amplify 200-rewrite → API GW → Lambda → DynamoDB. No CORS, no auth. `Cache-Control: no-store`.

### Response — always HTTP 200

```typescript
type LandingConfig = {
  /** Public MCP endpoint for this server. Always present.
   *  Example: "https://cost-coach.us-east.true-mcp.com/mcp" */
  mcpEndpoint: string;

  /** Null when sign-up is disabled (OAuth not enabled, or the 'landing'
   *  Cognito client has not been provisioned yet). Hide Sign-up UI in this case. */
  signup: SignupConfig | null;
};

type SignupConfig = {
  /** Cognito domain PREFIX — NOT the full hosted-UI host.
   *  Example: "pmcp-cost-coach-mncj4och"
   *  Browser composes: `https://${cognitoDomain}.auth.${cognitoRegion}.amazoncognito.com/signup` */
  cognitoDomain: string;

  /** FULL AWS region (e.g., "us-east-1"). NEVER the shortname ("us-east"). */
  cognitoRegion: string;

  /** Cognito app client ID for the 'landing' Day-1 client. Confidential client;
   *  browser only sends it in the authorization code request, never the secret. */
  landingClientId: string;

  /** Relative path to redirect to after Cognito callback. Default "/connect". */
  signupRedirectAfter: string;
};
```

### Example responses

**OAuth enabled (sign-up available):**
```json
{
  "mcpEndpoint": "https://cost-coach.us-east.true-mcp.com/mcp",
  "signup": {
    "cognitoDomain": "pmcp-cost-coach-mncj4och",
    "cognitoRegion": "us-east-1",
    "landingClientId": "hf8kf0cvpjc3jsh0rl6rh1f18",
    "signupRedirectAfter": "/connect"
  }
}
```

**OAuth disabled / pool not yet provisioned:**
```json
{
  "mcpEndpoint": "https://cost-coach.us-east.true-mcp.com/mcp",
  "signup": null
}
```

**Server unknown (misconfigured deploy):** HTTP 404 `{ "error": "server_not_found" }`. This should not happen in practice — `deploy-landing` always resolves the server before wiring the rewrite rule. Template treats 404 the same as `signup: null` AND falls back on a hand-assembled endpoint only if it has one (see `mcpEndpoint` fallback below).

### Why this shape (vs the superseded 4-key `/signup-config`)

- **Always-200** — browser fetch-then-render code paths don't have to branch on HTTP status just to decide whether to show a button. The `signup === null` check is idiomatic and colocated with the Sign-up UI rendering.
- **Unified `mcpEndpoint`** — the `/connect` page currently interpolates `MCP_SERVER_NAME` to hand-assemble `https://<name>.us-east.true-mcp.com` (CR-02 divergence D1). That's fragile: it hardcodes `.us-east.true-mcp.com`, breaking on `pmcp.run` (prod), `us-west`, and any future per-customer domain. Server-derived `mcpEndpoint` eliminates the hardcode and keeps the URL truthful when servers move regions.
- **One endpoint, one fetch, one cache** — `/signup/page.tsx` and `/connect/page.tsx` both need `mcpEndpoint`; unifying avoids a second fetch.

## Template changes

### 1. `lib/useLandingConfig.ts` — new, required shared hook

**Required**, not optional. All four consumer files below MUST route their fetch through this hook. The identical fetch+state pattern would otherwise end up copied four times, and any future fix (abort-on-unmount, retry policy, auth, telemetry) would have to land in four places — a guaranteed drift vector.

```typescript
'use client'
import { useEffect, useState } from 'react'

export type LandingConfig = {
  /** Public MCP endpoint — always present. */
  mcpEndpoint: string
  /** Sign-up config, or null when OAuth is not enabled / landing client missing. */
  signup: {
    cognitoDomain: string
    cognitoRegion: string
    landingClientId: string
    signupRedirectAfter: string
  } | null
}

/**
 * Resolved states:
 *   null      — still loading
 *   'error'   — fetch failed / 404 / unexpected shape
 *   LandingConfig — successful load (signup may itself be null when disabled)
 */
export type LandingConfigState = LandingConfig | null | 'error'

export function useLandingConfig(): LandingConfigState {
  const [cfg, setCfg] = useState<LandingConfigState>(null)
  useEffect(() => {
    let cancelled = false
    fetch('/landing-config', { cache: 'no-store' })
      .then((r) => (r.ok ? r.json() : Promise.reject(r.status)))
      .then((c) => { if (!cancelled) setCfg(c as LandingConfig) })
      .catch(() => { if (!cancelled) setCfg('error') })
    return () => { cancelled = true }
  }, [])
  return cfg
}
```

Consumers distinguish three states by type narrowing: `null` → loading, `'error'` → failed, `LandingConfig` → loaded (then branch on `cfg.signup`).

### 2. `app/signup/page.tsx`

```typescript
'use client'

import { useEffect } from 'react'
import { useLandingConfig } from '@/lib/useLandingConfig'

export default function Signup() {
  const cfg = useLandingConfig()

  useEffect(() => {
    if (cfg === null || cfg === 'error') return
    if (!cfg.signup) return  // rendered as "not available" below

    const { cognitoDomain, cognitoRegion, landingClientId } = cfg.signup
    const origin = window.location.origin
    const url = new URL(`https://${cognitoDomain}.auth.${cognitoRegion}.amazoncognito.com/signup`)
    url.searchParams.set('client_id', landingClientId)
    url.searchParams.set('response_type', 'code')
    url.searchParams.set('scope', 'openid email')
    url.searchParams.set('redirect_uri', `${origin}/signup/callback`)
    window.location.replace(url.toString())
  }, [cfg])

  const disabled = cfg === 'error' || (cfg && !cfg.signup)
  return (
    <main className="min-h-screen flex items-center justify-center bg-gray-50">
      <p className="text-gray-600">
        {disabled ? 'Sign-up is not available for this server.' : 'Redirecting to sign-up…'}
      </p>
    </main>
  )
}
```

### 3. `app/signup/callback/page.tsx`

```typescript
'use client'

import { useEffect } from 'react'
import { useRouter } from 'next/navigation'
import { useLandingConfig } from '@/lib/useLandingConfig'

export default function SignupCallback() {
  const router = useRouter()
  const cfg = useLandingConfig()

  useEffect(() => {
    if (cfg === null) return  // still loading
    const redirect = (cfg !== 'error' && cfg.signup?.signupRedirectAfter) || '/connect'
    router.replace(redirect)
  }, [cfg, router])

  return (
    <main className="min-h-screen flex items-center justify-center bg-gray-50">
      <p className="text-gray-600">Welcome! Redirecting…</p>
    </main>
  )
}
```

### 4. `app/connect/page.tsx`

Server components can't call `fetch('/landing-config')` without an absolute URL (no runtime origin at render time). Convert to a client component and use the hook:

```typescript
'use client'

import { useLandingConfig } from '@/lib/useLandingConfig'
import { ConnectSnippet } from './components/ConnectSnippet'

export default function Connect() {
  const cfg = useLandingConfig()

  if (cfg === null) {
    return <main className="p-8"><p className="text-gray-600">Loading connection instructions…</p></main>
  }
  if (cfg === 'error') {
    return <main className="p-8"><p className="text-gray-600">Connection instructions unavailable. Please try again.</p></main>
  }

  const endpoint = cfg.mcpEndpoint

  return (
    <main className="p-8">
      <h1>Connect your MCP client</h1>
      <p>Your account is ready. Pick your MCP client to get connected.</p>

      <ConnectSnippet
        title="Claude Desktop"
        config={{ mcpServers: { server: { url: endpoint } } }}
      />
      <ConnectSnippet
        title="Claude Code"
        command={`claude mcp add server ${endpoint}`}
      />
      <ConnectSnippet
        title="ChatGPT"
        instructions={`Open ChatGPT → Settings → Beta features → MCP Connectors → Add → ${endpoint}`}
      />
      <ConnectSnippet
        title="Generic MCP CLI"
        command={`npx mcp-client ${endpoint}`}
      />
    </main>
  )
}
```

Exact snippet text, client-specific config shape, and copy-to-clipboard interactivity are SDK team discretion. The contract the template owes the platform: **use `cfg.mcpEndpoint` verbatim — do not hand-assemble the URL from `MCP_SERVER_NAME` + hardcoded domain**. Header / marketing uses of `MCP_SERVER_NAME` in this same file are unaffected (see §0).

### 5. `components/Header.tsx`

Header currently renders the Sign-up button unconditionally. New behavior:

```typescript
'use client'

import Link from 'next/link'
import { useLandingConfig } from '@/lib/useLandingConfig'

export function Header() {
  const cfg = useLandingConfig()
  const signupEnabled = cfg !== null && cfg !== 'error' && cfg.signup !== null

  return (
    <header>
      {/* ... existing logo / nav / MCP_SERVER_NAME branding unchanged ... */}
      {signupEnabled && (
        <Link href="/signup" className="btn-primary">Sign up</Link>
      )}
    </header>
  )
}
```

Render the button only when the hook has resolved AND `cfg.signup !== null`. No button flash during the pre-fetch state (show nothing until resolved).

### 6. `NEXT_PUBLIC_*` env-var removal

All four `process.env.NEXT_PUBLIC_COGNITO_*` / `NEXT_PUBLIC_LANDING_CLIENT_ID` / `NEXT_PUBLIC_SIGNUP_REDIRECT_AFTER` reads in the template must be **deleted**, not defaulted. The platform actively strips these keys on every deploy — leaving the reads in place creates silent confusion for any developer who sees them in the template and assumes they still work. `MCP_SERVER_NAME` reads are NOT removed (see §0).

## Notes for SDK reviewers

- **Why not SSR?** The template uses `output: 'export'` to keep Amplify hosting cheap (static CDN delivery, no SSR Lambda). Client-side `fetch('/landing-config')` is the lowest-friction solution that respects that constraint. If the SDK team later ships an SSR variant, the landing-config fetch can move to `getServerSideProps` / a Route Handler and become invisible to the user — but that's a future evolution, not a blocker for 0.8.1.
- **Why always-200 instead of 404 on disabled?** The 4-key `/signup-config` predecessor returned 404 when sign-up was disabled. That made the Header component's "should I show Sign up?" decision couple to HTTP semantics, and it duplicated the fetch (once for the button, once for the `/signup` redirect) because the 404 body was empty. Single always-200 endpoint, one fetch, one state variable.
- **Why bundle `mcpEndpoint` here and not a separate `/mcp-info` endpoint?** The `/connect` page needs both `mcpEndpoint` and to know whether sign-up is available (to render the "Signed up? Great, here's how to connect" vs. "Here's how to connect if you're already a user" copy, at the SDK team's discretion). One fetch, one source of truth.
- **No `NEXT_PUBLIC_*` fallback.** Some CRs recommend a transition window where the template checks env vars first, then falls back to the fetch. We explicitly DO NOT want that here: the platform's strip guarantees the env vars are absent, and a fallback path creates two possible states the template has to reason about. The fetch is the only path.
