# Cross-AI Plan Review — Phase 90.1

## Codex Review

## Summary

The three plans are generally strong and well-aligned with Phase 90.1: they keep scope to config, docs, examples, and replay coverage; they reuse the SQL parity patterns; and they explicitly protect the known `base_url` landmine. The main weakness is Plan 01's handling of P901-EXAMPLE: a build-only Rust example with inline TOML may not fully satisfy "a runnable example config users can point at," especially because `tests/fixtures/london-tube.toml` is excluded from published crates.

### 90.1-01

**Strengths**
- Correctly treats this as pure config/example/test work with no runtime parser or assembly changes.
- Strong guardrails around `deny_unknown_fields`, URI exact matching, and the `base_url` string-replace anchor.
- Good fixture assertions: checking `cfg.resources` and `cfg.prompts` directly catches parse/schema regressions cheaply.
- Security posture is appropriate: placeholder API key, dev-only token secret guard, no real secrets.
- Build-only example avoids accidentally starting a long-running server in CI.

**Concerns**
- **MEDIUM:** The plan may not fully satisfy "runnable example config users can point at." `london_tube_min.rs` embeds TOML in Rust, and the enriched fixture lives under `tests/`, which the research says is excluded from crate publishing. That is not the same as a user-facing example config path.
- **MEDIUM:** The example verification uses `cargo build --example`, not `cargo run --example`. If `main` contains config parsing or dispatch assumptions that only fail at runtime, build-only will not catch them.
- **LOW:** Task 1 verification runs `london_tube_fixture` before the new resource/prompt asserts are added. It still validates parsing, but it does not prove the enrichment until Task 2.
- **LOW:** The acceptance criterion "absence of a real credential" is stated but not concretely specified. A grep for `${TFL_APP_KEY}` is good; proving absence of arbitrary secrets is harder.

**Suggestions**
- Add a user-facing config file, e.g. `crates/pmcp-openapi-server/examples/london-tube.toml`, or explicitly document why the fixture is intentionally the pointable config despite being test-only.
- If adding `examples/london-tube.toml`, make `london_tube_min.rs` read it or mirror it closely enough that drift is obvious.
- Prefer `cargo run --example london_tube_min -p pmcp-openapi-server` if the example exits immediately as planned.
- Add an assertion for all three resource URIs, not only `docs://london-tube/schema`, to catch prompt/resource drift earlier.

**Risk Assessment: MEDIUM.** The fixture/test work is low risk, but P901-EXAMPLE has an acceptance ambiguity that could leave the phase technically incomplete.

### 90.1-02

**Strengths**
- Correctly scoped to documentation only.
- Good insertion points and clear sync requirement between book and course.
- The TOML walkthrough requirements cover the real operator pitfalls: strict keys and exact `include_resources` URI matching.
- mdBook rendering is included, which is the right validation for documentation changes.

**Concerns**
- **LOW:** The plan asks docs to mention internal implementation details like `assemble.rs`, `builder.resources_arc()`, and `builder.prompt_arc()`. Useful for SDK maintainers, but possibly too implementation-heavy for user-facing docs.
- **LOW:** Verification calls `mdbook build` directly, while project constraints emphasize using `make`/project scripts where available.
- **LOW:** "Reuse EXACT TfL excerpts" can cause docs to become overly fixture-like.

**Suggestions**
- Keep the implementation wiring explanation, but phrase it primarily in user-facing terms: "the server registers static resources and prompts at startup from config."
- Use the project's preferred doc build commands if they exist, or explicitly state that direct `mdbook build` is the local chapter-render gate.
- Include a short note that resources are static markdown context, not executable scripts.

**Risk Assessment: LOW.** Straightforward documentation parity with good validation.

### 90.1-03

**Strengths**
- Clean dependency on Plan 01 only; Plan 02 is correctly independent.
- Minimal and focused: adds only `list_resources` and `list_prompts`, mirroring SQL exactly.
- Avoids unverified `get_prompt` support, which is the right conservative choice.
- The existing replay harness makes the new steps binding without extra Rust changes.
- Maintains offline/single-threaded discipline.

**Concerns**
- **LOW:** `exists path: resources` and `exists path: prompts` prove the fields exist, but may not prove the expected resource URI or prompt name is exposed. The served surface could theoretically list empty arrays and still pass depending on `exists` semantics.
- **LOW:** The acceptance criterion says `parity_replay.rs` unchanged, but Plan 01 already modifies that file. This means "unchanged by Plan 03," which could confuse execution tracking.

**Suggestions**
- If `mcp-tester` supports array/content assertions, assert `docs://london-tube/schema` and `start_code_mode` in the scenario output. If not, keep the current minimal SQL-parity check.
- Reword acceptance to "Plan 03 does not modify `parity_replay.rs`."
- Run the full `parity_replay` test after Plan 03, not only the named replay test, if time permits.

**Risk Assessment: LOW.** Tightly scoped and likely to pass. The only weakness is that scenario assertions may be shallow.

### Overall Risk

**LOW to MEDIUM.** The phase is well researched and the plans mostly execute the locked scope without unnecessary runtime changes. The main action item before execution is clarifying or strengthening P901-EXAMPLE by adding a real example `.toml` users can point at, or explicitly accepting the inline Rust example as the intended deliverable.

## Gemini Review

## 1. Summary
The implementation plans for Phase 90.1 are exceptionally well-structured and provide a surgical, low-risk path to bringing the OpenAPI built-in server to full showcase parity with the SQL server. By leveraging the existing "zero-runtime-code" capabilities of the `pmcp-server-toolkit`, the plans focus correctly on content authoring (TOML/Markdown) and testing (YAML/Rust). The strategy of mirroring the SQL "Gold Standard" patterns (`reference-config.toml`, `generated.yaml`, `sql_server_min.rs`) ensures that the developer experience remains consistent across both "Configuration-Only" server types.

## 2. Strengths
- **High-Signal Research:** The finding that the toolkit already supports all required config keys (Resources, Prompts, Annotations) correctly limits the scope to configuration and documentation, avoiding unnecessary code changes.
- **Landmine Awareness:** Plan 01 explicitly identifies and protects the brittle `base_url` string-replace anchor in `parity_replay.rs`.
- **Structural Parity:** The plans go beyond simple content updates by mirroring the *teaching shape* of the documentation (e.g., the `deny_unknown_fields` discipline note).
- **Verification Rigor:** The use of `mcp-tester` for offline replay parity (`list_resources`/`list_prompts`) provides empirical proof that the already-wired toolkit surface is actually reachable on the OpenAPI side.
- **Build-Only Example Safety:** Creating `london_tube_min.rs` as a build-only example prevents CI blocking on a live listener while still providing a pointable reference for users.

## 3. Concerns
- **URI Integrity Fragility (LOW):** Plan 01 Task 1 includes a manual check for `include_resources` URI alignment. This remains a manual synchronization point between `[[resources]]` and `[[prompts]]`.
- **Course Build Complexity (LOW):** Research Assumption A2 notes potential course-specific build nuances. Plan 02 assumes standard `mdbook build` works; if the course uses a different wrapper, the verification command might require a small adjustment.
- **Redundancy with Existing Examples (LOW):** `openapi_server_min.rs` already exists. The new `london_tube_min.rs` adds value but there is a slight risk of user confusion between the two "min" examples.

## 4. Suggestions
- **Cross-Reference in Docs:** In Plan 02, consider adding a specific "See also" link in the OpenAPI chapters pointing to the SQL server's `reference-config.toml`.
- **Validation check for `cost_hint`:** Since `AnnotationsDecl` only accepts specific strings for `cost_hint`, adding an explicit check in `london_tube_fixture` (Plan 01 Task 2) to verify these fields are correctly parsed (and not just present) would add a small layer of extra safety.
- **Prompt Body Verification:** Adding a single `get_prompt` step in `london-tube-scenarios.yaml` to assert that `start_code_mode` actually assembles its three resources would provide the "Full Showcase" depth promised by the phase goal.

## 5. Risk Assessment: LOW
**Justification:** The phase is purely additive and modifies no runtime logic. It extends existing, proven test harnesses and uses documentation patterns that have already been successful for the SQL server.

**Verdict:** The plans are approved for execution.

## Consensus Summary

Both reviewers independently confirm the plans are well-structured, correctly scoped to **zero-runtime-code** config/docs/test work, and faithfully mirror the proven SQL-server patterns. Overall risk converges on **LOW** (Gemini) to **LOW–MEDIUM** (Codex), with a single substantive disagreement on the example deliverable.

### Agreed Strengths
- **Landmine protection** for the byte-identical `base_url = "https://api.tfl.gov.uk"` anchor in `parity_replay.rs` (both call this out as a critical, well-handled detail).
- **Zero-runtime-code scoping** — the research finding that the toolkit already parses/wires resources/prompts correctly limits the blast radius (both).
- **Build-only example safety** — `london_tube_min.rs` as build-only avoids a live-listener CI hang (both).
- **Structural / teaching-shape parity** with the SQL chapters, not just content parity (both).
- **Empirical replay verification** via `mcp-tester` offline parity (both).

### Agreed Concerns
- **Shallow scenario assertions in Plan 03 (LOW, both):** `exists path: resources` / `exists path: prompts` prove the fields exist but not that the *expected* URI (`docs://london-tube/schema`) or prompt name (`start_code_mode`) is exposed — an empty array could theoretically pass. Both suggest asserting specific content if `mcp-tester` supports it; otherwise keep the minimal SQL-parity check. Plan 01's fixture asserts partially mitigate this.
- **URI-integrity is a manual sync point (LOW, both):** the `[[prompts]].include_resources` → `[[resources]].uri` match is verified by grep, not structurally enforced.

### Divergent Views — worth deciding before execution
- **The example deliverable (Codex MEDIUM vs. Gemini LOW):** Codex flags that "a runnable example config users can point at" may NOT be satisfied by a build-only inline-TOML Rust example, because `tests/fixtures/london-tube.toml` is excluded from published crates — it recommends adding a user-facing `crates/pmcp-openapi-server/examples/london-tube.toml` and/or switching the gate to `cargo run --example`. Gemini accepts the inline Rust example as sufficient (matching the SQL `sql_server_min.rs` precedent) and rates it LOW. **Decision needed:** either (a) add a pointable `examples/london-tube.toml` + document `pmcp-openapi-server --config <path>`, or (b) explicitly accept the inline Rust example as the intended deliverable and document why (SQL parity). The SQL analog uses the inline Rust example, so option (b) is defensible on parity grounds — but option (a) more literally satisfies "config users can point at."

### Optional enhancements (non-blocking, raised by one reviewer each)
- Assert all three resource URIs, not only `docs://london-tube/schema` (Codex).
- Value-check `cost_hint` parses to its allowed enum, not just present (Gemini).
- Add a `get_prompt` step to prove the prompt body assembles its resources — note this contradicts the locked RESEARCH decision to stop at `list_prompts` for SQL parity; treat as a deliberate depth-vs-parity tradeoff (Gemini).
- Reword Plan 03 acceptance "parity_replay.rs unchanged" → "Plan 03 does not modify parity_replay.rs" (Codex).
- Phrase doc wiring (`assemble.rs`, `resources_arc()`) in user-facing terms; add a "resources are inert markdown context, not executable scripts" note (Codex).

### Recommended action
No HIGH concerns. The plans are executable as-is. Before execution, resolve the **one divergent item** (example deliverable) — the cheapest robust choice is to add a pointable `examples/london-tube.toml` *and* keep the build-only Rust example, satisfying both readings. Run `/gsd:plan-phase 90.1 --reviews` to fold this (and the LOW assertion-depth suggestions) into the plans, or proceed and accept the inline-example reading.
