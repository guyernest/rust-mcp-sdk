# Phase 90.1 Plan 01: Enrich london-tube showcase fixture + ship pointable example & runnable min example Summary

Brought the OpenAPI `london-tube` showcase to SQL-config parity: the fixture now ships
three `[[resources]]`, a `start_code_mode` `[[prompts]]` block, and full per-tool
annotations (`cost_hint` "low"/"medium" + `open_world_hint`); a new published
`examples/london-tube.toml` is the pointable equivalent and a runnable
`london_tube_min.rs` demonstrates the resources/prompts-bearing config build-and-exit.

## What Was Built

### Task 1 — Enrich `tests/fixtures/london-tube.toml` (commit `59f01810`)
- Added `open_world_hint = true` + `cost_hint = "low"` to `get-tube-status` and
  `idempotent_hint`/`open_world_hint`/`cost_hint = "medium"` to `disrupted-lines-with-detail`.
- Added three `[[resources]]` blocks (`docs://london-tube/schema`,
  `docs://london-tube/examples`, `code-mode://learnings`) as inert TfL markdown
  (endpoints, statusSeverity<10 scale, lowercase line ids, Code Mode gotchas).
- Added a `start_code_mode` `[[prompts]]` block whose `include_resources` lists
  exactly the three resource URIs (Pitfall 2 / no orphan reference).
- `base_url = "https://api.tfl.gov.uk"` kept byte-identical (parity_replay anchor).

### Task 2 — Extend `london_tube_fixture` asserts (commit `ce7ac9f5`)
- Asserts all three resource URIs present (loop over the URI set with a descriptive
  failure that prints the collected URIs).
- Asserts the `start_code_mode` prompt present.
- cost_hint VALUE assert: `get-tube-status` annotations `cost_hint == Some("low")`
  PLUS an allowed-enum `matches!(.., Some("low"|"medium"|"high"))` check (not a mere
  `.is_some()` presence check).
- `temp_config_pointing_at` / `REFERENCE_BASE_URL` left untouched.

### Task 3 — Pointable config + runnable example (commit `2562e555`)
- `examples/london-tube.toml`: full-surface pointable showcase (2 tools w/ full
  annotations, the same 3 resources, the same `start_code_mode` prompt). Ships with
  the crate (`examples/` is not in Cargo.toml `exclude`). `${TFL_APP_KEY}` placeholder
  + dev-guarded inline `token_secret` only — no real credential.
- `examples/london_tube_min.rs`: `#[tokio::main]` build-and-exit example over a
  trimmed enriched CONFIG (one tool + one `[[resources]]` + `start_code_mode` prompt),
  prints `name / tools.len() / resources.len() / prompts.len()`, references
  `examples/london-tube.toml` in a comment, never serves.
- `parity_replay.rs::pointable_example_config_parses_and_validates`: smoke test that
  reads `examples/london-tube.toml`, parses via `from_toml_strict_validated`, and
  asserts the three URIs + prompt + `${TFL_APP_KEY}` placeholder presence.

## Threat Model Outcomes

- **T-90.1-01 (Information Disclosure):** mitigated — all three artifacts keep the
  `${TFL_APP_KEY}` placeholder and a dev-only guarded `token_secret`; the published
  `examples/london-tube.toml` smoke test asserts the placeholder is present.
- **T-90.1-02 (Tampering):** accepted — resource `content` is inert markdown served
  as context, never executed.
- **T-90.1-03 (Spoofing/Elevation):** mitigated — every TfL tool sets only
  `read_only_hint=true` (GET-only), no `destructive_hint`.
