## Assertion-Depth Investigation (the plan's conditional)

The plan asked to STRENGTHEN to per-URI/per-name assertions IF mcp-tester supports an
array/content or value-path assertion. Investigation of `crates/mcp-tester/src/scenario_executor.rs`:

- `Assertion::Contains` on an **array** path only matches array elements that are
  **strings equal** to the value (`scenario_executor.rs:556-566`). `list_resources`
  returns `resources` as an array of resource **objects** (`scenario_executor.rs:353-356`),
  and `list_prompts` returns `prompts` as an array of prompt objects
  (`scenario_executor.rs:379-382`). So a `contains` match on a URI/name is impossible —
  it returns `false` for non-string array elements.
- `get_value_at_path` does support indexed access (`resources[0].uri`,
  `scenario_executor.rs:766-788`), but an indexed `equals`/`matches` would be
  **order-dependent** — brittle to any change in resource/prompt list ordering.

Per the plan's documented fallback: because no robust order-independent per-URI/per-name
assertion is available, the minimal SQL-parity `success` + `exists` check is kept.
**Mitigation (noted per plan instruction):** Plan 01's fixture asserts in
`parity_replay.rs::london_tube_fixture` already verify all three resource URIs
(`docs://london-tube/schema`, `docs://london-tube/examples`, `code-mode://learnings`)
and the `start_code_mode` prompt name are present in the served config — so URI/name
specificity is covered at the fixture layer, and these scenario steps prove the
list-discovery surface exposes them.
