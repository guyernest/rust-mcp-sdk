## Summary

Phase 90.1 Plan 01 + 03 deliver: an enriched `tests/fixtures/london-tube.toml` (three
`[[resources]]` + one `start_code_mode` `[[prompts]]` + full per-tool annotations), a
user-pointable `examples/london-tube.toml` (ships with the published crate), a
runnable build-and-exit `examples/london_tube_min.rs`, extended `parity_replay.rs`
fixture asserts, and two new discovery steps in `london-tube-scenarios.yaml`.

Security posture is clean: no real credentials are committed. All three artifacts use
only the `${TFL_APP_KEY}` placeholder and dev-guarded inline `token_secret` values
correctly protected by `allow_inline_token_secret_for_dev = true`. Resource content is
inert static markdown with no executable payload. All TfL tools are correctly annotated
`read_only_hint = true` with no `destructive_hint`. The T-90.1-01/02/03 threat model
items are properly mitigated.

One warning-level issue was found: a minor URI cross-reference inconsistency in
`london_tube_min.rs` (the inline CONFIG's prompt references only one resource URI
while no matching resource exists in the CONFIG for the other two referenced by the
pointable example). The inline CONFIG is intentionally trimmed (documented in the
plan), but the prompt's single URI does match its single resource, so this is actually
consistent — however, the plan's "kept close enough to `examples/london-tube.toml` that
drift is obvious" goal raises a note about the divergence in surface size not being
flagged in the summary line. Two info items follow.

## Warnings

### WR-01: `london_tube_min.rs` summary println does not differentiate the trimmed CONFIG from the full pointable config

**File:** `crates/pmcp-openapi-server/examples/london_tube_min.rs:38-44`

**Issue:** The inline `CONFIG` in `london_tube_min.rs` deliberately holds a trimmed
subset: one tool, one resource (`docs://london-tube/schema`), and one prompt
(`start_code_mode` with `include_resources = ["docs://london-tube/schema"]`). The
`main` function prints:

```
london-tube assembled: 1 tool(s), 1 resource(s), 1 prompt(s) (build-only example)
```

This is correct for the trimmed CONFIG. However, the Phase 90.1-01 SUMMARY (line 89)
records the verification output as `"1 tool(s), 1 resource(s), 1 prompt(s)"` and
notes it as passing — which is accurate for the trimmed CONFIG.

The real risk is documentation drift: the doc-comment on line 15-17 says "that
pointable file carries the full two-tool / three-resource surface" and the header on
lines 1-17 accurately describes what `examples/london-tube.toml` provides. But someone
running `cargo run --example london_tube_min` and seeing "1 resource(s)" may expect 3
resources (matching the pointable file they were pointed to), causing confusion.

The plan explicitly calls for a trimmed CONFIG (plan Task 3 action B: "kept close
enough to `examples/london-tube.toml` that drift is obvious"), so this is within
design scope, but the output message does not indicate it is the trimmed variant. A
reader running both the example and the binary with the pointable config will see
different counts with no in-output explanation.

**Fix:** Add a note to the summary println that flags it as the trimmed (single-tool)
variant, matching the doc-comment intent:

```rust
println!(
    "{} assembled (trimmed example — full surface in examples/london-tube.toml): \
     {} tool(s), {} resource(s), {} prompt(s) (build-only example)",
    cfg.server.name,
    cfg.tools.len(),
    cfg.resources.len(),
    cfg.prompts.len()
);
```

This prevents user confusion when comparing example output to the pointable-config
count (2 tools, 3 resources, 1 prompt).

## Info

### IN-01: `pointable_example_config_parses_and_validates` does not assert cost_hint value for the pointable config

**File:** `crates/pmcp-openapi-server/tests/parity_replay.rs:57-85`

**Issue:** The `london_tube_fixture` test (lines 131-149) correctly asserts the
`cost_hint` value for `get-tube-status` equals `Some("low")` with an allowed-enum
cross-check. The `pointable_example_config_parses_and_validates` smoke test (lines
57-85) asserts the three resource URIs, the `start_code_mode` prompt, and the
`${TFL_APP_KEY}` placeholder — but does not assert `cost_hint` or any annotation
value for the pointable config. Since the plan's acceptance criterion for Task 3 does
not require a `cost_hint` assert in this smoke test, this is intentionally lighter than
the fixture assert. However, the pointable config's annotations are part of its
"showcase surface" claim, and a regression in `examples/london-tube.toml` that silently
drops or mutates `cost_hint` would not be caught by this test.

**Fix:** Add a `cost_hint` value assertion to `pointable_example_config_parses_and_validates`
mirroring the fixture assert, after the prompt assertion:

```rust
let status_tool = cfg.tools.iter().find(|t| t.name == "get-tube-status")
    .expect("pointable example ships get-tube-status");
let cost_hint = status_tool.annotations.as_ref().and_then(|a| a.cost_hint.as_deref());
assert_eq!(
    cost_hint, Some("low"),
    "pointable example get-tube-status cost_hint must be \"low\""
);
```

### IN-02: Scenario YAML comment line-number reference slightly misleading

**File:** `crates/pmcp-openapi-server/tests/fixtures/london-tube-scenarios.yaml:45`

**Issue:** The inline comment (added by Plan 03) states that URI/name specificity is
covered by Plan 01's fixture asserts "in parity_replay.rs (all three resource URIs +
the start_code_mode name)". This is accurate and useful. However, the comment does not
mention that the `pointable_example_config_parses_and_validates` test also asserts the
same three URIs for the published example config — a reader of the YAML may not know the
full assertion coverage picture. This is a minor documentation gap, not a correctness
issue.

The Plan 03 SUMMARY also reports "PASS (lines 31 < 49 < 57 < 73)" referring to the
`operation.type` lines rather than the step-name lines (step names appear two lines
earlier at 47 and 55 respectively). This is a labeling ambiguity in the summary
document, not in the committed files, and does not affect correctness.

**Fix:** Optionally extend the comment in the scenario YAML to mention both the fixture
assert and the `pointable_example_config_parses_and_validates` test as the URI-specificity
coverage layer. No functional change required.
