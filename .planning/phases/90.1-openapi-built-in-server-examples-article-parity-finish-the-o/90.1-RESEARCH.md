## Summary

Phase 90.1 brings the OpenAPI built-in server (`crates/pmcp-openapi-server`) to full
showcase parity with the SQL built-in server (`crates/pmcp-sql-server`) along three axes:
(1) the london-tube reference fixture grows from a thin 114-line parity fixture into a
fully-annotated showcase config mirroring SQL's 670-line `reference-config.toml`;
(2) the book/course chapters gain the one config-DEPTH section they currently skip —
a `[[resources]]` / `[[prompts]]` walkthrough; (3) the parity test + scenarios are
extended to assert the new resource/prompt surface resolves.

**The single most important finding — this is a PURE CONFIG + DOCS phase, zero runtime
code:** Both binaries parse the SAME `pmcp_server_toolkit::config::ServerConfig` struct.
`[[resources]]` (`ResourceDecl`), `[[prompts]]` (`PromptDecl`), and per-tool
`[tools.annotations]` (`AnnotationsDecl`) are ALREADY fully supported by the toolkit
parser `[VERIFIED: crates/pmcp-server-toolkit/src/config.rs:131-141, 701-785]`. The
OpenAPI binary's `assemble.rs` ALREADY wires `cfg.resources` → `StaticResourceHandler` →
`builder.resources_arc()` and `cfg.prompts` → `register_prompts` → `builder.prompt_arc()`
`[VERIFIED: crates/pmcp-openapi-server/src/assemble.rs:144-213, 264-295]`. Enriching
`london-tube.toml` requires NO parser change, NO struct change, NO assemble change. Every
field the SQL reference uses is already accepted on the OpenAPI side.

**Primary recommendation:** Treat this as a config-authoring + docs-authoring + test-extension
phase. Mirror `reference-config.toml`'s `[[resources]]`/`[[prompts]]`/`[tools.annotations]`
shapes onto london-tube (TfL = all read-only GET endpoints), add `list_resources`/`list_prompts`
steps to `london-tube-scenarios.yaml` (the exact steps SQL's `generated.yaml` has at lines 22-36),
add a `## Resources & Prompts (Code Mode Context)` section to both chapters, and add a runnable
example config. Gate via the existing `parity_replay.rs` harness — no new test file is needed,
extend the existing one.

### Pattern 1: `[[resources]]` block (mirror SQL `reference-config.toml`)
**What:** A statically-shipped resource — inline markdown content addressed by URI.
**When to use:** Schema docs, example scripts, and "learnings" that Code Mode loads as context.
**Target shape** `[VERIFIED: crates/pmcp-sql-server/tests/fixtures/reference-config.toml:269-415]`:
```toml
[[resources]]
uri = "docs://london-tube/schema"
name = "TfL Line API Schema"
description = "Endpoints, response shapes, and line ids for the TfL Line API"
mime_type = "text/markdown"
content = """
# TfL Line API (subset)
...endpoints, statusSeverity scale, sample line ids (victoria, central)...
"""
```
`ResourceDecl` fields accepted `[VERIFIED: config.rs:765-785]`: `uri` (String), `name`
(Option<String>), `description` (Option<String>), `mime_type` (Option<String>),
`content` (Option<String>). `#[serde(deny_unknown_fields)]` — any extra key is a hard
parse error.

### Pattern 2: `[[prompts]]` block — single Code Mode entry point
**What:** A prompt that bundles resource URIs into one assembled context body.
**Target shape** `[VERIFIED: reference-config.toml:661-670]`:
```toml
[[prompts]]
name = "start_code_mode"
description = "Load all context needed for Code Mode script generation"
include_resources = [
    "docs://london-tube/schema",
    "docs://london-tube/examples",
    "code-mode://learnings",
]
```
`PromptDecl` fields `[VERIFIED: config.rs:730-743]`: `name` (String), `description`
(Option<String>), `include_resources` (Vec<String>), `arguments` (Vec<PromptArgumentDecl>).
**Landmine:** `include_resources` URIs must reference resources that actually exist in the
config — `register_prompts` resolves them against the `StaticResourceHandler`
`[VERIFIED: assemble.rs:204-213]`. A URI typo points at a missing resource.

### Pattern 3: per-tool `[tools.annotations]` (TfL = read-only)
**Target shape** `[VERIFIED: reference-config.toml:197-200, 224-227, 256-259]`:
```toml
[tools.annotations]
read_only_hint = true
idempotent_hint = true
cost_hint = "low"      # "low" | "medium" | "high"
```
`AnnotationsDecl` fields `[VERIFIED: config.rs:701-721]`:
`read_only_hint` (bool), `destructive_hint` (bool), `idempotent_hint` (bool),
`open_world_hint` (bool), `cost_hint` (Option<String>). **All TfL tools are GET endpoints →
read-only, non-destructive.** `open_world_hint = true` is defensible for both tools (they
hit an external live API). Suggested mapping:
- `get-tube-status`: `read_only_hint=true, idempotent_hint=true, open_world_hint=true, cost_hint="low"`
- `disrupted-lines-with-detail`: `read_only_hint=true, idempotent_hint=true, open_world_hint=true, cost_hint="medium"` (multi-call fan-out)

The current fixture already has minimal annotations (`read_only_hint`/`idempotent_hint`);
enrichment adds `cost_hint` (+ optionally `open_world_hint`).

### Pattern 4: scenario steps for resource/prompt parity
**Target shape** `[VERIFIED: crates/pmcp-sql-server/tests/fixtures/generated.yaml:22-36]`:
```yaml
- name: List available resources
  operation:
    type: list_resources
  assertions:
  - type: success
  - type: exists
    path: resources

- name: List available prompts
  operation:
    type: list_prompts
  assertions:
  - type: success
  - type: exists
    path: prompts
```
Optionally add a `get_prompt` step asserting `start_code_mode` resolves (mcp-tester
supports prompt retrieval — verify the operation name against `mcp-tester` 0.7.0 before
authoring; `list_prompts` + invocation is the SQL-proven minimum).

### Anti-Patterns to Avoid
- **Touching the toolkit parser or `assemble.rs`:** The surface already works. Any struct
  change is out of scope and breaks the "no runtime code" decision.
- **Byte-copying `reference-config.toml`:** london-tube is HTTP/OpenAPI, not SQL. There is
  no `[database]`, no `[[database.tables]]`, no `[shared_policy_store]`, no SQL in resources.
  Mirror the STRUCTURE (resources/prompts/annotations), not the SQL content.
- **Adding a backend round-trip to prove resources:** Resources are static inline content;
  `list_resources` + `exists` is the parity proof. No new wiremock mounts needed for that.
- **Inline real secrets:** Keep `${TFL_APP_KEY}` placeholder + the dev-only inline
  `token_secret` guarded by `allow_inline_token_secret_for_dev = true` (already present).

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Resource/prompt config parsing | A custom TOML reader or new structs | Existing `ResourceDecl`/`PromptDecl`/`AnnotationsDecl` | Already parse with `deny_unknown_fields`; struct change is out of scope |

| Resource-list assertions | Manual JSON walking | mcp-tester `list_resources`/`list_prompts` ops | SQL `generated.yaml` proves the exact step shape |

reference. The work is authoring content into a config + two chapters + a scenario file, then
asserting the (already-wired) surface lists.

## Common Pitfalls

### Pitfall 1: `deny_unknown_fields` makes any key typo a hard failure
**What goes wrong:** A misspelled resource/prompt/annotation key fails `from_toml_strict_validated`.
**Why it happens:** Every section is `#[serde(deny_unknown_fields)]` `[VERIFIED: config.rs:101, 766]`.
**How to avoid:** Use ONLY the documented fields (listed in Patterns 1-3). Run the fixture-validity
test (`london_tube_fixture`) immediately after editing.
**Warning signs:** `ToolkitError::Parse` mentioning an unknown field.

### Pitfall 2: prompt `include_resources` referencing a non-existent URI
**What goes wrong:** `start_code_mode` includes a resource URI that no `[[resources]]` block defines.
**Why it happens:** `register_prompts` resolves URIs against the resource handler `[VERIFIED: assemble.rs:204-213]`.
**How to avoid:** Keep the `include_resources` list in exact sync with the `[[resources]]` `uri` values.
**Warning signs:** prompt resolves to an empty/partial body at replay time.

### Pitfall 3: the `parity_replay.rs` base_url string-replace anchor
**What goes wrong:** Enrichment that changes the `base_url = "https://api.tfl.gov.uk"` line breaks
`temp_config_pointing_at` (it asserts that exact string is present) `[VERIFIED: parity_replay.rs:144-151]`.
**How to avoid:** Leave the `[backend] base_url` line byte-identical. Add new blocks; don't reformat existing ones.
**Warning signs:** `"london-tube.toml must contain the base_url line to override"` assertion fails.

### Pitfall 4: the offline replay test must stay offline + single-threaded
**What goes wrong:** New scenario steps that hit the real TfL API or rely on shared process env break CI.
**How to avoid:** Resources/prompts are static — assert via `list_resources`/`list_prompts` (no backend).
Run with `--test-threads=1` (the harness uses an ephemeral port + per-process `TFL_APP_KEY` env).
**Warning signs:** flaky tests, or a step that needs network.

### Pitfall 5: `london-tube.toml` ships in NO crate tarball — but tests still read it
**What goes wrong:** Forgetting the fixture is excluded from the published crate (`exclude = ["tests/"]`
in `Cargo.toml` `[VERIFIED: crates/pmcp-openapi-server/Cargo.toml]`).
**How to avoid:** This is fine — fixtures are test-only. The runnable EXAMPLE (separate from the
fixture) is what ships and demonstrates usage. Don't try to make the fixture a published artifact.
