## ProvenanceClass model (net-new design — pre-decided here to remove ad-hoc detection)
Define `pub enum ProvenanceClass { ExcelTrusted, NonExcel, UmyaFabricated, UnknownStale }`.
Classification reads ORIGINAL bytes (docProps/app.xml `<Application>`/`<AppVersion>` + xl/workbook.xml
`calcPr@calcId`):
- ExcelTrusted — anchored `<Application>` starts_with "Microsoft Excel" AND a positive Excel marker
  is present (an `<AppVersion>` build string and a calcId that is NOT the umya sentinel) → ACCEPT.
- UmyaFabricated — `<Application>` says "Microsoft Excel" but the umya fabrication signal is present
  (calcId == 122211 sentinel AND/OR the `<AppVersion>` Excel build string is absent) → REFUSE
  (oracle/non-excel-app, or a distinct oracle/umya-fabricated reason-code).
- NonExcel — `<Application>` is not "Microsoft Excel" → REFUSE (existing behavior).
- UnknownStale — app.xml unreadable/absent but ZIP is otherwise valid → REFUSE fail-closed (do NOT
  default to trusted; the refuse decision and reason-code derive from this enum, never an ad-hoc check).
False-positive policy (document in code + SUMMARY): the ONLY path to ExcelTrusted requires BOTH the
anchored name AND a positive Excel marker; a real Excel file always carries an AppVersion build string,
so genuine Excel saves are not refused. If a real Excel file is ever observed with a sentinel calcId,
the positive-AppVersion marker still admits it (the sentinel alone never refuses).

## Trusted-fixture override (test paths only — addresses both-reviewer fixture risk)
Add a `TrustedFixtureMarker` mechanism: a committed fixture may ship a sibling
`<fixture>.provenance-override.json` (or an explicit `ProvenanceClass::ExcelTrusted` override passed
ONLY via a `#[cfg(test)]`/test-API entry) that lets reconcile tests run against fixtures whose authoring
tool (rust_xlsxwriter) does not write Excel provenance. The override is REFUSED on the production
compile path — production always classifies from raw bytes. A test MUST assert the override does not
weaken production refusal (see Task 2 behavior `override_does_not_weaken_production`).

## ZIP/XML hard limits (fail-closed DoS guards)
MAX_ZIP_ENTRY_BYTES, MAX_TOTAL_DECOMPRESSED_BYTES, MAX_XML_DEPTH, MAX_CELL_COUNT — each exceedance →
a typed oracle/* Error finding (PartTooLarge / DecompressBomb / XmlTooDeep / TooManyCells), never a panic
or unbounded allocation.
