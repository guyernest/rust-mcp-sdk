## D-09 auto-corpus concretization (pre-decided — Codex HIGH/MEDIUM; implement exactly)
- EVALUATION API: replay each case through the pure-Rust `sheet_ir` executor + `scalar_eval` (the same
  evaluator the reconcile path uses in Plan 04 — NAME it explicitly: `sheet_ir::evaluate_bundle(ir, inputs)
  -> outputs`, or the lighthouse's equivalent bundle-evaluation entry). Run BOTH the prior accepted
  version's IR and the candidate's IR through it; the prior's outputs ARE the golden (captured automatically).
  Do NOT assume an external evaluator exists — use the in-crate pure-Rust path.
- NUMERIC-BOUNDARY GENERATION: for each numeric input, generate cases at {default, default−step, default+step,
  declared min (if any), declared max (if any)}. `step` = the input's declared increment if the manifest
  carries one, else a documented heuristic (1 unit for integers; 1% of |default| for floats, min 0.01).
  Document the heuristic in code + SUMMARY.
- GRID CAP: MAX_CORPUS_CASES = 50. Generation order is deterministic: the default case is ALWAYS first,
  then one case per enum member (others held at default), then numeric-boundary cases, truncated
  deterministically at 50 (stable input ordering). corpus_grid_is_bounded asserts ≤ 50 and never combinatorial.
- APPROVAL STORAGE: ApprovalRecords are written under `<out_root>/<bundle_id>/approvals/<fingerprint>.json`.
  ATOMIC WRITE: write to a temp file in the same dir, fsync where practical, then rename over the target
  (no partial files on crash). New bundle dirs likewise: write to a temp dir, then rename to
  `<bundle_id>@<version>/`.
