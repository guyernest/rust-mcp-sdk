<objective>
Fix mcp-preview's tool result delivery to use the widget readiness signal instead of fragile setTimeout, and emit both short-form and long-form method names for maximum compatibility.

Purpose: The current mcp-preview sends `ui/notifications/tool-result` (long form only) and uses a 300ms setTimeout before delivery. This is fragile (race condition on first load) and incompatible with widgets using the App class (which handles short form). By waiting for `ui/notifications/initialized` and sending both forms, mcp-preview becomes a reliable development proxy for both ChatGPT and standard MCP Apps hosts.

Output: Updated index.html with readiness-based delivery and dual method name emission.
</objective>

  <action>
In `crates/mcp-preview/assets/index.html`, locate the tool result delivery section (around lines 1630-1680) and make these changes:

1. **Add readiness signal listener**: Before `await this.loadResourceWidget(widgetUri)`, create a Promise that resolves when the widget sends `ui/notifications/initialized`:
   ```javascript
   const readyPromise = new Promise((resolve) => {
     const handler = (event) => {
       const msg = event.data;
       if (msg && msg.jsonrpc === '2.0' && msg.method === 'ui/notifications/initialized') {
         window.removeEventListener('message', handler);
         resolve();
       }
     };
     window.addEventListener('message', handler);
     // Fallback timeout: some widgets may not send initialized (legacy)
     setTimeout(() => {
       window.removeEventListener('message', handler);
       resolve();
     }, 3000);
   });
   ```

2. **Replace setTimeout with readiness await**: Change the `setTimeout(deliverToolResult, 300)` to:
   ```javascript
   await this.loadResourceWidget(widgetUri);
   await readyPromise;
   deliverToolResult();
   ```

3. **Update deliverToolResult to send BOTH method forms**: In the `deliverToolResult` function, change the postMessage call to send short form as primary:
   - Send `ui/toolResult` (short form) first -- this is what App class and ext-apps SDK listen for
   - Also send `ui/notifications/tool-result` (long form) as fallback for older widgets
   - Include `content` in params alongside `structuredContent` (ext-apps spec sends both)

   Updated postMessage block:
   ```javascript
   // Short form (primary -- matches ext-apps SDK App class)
   fr.contentWindow.postMessage({
     jsonrpc: '2.0',
     method: 'ui/toolResult',
     params: { structuredContent: result.structuredContent, content: result.content, _meta: responseMeta }
   }, '*');
   // Long form (fallback -- for older injected bridge scripts)
   fr.contentWindow.postMessage({
     jsonrpc: '2.0',
     method: 'ui/notifications/tool-result',
     params: { structuredContent: result.structuredContent }
   }, '*');
   ```

4. **Keep everything else unchanged**: The AppBridge delivery (ChatGPT mode lines 1641-1647), the `window.openai` property injection (lines 1661-1666), and the `widgetAlreadyLoaded` fast path (line 1670-1673) remain exactly as-is.

5. Also update the `wrapWidgetHtml` ready signal (around line 1875) from `setTimeout(() => { preview.logEvent('widgetReady', {}); }, 100)` to fire immediately after connect() resolves -- it currently uses a 100ms delay which is less fragile but still unnecessary given the init handshake.
