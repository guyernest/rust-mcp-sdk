  <name>Task 1: Add Bridge diagnostics tab to mcp-preview</name>
  <files>crates/mcp-preview/assets/index.html</files>
  <action>
Add a dedicated "Bridge" tab to mcp-preview's DevTools panel. This is a substantial but self-contained UI addition to the existing SPA.

**Tab UI structure:**
1. Add a "Bridge" button to the existing devtools tab bar (alongside Console, State, Protocol, Events).
2. Create a `bridge-tab` content div with three sections:
   a. **Mode Indicator**: Shows current preview mode (standard/chatgpt) as a read-only badge. No mode switching.
   b. **PostMessage Traffic Log**: Scrollable table/list showing every postMessage between host and widget iframe. Columns: timestamp (HH:MM:SS.mmm), direction arrow (host->widget or widget->host), method name, payload preview (truncated to 120 chars, click to expand full JSON).
   c. **Handshake Trace**: Step-by-step status visualization showing: (1) Widget iframe loaded, (2) Widget sends ui/initialize request, (3) Host responds with hostContext, (4) Widget sends ui/notifications/initialized. Each step shows a green checkmark when completed, grey when pending.

**Data capture implementation:**
1. Add a `BridgeDiagnostics` class to the PreviewRuntime:
   - `log` array: stores message entries `{ timestamp, direction, method, params, raw }`
   - `handshakeState` object: tracks which handshake steps have completed
   - `startCapture(iframe)`: installs a message listener on `window` that logs all messages from/to the widget iframe
   - `logOutgoing(method, params)`: called from deliverToolResult and AppBridge notifications to log host->widget messages
   - `render()`: updates the Bridge tab DOM with current log and handshake state

2. Hook into existing message flow:
   - In the existing `window.addEventListener('message', ...)` handler (the one that processes requests from the widget iframe), add a call to `this.bridgeDiagnostics.logIncoming(event)` BEFORE processing.
   - In `deliverToolResult()`, call `this.bridgeDiagnostics.logOutgoing('ui/toolResult', params)` after each postMessage send.
   - In `AppBridge.sendToolResult/sendToolInput` calls, add corresponding outgoing logs.
   - Track handshake steps: log when iframe loads (widgetLoaded event), when ui/initialize request arrives, when host responds, when ui/notifications/initialized notification arrives.

3. **Styling**: Match existing DevTools panel styling (dark background #1e1e2e, monospace font for log entries, same color scheme as Console tab). Use green (#4ade80) for host->widget arrows, blue (#60a5fa) for widget->host arrows. Handshake checkmarks use green (#4ade80) for complete, grey (#6b7280) for pending.

4. **Tab switching**: Follow the existing pattern used by Console/State/Protocol/Events tabs.

5. Keep the tab content lightweight -- no external dependencies, pure DOM manipulation matching the existing codebase style.
