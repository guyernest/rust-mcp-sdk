**New questions to ADD (replace outdated ones or append):**

1. **WidgetDir pattern** — What is WidgetDir and how does it differ from the old UIResourceBuilder approach? (Answer: WidgetDir serves HTML files directly from a directory on disk; no Rust code needed to define UI content. Files are authored as plain HTML and served automatically.)

2. **cargo pmcp app new** — What does `cargo pmcp app new <name>` generate? (Answer: scaffolds a new widget project with HTML template, manifest, and development configuration ready for `cargo pmcp app preview`)

3. **cargo pmcp app build** — What does `cargo pmcp app build` produce? (Answer: compiles and bundles the widget HTML/assets into the format expected by the MCP server for production serving)

4. **cargo pmcp app preview** — What does the preview workflow provide? (Answer: hot-reload development server that re-serves the widget on file changes so developers can iterate without restarting the MCP server)

5. **window.mcpBridge** — What is window.mcpBridge and when is it available? (Answer: a convenience API injected by MCP-aware hosts that wraps postMessage calls; available in Claude Desktop and MCP Apps hosts but NOT in raw iframe embedding)

6. **Adapter pattern** — What problem does the multi-platform adapter pattern solve? (Answer: different hosts — ChatGPT via window.openai, Claude via window.mcpBridge, generic MCP via postMessage — have different APIs; the adapter pattern provides a single interface that delegates to the correct host API at runtime)

7. **4-step common architecture** — What are the four steps of the common MCP Apps architecture pattern? (Answer: 1) detect host environment, 2) create platform adapter, 3) call tools through adapter, 4) render results in HTML)
