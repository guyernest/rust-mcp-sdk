**4. ## Chess: Stateless Game Widget** (## Chess: Stateless Game Widget)
- Teaching framing: "The chess example demonstrates the simplest and most recommended pattern: stateless tools. The widget owns ALL state. The server just validates and processes."
- ### Architecture
  - ASCII diagram: Widget (board.html) -> mcpBridge.callTool("chess_move", { state, move }) -> Server validates -> returns new GameState -> Widget updates board
  - "Why stateless? No sessions, no state cleanup, no scaling problems. The widget is the source of truth."
- ### Run It
  - ```bash
    cd examples/mcp-apps-chess
    cargo run
    # Server starts on http://localhost:3000
    cargo pmcp preview --url http://localhost:3000 --open
    ```
  - "Click a piece to see valid moves highlighted. Click a destination to make the move."
- ### Explore the Tools
  - Tools table: chess_new_game, chess_move, chess_valid_moves (input/output for each)
  - Show the GameState struct — "This entire struct travels with every request"
  - Show one tool handler (chess_move) with teaching annotations
  - "Notice: the handler is a synchronous function. No database, no sessions."
- ### Explore the Widget
  - Brief walkthrough of board.html structure
  - How it uses mcpBridge.callTool() for each interaction
  - "Try this:" — start a game, make a few moves, and watch the state object grow in the browser console
- ### Explore the Server
  - Show the ServerBuilder setup with tool_typed_sync_with_description
  - Show the ResourceHandler pattern (ChessResources with ChatGptAdapter + WidgetDir)
  - "session_id_generator: None — because it's stateless, no sessions needed"
- ### Key Takeaway
  - "The stateless pattern is the recommended default. Let the widget own state unless you have a specific reason to persist it on the server."

**7. ## The Common Pattern** (## The Common Pattern)
- Teaching framing: "All three examples share the same architecture. Here it is, distilled into four steps."
- ### Step 1: Define Tool Input Types
  ```rust
  #[derive(Deserialize, JsonSchema)]
  struct MyToolInput {
      query: String,
  }
  ```
  - "Derive Deserialize for JSON parsing. Derive JsonSchema for automatic schema generation."
- ### Step 2: Write Tool Handlers
  ```rust
  fn my_handler(input: MyToolInput, _extra: RequestHandlerExtra) -> Result<Value> {
      Ok(json!({ "result": "data" }))
  }
  ```
  - "Pure functions: input in, JSON out. No async needed for most tools."
- ### Step 3: Create ResourceHandler with Adapter + WidgetDir
  ```rust
  struct AppResources {
      chatgpt_adapter: ChatGptAdapter,
      widget_dir: WidgetDir,
  }
  ```
  - "The three-step read pattern: extract name → read from disk → transform with adapter."
- ### Step 4: Build and Run
  ```rust
  let server = ServerBuilder::new()
      .name("my-server")
      .version("1.0.0")
      .tool_typed_sync_with_description("my_tool", "Description", my_handler)
      .resources(AppResources::new(widgets_path))
      .build()?;
  ```
  - "That's it. Four steps. Every widget-based MCP server follows this pattern."
