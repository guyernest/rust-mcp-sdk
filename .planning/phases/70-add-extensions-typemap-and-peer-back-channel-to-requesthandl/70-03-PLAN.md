//! Peer back-channel trait for server-to-client RPCs from inside request handlers.
//!
//! Phase 70 / PARITY-HANDLER-01 (HANDLER-05 half).
//!
//! Implementations route outbound RPCs (`sampling/createMessage`, `roots/list`,
//! `notifications/progress`) to the client that originated the current request.
//! The trait is object-safe so `RequestHandlerExtra` can hold
//! `Option<Arc<dyn PeerHandle>>`.
//!
//! This module is non-wasm only; on wasm32 targets the server dispatch path does
//! not carry a peer, and handlers should treat `extra.peer()` returning `None`
//! as the normal case.

#![cfg(not(target_arch = "wasm32"))]

use crate::error::Result;
use crate::server::roots::ListRootsResult;
use crate::types::notifications::ProgressToken;
use crate::types::sampling::{CreateMessageParams, CreateMessageResult};
use async_trait::async_trait;

/// Server-to-client back-channel accessible from inside request handlers.
///
/// # Example
///
/// ```rust,no_run
/// use pmcp::{PeerHandle, RequestHandlerExtra};
/// # async fn inside_a_tool_handler(extra: RequestHandlerExtra) -> pmcp::Result<()> {
/// if let Some(peer) = extra.peer() {
///     let _roots = peer.list_roots().await?;
/// }
/// # Ok(())
/// # }
/// ```
#[async_trait]
pub trait PeerHandle: Send + Sync {
    /// Request the client to sample its LLM (`sampling/createMessage`).
    async fn sample(&self, params: CreateMessageParams) -> Result<CreateMessageResult>;

    /// Request the client's root list (`roots/list`).
    async fn list_roots(&self) -> Result<ListRootsResult>;

    /// Send a progress notification (`notifications/progress`).
    ///
    /// Returns `Ok(())` silently when no progress reporter is configured —
    /// matches the existing `RequestHandlerExtra::report_progress` no-op guard.
    async fn progress_notify(
        &self,
        token: ProgressToken,
        progress: f64,
        total: Option<f64>,
        message: Option<String>,
    ) -> Result<()>;
}
```

```rust
//! Concrete `PeerHandle` implementation that delegates to the
//! `ServerRequestDispatcher` (Phase 70 / PARITY-HANDLER-01).
//!
//! `DispatchPeerHandle` does NOT own a channel. It holds an
//! `Arc<ServerRequestDispatcher>` and delegates every outbound RPC to
//! `dispatcher.dispatch(...)`. The dispatcher owns the correlation layer
//! (pending oneshot map keyed by correlation id) and the drain-to-transport
//! task. This avoids the Codex-review-Finding-2 anti-pattern of ad-hoc
//! per-site channel construction.
//!
//! Deserialization: the dispatcher returns `serde_json::Value`; we parse
//! into the typed result here.

#![cfg(not(target_arch = "wasm32"))]

use std::sync::Arc;

use async_trait::async_trait;

use crate::error::{Error, ErrorCode, Result};
use crate::server::roots::ListRootsResult;
use crate::server::server_request_dispatcher::ServerRequestDispatcher;
use crate::server::ServerRequest;
use crate::shared::peer::PeerHandle;
use crate::types::notifications::ProgressToken;
use crate::types::sampling::{CreateMessageParams, CreateMessageResult};

/// `PeerHandle` that delegates outbound RPCs to a shared `ServerRequestDispatcher`.
pub(crate) struct DispatchPeerHandle {
    dispatcher: Arc<ServerRequestDispatcher>,
}

impl DispatchPeerHandle {
    pub(crate) fn new(dispatcher: Arc<ServerRequestDispatcher>) -> Self {
        Self { dispatcher }
    }
}

#[async_trait]
impl PeerHandle for DispatchPeerHandle {
    async fn sample(&self, params: CreateMessageParams) -> Result<CreateMessageResult> {
        let value = self
            .dispatcher
            .dispatch(ServerRequest::CreateMessage(Box::new(params)))
            .await?;
        serde_json::from_value::<CreateMessageResult>(value).map_err(|e| {
            Error::protocol(
                ErrorCode::INTERNAL_ERROR,
                format!("Invalid sample response: {e}"),
            )
        })
    }

    async fn list_roots(&self) -> Result<ListRootsResult> {
        let value = self
            .dispatcher
            .dispatch(ServerRequest::ListRoots)
            .await?;
        serde_json::from_value::<ListRootsResult>(value).map_err(|e| {
            Error::protocol(
                ErrorCode::INTERNAL_ERROR,
                format!("Invalid list_roots response: {e}"),
            )
        })
    }

    async fn progress_notify(
        &self,
        _token: ProgressToken,
        _progress: f64,
        _total: Option<f64>,
        _message: Option<String>,
    ) -> Result<()> {
        // Progress is a notification (one-way, no response) not a request/response.
        // The existing `Server::notification_tx: Sender<Notification>` channel is
        // the right vehicle, but DispatchPeerHandle doesn't hold a clone. For this
        // phase we preserve the existing `RequestHandlerExtra::report_progress`
        // no-op behavior: return Ok(()) silently. Future work (follow-on phase)
        // can plumb notification_tx through DispatchPeerHandle for live progress.
        Ok(())
    }
}

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| handler → client | Untrusted outbound RPC crossing from the server's tool/prompt/resource handler back to the originating client via `peer.sample()` / `peer.list_roots()`. |
| client → handler (response path) | Untrusted CreateMessageResult / ListRootsResult JSON arriving back through `dispatcher.handle_response`. |
| shared dispatcher → many peer handles | All DispatchPeerHandle instances in a given Server share one Arc<ServerRequestDispatcher>. Concurrency isolation is enforced by the dispatcher's internal RwLock + unique correlation ids. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-70-02 | Tampering | Per-request DispatchPeerHandle session isolation | mitigate | Each `Server` instance owns its own `ServerRequestDispatcher` bound to its own transport. Peer handles constructed under one Server cannot reach another Server's dispatcher — the `Arc<ServerRequestDispatcher>` is not shared across Server instances. Integration test `test_peer_sample_round_trip_through_dispatcher` proves a peer handle routes its dispatch to the dispatcher it was constructed with. Cross-session confusion requires cross-process access, which is out of the threat model. session_id is not populated on RequestHandlerExtra today; documented as a known limitation (follow-on phase can thread it through `ProtocolHandler::handle_request` when rmcp parity for session-scoped auth becomes a phase goal). |
| T-70-04 | Denial of Service | Malformed `CreateMessageResult` / `ListRootsResult` JSON from adversarial client | mitigate | `serde_json::from_value::<CreateMessageResult>(value)` returns `Err(serde_json::Error)` on malformed JSON; DispatchPeerHandle converts to a protocol error (INTERNAL_ERROR) rather than panicking. Plan 04 adds a fuzz target exercising this boundary explicitly. |
| T-70-05 | Denial of Service | Peer.sample()/list_roots() timeout | mitigate | Timeout is owned by the dispatcher (DEFAULT_DISPATCH_TIMEOUT = 60s, configurable via `with_timeout`). Unit test `test_peer_sample_propagates_dispatcher_timeout` asserts a short-timeout peer returns Err within 500ms. Pending-map cleanup is asserted by Plan 02's `test_dispatcher_timeout_cleans_pending`. |
| T-70-07 | Information Disclosure | DispatchPeerHandle in Debug output of RequestHandlerExtra | mitigate | Debug impl on RequestHandlerExtra formats peer as `"Arc<dyn PeerHandle>"` placeholder (Task 2 Step 3 Debug impl update) — does not expose the inner dispatcher, pending map, or transport state. |
| T-70-08 | Elevation of Privilege | Peer inherits caller's auth scope but carries no explicit auth check | accept | Peer access is inherited from the tool's authorization context. Tool-level authz runs at `src/server/core.rs:398-407` BEFORE the dispatch site wires peer — an unauthorized caller never reaches the handler body and therefore never sees `extra.peer()`. Rustdoc on `PeerHandle::sample` explains that peer calls inherit the originating tool's auth scope. |
| T-70-17 | Denial of Service | Dispatcher shared across all peers → one slow client backs up all in-flight dispatches | accept | The dispatcher's outbound channel is bounded at 100. A slow or disconnected client causes `dispatch()` calls to timeout within 60s (configurable). Tradeoff is explicit: one dispatcher = simpler correlation + single-transport semantics (a Server has one transport, so this matches the runtime topology). Per-session dispatchers are future work if multi-tenant isolation becomes a phase goal. |
</threat_model>
