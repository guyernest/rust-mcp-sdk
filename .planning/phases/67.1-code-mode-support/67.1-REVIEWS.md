# Cross-AI Plan Review — Phase 67.1

## Gemini Review

This review evaluates the 6-plan roadmap for implementing **Phase 67.1: Code Mode Support** in the PMCP SDK.

### 1. Summary
The implementation strategy is logically sound, moving from foundational crate migration to security hardening, developer experience (macros), and finally rigorous verification. The use of the `validate → approve → execute` pattern is a high-signal security feature for MCP servers. The plans correctly identify and address critical technical hurdles such as dependency mismatches (hmac/sha2), edition downgrades, and the need for `secrecy` integration. By prioritizing `trybuild` for macro testing and `cargo-fuzz` for parser stability, the roadmap aligns well with the "Toyota Way" and high-quality standards of the PMCP project.

### 2. Strengths
* **Security-First Foundation:** Integrating `secrecy::SecretBox` and `zeroize` at the core of `TokenSecret` (Plan 02) ensures that the high-level API is "secure by default" before the derive macro is even built.
* **Exhaustive Macro Testing:** Using `trybuild` with both compile-pass and specific compile-fail cases (Plan 04) is the gold standard for proc-macro development, ensuring clear error messages for SDK consumers.
* **Pragmatic Trait Design:** The `CodeExecutor` trait (Plan 03) provides a clean abstraction that hides the complexity of specific runtimes (JS, SQL, etc.) from the server logic.
* **High-Assurance Validation:** Plan 05 (Property tests + Fuzzing) is essential for security-sensitive code. Fuzzing the GraphQL/JS parsers specifically targets the highest risk area: untrusted input from LLMs.
* **Re-export Strategy:** `pub use async_trait::async_trait;` (Plan 02) is a sophisticated choice to prevent "dependency hell" for downstream users where macro-generated code might otherwise conflict with local `async-trait` versions.

### 3. Concerns
* **Macro Attribute Mapping (Plan 04):** **Severity: MEDIUM.** The plan mentions `#[derive(CodeMode)]` but doesn't explicitly define how it identifies which fields are the `PolicyEvaluator` and `CodeExecutor`. If the macro relies on fixed field names, it will be brittle.
* **Edition Downgrade Risks (Plan 01):** **Severity: LOW.** While the plan notes that no 2024 features are used, downgrading from 2024 to 2021 can occasionally trigger subtle changes in macro expansion or prelude imports.
* **Token Secret Leakage via Serialization (Plan 02/05):** **Severity: HIGH.** While `SecretBox` prevents `Debug` leaks, if the `TokenSecret` or the struct containing it derives `serde::Serialize` (common in MCP tools), the secret might be leaked in JSON responses unless explicitly skipped with `#[serde(skip)]`.
* **Code Mode "Execute" Tool Safety (Plan 04):** **Severity: MEDIUM.** If `register_code_mode_tools` automatically adds an `execute` tool, there must be a mechanism to ensure the tool's input (the code + token) is strictly validated *within* the generated tool handler before reaching the `CodeExecutor`.

### 4. Suggestions
* **Attribute Specification:** For Plan 04, define attributes like `#[code_mode(evaluator = "my_eval")]` or `#[code_mode(config = "mode_cfg")]` to allow developers flexibility in their struct layouts.
* **Negative Token Fuzzing:** In Plan 05, add a specific fuzz target for HMAC verification that attempts to "brute force" or find collisions by bit-flipping valid tokens.
* **Serde Protection:** In Plan 02, ensure the `TokenSecret` newtype explicitly does *not* implement `Serialize` or `Deserialize` (or provides a safe, no-op implementation) to prevent accidental JSON leakage.
* **Workspace Integration:** In Plan 01, explicitly add a step to update the root `Cargo.toml` `[workspace.members]` and `[workspace.dependencies]` to include the new crates. This ensures `make quality-gate` actually sees them immediately.
* **Feature Flag Cascading:** Ensure the `pmcp` root crate has a `code-mode` feature that enables the sub-crates, allowing users to opt-in to the extra dependencies (SWC, HMAC, etc.).

### 5. Risk Assessment
**Overall Risk: MEDIUM**

**Justification:**
The logic is sound and the technical choices (HMAC approval, `secrecy` crate, `trybuild`) are robust. The primary risk is **Integration Complexity**. Moving code from a sibling repo while simultaneously downgrading editions, upgrading crypto dependencies, and wrapping it all in a proc-macro creates multiple "moving parts." However, the inclusion of Wave 4 (Fuzzing and Property tests) acts as a strong safety net to catch regressions or security flaws introduced during the move. If Wave 4 is executed thoroughly, the final product should be production-ready.

## Codex Review

### Cross-Plan Observations
- The biggest structural gap is workspace integration: Plan 01 needs to own root `Cargo.toml` workspace/member/feature/docs.rs wiring, not just crate copy + local `Cargo.toml` edits.
- Plan 04's generated API shape is underspecified against this repo's actual builder style. `ServerCoreBuilder` is a by-value fluent builder, so a `register_code_mode_tools(builder)` function must return the builder or take `&mut` explicitly.
- Plan 05 should extend the existing root `fuzz/` crate and repo testing conventions, not invent crate-local fuzz infrastructure.
- Plan 06 is mis-ordered. The example depends on Plan 02 (`NoopPolicyEvaluator`, `TokenSecret`) and Plan 03 (`CodeExecutor`), not only Plan 04.

### Plan 01: Crate Scaffolding + Source Move
**Risk: MEDIUM-HIGH** — Base dependency for all later plans. Workspace-level wiring gaps (HIGH): root `[workspace].members`, features, docs.rs metadata, publish dry-run not in acceptance. Source fidelity check insufficient if original security invariants aren't fully test-covered.

### Plan 02: Security Hardening
**Risk: HIGH** — Most security-critical plan. TokenSecret must explicitly forbid `Debug`, `Display`, `Clone`, serde traits. Secret copies may leak if HMAC setup clones bytes into ordinary buffers. NoopPolicyEvaluator normalization risk. Needs constant-time token verification.

### Plan 03: CodeExecutor Trait
**Risk: MEDIUM-HIGH** — `execute(code, variables)` may be too narrow (no approval claims, timeout/cancellation, request metadata). "Explore blanket impl or defer" too vague for public API. `serde_json::Value` I/O weak contracts. No plan for error taxonomy.

### Plan 04: Derive Macro
**Risk: HIGH** — Builder signature underspecified (by-value vs `&mut`). Compile-fail coverage misses duplicate tool names, generic/lifetime edge cases, visibility/path hygiene, non-`'static` state. No explicit diagnostics/spans quality. Generated code may break under reduced-feature builds.

### Plan 05: Property Tests + Fuzz + Contracts
**Risk: MEDIUM-HIGH** — Fuzz targets should extend root `fuzz/` crate, not crate-local. Properties too positive-path; missing tampering, expiry, wrong-secret, wrong-code, Unicode normalization. `pmat comply check` ambiguity.

### Plan 06: Example + Docs + Quality-Gate
**Risk: MEDIUM** — Dependency ordering wrong (needs 02+03+04, not just 04). Single NoopPolicyEvaluator example normalizes insecure defaults. Missing `cargo publish --dry-run` acceptance. SECURITY.md needs operational guidance.

## Consensus Summary

### Agreed Strengths
- **Security-first design** — both reviewers praise the `secrecy` + `zeroize` foundation as the right approach for HMAC token secrets
- **trybuild testing for proc macro** — gold standard for derive macro development; praised by both
- **CodeExecutor abstraction** — clean one-trait-fits-all design recognized as pragmatic
- **Fuzz + property testing strategy** — appropriate for LLM-input-facing parsers; both see it as essential
- **async_trait re-export** — prevents downstream version conflicts; both call it a good DX move

### Agreed Concerns
1. **[HIGH] TokenSecret serde/serialization leakage** — Both flagged: if the struct derives `Serialize` (common in MCP tools), the secret could leak in JSON. Must explicitly forbid `Serialize`, `Deserialize`, `Debug`, `Display`, `Clone` on `TokenSecret`. Must audit all sites that extract raw bytes from the secret.
2. **[HIGH] Derive macro field identification / builder API** — Both flagged: Plan 04 doesn't specify how the macro identifies which fields are `PolicyEvaluator`, `CodeExecutor`, etc. (Gemini: attribute annotations needed; Codex: builder return type underspecified for by-value fluent API). Fixed field names are brittle.
3. **[HIGH] Plan 06 dependency ordering** — Codex flagged as wrong (depends on only Plan 04 but needs 02+03+04). Gemini didn't explicitly flag this but both agree the example is the primary onboarding path and must demonstrate the full security model.
4. **[MEDIUM] NoopPolicyEvaluator normalization risk** — Both warn that making the "allow-all" evaluator too prominent in docs/examples could normalize insecure defaults. Must be documented as test-only, and the example should show both success and rejection paths.
5. **[MEDIUM] Workspace-level integration gaps in Plan 01** — Root `Cargo.toml` workspace members, feature cascading, docs.rs metadata, and `cargo publish --dry-run` not explicitly covered as acceptance criteria.

### Divergent Views
1. **CodeExecutor trait surface area** — Codex wants `ExecutionContext` parameter (timeout, cancellation, metadata); Gemini accepts the simple `execute(code, variables)` API. *Resolution needed: is the simple API sufficient for v0.1.0, or will it box the project into a weak contract?*
2. **Fuzz infrastructure location** — Codex insists fuzz targets belong in the existing root `fuzz/` crate; Gemini doesn't mention this. *Resolution needed: check if a root fuzz/ crate exists and follow the existing pattern.*
3. **Overall risk calibration** — Gemini rates overall MEDIUM; Codex rates Plans 02 and 04 as HIGH individually. The difference reflects Codex's heavier weight on API-surface-hardening and downstream adoption risk.
4. **Attribute-based field mapping vs. fixed names** — Gemini suggests `#[code_mode(evaluator = "my_eval")]` attribute annotations; Codex doesn't propose a specific solution but flags fixed names as fragile. *Resolution needed: pick one approach before execution.*
