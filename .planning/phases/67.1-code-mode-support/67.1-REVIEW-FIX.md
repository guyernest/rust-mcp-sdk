## Fixed Issues

### CR-02: TokenSecret::new silently accepts empty secrets

**Files modified:** `crates/pmcp-code-mode/src/token.rs`, `crates/pmcp-code-mode/src/validation.rs`, `crates/pmcp-code-mode/tests/property_tests.rs`
**Commit:** 9a1c1fc6
**Applied fix:** Added minimum 16-byte length assertion in `HmacTokenGenerator::new` (alternative fix per review guidance -- less disruptive than changing `TokenSecret::new` API). Updated all test secrets from 11-15 bytes to 16 bytes (`b"test-secret-key!"`). Added `test_empty_secret_rejected` and `test_short_secret_rejected` tests.

### CR-03: Derive macro hard-codes static validation context

**Files modified:** `crates/pmcp-code-mode-derive/src/lib.rs`
**Commit:** da06bc98
**Status:** fixed: partial_fix (documentation only)
**Applied fix:** Added prominent WARNING comment in the generated code explaining that placeholder values are used and tokens are not session-bound. Updated crate-level documentation with a "Known Limitation (v0.1.0)" section documenting the issue and recommending workarounds. API surface unchanged per instructions.

### WR-01: validate_graphql_query never enforces blocked_queries or allowed_queries

**Files modified:** `crates/pmcp-code-mode/src/validation.rs`
**Commit:** 974ac1c0
**Applied fix:** Added query (read) authorization checks mirroring the mutation enforcement block. Checks `blocked_queries` first (blocklist rejects), then `allowed_queries` (allowlist rejects if non-empty and query not listed). Added two new tests: `test_blocked_query_rejected` and `test_allowed_queries_enforced`.

### WR-03: Subscription operations classified as "Write" in Cedar policy

**Files modified:** `crates/pmcp-code-mode/src/policy/cedar.rs`
**Commit:** a0aa24b9
**Applied fix:** Changed `"subscription" => "Write"` to `"subscription" => "Read"` in the Cedar action mapping, consistent with `GraphQLValidator::to_code_type` which already treats subscriptions as read operations.

### WR-04: normalize_path_to_pattern UUID detection is overly broad

**Files modified:** `crates/pmcp-code-mode/src/policy/types.rs`
**Commit:** 865d76c4
**Applied fix:** Extracted `is_uuid_like()` function that validates the 8-4-4-4-12 hex pattern instead of just checking `len == 36 && contains('-')`. Applied to both `normalize_path_to_pattern` and `normalize_operation_format` functions.

### WR-05: is_policy_configured default returns true

**Files modified:** `crates/pmcp-code-mode/src/handler.rs`
**Commit:** 364d055e
**Applied fix:** Flipped default from `true` to `false` (safe default). Updated doc comment to explain that implementations with a configured policy evaluator MUST override to return `true`. No tests relied on the `true` default.

### WR-06: payload_bytes uses Debug formatting for RiskLevel

**Files modified:** `crates/pmcp-code-mode/src/token.rs`
**Commit:** 3d83c74b
**Applied fix:** Changed `{:?}` (Debug) to `{}` (Display) for `risk_level` in the HMAC payload format string. Added doc comment noting this is a breaking change for pre-existing tokens (acceptable for v0.1.0 pre-release). Display produces stable all-caps strings ("LOW", "MEDIUM", "HIGH", "CRITICAL").

### WR-07: from_token_secret Vec copy not zeroized

**Files modified:** `crates/pmcp-code-mode/src/validation.rs`
**Commit:** a554a7c8
**Status:** fixed: partial_fix (documentation only)
**Applied fix:** Updated the doc comment for `from_token_secret` to honestly describe the unzeroized `Vec<u8>` intermediate copy limitation. Removed misleading "secure entry point" language. Documented that `TokenSecret::from_env` is preferred for maximum security, and that a future `HmacTokenGenerator::from_secret_ref` constructor will address the limitation.

## Skipped Issues

### CR-01: verify_code panics on hash strings shorter than 12 bytes

**File:** `crates/pmcp-code-mode/src/token.rs:272-275`
**Reason:** Already fixed in a prior pass. The current code (lines 269-278) uses bounds checking (`if token.code_hash.len() >= 12`) instead of direct slice indexing, which prevents the panic on short/malformed hashes from deserialized tokens.
**Original issue:** `token.code_hash[..12]` uses direct byte-index slicing that panics on short strings from deserialized tokens.

### WR-02: validate_graphql_query_async double-validates when no policy evaluator

**File:** `crates/pmcp-code-mode/src/validation.rs:317-319`
**Reason:** Skipped per instructions -- this is a refactoring task that risks breaking the validation pipeline. The double validation is a performance issue, not a correctness issue.
**Original issue:** When no policy evaluator is configured, the async variant re-runs the full parse and depth check by calling the sync `validate_graphql_query` method.
