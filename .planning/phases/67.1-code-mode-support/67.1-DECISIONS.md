## Context

Code Mode is a validate-then-execute pattern: an LLM client generates structured code (GraphQL, SQL, JavaScript-over-OpenAPI), the server validates it against policies and issues an approval token, then the client calls execute with code + token. This pattern exists today in `pmcp-run/built-in/shared/pmcp-code-mode` (used by OpenAPI, SQL, GraphQL, MCP-API, AWS Billing, Graph-RAG servers) and needs to become a first-class PMCP SDK primitive so external developers can adopt it consistently.

**Urgency:** Blocker for an upcoming MCP server launch. Takes precedence over continuing the docs-cleanup milestone (v2.1 Phase 68 is deferred behind 67.1).

---

## Decisions

### D1. Repo topology — **Option (a): move `pmcp-code-mode` into rust-mcp-sdk workspace**

- `pmcp-code-mode` becomes a first-class SDK crate under `crates/pmcp-code-mode/`.
- `pmcp-code-mode-derive` (proc macro) lands alongside it at `crates/pmcp-code-mode-derive/`.
- `pmcp-avp` (AWS Verified Permissions plugin) stays **out of scope** for this phase — it remains downstream in pmcp-run because of the `aws-sdk-verifiedpermissions` dependency and platform coupling.
- Rationale: SDK becomes the code-mode authority; publishing discipline consolidated in one workspace; third-party developers get consistent dependency story.

### D2. Version targets — **pmcp v2.3.0** (already bumped), new crates start at **0.1.0**

- Root `pmcp` is already `2.3.0` in Cargo.toml. Phase 67.1 will be delivered under v2.3.0.
- Spec's `pmcp = "2.0.2"` reference is outdated — plan must target **2.3.0**.
- New crates: `pmcp-code-mode = "0.1.0"`, `pmcp-code-mode-derive = "0.1.0"`.
- Publishing order update: `pmcp-widget-utils → pmcp → pmcp-code-mode → pmcp-code-mode-derive → mcp-tester → mcp-preview → cargo-pmcp`.
- Downstream `mcp-tester` (0.5.0), `mcp-preview` (0.3.0), `cargo-pmcp` (0.6.0) only bump if they consume the new crates.

### D3. Builder ergonomics — **keep it simple**

- **No `.code_mode(&server)` extension on `ServerBuilder`**. Avoid coupling `pmcp` core to code-mode concepts.
- Use the neutral `register_code_mode_tools(builder)` pattern from SDK_DESIGN_SPEC §5.
- Target audience: SDK is for **developers**, pmcp.run is for **administrators**. Developers wire the tools explicitly in `main()`; admins configure policies via the pmcp.run UI.
- First version now, advanced sugar (builder ext trait, config files, feature discovery) deferred to later phases.

### D4. `CodeExecutor` high-level trait — **adopt**

- Implement the single high-level `CodeExecutor` trait from SDK_DESIGN_SPEC §7. Collapses the three existing internal executor traits (`HttpExecutor`, `SdkExecutor`, `McpExecutor`) into one server-facing interface.
- Optimize for developer experience: the derive macro only needs to know "validated code → result". Implementation patterns A-D from the spec (SQL direct, JS+HTTP, JS+SDK, JS+MCP) are all valid and covered.
- **Stretch goal:** explore whether `PlanExecutor` can impl `CodeExecutor` via a blanket impl, collapsing the "~30 lines of wiring" patterns. If trivial, include in this phase; if it ripples, defer.

### D5. Token secret handling — **security is not optional**

- Do **not** store `token_secret` as a plain `Vec<u8>` field.
- Use `secrecy::SecretVec<u8>` or a newtype wrapper (`TokenSecret`) that:
  - Implements `Drop` with zeroization (`zeroize` crate)
  - Blocks `Debug` / `Display` printing
  - Guards against accidental clone/log exposure
- Add `secrecy` (+ `zeroize`) as dependencies of `pmcp-code-mode`.
- Document the threat model in the crate README and in a SECURITY.md.

### D6. Contract-first development — **mandatory**

- Write contract YAML under `../provable-contracts/contracts/pmcp-code-mode/` and `../provable-contracts/contracts/pmcp-code-mode-derive/` **before** implementation.
- Minimum contract coverage:
  - `PolicyEvaluator` trait invariants (`evaluate_operation`, `evaluate_script`, default-deny semantics)
  - `CodeExecutor` trait invariants (token pre-verified, errors propagated)
  - Derive-macro expansion contracts (required fields present, `Send + Sync` enforced, token verified before `execute`)
  - HMAC token generation/verification contract (bind code hash to approval)
- Run `pmat comply check` before and after implementation.

### D7. ALWAYS testing (fuzz, property, unit, example) — **per crate, with pragmatic scope**

- **Core `pmcp-code-mode`**:
  - Unit: existing tests migrate + cover new `CodeExecutor`, `NoopPolicyEvaluator`, `TokenSecret`
  - Property: HMAC round-trip invariants, validation pipeline determinism, policy-evaluator default-deny
  - Fuzz: `cargo fuzz` targets for GraphQL parser input, JavaScript parser input, token verification
  - Example: end-to-end `examples/XX_code_mode_graphql.rs` (or similar) with `NoopPolicyEvaluator`
- **`pmcp-code-mode-derive`**:
  - Unit: `trybuild` compile-pass and compile-fail snapshots (missing fields, non-`Send` fields, wrong types)
  - Property: for any valid struct shape in a generator, macro emits compiling code (if `trybuild` quickcheck integration is manageable; otherwise skip and document)
  - Fuzz: **skip** — macro-input fuzzing has low ROI without a full compiler harness. Fuzzing lives in the core crate.
  - Example: a complete server example showing `#[derive(CodeMode)] → register_code_mode_tools → validate → execute` round trip
- Pragmatic flexibility: if a property/fuzz target proves infeasible without a dedicated server, document the gap in the phase VERIFICATION.md rather than blocking on it.

### D8. "NOT YET" spec items are deliverables — **reference existing pmcp-run servers for details**

Spec §11 flags these as not-yet-in-codebase; Phase 67.1 delivers them:

- `pub use async_trait::async_trait;` re-export in `pmcp-code-mode/src/lib.rs`
- `NoopPolicyEvaluator` struct + impl
- `register_code_mode_tools(builder)` method (emitted by derive macro)
- `CodeExecutor` trait definition
- `TokenSecret` newtype (new, per D5)

**Reference implementations to consult** in pmcp-run for JavaScript-subset details, SQL validation specifics, MCP composition patterns, policy annotation handling:

- `pmcp-run/built-in/shared/pmcp-code-mode/src/` — the canonical source
- `pmcp-run/built-in/sql-api/servers/open-images` — reference OpenAPI-style server using Code Mode
- `pmcp-run/built-in/aws-billing-lambda/src/code_mode.rs` and `src/main.rs` — reference SDK-backed Code Mode server (manual wiring pattern the derive macro replaces)

### D9. Cross-repo migration — **SDK ships first, pmcp-run switches after**

- Phase 67.1 delivers the crates into rust-mcp-sdk and publishes them to crates.io.
- pmcp-run will then migrate its `built-in/shared/pmcp-code-mode` consumers to depend on the published `pmcp-code-mode` and `pmcp-code-mode-derive` crates.
- The old path (`pmcp-run/built-in/shared/pmcp-code-mode/`) may remain temporarily for parity verification but is not the source of truth after this phase ships.
- **Out of scope for 67.1:** the actual pmcp-run switchover — that's downstream work in a separate pmcp-run PR.

### D10. Scope — **split into multiple plans as needed**

- This is a legitimate `(INSERTED)` phase (blocks an imminent server launch), but the scope is larger than a typical urgent hotfix.
- Expected plan breakdown (the planner will finalize):
  1. **67.1-01:** Crate scaffolding + move `pmcp-code-mode` source into rust-mcp-sdk workspace (build green, existing tests pass)
  2. **67.1-02:** Security hardening — `TokenSecret` newtype, `NoopPolicyEvaluator`, `async_trait` re-export
  3. **67.1-03:** `CodeExecutor` trait + optional `PlanExecutor` blanket impl exploration
  4. **67.1-04:** `pmcp-code-mode-derive` proc macro crate — `#[derive(CodeMode)]`, `register_code_mode_tools`, `trybuild` tests
  5. **67.1-05:** Contract YAMLs, `pmat comply check`, property + fuzz targets
  6. **67.1-06:** End-to-end example + docs + CRATE-README + publish prep (publish itself happens in a release phase, not here)

Plans 01-04 can run in waves (01 blocks all; 02/03/04 can parallelize after 01; 05/06 close out).
