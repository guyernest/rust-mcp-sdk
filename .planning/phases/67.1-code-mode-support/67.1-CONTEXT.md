# Phase 67.1: Code Mode Support - Context

**Gathered:** 2026-04-11
**Status:** Ready for planning
**Source:** Design review discussion of `pmcp-run/built-in/shared/pmcp-code-mode/SDK_DESIGN_SPEC.md` + locked decisions in `67.1-DECISIONS.md`

<domain>
## Phase Boundary

This phase moves the Code Mode framework from pmcp-run's internal `built-in/shared/pmcp-code-mode` crate into the rust-mcp-sdk workspace as a first-class SDK primitive, adds a proc-macro derive crate, hardens token secret handling, introduces a high-level `CodeExecutor` trait, and delivers an end-to-end worked example.

**In scope:**
- Move + adapt `pmcp-code-mode` source into `crates/pmcp-code-mode/` (workspace member)
- New `crates/pmcp-code-mode-derive/` proc macro crate (`#[derive(CodeMode)]`)
- New `CodeExecutor` trait (high-level, replaces per-server executor glue)
- `TokenSecret` newtype with `secrecy` + `zeroize`
- `NoopPolicyEvaluator` for tests/local dev
- `async_trait` re-export from `pmcp-code-mode`
- Worked example: `XX_code_mode_graphql.rs`
- Contract YAMLs + `pmat comply check`
- Property/fuzz test coverage for core crate
- `trybuild` compile-pass/fail snapshots for derive macro
- Both crates slotted into publish order, `make quality-gate` green

**Out of scope:**
- `pmcp-avp` (AWS Verified Permissions plugin) — stays in pmcp-run, has `aws-sdk-verifiedpermissions` dep
- pmcp-run consumer migration — downstream work after SDK publishes
- `.code_mode()` builder sugar on `ServerBuilder` — deferred; use `register_code_mode_tools(builder)` pattern
- Runtime data-format changes to `pmcp` core crate

</domain>

<decisions>
## Implementation Decisions

### D-01: Crate location — `crates/pmcp-code-mode/` and `crates/pmcp-code-mode-derive/`
Move source from `pmcp-run/built-in/shared/pmcp-code-mode/` into `crates/pmcp-code-mode/`. Derive macro goes in sibling `crates/pmcp-code-mode-derive/`. Both become workspace members in root `Cargo.toml`.

### D-02: Version targets — pmcp v2.3.0 (already bumped), new crates at 0.1.0
Root `pmcp` is v2.3.0. New crates: `pmcp-code-mode = "0.1.0"`, `pmcp-code-mode-derive = "0.1.0"`.

### D-03: Builder ergonomics — `register_code_mode_tools(builder)` (no `.code_mode()`)
Derive macro emits a `register_code_mode_tools(builder: ServerBuilder) -> ServerBuilder` method. No coupling between `pmcp` core and code-mode concepts. Developers wire tools explicitly in `main()`.

### D-04: `CodeExecutor` high-level trait adopted
Single `execute(code, variables) -> Result<Value, ExecutionError>` method. Collapses `HttpExecutor`, `SdkExecutor`, `McpExecutor` into one server-facing interface. Stretch: explore `PlanExecutor` blanket impl.

### D-05: `TokenSecret` newtype with `secrecy` + `zeroize`
Replace plain `Vec<u8>` with a type that zeroes on drop, blocks Debug/Display. Add `secrecy` and `zeroize` as deps. Document threat model.

### D-06: `NoopPolicyEvaluator` in `pmcp-code-mode`
Always-allow evaluator for tests and local development. Implements `PolicyEvaluator` trait.

### D-07: `async_trait` re-export
`pmcp-code-mode/src/lib.rs` adds `pub use async_trait::async_trait;`. Derive macro emits `#[pmcp_code_mode::async_trait]` to avoid version conflicts.

### D-08: Derive macro generates `Send + Sync` static assertion
`const _: fn() = || { fn assert_send_sync<T: Send + Sync>() {} assert_send_sync::<MyServer>(); };` emitted by macro.

### D-09: Feature flags preserved from source
`cedar`, `openapi-code-mode`, `js-runtime`, `mcp-code-mode` — same flag names as the pmcp-run source. May rename for namespace clarity (planner decides).

### D-10: Contract-first per CLAUDE.md
Contract YAMLs under `../provable-contracts/contracts/pmcp-code-mode/` and `../provable-contracts/contracts/pmcp-code-mode-derive/`. `pmat comply check` must pass.

### D-11: Testing strategy
Core crate: unit + property (HMAC round-trip, validation determinism, default-deny) + fuzz (GraphQL parser, JS parser, token verification) + example.
Derive crate: `trybuild` compile-pass/fail + example. Fuzz skipped for macro input (documented gap).

### D-12: Publishing order
`pmcp-widget-utils → pmcp → pmcp-code-mode → pmcp-code-mode-derive → mcp-tester → mcp-preview → cargo-pmcp`

### Claude's Discretion
- Exact crate-move mechanism (`git mv` vs fresh import with attribution)
- Whether feature flags get namespace prefixes (`code-mode-openapi` vs `openapi-code-mode`)
- Whether derive macro goes as sibling crate or nested under `pmcp-code-mode`
- `trybuild` fixture structure and dev-deps strategy
- Example numbering (next available `XX_` prefix in `examples/`)
- Whether `PlanExecutor` blanket impl is feasible or deferred

</decisions>

<specifics>
## Specific Ideas

### From SDK_DESIGN_SPEC.md
- Macro emits `register_code_mode_tools(builder: ServerBuilder) -> ServerBuilder` that registers both `validate_code` and `execute_code` tools
- `ValidateCodeState` and `ExecuteCodeState` are generated internal types holding pipeline + token secret + policy evaluator refs
- HMAC token binds code hash to validation result — `HmacTokenGenerator::new(secret)` / `verify_token(token, code)`
- `ValidationPipeline::new(config, token_secret)` — signature verified against aws-billing usage
- Cedar policy evaluation is behind `cedar` feature flag, uses `cedar-policy = "4.8"`
- Runtime detection pattern: `POLICY_STORE_ID` env var selects AVP vs local Cedar

### From existing server implementations
- `pmcp-run/built-in/aws-billing-lambda/src/code_mode.rs` — Manual wiring pattern the derive macro replaces
- `pmcp-run/built-in/sql-api/servers/open-images` — Reference OpenAPI-style server using Code Mode
- Pattern A (SQL): direct execution, no JS runtime — `SqlValidator::new().validate(code, &config)`
- Pattern B (OpenAPI/GraphQL): `PlanExecutor::new(http_executor, exec_config)` + `PlanCompiler::compile(code)`
- Pattern C (AWS Billing): `PlanExecutor::with_sdk_executor(sdk_executor, exec_config)`
- Pattern D (MCP-API): `PlanExecutor::with_mcp_executor(mcp_executor, exec_config)`

</specifics>

<deferred>
## Deferred Ideas

- `.code_mode(&server)` builder extension trait on `ServerBuilder` — ergonomic sugar, but couples SDK core to code-mode; deferred to a later phase
- `pmcp-avp` crate (AWS Verified Permissions plugin) — separate crate with `aws-sdk-verifiedpermissions` dep, stays in pmcp-run for now
- pmcp-run consumer migration — the downstream switchover from internal to SDK crates
- Feature flag renaming (e.g., `code-mode-openapi` instead of `openapi-code-mode`) — only if there's a naming collision, otherwise preserve compat
- `PlanExecutor` blanket impl for `CodeExecutor` — explore during implementation, defer if non-trivial
- Batch policy evaluation optimization — the trait has `batch_evaluate` default impl, optimize later
- AVP Cedar schema management for external servers — platform concern, not SDK

</deferred>
