## Summary

Phase 67.1 moves `pmcp-code-mode` (15,583 lines, 149 existing tests) from `pmcp-run/built-in/shared/` into the `rust-mcp-sdk` workspace as `crates/pmcp-code-mode/`, adds a `crates/pmcp-code-mode-derive/` proc macro crate with `#[derive(CodeMode)]`, introduces a `TokenSecret` newtype using `secrecy` + `zeroize`, adds `CodeExecutor` high-level trait, and delivers `NoopPolicyEvaluator`, `async_trait` re-export, property/fuzz tests, trybuild snapshots, and an end-to-end example.

The critical technical risks are: (1) crypto dependency version mismatches between the source (`sha2 0.10`, `hmac 0.12`) and the SDK workspace (`sha2 0.11`) -- these MUST be upgraded to `sha2 0.11` + `hmac 0.13` which are confirmed compatible [VERIFIED: docs.rs/hmac/0.13.0]; (2) edition mismatch -- source uses `edition = "2024"` but ALL workspace crates use `edition = "2021"` -- must downgrade; (3) `pmcp` version -- source pins `pmcp = "2.0.2"` but workspace is at `2.3.0` -- must use path dependency; (4) SWC crate pinning -- parser/ast/visit/common must remain version-locked as a set (32/19/19/18) because SWC publishes tightly coupled breaking releases.

**Primary recommendation:** Fresh import (not `git mv`) into `crates/pmcp-code-mode/` with edition downgrade to 2021, crypto dep alignment to workspace versions, pmcp path dependency, and `cedar-policy` bump to 4.9. Derive macro follows `pmcp-macros` patterns exactly (syn 2.0, quote, darling, trybuild UI tests).

### Pattern 5: TokenSecret Newtype

**What:** Wraps token secret bytes with zeroize-on-drop and blocks Debug/Display.
**When to use:** Replaces all `Vec<u8>` token secret storage.

```rust
// Source: secrecy 0.10.3 API [VERIFIED: docs.rs/secrecy/0.10.3]
use secrecy::{ExposeSecret, SecretSlice};
use zeroize::Zeroize;

/// Zeroizing wrapper for HMAC token secrets.
///
/// - Zeroes memory on drop via `zeroize`
/// - Blocks Debug/Display to prevent logging
/// - Cannot be accidentally cloned or compared
pub struct TokenSecret {
    inner: Vec<u8>,  // Uses zeroize::Zeroizing<Vec<u8>> for auto-zeroize
}

// Alternative using secrecy directly:
pub struct TokenSecret(secrecy::SecretBox<[u8]>);

impl TokenSecret {
    pub fn new(secret: Vec<u8>) -> Self {
        Self(secrecy::SecretBox::new(Box::from(secret.as_slice())))
    }

    /// Expose the secret bytes for HMAC operations.
    /// Callers MUST NOT log or persist the returned slice.
    pub fn expose_secret(&self) -> &[u8] {
        self.0.expose_secret()
    }
}

impl Drop for TokenSecret {
    fn drop(&mut self) {
        // SecretBox handles zeroize on drop automatically
    }
}
```

**Design choice:** Use `secrecy::SecretBox<[u8]>` (aliased as `SecretSlice`) internally. This provides: (1) no Debug/Display impl, (2) zeroize on drop, (3) `ExposeSecret` for controlled access. The `TokenSecret` newtype adds domain-specific constructors (`from_env`, `new`). [VERIFIED: secrecy docs.rs]

### Anti-Patterns to Avoid

- **Token secret as `Vec<u8>`:** The whole point of D-05. Plain Vec leaks via Debug, Clone, log.
- **Direct `async_trait` dependency in derive macro output:** Use `#[pmcp_code_mode::async_trait]` to avoid version conflicts between the server's async-trait and the crate's.
- **Coupling `pmcp` core to code-mode:** No changes to root `pmcp` crate. The derive macro generates code that calls `builder.tool()` -- this is the standard registration API.
- **Edition 2024 in workspace:** All workspace crates use 2021. Breaking this convention causes subtle issues with `gen` keyword reservation and other edition-specific behavior.
- **Upgrading SWC versions:** SWC 32->38 is 6 major versions with breaking changes. Keep pinned at source versions.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Secret memory zeroing | Custom `Drop` with `ptr::write_volatile` | `secrecy` + `zeroize` crates | Compiler can optimize away hand-written zeroing; zeroize uses `core::ptr::write_volatile` correctly |
| HMAC-SHA256 | Custom HMAC implementation | `hmac` + `sha2` crates | Cryptographic primitives must be audited implementations |

## Common Pitfalls

### Pitfall 1: Crypto Dependency Version Mismatch

**What goes wrong:** Source uses `sha2 = "0.10"` and `hmac = "0.12"`. The SDK workspace uses `sha2 = "0.11"`. If you keep 0.10, the `Sha256` type from pmcp-code-mode is a DIFFERENT type than the one in pmcp core. Cargo resolves both versions but they can't interoperate.
**Why it happens:** The `hmac 0.12` crate depends on `digest ^0.10` which is incompatible with `sha2 0.11` (uses `digest ^0.11`).
**How to avoid:** Upgrade to `hmac = "0.13"` + `sha2 = "0.11"`. hmac 0.13 uses `digest ^0.11.2`, confirmed compatible. [VERIFIED: docs.rs/hmac/0.13.0]
**Warning signs:** Duplicate `sha2` in `cargo tree`, "expected sha2::Sha256, found sha2::Sha256" type errors.

### Pitfall 2: hmac 0.12 -> 0.13 API Changes

**What goes wrong:** `HmacSha256::new_from_slice()` might have a different import path or API.
**Why it happens:** Major version bump in hmac.
**How to avoid:** The `Hmac<Sha256>` pattern and `Mac` trait should be stable across 0.12->0.13, but verify the `hmac::Mac` trait still provides `new_from_slice`, `update`, `finalize`, `verify_slice`. Test compilation immediately after the version bump.
**Warning signs:** Import errors on `hmac::Mac`, `HmacSha256` type alias failures.

### Pitfall 3: Edition 2024 vs 2021

**What goes wrong:** Source crate uses `edition = "2024"`. Workspace uses 2021 everywhere. If you leave it at 2024, you get inconsistent behavior and potential clippy differences.
**Why it happens:** The source was developed independently. Edition 2024 reserves `gen` as a keyword and changes some default behavior.
**How to avoid:** Change to `edition = "2021"` immediately. Grep confirmed no edition-2024-specific features used in the source. [VERIFIED: no `gen` keyword or `unsafe extern` usage]
**Warning signs:** Clippy lint differences, unexpected keyword reservation errors.

### Pitfall 4: Workspace Lint Scope

**What goes wrong:** `make quality-gate` runs `cargo clippy --features "full" --lib --tests` which uses `--features` (root crate only) NOT `--workspace`. New workspace member crates might not get linted.
**Why it happens:** The Makefile lint target was written before workspace members were common.
**How to avoid:** After adding new crates, verify that `cargo build --all-features` (which IS workspace-wide) catches them. Also run `cargo clippy -p pmcp-code-mode` explicitly. The `cargo build --all-features` in the `build` target IS workspace-wide (default behavior). [VERIFIED: Makefile analysis]
**Warning signs:** Clippy passes locally but the new crate has warnings that aren't caught.

### Pitfall 5: Derive Macro Testing Without Server Runtime

**What goes wrong:** trybuild tests for compile-fail cases work fine, but compile-pass tests need a minimal `pmcp` server setup which requires tokio runtime.
**Why it happens:** The derive macro generates code that references `pmcp::server::builder::ServerCoreBuilder` and tool handler types.
**How to avoid:** Dev-dependencies of the derive crate must include `pmcp` with relevant features, `tokio`, and `pmcp-code-mode`. The trybuild fixtures need to be self-contained `fn main()` programs. Follow the pattern from `pmcp-macros/tests/ui/` exactly. [VERIFIED: pmcp-macros test structure]
**Warning signs:** trybuild tests fail with "unresolved import" rather than the expected error.

### Pitfall 6: SecretBox Trait Bound Constraints

**What goes wrong:** `SecretBox<[u8]>` doesn't implement `Clone`, `Debug`, `Display`, `PartialEq`, or `Serialize`. Any code path that requires these on the token secret will fail to compile.
**Why it happens:** This is by design -- the whole point of secrecy is to prevent these operations.
**How to avoid:** The `HmacTokenGenerator` needs `expose_secret()` to access the raw bytes for HMAC signing. The `ValidationPipeline::new()` currently takes `Vec<u8>` -- it needs to accept `TokenSecret` or `impl AsRef<[u8]>`. Plan the API migration carefully: `TokenSecret` wraps the secret, `expose_secret()` gives `&[u8]` for crypto ops.
**Warning signs:** "the trait `Clone` is not implemented for `SecretBox<[u8]>`" errors, especially in the derive macro generated state structs.

### Pitfall 7: Workspace Members List

**What goes wrong:** Forgetting to add new crates to the `[workspace] members` list in root Cargo.toml.
**Why it happens:** Easy to overlook.
**How to avoid:** First task in Plan 01 must add `"crates/pmcp-code-mode"` and `"crates/pmcp-code-mode-derive"` to the members list. [VERIFIED: current members list in Cargo.toml line 503]
**Warning signs:** "not a workspace member" errors, crate not found.

## Code Examples

### TokenSecret Newtype Implementation

```rust
// Source: secrecy 0.10.3 API [VERIFIED: docs.rs/secrecy/0.10.3]
use secrecy::{ExposeSecret, SecretBox};

/// Zeroizing wrapper for HMAC token secrets.
///
/// ## Security Properties
/// - Memory is zeroed on drop via `zeroize`
/// - No `Debug`, `Display`, `Clone`, or `PartialEq` implementations
/// - Secret bytes accessed only via `expose_secret()` which returns `&[u8]`
///
/// ## Threat Model
/// Protects against: accidental logging, memory dumps after drop,
/// clone-and-forget patterns, comparison side channels.
/// Does NOT protect against: active memory forensics while the secret
/// is in use, side-channel attacks on the HMAC computation itself.
pub struct TokenSecret(SecretBox<[u8]>);

impl TokenSecret {
    /// Create from raw bytes. The input Vec is consumed and will NOT be zeroed.
    /// For maximum security, use `from_env()` which reads directly into the wrapper.
    pub fn new(secret: Vec<u8>) -> Self {
        Self(SecretBox::new(Box::from(secret.as_slice())))
    }

    /// Read from an environment variable.
    pub fn from_env(var: &str) -> Result<Self, std::env::VarError> {
        let val = std::env::var(var)?;
        Ok(Self::new(val.into_bytes()))
    }

    /// Expose the secret bytes for cryptographic operations.
    pub fn expose_secret(&self) -> &[u8] {
        self.0.expose_secret()
    }
}
```

### HmacTokenGenerator with TokenSecret

```rust
// Adapted from source token.rs [VERIFIED: source code]
pub struct HmacTokenGenerator {
    secret: TokenSecret,
}

impl HmacTokenGenerator {
    pub fn new(secret: TokenSecret) -> Self {
        Self { secret }
    }

    fn sign(&self, payload: &[u8]) -> String {
        // hmac 0.13 API -- uses digest 0.11 trait
        let mut mac = HmacSha256::new_from_slice(self.secret.expose_secret())
            .expect("HMAC can take key of any size");
        mac.update(payload);
        hex::encode(mac.finalize().into_bytes())
    }
}
```

## Open Questions

1. **Fresh import vs git mv**
   - What we know: `git mv` preserves history but the crate is changing repo (pmcp-run -> rust-mcp-sdk). There's no direct `git mv` across repos.
   - What's unclear: Whether attribution header in files is sufficient for license compliance
   - Recommendation: Fresh import. Copy files, add attribution comment in lib.rs referencing the pmcp-run origin. This is cleaner than trying to splice git history.

2. **PlanExecutor blanket impl for CodeExecutor**
   - What we know: Patterns B/C/D all use `PlanExecutor` with different underlying executors. A blanket impl would collapse ~30 lines per server.
   - What's unclear: Whether `PlanExecutor`'s `execute(&mut self, ...)` signature (mutable self) is compatible with `CodeExecutor`'s `execute(&self, ...)` (shared ref).
   - Recommendation: Explore in Plan 03. If `PlanExecutor` requires `&mut self`, the blanket impl needs an internal `Mutex` or `RefCell` -- probably not worth it. Document as deferred if so.

3. **Makefile lint scope for new crates**
   - What we know: `make lint` uses `--features "full"` not `--workspace`. New crates may not be linted.
   - What's unclear: Whether `cargo build --all-features` (which IS workspace-wide) implicitly covers clippy via RUSTFLAGS.
   - Recommendation: After Plan 01, verify by deliberately introducing a clippy warning in the new crate and checking whether `make quality-gate` catches it. If not, Plan 06 should add explicit `-p pmcp-code-mode -p pmcp-code-mode-derive` to the lint target.

4. **Whether `pmcp-code-mode` should depend on `pmcp` at all**
   - What we know: The source depends on `pmcp` for `ToolInfo` type (used in handler.rs `build_tools()`)
   - What's unclear: Whether we should invert this so pmcp-code-mode is independent and the derive macro depends on both
   - Recommendation: Keep the dependency. `ToolInfo` is core MCP protocol type. The alternative (duplicating it) is worse.

### Known Threat Patterns for Code Mode

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Token replay | Spoofing | request_id UUID in token, TTL expiry (default 5min) |
| Code modification after validation | Tampering | SHA-256 code hash bound in HMAC-signed token |
| Secret key exposure in logs | Information Disclosure | TokenSecret blocks Debug/Display, secrecy crate |
| Policy bypass | Elevation of Privilege | PolicyEvaluator default-deny on evaluate_script |
| Context drift (schema change between validate/execute) | Tampering | context_hash in token binds schema+permissions hash |
| Cross-session token usage | Spoofing | session_id bound in token |
| Secret key in memory after use | Information Disclosure | zeroize-on-drop via secrecy |
