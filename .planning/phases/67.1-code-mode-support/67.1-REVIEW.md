# Phase 67.1: Code Review Report

**Reviewed:** 2026-04-11
**Depth:** standard
**Files Reviewed:** 19
**Status:** issues_found

## Summary

This review covers the `pmcp-code-mode` and `pmcp-code-mode-derive` crates added in Phase 67.1. The crates provide a Code Mode validation and execution framework: HMAC-signed approval tokens, a GraphQL validation pipeline, Cedar policy evaluation, schema exposure architecture, and a derive macro for registering tools on `ServerBuilder`.

The overall architecture is sound and the security design for `TokenSecret` is well thought out. However, there are three critical issues that must be resolved before this code ships to production:

1. The `verify_code` method panics on short hash strings (< 12 hex chars), which is reachable from the fuzz target.
2. The `TokenSecret::new` constructor accepts empty secrets without error, allowing silent HMAC key misuse.
3. The derive macro hard-codes `"anonymous"/"session"/"schema"/"perms"` as the validation context, meaning every request uses the same context hash and the token is effectively session-agnostic.

Seven warning-level issues cover logic gaps in the validation pipeline (most notably, `validate_graphql_query` never enforces `blocked_queries`/`allowed_queries`) and an action-classification gap for subscriptions.

## Critical Issues

### CR-01: `verify_code` panics on hash strings shorter than 12 bytes

**File:** `crates/pmcp-code-mode/src/token.rs:272-275`

**Issue:** `token.code_hash[..12]` and `current_hash[..12]` use direct byte-index slicing. Both values are hex-encoded SHA-256 hashes (always 64 chars) for tokens generated by this library. However, `ApprovalToken` is `Deserialize`, so an attacker or the fuzz harness can decode a JSON token where `code_hash` is an arbitrary short string. When `verify_code` is called on such a token, the slice panics at runtime.

The fuzz target in `fuzz/fuzz_targets/fuzz_token_code_mode.rs` exercises exactly this path: it decodes an arbitrary base64 token and calls `verify_code`, so this panic is reachable in fuzzing and in any server that receives untrusted token strings.

**Fix:**
```rust
fn verify_code(&self, code: &str, token: &ApprovalToken) -> Result<(), ExecutionError> {
    let current_hash = hash_code(code);
    if current_hash != token.code_hash {
        // Avoid panicking on short/malformed hashes from deserialized tokens.
        let expected_prefix = token.code_hash.get(..12).unwrap_or(&token.code_hash);
        let actual_prefix  = current_hash.get(..12).unwrap_or(&current_hash);
        return Err(ExecutionError::CodeMismatch {
            expected_hash: expected_prefix.to_string(),
            actual_hash:   actual_prefix.to_string(),
        });
    }
    Ok(())
}
```

### CR-02: `TokenSecret::new` silently accepts empty secrets

**File:** `crates/pmcp-code-mode/src/token.rs:52-55`

**Issue:** `TokenSecret::new` accepts any `impl Into<Vec<u8>>`, including an empty byte slice. An empty HMAC key is valid from the `hmac` crate's perspective, but it makes the token trivially forgeable: any caller who passes `b""` or an empty `Vec` gets HMAC signatures that are determined solely by the message. There is no runtime error. The `from_env` constructor also silently yields an empty secret when the environment variable is set to an empty string.

This is especially dangerous because the generated derive code calls `ValidationPipeline::from_token_secret` with the user-supplied `token_secret` field, and there is no check anywhere in the pipeline that the secret is non-empty.

**Fix:**
```rust
pub fn new(secret: impl Into<Vec<u8>>) -> Result<Self, SecretError> {
    let bytes: Vec<u8> = secret.into();
    if bytes.len() < 16 {
        return Err(SecretError::TooShort { len: bytes.len() });
    }
    Ok(Self(SecretBox::new(Box::from(bytes.as_slice()))))
}

#[derive(Debug, thiserror::Error)]
pub enum SecretError {
    #[error("Token secret is too short ({len} bytes); minimum is 16 bytes")]
    TooShort { len: usize },
}
```

Alternatively, enforce minimum length only in `HmacTokenGenerator::new` if changing `TokenSecret::new` signature is too disruptive:
```rust
pub fn new(secret: TokenSecret) -> Self {
    assert!(
        secret.expose_secret().len() >= 16,
        "HMAC token secret must be at least 16 bytes"
    );
    Self { secret }
}
```

### CR-03: Derive macro hard-codes static validation context, making tokens session-independent

**File:** `crates/pmcp-code-mode-derive/src/lib.rs:189-196`

**Issue:** The generated `ValidateCodeHandler::handle` method creates the `ValidationContext` with literal string constants:

```rust
let context = pmcp_code_mode::ValidationContext::new(
    "anonymous",
    "session",
    "schema",
    "perms",
);
```

Because `context_hash` is derived from `schema_hash` and `permissions_hash`, every token produced by the derive-generated handler embeds the identical context hash `sha256("schema|perms")`. The approval token's documented purpose is to bind the code hash to a specific user+session+schema combination, so that a token issued in session A cannot be replayed in session B with a different schema or different user permissions.

With the current hard-coded values, the session binding and schema binding are entirely absent. An attacker who obtains a valid token can replay it across different users and different schema versions, as long as the token has not expired.

The `RequestHandlerExtra` parameter is already available in the handler signature (`_extra: pmcp::RequestHandlerExtra`) but is unused. The user ID and session ID should be extracted from the MCP request context or connection metadata and threaded through.

**Fix (minimum viable):** At minimum, document clearly in generated code that these are placeholder values and the user MUST override the `ValidateCodeHandler::handle` implementation to supply real user/session context. Better: propagate real context from `RequestHandlerExtra` or require the struct to provide a method that returns user/session from its own fields.

```rust
// In the generated code, extract from extra or from the parent struct:
let user_id = _extra.client_id().unwrap_or("anonymous");
let session_id = _extra.request_id().as_deref().unwrap_or("session");
let context = pmcp_code_mode::ValidationContext::new(
    user_id,
    session_id,
    "schema",   // still a limitation -- schema hash must come from the struct
    "perms",    // still a limitation -- permissions hash must come from the struct
);
```

A proper fix requires `CodeModeHandler` or `CodeModeOpts` to capture a `fn(&RequestHandlerExtra) -> ValidationContext` callback, or the validated struct to expose a `context_for_request` method.

## Warnings

### WR-01: `validate_graphql_query` never enforces `blocked_queries` or `allowed_queries`

**File:** `crates/pmcp-code-mode/src/validation.rs:174-253`

**Issue:** `CodeModeConfig` exposes `blocked_queries: HashSet<String>` and `allowed_queries: HashSet<String>` fields (config.rs lines 43-44), and the documentation implies they are analogous to `blocked_mutations`/`allowed_mutations`. However, `validate_graphql_query` has no code path that checks these sets for read operations. A server operator who sets `blocked_queries: ["sensitiveReport"]` expecting it to block that query field will see no effect.

**Fix:** After extracting `query_info`, add a symmetric check for `query_info.root_fields` against `blocked_queries` and `allowed_queries`, mirroring the mutation enforcement block starting at line 202.

### WR-02: `validate_graphql_query_async` double-validates when no policy evaluator is configured

**File:** `crates/pmcp-code-mode/src/validation.rs:317-319`

**Issue:** When the `else` branch triggers (no policy evaluator configured), the code calls `self.validate_graphql_query(query, context)`. That method begins by re-running `self.graphql_validator.validate(query)` again from scratch (line 199). The full parse and depth check are duplicated. This is wasteful and could produce subtly different timing behavior between the two paths. More importantly, `query_info` has already been computed at line 280 in the async variant but is discarded before the fallback call.

**Fix:** Pass the already-computed `query_info` to a shared `complete_validation` call, or restructure the fallback to call `complete_validation` directly with the parsed `query_info` rather than re-invoking `validate_graphql_query`.

### WR-03: Subscription operations classified as "Write" in Cedar policy action mapping

**File:** `crates/pmcp-code-mode/src/policy/cedar.rs:233`

**Issue:** The Cedar evaluator maps `"subscription"` to the `"Write"` action:

```rust
"subscription" => "Write",
```

Subscriptions are read-only streaming operations. Classifying them as `Write` means they will be denied by default on servers where `allow_write: false` (the default), even though the subscription does not mutate data. This is an incorrect access control decision that will silently block legitimate read-only subscriptions.

**Fix:**
```rust
"subscription" => "Read",
```

The `GraphQLValidator::to_code_type` method already treats subscriptions as `GraphQLQuery` (graphql.rs line 342), so the Cedar evaluator should be consistent.

### WR-04: `normalize_path_to_pattern` UUID detection is overly broad (false positive)

**File:** `crates/pmcp-code-mode/src/policy/types.rs:344-355`

**Issue:** UUID detection uses: `segment.len() == 36 && segment.contains('-')`. Any 36-character path segment containing a hyphen will be treated as a UUID and replaced with `*`. Path segments like `google-oauth2-callback` (34 chars) might not trigger this, but any 36-char segment with a hyphen—including version strings like `openapi-schema-v3-draft-2020-12`—will be wrongly normalized to `*`, causing allowlist/blocklist mismatches.

**Fix:** Use a proper UUID regex or at minimum check that the segment is exactly 36 chars AND matches the UUID hex pattern `[0-9a-fA-F]{8}-...-`:
```rust
fn is_uuid_like(segment: &str) -> bool {
    if segment.len() != 36 { return false; }
    let parts: Vec<&str> = segment.split('-').collect();
    matches!(parts.as_slice(), [a, b, c, d, e]
        if a.len() == 8 && b.len() == 4 && c.len() == 4
        && d.len() == 4 && e.len() == 12
        && segment.chars().all(|ch| ch.is_ascii_hexdigit() || ch == '-'))
}
```

### WR-05: `is_policy_configured` default returns `true`, masking misconfigured production servers

**File:** `crates/pmcp-code-mode/src/handler.rs:179-181`

**Issue:** The default implementation of `is_policy_configured()` returns `true`:

```rust
fn is_policy_configured(&self) -> bool {
    true
}
```

The comment says "default returns `true` for backward compatibility with tests. Production implementations MUST override this." This inverts the safe default: a production implementation that forgets to override `is_policy_configured` will appear to have a policy evaluator when it does not, bypassing the gating check in `handle_tool` (line 224). The consequence is that the server falls through to the no-policy code path that only performs basic config checks, without any user-visible error.

**Fix:** Flip the default to `false`, and update tests that rely on the `true` default to explicitly override it. This is the safe default: if the implementer does not declare a policy, the tool is blocked rather than silently passing basic-config-only checks.

### WR-06: `payload_bytes` uses `{:?}` (Debug) formatting for `RiskLevel`, making the HMAC payload format fragile

**File:** `crates/pmcp-code-mode/src/token.rs:129-141`

**Issue:** The HMAC payload is formatted as:

```rust
"{:?}",  // <-- Debug format, currently "Low" / "Medium" / "High" / "Critical"
self.risk_level,
```

If `RiskLevel`'s `Debug` implementation ever changes (e.g., through a `#[derive(Debug)]` renaming or adding a new variant that changes the internal representation), existing issued tokens become unverifiable. The `Display` implementation already exists and produces stable all-caps strings (`"LOW"`, `"MEDIUM"`, etc.). Use `{}` (Display) instead.

**Fix:**
```rust
format!(
    "{}|{}|{}|{}|{}|{}|{}|{}|{}",
    self.request_id,
    self.code_hash,
    self.user_id,
    self.session_id,
    self.server_id,
    self.context_hash,
    self.risk_level,  // uses Display: "LOW", "MEDIUM", etc.
    self.created_at,
    self.expires_at,
)
```

### WR-07: `from_token_secret` exposes the secret bytes to create a `Vec<u8>` copy outside `SecretBox`

**File:** `crates/pmcp-code-mode/src/validation.rs:122-124`

**Issue:**
```rust
pub fn from_token_secret(config: CodeModeConfig, secret: &TokenSecret) -> Self {
    Self::new(config, secret.expose_secret().to_vec())
}
```

`secret.expose_secret().to_vec()` allocates a new `Vec<u8>` on the heap that is not zeroized on drop (plain `Vec` has no zeroize). This Vec is passed into `Self::new`, which passes it into `HmacTokenGenerator::new_from_bytes`, which wraps it in a new `TokenSecret`. The intermediate `Vec<u8>` lives for the duration of the `new` call but is never zeroed.

The comment in `TokenSecret::new` (token.rs line 51) acknowledges this: "The original Vec is NOT zeroed -- callers should use `from_env()` for maximum security." However, `from_token_secret` is documented as "the secure entry point for production callers and derive macro generated code," which is misleading given this limitation.

**Fix:** Either update the documentation to be honest about the limitation, or add a `from_token_secret` convenience that avoids the intermediate allocation by passing `&TokenSecret` directly to `HmacTokenGenerator`:
```rust
pub fn from_token_secret(config: CodeModeConfig, secret: &TokenSecret) -> Self {
    Self {
        token_generator: HmacTokenGenerator::new(TokenSecret::new(
            secret.expose_secret().to_vec()  // same limitation, but at least rename the constructor
        )),
        ...
    }
}
```

The cleaner fix is to add a `HmacTokenGenerator::from_secret_ref(secret: &TokenSecret)` constructor that creates the HMAC key directly from the slice without going through a Vec.

## Info

### IN-01: `field_name_to_type` heuristic produces incorrect type names for camelCase fields

**File:** `crates/pmcp-code-mode/src/graphql.rs:349-363`

**Issue:** The de-pluralization heuristic removes a trailing `s`:
- `"users"` → `"User"` (correct)
- `"orderItems"` → `"OrderItem"` (correct)
- `"address"` → `"Addres"` (WRONG — `'s'` stripped from non-plural)
- `"status"` → `"Statu"` (WRONG)

These wrong type names flow into `types_accessed` in `GraphQLQueryInfo`, which propagates to the Cedar `OperationEntity`'s `accessedTypes` set. Cedar policies that restrict by type name would match against incorrect strings.

Since this is labeled an "heuristic" in the source code, it may be acceptable, but it should either be flagged as known-approximate or improved.

**Fix:** Skip de-pluralization for known false positives, or use a simple allowlist of common GraphQL plural patterns (`"es"`, `"ies"`, `"s"` only if preceded by a consonant).

### IN-02: `ExecutionError::LoopContinue` and `LoopBreak` are control-flow signals exposed as public API

**File:** `crates/pmcp-code-mode/src/types.rs:487-494`

**Issue:** Two variants of `ExecutionError` are control-flow signals used internally by the JS runtime, not genuine error conditions:

```rust
/// Loop continue signal (not a real error, used for control flow)
#[error("Loop continue")]
LoopContinue,

/// Loop break signal (not a real error, used for control flow)
#[error("Loop break")]
LoopBreak,
```

Because `ExecutionError` is part of the public `CodeExecutor` trait (`execute` returns `Result<Value, ExecutionError>`), external implementors of `CodeExecutor` may accidentally match or produce these variants, and downstream consumers may hit confusing error messages ("Loop continue") when these leak across boundaries.

**Fix:** Move these variants to an internal `RuntimeControlFlow` enum used only within the `js-runtime` feature, gated behind `#[cfg(feature = "js-runtime")]`.

### IN-03: Cedar entity construction panics on `EntityId` with slash characters

**File:** `crates/pmcp-code-mode/src/policy/cedar.rs:85-86`

**Issue:**
```rust
EntityId::from_str(&operation.id).expect("valid entity id")
```

`OperationEntity::id` is set to `query_info.operation_name` or `"anonymous"`. GraphQL operation names are typically simple identifiers, but `OperationEntity::id` is also constructed from path strings in `ScriptEntity` (policy/types.rs line 298):
```rust
format!("{}:{}", format!("{:?}", c.method).to_uppercase(), c.path)
```

Cedar `EntityId` may not accept all characters. If the Cedar crate rejects the id, the `expect` call panics. This should return a `PolicyEvaluationError` instead.

**Fix:** Replace `.expect("valid entity id")` with `.map_err(|e| PolicyEvaluationError::ConfigError(format!("Invalid entity id '{}': {}", operation.id, e)))?` and propagate the error.

### IN-04: `sha2` and `hmac` version constraints use yanked or mismatched crate versions

**File:** `crates/pmcp-code-mode/Cargo.toml:26-28`

**Issue:** The `Cargo.toml` specifies:
- `sha2 = "0.11"` — the latest published `sha2` version is `0.10.x`; `0.11` does not exist and will fail to resolve.
- `hmac = "0.13"` — similarly, the latest `hmac` is `0.12.x`.

These constraints would prevent the crate from compiling in a clean environment. (This may be correct if a pre-release is intended, but it is unusual and should be verified.)

**Fix:** Verify the published crate versions and pin to the correct ones:
```toml
sha2 = "0.10"
hmac = "0.12"
```

### IN-05: Dead code — `validate_code_impl`'s `variables` parameter is never used in GraphQL path

**File:** `crates/pmcp-code-mode/src/handler.rs:158-165`

**Issue:** The `CodeModeHandler::validate_code_impl` signature accepts `variables: Option<&Value>`. However, the GraphQL validation pipeline (`ValidationPipeline::validate_graphql_query`) takes only `(query: &str, context: &ValidationContext)` and never accepts variables. The `ValidateCodeInput` struct captures `variables`, but the only thing done with them in `handle_validate_code` (lines 260-272) is pass them along to `validate_code_impl`, where they are unused in the GraphQL case.

This is not a bug per se but misleads implementors into thinking variables influence validation.

**Fix:** Either thread the variables into the validation context (for per-variable policy checks), or remove the parameter from `validate_code_impl` and document that variables are only relevant at execution time.
