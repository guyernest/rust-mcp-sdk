<task type="auto" tdd="true">
  <name>Task 2: Add deterministic workflow renderer</name>
  <files>cargo-pmcp/src/commands/github/workflow.rs, cargo-pmcp/src/commands/github/mod.rs</files>
  <read_first>
    - cargo-pmcp/src/commands/github/target.rs
    - cargo-pmcp/src/commands/deploy/mod.rs
    - cargo-pmcp/docs/commands/deploy.md
  </read_first>
  <action>
    Create `workflow.rs` with `pub struct WorkflowSpec` carrying `target: GitHubDeployTarget`, `branch: String`, `environment: Option<String>`, `workflow_name: String`, and `use_locked: bool`. Implement `pub fn render_workflow(spec: &WorkflowSpec) -> String`. The renderer must output YAML with `name: Deploy MCP server`, `on.push.branches` containing the branch, `workflow_dispatch`, `permissions.contents: read`, and for pmcp-run only `permissions.id-token: write`. The deploy step must run exactly `cargo pmcp deploy --target-type {target} --non-interactive`; if `use_locked` is true, include `cargo install cargo-pmcp --locked`, otherwise `cargo install cargo-pmcp`. Do not serialize a second deployment config path.
  </action>
  <verify>
    <automated>cargo test -p cargo-pmcp github_workflow</automated>
  </verify>
  <acceptance_criteria>
    - `render_workflow` output contains `cargo pmcp deploy --target-type pmcp-run --non-interactive` for pmcp-run
    - pmcp-run rendered YAML contains `id-token: write`
    - aws-lambda rendered YAML does not contain `id-token: write`
    - Two calls to `render_workflow` with equal `WorkflowSpec` values produce equal strings
  </acceptance_criteria>
</task>
