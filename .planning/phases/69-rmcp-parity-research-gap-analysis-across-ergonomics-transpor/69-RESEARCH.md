## Executive Summary

The matrix contains 32 rows across 6 surfaces: **4 High-severity gaps**, **7 Medium-severity gaps**, and **21 Low-severity rows** (including Parity rows and Strength-to-preserve rows where pmcp already matches or exceeds rmcp).

**High-severity rows:** MACRO-02 (tool-description rustdoc fallback), HANDLER-02 (missing request-scoped `Extensions` typemap on `RequestHandlerExtra`), HANDLER-05 (missing peer handle in `RequestHandlerExtra` for server-to-client RPCs from inside handlers), CLIENT-02 (typed `call_tool_typed` and `list_all_*` auto-pagination convenience methods). These are the four gaps that affect the typical first-hour user path AND admit a clean articulable fix.

**Gap pressure concentrates on Handler Signatures + State/Extra Injection and Client-Side Ergonomics** — together those two surfaces account for 3 of the 4 High-severity findings, suggesting Plan 02 should prioritize follow-on phase proposals targeting the `RequestHandlerExtra` extension surface and the client typed-request/auto-pagination helpers. Tool Macros contributes the fourth High (MACRO-02) and is a narrower, single-fix proposal.

**pmcp strengths identified for preservation:** `ServerCoreBuilder` fluent builder (BUILDER-01), `State<T>` + impl-block dual state-injection paths (MACRO-05 / HANDLER-04), `trybuild` compile-fail test discipline in pmcp-macros (MACRO-06), the `#[mcp_resource]` macro with URI-template matching (MACRO-07), `Client::new` one-line client construction (CLIENT-01), task-aware `ToolCallResponse` enum + `call_tool_and_poll` (CLIENT-05), the crate-root `Result<T>` alias (ERR-04), and the `Other(#[from] anyhow::Error)` zero-friction `?`-operator variant (ERR-03). Future proposals must not regress any of these.

Each High-severity gap is proposed as a follow-on phase in `69-PROPOSALS.md` (Plan 02 output), cited by Row ID.

## Methodology Notes

**Research approach:** Source and docs reading only (per D-03). No hands-on "build a tiny example in both SDKs" — the depth-to-cost ratio was judged insufficient to triple research time. All rmcp evidence comes from reading the `modelcontextprotocol/rust-sdk` source tree at the pinned git tag, rmcp-macros crate docstrings and attribute tables, and rustdoc where a single source line was not cleanly attributable. All pmcp evidence comes from reading local source files in the `rust-mcp-sdk` checkout at the pinned HEAD commit.

**Evidence standard:**
- pmcp column: `path/to/file.rs:LINENO` (or `path:LINE-RANGE`) followed by a baseline tag — either `[v2.3.0]` (the API was present in the published v2.3.0 release) or `[main]` (post-v2.3.0 main-only, present on the `feat/sql-code-mode` branch at commit dbaee6cc but not yet shipped to crates.io). All pmcp citations in this document end with `[v2.3.0]` — no `[main]` citations were required, which indicates the compared surface is stable at the v2.3.0 release.
- rmcp column: GitHub blob URL with `#L<N>` line fragment pinned to the `rmcp-v1.5.0` tag, targeting `crates/rmcp/src/...` or `crates/rmcp-macros/src/...` workspace paths. The 80/20 acceptance bar (≥80% blob URLs, ≤20% docs.rs anchors) was met — 100% of rmcp evidence citations in this document are GitHub blob URLs.

**Row ID convention:** `<SURFACE>-NN` where `<SURFACE>` is one of `MACRO` / `BUILDER` / `TYPED` / `HANDLER` / `CLIENT` / `ERR` (matching the H2 section prefix) and `NN` is a two-digit zero-padded counter starting at `01` per surface, assigned in document order. IDs are persistent once written — if a row is deleted in a later revision, its ID is retired rather than reused, and the deletion should be noted here rather than fixed by renumbering. Plan 02 proposals cite these Row IDs verbatim in their `### Rationale` subsection; Plan 03's quality gate compares High-severity Row IDs against proposal citations for bijection.

**Severity rubric (D-14, verbatim):**
- **High:** affects the typical first-hour user path AND a clean specific fix can be articulated.
- **Medium:** affects less common paths, or has no single clean fix.
- **Low:** cosmetic or theoretical. "Parity" rows and "Strength to preserve" rows are always Low.

**Out-of-scope reminder:** Tower / service integration (D-11) was handled in Phase 56 and is NOT compared here. Transport construction API shape (D-12) is deferred — the matrix did not surface it as a High-severity ergonomics gap, so it remains outside Phase 69's proposal scope. If a future hands-on spike reveals a High-severity transport-construction ergonomics issue, it can enter the pipeline via `69-PROPOSALS.md` amendment or a separate phase proposal.

**Reproducibility note:** Running this research against a different rmcp version or a different pmcp commit will produce different line numbers, and potentially different gap severities as either SDK evolves. The pinned versions in the document header (rmcp 1.5.0 at tag `rmcp-v1.5.0`; pmcp v2.3.0 + main at commit `dbaee6cc`) are the canonical baseline for this document. Plan 02 proposals derived from this matrix remain valid as long as the cited Row IDs still map to the gaps described; if rmcp publishes a new version that closes a High-severity gap on its side, the corresponding pmcp proposal should be re-evaluated rather than auto-executed.
