## (1) Two-sided passthrough proof (recorded)
- **`Some("contoso-user-tok")` (4th arg) -> PASS.** Full suite green: 3 passed, 1 ignored.
- **`None` (4th arg) -> FAIL.** `contoso_m365_parity_through_real_binary_path` fails: the wiremock Graph mock REQUIRES `Authorization: Bearer contoso-user-tok` on its matchers, so with no forwarded bearer every backend call 404s, the replay steps fail, and the test panics. Restored to `Some` -> green.
- This proves the server FORWARDS the user's token (oauth_passthrough) and never substitutes a server-held credential — the X-API-Key header (also injected) is explicitly NOT treated as the credential.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1/3 - Blocking bug] `oauth_passthrough` config type did not deserialize**
- **Found during:** Task 1 (`contoso_m365_fixture` + the real-binary path both failed at config parse).
- **Issue:** `AuthConfig` uses `#[serde(tag = "type", rename_all = "snake_case")]`, which derives the tags `o_auth_passthrough` and `o_auth2_client_credentials`. But the README auth table and the enum doc comment (line ~56) document `type = "oauth_passthrough"` / `oauth2_client_credentials`. The Plan 01 fixture used the documented no-underscore form, so `from_toml_strict_validated` errored with `unknown variant oauth_passthrough`. Plan 01's SUMMARY claimed it parsed; it did not (Plan 01 never ran the strict parse on the passthrough variant).
- **Fix:** Added `#[serde(alias = "oauth_passthrough")]` and `#[serde(alias = "oauth2_client_credentials")]` to the two variants so the documented config form deserializes (additive; existing tests still pass).
- **Files modified:** `crates/pmcp-server-toolkit/src/http/auth.rs`
- **Commit:** `6be8aec8`

**2. [Rule 1 - Bug] Both Contoso script tools produced wrong/empty addresses in the accurate JS engine**
- **Found during:** Task 1 (tool calls 404'd: `get_customer` requested `range(address='A2.0:D2.0')`; `get_customer_orders` requested `range(address='null')`).
- **Issue:** Two pmcp-code-mode engine limitations: (a) numeric arithmetic renders as a float — `A${idx+1}` where `idx+1 = 2.0` produces `"A2.0:D2.0"` (`add_values` makes `serde_json::Number::from_f64(2.0)` -> `"2.0"`); (b) `evaluate_array_index` supports only `(Array, Number)`, so the object string-index `blocks[args.customer_id]` returned `null`.
- **Fix:** Rewrote both scripts (in `tests/fixtures/contoso-m365.toml`, and identically in the two example files) to map `customer_id` to a LITERAL range-address string via an `if`/`else if` chain, then a single `api.get` template literal. The literal addresses are the exact canonical-json values. No arithmetic, no object indexing — both engine-safe constructs already proven by london-tube's script tool.
- **Files modified:** `crates/pmcp-openapi-server/tests/fixtures/contoso-m365.toml` (+ `examples/contoso-m365.toml`, `examples/contoso_m365_min.rs` in Task 2).
- **Commit:** `6be8aec8` (fixture) / `554b9b7e` (examples)
