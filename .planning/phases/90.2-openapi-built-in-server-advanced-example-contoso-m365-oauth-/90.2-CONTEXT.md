<decisions>
## Implementation Decisions

### Workbook data model
- **D-01:** Lean business shape. **Customers** sheet columns: `customer_id`, `name`, `segment`, `region`. **Orders** sheet columns: `order_id`, `customer_id`, `order_date`, `amount`. Enough for the headline query AND richer Code Mode demos (by-region rollups, top-segment), while keeping the fixture small and deterministic. Not minimal (would starve Code Mode demos), not richer (avoids fixture-maintenance cost).

### Customer-id → Excel range addressing
- **D-02:** Simple sequential **row convention** — `customer_id` maps to a known row, and the tool templates the Excel range address from the id (e.g., `C007` → row = id index + header offset → `range(address='A8:D8')`). Obviously a demo, smallest fixture, stays within the single range-read Graph op. Rejected the "realistic Excel-table lookup" alternative (more fixture/spec surface, risks needing a second Graph op → feature gap).
- **D-03:** `get_customer_orders(customer_id)` returns that customer's orders by reading an **addressable contiguous orders block** for the id. The Orders sheet must be laid out (grouped/sorted by `customer_id`) so each customer's orders form a contiguous, address-derivable range. Exact block addressing (fixed orders-per-customer block size vs a small per-id offset lookup) is finalized in planning — keep it within range-read-only.
- **D-04:** No server-side filtering in either tool — the connector is a thin HTTP mapper. All joining/filtering/aggregation (incl. the "bought >100 in last 3 months" headline) is Code Mode's job.

### Security-story docs depth
- **D-05:** Concise — a **config walkthrough + short governance note** (1-2 paragraphs): admin consents once to a bounded scope (the ceiling of what the server may request); the signed-in user's forwarded token governs per-file access; the server holds no standing credentials and can only ever act as the calling user. Match 90.1 chapter depth. NOT a full Mermaid sequence diagram or a dedicated threat-framing subsection (deferred — see Deferred Ideas).

### Contoso framing & naming
- **D-06:** Lean into the fictional **Contoso** org: a named SharePoint site (e.g. "Sales"), a workbook like `Customers.xlsx`, and recognizable Microsoft-canonical fictional customer names (Northwind Traders, Fabrikam, Tailspin Toys, etc.) for a polished business feel. Trivial extra cost (nicer fixture values).

### Claude's Discretion
- Exact wiremock response JSON shapes, range-address strings, and the curated-spec operationId/parameter names — within the decisions above.
- Number of sample customers/orders (enough for ≥1 match and ≥1 non-match on the headline query, per P902-FIXTURE; a handful is fine).
- Chapter filenames / nav placement (suggested `openapi-graph-m365.md` in both books).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `OAuthPassthroughAuth` + its unit test: passthrough forwarding is fully implemented and tested — this phase only *configures* it (no code).
- `TokenCaptureAuthProvider` (assemble.rs): captures the inbound MCP `Authorization` header into `AuthContext` and threads it to the outbound forward — the full delegated chain already works in the binary.
- 90.1 `parity_replay.rs`: the offline wiremock harness (stand up a mock backend, drive the real binary, replay a scenario, gate per-step) is the direct template for P902-PARITY.
- `validate_code` scenario step (london-tube-scenarios.yaml): the pattern for invoking Code Mode in the replay (script passed as `code` arg). NOTE: london-tube only asserts `success`, not a computed value — see Research Questions for the deterministic-result assertion.
- `london_tube_min.rs` / `examples/london-tube.toml`: build-and-exit example + pointable-config templates.

### Established Patterns
- Config is additive TOML with `deny_unknown_fields` — misspelled keys are hard parse errors; use exact field names from `config.rs`.
- Fixtures use `${VAR}` / dev-guarded inline `token_secret` placeholders, never real secrets; tests assert the *resolved* value reached the backend (london-tube asserts `app_key=dummy`), never the literal `${...}`.
- Offline tests run single-threaded (`--test-threads=1`) due to per-process env + ephemeral port.

### Integration Points
- The new fixture/config plug into the existing `pmcp-openapi-server` build path (`from_toml_strict_validated` → `dispatch` → `build_server`) unchanged.
- New chapters wire into `pmcp-book/src/SUMMARY.md` and `pmcp-course/src/SUMMARY.md`.

### Research Questions (for gsd-phase-researcher to resolve — NOT user decisions)
- **RQ-1 (passthrough proof):** Can mcp-tester's `ScenarioExecutor`/`ServerTester` attach an *inbound* `Authorization: Bearer` to its MCP session/tool calls so passthrough can be asserted in the YAML replay? Scout found no inbound-auth hook in the scenario layer (the OAuth refs there are mcp-tester's own *client-side* connection auth). If not available → assert passthrough via a Rust integration test that builds the server with `TokenCaptureAuthProvider` + a known inbound token and asserts the wiremock saw the forwarded bearer (extend the `auth.rs:791` pattern to the real binary path).
- **RQ-2 (Code Mode result assertion):** Can a `validate_code`/code-mode scenario step assert a *computed result* (the deterministic customer set), or must P902-CODEMODE be a Rust test that invokes the code-mode executor and checks the returned output? Confirm which code-mode tool executes-and-returns (vs `validate_code` which may only validate).
- **RQ-3 (Graph range addressing):** Confirm the exact Graph Excel range-read URL/param shape (`/drives/{drive-id}/items/{item-id}/workbook/worksheets/{worksheet-id}/range(address='…')`) and that an id→row-address templating + a contiguous orders-block read both fit a single curated range-read op (no second Graph op, staying within scope).
- **RQ-4 (curated spec format):** Confirm the OpenAPI shape `OpenApiSchema::parse` accepts for path-templated drive/item/worksheet/address parameters.

<specifics>
## Specific Ideas

- Headline Code Mode query (the demo's hero): "customers who bought more than 100 in the last 3 months" — joins Orders→Customers on `customer_id`, filters `order_date` to the trailing 3 months **of a fixed reference date pinned in the fixture** (never wall-clock `now`), sums `amount` per customer, returns those with total > 100.
- Tool examples: `get_customer(customer_id="C007") → {customer_id, name, segment, region}`; `get_customer_orders(customer_id="C007") → [{order_id, order_date, amount}, …]`.
- Contoso flavor: org "Contoso", SharePoint "Sales" site, `Customers.xlsx`, sample customers Northwind Traders / Fabrikam / Tailspin Toys.

</specifics>
