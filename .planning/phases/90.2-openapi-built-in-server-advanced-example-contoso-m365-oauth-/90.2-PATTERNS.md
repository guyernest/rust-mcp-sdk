# Phase 90.2: OpenAPI Built-In Server — Advanced Example (Contoso M365) - Pattern Map

**Mapped:** 2026-05-30
**Files analyzed:** 10 new/modified
**Analogs found:** 10 / 10 (all exact — this phase is a re-skin of 90.1)

This phase has **no `src/` changes**. Every new file copies a 90.1 london-tube
analog and re-skins it for Contoso/Graph/oauth_passthrough. The three genuinely
new things to apply to the copies are: (1) `[backend.auth] type = "oauth_passthrough"`
instead of `api_key`, (2) **script tools** that do `customer_id → Excel range
address` arithmetic in JS (the connector does literal `str::replace`, never math),
and (3) the **Rust** Code Mode test driving `JsCodeExecutor` directly (the YAML
`execute_code` path can't thread the HMAC approval token).

## Pattern Assignments

### `tests/fixtures/contoso-m365.toml` (config, request-response)

**Analog:** `crates/pmcp-openapi-server/tests/fixtures/london-tube.toml`

Copy the whole structure verbatim: `[server]` → `[backend]` → `[backend.auth]` →
`[backend.http]` → two `[[tools]]` (with `[[tools.parameters]]` + `[tools.annotations]`)
→ three `[[resources]]` → one `[[prompts]]` → `[code_mode]`. Re-skin per the
differences below.

**`[backend]` + auth — THE KEY DIFFERENCE.** london-tube uses api_key (lines 36-47):
```toml
[backend]
base_url = "https://api.tfl.gov.uk"
[backend.auth]
type = "api_key"
query_params = { app_key = "${TFL_APP_KEY}" }
required = false
```
Contoso uses **oauth_passthrough + the Graph base_url** (field names verified from
`AuthConfig::OAuthPassthrough` at `crates/pmcp-server-toolkit/src/http/auth.rs:122-130`
— `target_header` defaults `"Authorization"`, `required` defaults `true`):
```toml
[backend]
base_url = "https://graph.microsoft.com/v1.0"
[backend.auth]
type = "oauth_passthrough"
target_header = "Authorization"
required = true
```

**Tools — SCRIPT tools (the address math must be in JS).** london-tube's tool 1 is
single-call (lines 60-70, `path = "/Line/Mode/tube/Status"`), tool 2 is script
(lines 82-101). Contoso's **both** tools are script tools because `customer_id →
row → range(address='A{row}:D{row}')` is arithmetic the connector cannot do
(connector does literal `str::replace` only — `client.rs:158`). Copy the script-tool
shape from london-tube tool 2 (the `script = """..."""` + `[[tools.parameters]]` +
`[tools.annotations]` block, lines 82-107) and re-skin:
```toml
[[tools]]
name = "get_customer"
description = "Fetch one customer row (customer_id, name, segment, region) from the Customers sheet."
script = """
const idx = parseInt(args.customer_id.slice(1), 10);   // 'C007' -> 7
const row = idx + 1;                                    // header offset (row 1 = header)
const addr = `A${row}:D${row}`;
const r = await api.get(`/drives/CONTOSO_DRIVE/items/CUSTOMERS_ITEM/workbook/worksheets/Customers/range(address='${addr}')?$select=values`);
return r.values;
"""
[[tools.parameters]]
name = "customer_id"
type = "string"
description = "Customer id, e.g. 'C007'."
required = true
[tools.annotations]
read_only_hint = true
idempotent_hint = true
open_world_hint = true
cost_hint = "low"
```
Second tool `get_customer_orders` is the same shape but reads a contiguous Orders
block (D-03 — the Orders sheet is grouped by `customer_id`). Note the JS engine
has **no `Date` builtin** — any date logic stays in Code Mode, not here.

**Resources + prompt — copy the exact-URI-match discipline.** london-tube ships
three `[[resources]]` (lines 118-192) whose URIs are referenced verbatim by the
`start_code_mode` `[[prompts]] include_resources` (lines 201-208). Mirror this with
Contoso URIs (e.g. `docs://contoso-m365/schema`, `docs://contoso-m365/examples`,
`code-mode://learnings`). **Pitfall:** every URI in `include_resources` must equal
a `[[resources]] uri` exactly or `register_prompts` fails at assembly
(`assemble.rs` register_prompts; RESEARCH Pitfall 3).

**`[code_mode]` — copy verbatim, re-skin secret string** (london-tube lines 214-220):
```toml
[code_mode]
enabled = true
token_secret = "contoso-m365-parity-dev-secret-32bytes"
allow_inline_token_secret_for_dev = true
```

**deny_unknown_fields field names** (RESEARCH Pitfall 2, verified `config.rs:578-784`):
`ParamDecl` = `name`/`type`/`description`/`required`/`default`/`max_length`/`minimum`/`maximum`/`enum`;
`ToolDecl` = `name`/`description`/`path`/`method`/`base_url`/`script`/`parameters`;
`ResourceDecl` = `uri`/`name`/`description`/`mime_type`/`content`. A misspelled key is a hard parse error.

**No real secrets:** any tenant/client id is a `${VAR}` placeholder; tests assert the
*resolved* value, never the literal `${...}` (london-tube asserts `app_key=dummy`).

### `tests/fixtures/contoso-m365-api.yaml` (config, curated spec)

**Analog:** `crates/pmcp-openapi-server/tests/fixtures/london-tube-api.yaml` (55 lines)

Copy the minimal OpenAPI 3.0 shape. london-tube declares two paths with `in: path`
+ `in: query` params (lines 14-55). Contoso declares ONE op — the Graph range-read:
```yaml
openapi: 3.0.0
info:
  title: Contoso M365 — Excel range-read subset (Microsoft Graph)
  version: "1.0.0"
servers:
  - url: https://graph.microsoft.com/v1.0
paths:
  /drives/{drive-id}/items/{item-id}/workbook/worksheets/{worksheet-id}/range(address='{address}'):
    get:
      operationId: getWorksheetRange
      parameters:
        - { name: drive-id,     in: path,  required: true,  schema: { type: string } }
        - { name: item-id,      in: path,  required: true,  schema: { type: string } }
        - { name: worksheet-id, in: path,  required: true,  schema: { type: string } }
        - { name: address,      in: path,  required: true,  schema: { type: string } }
        - { name: $select,      in: query, required: false, schema: { type: string } }
      responses:
        "200": { description: A workbookRange object with the requested values. }
```
**DIFFERENCE / RISK (RESEARCH A1):** the OData `range(address='{address}')` path
segment is non-standard. The fixture-validity test MUST parse it through
`OpenApiSchema::parse` (london-tube does this at `parity_replay.rs:167`). If
`parse` rejects the parenthesized segment, fall back to a plain `.../range` path —
the spec is **context only, not used for dispatch** (tools dispatch from `[[tools]]`
script), so a simplified path still satisfies P902-SPEC.

### `tests/contoso_m365_code_mode.rs` (test, integration — transform)

**Analog:** `crates/pmcp-openapi-server/tests/oauth_passthrough_e2e.rs`

`execute_code` requires an HMAC approval token a flat scenario can't thread. Copy
the three helpers verbatim and drive `JsCodeExecutor` directly:

- `passthrough_exec(base_url, required)` (lines 36-46) — builds the passthrough
  `HttpCodeExecutor` via `create_passthrough_auth_provider` + `AuthConfig::OAuthPassthrough`.
- `extra_with_token(token)` (lines 50-61) — builds a `RequestHandlerExtra` carrying
  the captured inbound `Authorization` (mirrors `TokenCaptureAuthProvider`).
- `code_mode_executor(base, extra)` (lines 66-72) — `request_executor_from_extra(base, extra)`
  then `JsCodeExecutor::new(http, ExecutionConfig::default())`.

The test body mirrors `oauth_passthrough_e2e_required_true_token_present_forwards_and_succeeds`
(lines 82-105): mount wiremock for the Customers + Orders range reads (each
`.and(header("authorization", "Bearer contoso-user-tok"))`), build
`passthrough_exec` + `extra_with_token(Some("Bearer contoso-user-tok"))` +
`code_mode_executor`, then `executor.execute(HEADLINE, None).await`.

**DIFFERENCE — the headline script computes a deterministic set with a PINNED
reference date** (the JS engine has NO `Date` builtin — RESEARCH Pitfall 1/A4):
```rust
const HEADLINE: &str = r#"
const REF = "2026-05-30";   // pinned reference date literal — NEVER wall-clock `now`
const customers = await api.get("/drives/.../worksheets/Customers/range(address='A2:D9')?$select=values");
const orders    = await api.get("/drives/.../worksheets/Orders/range(address='A2:D40')?$select=values");
// join Orders->Customers on customer_id, filter order_date in [REF-3mo, REF]
// (string/lexicographic ISO compare or integer month math), sum amount, total > 100
return result;
"#;
// ... assert_eq!(result, json!(["C007","C012"]));   // exact deterministic set from the fixture
```
Gate the file `#![cfg(feature = "openapi-code-mode")]` (line 22). Prefix fns
`contoso_m365_code_mode_*`. The fixture (mounted here) MUST contain ≥1 match and
≥1 non-match so the asserted set is non-trivial (P902-FIXTURE).

## Shared Patterns

### oauth_passthrough backend auth (the phase headline)
**Source:** `crates/pmcp-server-toolkit/src/http/auth.rs:122-130` (`AuthConfig::OAuthPassthrough`)
**Apply to:** `tests/fixtures/contoso-m365.toml`, `examples/contoso-m365.toml`, both `CONFIG` consts, both docs.
```toml
[backend.auth]
type = "oauth_passthrough"
target_header = "Authorization"   # optional, defaults "Authorization"
required = true                   # optional, defaults true
```
The chain `TokenCaptureAuthProvider → AuthContext → HttpCodeExecutor → outbound forward`
already ships (`assemble.rs:22-31,86-115`). This phase only CONFIGURES it — no `src/` edit.

### Script tools (connector does no arithmetic)
**Source:** `crates/pmcp-openapi-server/tests/fixtures/london-tube.toml:82-101` (script tool shape)
**Apply to:** both Contoso tools.
The `customer_id → row → range(address=...)` math MUST be in the JS `script` body
(connector does literal `str::replace`, `client.rs:158`). The JS engine supports
`Math.*`/`parseInt`/array methods but NOT `Date` (RESEARCH Pitfall 1). Bind `api.get`
to a `const` before `return` (engine-accuracy rule baked into the london-tube resource docs).

### Code Mode context surface (resources + prompt, exact-URI-match)
**Source:** `crates/pmcp-openapi-server/tests/fixtures/london-tube.toml:118-208` + helper `parity_replay.rs:56-73`
**Apply to:** both configs; assert via a copied `assert_contoso_code_mode_surface(cfg, label)` helper used for BOTH the fixture and the pointable example (prevents drift).
Every `[[prompts]] include_resources` URI must equal a `[[resources]] uri` exactly (Pitfall 3).

### Inbound-bearer passthrough proof (test seam)
**Source:** `crates/mcp-tester/src/tester.rs:96-101` (`api_key` arg → `Authorization: Bearer` in `extra_headers`)
**Apply to:** `tests/contoso_m365_parity.rs` (pass `Some("contoso-user-tok")` as the 4th `ServerTester::new` arg) and assert the forwarded header at the wiremock mock + on recorded requests.

### dev-guarded inline token_secret (offline tests need no env)
**Source:** `crates/pmcp-openapi-server/tests/fixtures/london-tube.toml:214-220`
**Apply to:** every config + embedded `CONFIG` const.
```toml
[code_mode]
enabled = true
token_secret = "contoso-m365-parity-dev-secret-32bytes"
allow_inline_token_secret_for_dev = true   # DEV-ONLY — never in production
```

### No real secrets + assert-resolved-not-literal
**Source:** `crates/pmcp-openapi-server/tests/parity_replay.rs:91-94,325-337`
**Apply to:** all configs + tests. Any tenant/client id is `${VAR}`; tests assert the resolved value (or the inbound bearer), never the literal `${...}`.
