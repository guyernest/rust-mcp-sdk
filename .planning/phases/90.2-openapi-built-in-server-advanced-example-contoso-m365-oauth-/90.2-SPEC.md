# Phase 90.2: OpenAPI Built-In Server — Advanced Example (Contoso M365: OAuth passthrough + Excel-over-Graph) — Specification

**Created:** 2026-05-30
**Ambiguity score:** 0.17 (gate: ≤ 0.20)
**Requirements:** 8 locked

## Goal

Ship a second OpenAPI built-in-server showcase — a fictional **Contoso M365** instance over Microsoft Graph — that demonstrates per-user delegated OAuth (`oauth_passthrough`) reading a two-sheet (Customers / Orders) Excel workbook, with two explicit id-keyed tools (`get_customer`, `get_customer_orders`) and a replayed Code Mode workflow that computes "customers who bought more than 100 in the last 3 months" — proving the "keep your existing Excel files, connect them to AI via MCP" story end-to-end **offline**, as config + curated spec + fixture + docs (no new feature code).

## Background

Phase 90 built `pmcp-openapi-server` (config-driven OpenAPI→MCP). Phase 90.1 delivered the **london-tube** showcase, which demonstrates `api_key` (query-param) backend auth only. The toolkit already ships the other auth variants and the full delegated chain, but **no end-to-end example exercises any OAuth path**:

- `AuthConfig::OAuthPassthrough` — `crates/pmcp-server-toolkit/src/http/auth.rs:123`
- Full passthrough wiring `TokenCaptureAuthProvider → AuthContext → HttpCodeExecutor::with_inbound_token → outbound forward` — `crates/pmcp-openapi-server/src/assemble.rs:22-31,93` (Phase 90 Plan 90-10)
- Passthrough unit test — `crates/pmcp-server-toolkit/src/http/auth.rs:791 test_passthrough_forwards_inbound_token`

So passthrough works at the unit level but is never shown as a runnable business example. The connector is a **thin HTTP mapper** — it does NOT filter rows server-side and does NOT follow `@odata.nextLink` pagination (confirmed: no pagination code in `pmcp-server-toolkit/src` or `pmcp-openapi-server/src`). This phase mirrors the 90.1 london-tube structure (fixture + pointable example + book/course chapters + offline `parity_replay`) for a Graph/Excel instance, with the headline being delegated OAuth + Excel automation.

The work is config + curated spec + fixture + docs. The only code is example/fixture/test Rust mirroring 90.1's `london_tube_min.rs` / `parity_replay.rs` — no connector or toolkit feature change.

## Requirements

1. **P902-SPEC — curated Graph OpenAPI spec**: A hand-authored, trimmed Graph OpenAPI description covering only the Excel worksheet range-read operation that backs the two tools.
   - Current: No Graph OpenAPI spec is vendored anywhere in the repo.
   - Target: A small spec (`contoso-m365.yaml` or equivalent) describing the Excel range-read op (`GET /drives/{drive-id}/items/{item-id}/workbook/worksheets/{worksheet-id}/range(address='…')` shape), NOT the full Graph metadata. The demo workbook's drive/item ids are pinned in config.
   - Acceptance: The spec file exists, parses through the toolkit's `OpenApiSchema::parse`, and contains exactly the range-read operation(s) needed by the two tools (no SharePoint file-listing/content ops, no write ops).

2. **P902-CONFIG — Contoso M365 fixture config**: A TOML config with `oauth_passthrough` backend auth and the two id-keyed tools.
   - Current: Only `london-tube.toml` (api_key) exists as an OpenAPI fixture.
   - Target: `tests/fixtures/contoso-m365.toml` with `[backend.auth] type = "oauth_passthrough"`, `base_url = "https://graph.microsoft.com/v1.0"`, two `[[tools]]` (`get_customer`, `get_customer_orders`) each taking a `customer_id` parameter, a `[code_mode]` block, and `[[resources]]`/`[[prompts]]` for Code Mode context (mirroring london-tube's resource/prompt surface).
   - Acceptance: `ServerConfig::from_toml_strict_validated` accepts the file; `grep` confirms `type = "oauth_passthrough"`, both tool names, and a `customer_id` parameter on each tool; no real credential present (only env-ref placeholders for any client/tenant ids in docs/config).

3. **P902-FIXTURE — date-pinned Customers/Orders workbook data**: Deterministic mock Graph responses representing the two-sheet workbook.
   - Current: No Graph/Excel mock data exists.
   - Target: wiremock mock response bodies for the range-read op returning a fixed Customers sheet and Orders sheet (orders reference customer ids). All order dates are pinned relative to a **fixed reference date embedded in the fixture** (not wall-clock `now`), so the "last 3 months" query is reproducible forever.
   - Acceptance: The fixture defines ≥1 customer that satisfies "bought >100 in the last 3 months" relative to the pinned reference date and ≥1 that does not, so the headline query has a non-trivial, deterministic expected result.

4. **P902-PARITY — offline tool + passthrough parity test**: An offline `parity_replay`-style test proving the two tools work and the user token is forwarded.
   - Current: `parity_replay.rs` covers london-tube (api_key) only.
   - Target: An offline wiremock-backed test driving the real binary path that (a) calls `get_customer`/`get_customer_orders` for a known `customer_id` and asserts the returned rows match the fixture, and (b) asserts the **inbound user bearer was forwarded** to the Graph backend (the mock requires/echoes `Authorization: Bearer <token>` and the test asserts the forwarded value, proving delegated passthrough — never a server-held credential).
   - Acceptance: The test runs single-threaded with no live network and passes; it fails if the forwarded `Authorization` header is absent or wrong; no real Graph endpoint is contacted in CI.

5. **P902-CODEMODE — replayed headline Code Mode workflow**: A runnable/offline-replayed Code Mode workflow computing the headline business query.
   - Current: No Code Mode business-aggregation workflow is exercised by any test.
   - Target: An offline-replayed Code Mode workflow that loads customers + orders via the tools, joins on `customer_id`, filters orders to the trailing 3 months of the pinned reference date, sums per customer, and returns the set with total > 100.
   - Acceptance: The workflow runs offline and asserts the exact deterministic set of customer ids expected from the fixture (P902-FIXTURE); changing the fixture's amounts/dates changes the asserted set; uses the pinned reference date, not wall-clock time.

6. **P902-EXAMPLE — pointable config + runnable build-only example**: A published, user-pointable config and a `cargo run` example, mirroring 90.1.
   - Current: 90.1 ships `examples/london-tube.toml` + `examples/london_tube_min.rs`; nothing for Graph.
   - Target: `crates/pmcp-openapi-server/examples/contoso-m365.toml` (ships with the published crate; same passthrough/tools/resources/prompts surface as the fixture) + `crates/pmcp-openapi-server/examples/contoso_m365_min.rs` (build-and-exit, references the pointable config, prints a summary, does not serve).
   - Acceptance: `cargo run --example contoso_m365_min -p pmcp-openapi-server` exits 0 without hanging; a smoke test parses `examples/contoso-m365.toml` through `from_toml_strict_validated`; no real credential in either artifact.

7. **P902-DOCS-BOOK — dedicated book chapter**: A new advanced-example chapter in pmcp-book.
   - Current: `pmcp-book/src/openapi-built-in-server.md` covers the basic OpenAPI server; no Graph/advanced chapter.
   - Target: A new chapter (e.g. `pmcp-book/src/openapi-graph-m365.md`) wired into `SUMMARY.md`, telling the governance story (admin consents once to a bounded scope; the signed-in user's forwarded token governs per-file access; server holds no standing credentials), the business-analyst-curates-the-API-slice point, the Excel-automation narrative, and a walkthrough of the two tools + the Code Mode headline query.
   - Acceptance: `make book` exits 0 with the new chapter in the nav; the chapter contains the governance explanation, the `oauth_passthrough` config block, both tool names, and the headline Code Mode query.

8. **P902-DOCS-COURSE — dedicated course chapter**: The matching chapter in pmcp-course (course voice).
   - Current: `pmcp-course/src/openapi-built-in-server.md` exists; no Graph/advanced chapter.
   - Target: A new course chapter (e.g. `pmcp-course/src/openapi-graph-m365.md`) wired into the course `SUMMARY.md`, same substance as the book chapter in the tighter course voice.
   - Acceptance: `mdbook build` in `pmcp-course` exits 0 with the new chapter in the nav; same required-content greps as P902-DOCS-BOOK pass.

## Boundaries

**In scope:**
- Curated/trimmed Graph OpenAPI spec for the Excel worksheet range-read op (P902-SPEC)
- `tests/fixtures/contoso-m365.toml` with `oauth_passthrough` + `get_customer`/`get_customer_orders` (id-keyed) + code_mode + resources/prompts (P902-CONFIG)
- Deterministic, date-pinned Customers/Orders workbook mock data (P902-FIXTURE)
- Offline parity test: tool correctness + forwarded-bearer passthrough assertion (P902-PARITY)
- Offline-replayed Code Mode workflow for "bought >100 in last 3 months" with a deterministic asserted result (P902-CODEMODE)
- Pointable `examples/contoso-m365.toml` + build-only `examples/contoso_m365_min.rs` (P902-EXAMPLE)
- New dedicated book + course chapters with the governance + Excel narrative (P902-DOCS-BOOK, P902-DOCS-COURSE)
- An `#[ignore]`+env-gated live Graph test scaffold (best-effort, like `parity_live_tfl`)

**Out of scope:**
- Any Excel **write** / workbook-session path — read-only only; write-back is a separate future spike (the connector is read-oriented and the workbook-session path is unwired)
- `@odata.nextLink` pagination — the connector does not support it and a single demo workbook does not need it; including it would make this a feature build
- SharePoint/OneDrive file-discovery ops (list document library, file content) — the workbook is addressed by pinned drive/item ids; navigation-first workflows are excluded to keep the curated spec minimal
- `oauth2_client_credentials` (app-only) as a delivered config — passthrough is the only hero; client-credentials may be mentioned in at most one doc sentence but is not built or tested here
- Server-side row filtering in the two tools — the connector is a thin HTTP mapper; all filtering/joining/aggregation is Code Mode's job
- A **required** live Graph test — live testing is optional/best-effort; the offline parity is the completion gate (no real Azure AD tenant, app registration, or admin consent needed to verify this phase)
- Any change to `AuthConfig`, the passthrough chain, the connector, or other toolkit/`pmcp-openapi-server` `src/` code — this phase is config + fixture + docs + example/test Rust only

## Constraints

- **No new feature code**: only `examples/` + `tests/` Rust (mirroring 90.1), config, curated spec, and docs. Any need to touch `src/` invalidates the "config + fixture + docs" premise and must be raised as a separate spike.
- **Deterministic over wall-clock**: the "last 3 months" filter MUST resolve against a fixed reference date embedded in the fixture, never `now`, or the Code Mode test rots.
- **Offline + single-threaded**: parity/Code Mode tests use wiremock, no live network in CI, run under `--test-threads=1` (per-process env + ephemeral port), matching the 90.1 `parity_replay` pattern.
- **No real secrets**: no real Graph client id/secret/tenant or user token committed; use env-ref placeholders (`${...}`/`env:VAR`) and a dev-guarded inline token_secret where 90.1 does, asserting placeholders (not literals) in tests.
- **Structural mirror of 90.1**: fixture + pointable example + `*_min.rs` + parity test + book/course chapters, to keep the two showcases consistent.
- **Quality gates**: example/test Rust must pass `make quality-gate` (fmt, clippy, cog ≤25, zero SATD) like all repo code.
