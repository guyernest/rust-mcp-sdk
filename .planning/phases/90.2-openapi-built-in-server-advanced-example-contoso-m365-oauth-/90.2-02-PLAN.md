<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client -> server | inbound `Authorization: Bearer` is untrusted client input; captured per-request, never stored |
| server -> Graph backend | the forwarded user token is the ONLY credential; the server holds no standing credential |
| repo commit -> published crate | examples/ ships in the tarball — highest-exposure artifact for a leaked secret |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-90.2-passthrough-02 | Spoofing / Elevation of Privilege | oauth_passthrough relay | mitigate | The parity test ASSERTS the forwarded `Authorization: Bearer contoso-user-tok` on every recorded backend request and FAILS if absent/wrong (two-sided proof recorded) — guarantees the server forwards the user's token and never substitutes a server-held credential. X-API-Key (also injected by ServerTester) is explicitly NOT treated as the delegated credential. |
| T-90.2-secrets-02 | Information Disclosure | examples/contoso-m365.toml + contoso_m365_min.rs (SHIPPED) | mitigate | `${VAR}` placeholders for any tenant/client/secret VALUE + the single dev-guarded inline `token_secret`; acceptance grep rejects JWT-shaped/GUID-shaped real credential literals while explicitly permitting `${VAR}` placeholders. |
| T-90.2-content-02 | Tampering | wiremock canned responses | accept | Test-only inert JSON authored by us, loaded from the canonical workbook fixture; no live Graph contacted in CI. |
