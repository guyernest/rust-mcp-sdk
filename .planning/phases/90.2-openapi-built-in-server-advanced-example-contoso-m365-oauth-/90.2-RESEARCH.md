## Summary

Phase 90.2 is a **pure config + curated-spec + fixture + docs + example/test-Rust** phase that mirrors the just-completed Phase 90.1 london-tube showcase. Every piece of machinery it needs — `AuthConfig::OAuthPassthrough`, the `TokenCaptureAuthProvider → AuthContext → HttpCodeExecutor → outbound forward` chain, single-call tool path/query parameter mapping, the wiremock offline parity harness, and the Code Mode `validate_code`/`execute_code` engine — already ships and is unit-tested. The phase **configures** that machinery for a fictional Contoso M365 instance; it touches **no `src/`**. [VERIFIED: codebase reads of auth.rs, assemble.rs, tools.rs, client.rs, code_mode.rs, parity_replay.rs, oauth_passthrough_e2e.rs]

All four research questions resolve cleanly with file:line evidence:

- **RQ-1 (passthrough proof) — RESOLVED, the YAML replay path CAN assert it.** `ServerTester::new`'s 4th argument `api_key: Option<&str>` injects `Authorization: Bearer <key>` into the streamable-HTTP transport's `extra_headers` on every MCP request (`crates/mcp-tester/src/tester.rs:96-101,154-162`). The server's `TokenCaptureAuthProvider` captures that inbound `Authorization` header into `AuthContext.token` (`crates/pmcp-openapi-server/src/assemble.rs:97-110`), and the passthrough provider forwards it to the Graph backend. So a parity test can pass `Some("contoso-user-tok")` to `ServerTester::new` and assert the wiremock Graph mock **required** `header("authorization", "Bearer contoso-user-tok")`. A standalone Rust integration test (the `oauth_passthrough_e2e.rs` pattern) is a viable belt-and-suspenders alternative but is **not required** — the replay path suffices.
- **RQ-2 (Code Mode result assertion) — `validate_code` only validates; `execute_code` runs+returns but requires an approval token.** A `validate_code` YAML step (london-tube's pattern) asserts `success` only, not a computed value. `execute_code` runs the script and returns the computed result (`crates/pmcp-server-toolkit/src/code_mode.rs:572-623`) **but verifies an HMAC `approval_token` + code-hash minted by `validate_code` first** — a two-call dance a flat YAML scenario can't thread. **Therefore P902-CODEMODE must be a Rust test** that drives the `JsCodeExecutor` directly over a token-threaded `HttpCodeExecutor` (the exact `oauth_passthrough_e2e.rs:63-72,99-103` seam) and asserts the deterministic customer set.
- **RQ-3 / RQ-4 (Graph range addressing + curated spec) — RESOLVED.** Graph's range-read URL is `GET /drives/{drive-id}/items/{item-id}/workbook/worksheets/{worksheet-id}/range(address='A1:D8')?$select=values` [CITED: learn.microsoft.com/graph/api/range-get]. The toolkit's single-call synthesizer makes every `{...}` path-template segment a **required path param** and every other declared param a **query param** (`crates/pmcp-server-toolkit/src/tools.rs:499-537`); path substitution is a literal `str::replace` with **no URL re-encoding** (`crates/pmcp-server-toolkit/src/http/client.rs:150-163`, `join_url` at `http/mod.rs:76`). Because the connector does **no arithmetic**, `customer_id`→row math cannot happen in the connector — so the demo's cleanest fit is a **per-tool literal path** where the worksheet name encodes the sheet and the range address is templated by a parameter (see Architecture Patterns).

**Primary recommendation:** Copy the 90.1 file set verbatim and re-skin it for Contoso. The single genuinely new design decision is how `get_customer(customer_id)` and `get_customer_orders(customer_id)` map a `customer_id` onto a Graph range address without connector arithmetic — resolve this by giving each tool a fixed worksheet-scoped path and making the **range address a templated path/query parameter the tool fills from `customer_id`**, OR (simpler, recommended) make each tool a **script tool** whose JS body templates the address (Code Mode CAN do arithmetic). The Rust Code Mode test (P902-CODEMODE) is non-negotiable per RQ-2.

### Pattern 2: id-keyed tool → Graph range address (RQ-3) — two viable shapes
**RECOMMENDED — script tool (JS templates the address; connector does no arithmetic):**
```toml
# get_customer: customer_id → row → range(address='A{row}:D{row}'). The JS engine
# does the arithmetic (Math.* is available; build the address string, then api.get).
[[tools]]
name = "get_customer"
description = "Fetch one customer row (customer_id, name, segment, region) from the Customers sheet."
script = """
const idx = parseInt(args.customer_id.slice(1), 10);   // 'C007' → 7
const row = idx + 1;                                    // header offset (row 1 = header)
const addr = `A${row}:D${row}`;
const r = await api.get(`/drives/CONTOSO_DRIVE/items/CUSTOMERS_ITEM/workbook/worksheets/Customers/range(address='${addr}')?$select=values`);
return r.values;
"""
[[tools.parameters]]
name = "customer_id"
type = "string"
description = "Customer id, e.g. 'C007'."
required = true
```
**ALTERNATIVE — single-call tool with a templated range param** (only if the planner wants a pure single-call op): make the range a `{...}` path segment so the synthesizer treats it as a path param (tools.rs:501-515). Note `customer_id` arithmetic must then happen client-side (the model fills the address), which is less of a clean "id-keyed" tool. The recommended script-tool shape keeps the tool truly `customer_id`-keyed.

**Verified mechanics:** `build_operation` (tools.rs:499-537) — `{...}` segments → required path params; other declared params → query params. Substitution is literal `str::replace` (client.rs:158-159) with no re-encode (`join_url`, mod.rs:76-83). reqwest performs URL parsing/encoding on send, so the OData `range(address='A8:D8')` segment (parens, quotes, colon) is carried through. **Smoke-test the spec + a sample call in the parity test** to confirm reqwest accepts the literal address (Pitfall 4).

### Pattern 5: Deterministic Code Mode headline query (RQ-2, P902-CODEMODE) — Rust test
```rust
// Source: oauth_passthrough_e2e.rs:36-103 — drive JsCodeExecutor over a token-threaded HttpCodeExecutor.
// The JS engine has NO Date builtin (Pitfall 1) → the reference date is a literal in the script.
const HEADLINE: &str = r#"
const REF = "2026-05-30";                       // pinned reference date (literal, never `now`)
const customers = await api.get("/drives/.../worksheets/Customers/range(address='A2:D9')?$select=values");
const orders    = await api.get("/drives/.../worksheets/Orders/range(address='A2:D40')?$select=values");

return result;   // deterministic set of customer ids
"#;
// ...build passthrough_exec + extra_with_token(Some("Bearer contoso-user-tok")),
// JsCodeExecutor::new(...).execute(HEADLINE, None) → assert_eq!(result, json!(["C007","C012"]));
```
**Date handling:** because the engine has no `Date`/`Date.now()` (verified — only Rust `Instant::now()` exists), the script compares `order_date` strings against the pinned `REF` constant using string/lexicographic comparison on ISO `YYYY-MM-DD` dates, or simple integer month math. The "last 3 months" window is `[REF - 3 months, REF]` computed from the literal — never wall-clock.

### Anti-Patterns to Avoid
- **Putting `customer_id`→row arithmetic in the single-call connector:** the connector does literal `str::replace` only (client.rs:158) — no math. Use a script tool or supply the address as a param.
- **Asserting a computed Code Mode result via a YAML `execute_code` step:** `execute_code` requires an approval token (code_mode.rs:589-616); YAML can't mint+thread it. Use the Rust test.
- **Using wall-clock `now` anywhere in the headline query:** the JS engine has no Date; even if it did, the fixture would rot. Pin a reference date literal.
- **Re-encoding the range address by hand:** don't double-encode; let reqwest handle it. Just verify the call lands via the parity test.
- **Adding a second Graph op for orders:** D-03 keeps it range-read-only; lay the Orders sheet out contiguously by `customer_id` and address a block.

## Common Pitfalls

### Pitfall 1: Assuming the JS engine has `Date`/`Date.now()`
**What goes wrong:** Authoring the headline query with `new Date()` / `Date.now()` — it won't evaluate (no such builtin), or if added later, the test rots.
**Why it happens:** JS habit; the engine supports `Math.*`, `parseFloat`, `Object.keys`, array methods (eval.rs:1197-1250) but NOT Date (verified: only Rust `Instant::now()` exists in the engine).
**How to avoid:** Embed the reference date as a string literal in the script; compare ISO `YYYY-MM-DD` order dates lexicographically or with integer month math.
**Warning signs:** A test that passes today but would change result tomorrow; any `now`/`Date` token in a fixture or script.

### Pitfall 2: deny_unknown_fields — exact field names
**What goes wrong:** A misspelled key (`type` vs `type_`, `parameters` vs `params`, `mime_type` vs `mimetype`) is a hard parse error, not a warning.
**Why it happens:** Every config struct carries `#[serde(deny_unknown_fields)]` (config.rs:8,76,101,…).
**How to avoid:** Use exact names — `ParamDecl`: `name`, `type` (renamed from `param_type`), `description`, `required`, `default`, `max_length`, `minimum`, `maximum`, `enum` (config.rs:670-699). `ToolDecl`: `name`, `description`, `path`, `method`, `base_url`, `script`, `parameters` (config.rs:579-614). `ResourceDecl`: `uri`, `name`, `description`, `mime_type`, `content` (config.rs:767-784). `PromptDecl`: `name`, `description`, `include_resources` (config.rs:730-739). `AnnotationsDecl`: `read_only_hint`, `destructive_hint`, `idempotent_hint`, `open_world_hint`, `cost_hint` (config.rs:705-720). `[backend.auth] type = "oauth_passthrough"` with `target_header`, `required`.
**Warning signs:** `from_toml_strict_validated` returns an "unknown field" error.

### Pitfall 3: include_resources must match a defined resource URI
**What goes wrong:** A `start_code_mode` prompt referencing a typo'd resource URI fails resolution at assembly (register_prompts resolves each against the StaticResourceHandler — assemble.rs:204-213).
**Why it happens:** Same as 90.1's "Pitfall 2" baked into london-tube.toml.
**How to avoid:** Every URI in `[[prompts]] include_resources` must equal a `[[resources]] uri` exactly. Mirror 90.1: define a `assert_contoso_code_mode_surface` helper asserting the resource URIs + prompt name in BOTH fixture and pointable example (parity_replay.rs:56-73).

### Pitfall 4: Range-address path segment + reqwest URL parsing
**What goes wrong:** The OData `range(address='A8:D8')` segment contains `'`, `(`, `)`, `=`, `:` — if substituted into a `{...}` path param it is NOT re-encoded by the toolkit (client.rs:158, join_url mod.rs:76); reqwest must accept it.
**Why it happens:** `join_url` is plain `format!` concatenation; no percent-encoding.
**How to avoid:** Prefer the **script-tool** shape (the address is built inside JS and passed to `api.get` as a literal path — proven to work for london-tube's templated `/Line/${id}/Disruption`). The parity test's wiremock `path(...)` matcher confirms the request lands; if the address must be a path param of a single-call tool, smoke-test one call in the parity test before relying on it.
**Warning signs:** A 404 at the mock, or a reqwest URL-parse error, on the range call.

### Pitfall 5: positional verify-filter test-fn naming
**What goes wrong:** GSD's verify step runs a positional name filter; mismatched test-fn names won't be selected.
**Why it happens:** Documented in oauth_passthrough_e2e.rs:18-20 ("test fns are `oauth_passthrough_e2e_`-prefixed so the positional verify filter resolves").
**How to avoid:** Prefix the parity test fns `contoso_m365_*` and the code-mode test fns `contoso_m365_code_mode_*` consistently with the file stem.

### Pitfall 6: tests/ is excluded from the crate tarball
**What goes wrong:** Putting the pointable config under `tests/` means users can't `pmcp-openapi-server --config <it>` from the published crate.
**Why it happens:** `Cargo.toml:13` `exclude = [..., "tests/"]`; `examples/` is NOT excluded.
**How to avoid:** The pointable `contoso-m365.toml` MUST live under `examples/` (the fixture copy under `tests/fixtures/` is for the offline test only). Add a smoke test parsing `examples/contoso-m365.toml` via `from_toml_strict_validated` (parity_replay.rs:80-95 pattern).

## Open Questions

1. **Does `OpenApiSchema::parse` accept the OData `(address='{address}')` path segment?**
   - What we know: it accepts standard `{param}` path templates (london-tube-api.yaml). The parenthesized OData function segment is non-standard OpenAPI.
   - What's unclear: whether the parser tolerates `(`/`'`/`=` in a path key.
   - Recommendation: Wave 0 — add a fixture-validity test that parses `contoso-m365-api.yaml`; if it rejects the OData segment, simplify the path to `.../range` and document the address as a query/`$select`-style parameter. The spec is served as context, not used for routing, so this is low-risk.

2. **Single-call tool vs script tool for `get_customer`/`get_customer_orders`?**
   - What we know: connector does no arithmetic; script tools (JS) can template the address (RQ-3).
   - What's unclear: whether the planner wants "id-keyed single-call tools" literally or accepts script tools that take `customer_id`.
   - Recommendation: use **script tools** (recommended in Pattern 2) — they keep the tool truly `customer_id`-keyed while doing the row→address math in JS. This matches D-02/D-03 ("the tool templates the Excel range address from the id").

## Security Domain

### Applicable ASVS Categories
| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | `oauth_passthrough` — delegated per-user bearer; server holds NO standing credential (the governance story, D-05) |
| V3 Session Management | no | Stateless MCP request; token captured per-request into `AuthContext`, not stored |
| V4 Access Control | yes (narrative) | Admin consents once to a bounded scope (ceiling); the forwarded user token governs per-file access — documented, not code |
| V5 Input Validation | yes | `customer_id` bounded by tool input schema (`additionalProperties:false`, tools.rs:556-558); range address built from validated id |
| V6 Cryptography | yes | HMAC approval-token (code_mode) — toolkit-owned, never hand-rolled; dev-guarded inline `token_secret` for offline tests only |

### Known Threat Patterns for oauth_passthrough + Graph
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Token redirected to attacker header | Tampering | `target_header` is operator-fixed in committed config; client controls only the VALUE (auth.rs WR-04 doc) |
| Header injection via malformed token | Tampering | `HeaderValue::try_from` rejects control chars (auth.rs:442-444) — malformed token rejected, not relayed |
| Real credential leaking into repo | Info Disclosure | No real Graph client/tenant/user token committed; `${VAR}` placeholders + dev-guarded inline secret; tests assert resolved (not literal) values |
| Server acting beyond user's rights | Elevation of Privilege | Server holds no standing credential; can only ever act AS the calling user (the governance note, D-05) |
