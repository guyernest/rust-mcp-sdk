## Warnings

### WR-01: No regression test for the `oauth_passthrough` alias deserialization

**File:** `crates/pmcp-server-toolkit/src/http/auth.rs:109,134`

**Issue:** The fix that motivated the only `src/` change in this phase — adding
`#[serde(alias = "oauth_passthrough")]` and `#[serde(alias =
"oauth2_client_credentials")]` — has no unit test that directly asserts the alias
forms deserialize correctly. The existing `test_auth_config_deserializes_snake_case_tag`
test (line 946) only exercises `"bearer"`, not either of the two new aliases. The
integration path through `ServerConfig::from_toml_strict_validated` in
`contoso_m365_fixture` provides indirect coverage, but that is a test-binary
dependency, not a focused unit assertion in `auth.rs` itself. If a future refactor
accidentally removes the alias attributes, no fast unit-level test will catch it.

**Alias vs rename analysis:** Using `#[serde(alias)]` (rather than changing the
variant name or adding `#[serde(rename)]`) is the correct choice here. `alias` is
purely additive: it accepts the documented spelling during deserialization while
leaving the canonical serialized form as the `rename_all = "snake_case"`-derived
string (`o_auth_passthrough`). A `#[serde(rename)]` would instead CHANGE the
canonical serialization output — any configs that already use the derived
`o_auth_passthrough` form (if any exist) would break, and the semantic tag embedded
in serialized `AuthConfig` values would silently change. The alias approach is the
safer, backward-compatible fix.

**Fix:** Add two `#[test]` cases to the existing test block, one per alias:

```rust
#[test]
fn test_auth_config_deserializes_oauth_passthrough_alias() {
    let toml_src = r#"type = "oauth_passthrough"
target_header = "Authorization"
required = true
"#;
    let cfg: AuthConfig = toml::from_str(toml_src).unwrap();
    assert!(matches!(
        cfg,
        AuthConfig::OAuthPassthrough { required: true, .. }
    ));
}

#[test]
fn test_auth_config_deserializes_oauth2_client_credentials_alias() {
    let toml_src = r#"type = "oauth2_client_credentials"
token_url = "https://login.example.com/token"
client_id = "cid"
client_secret = "secret"
"#;
    let cfg: AuthConfig = toml::from_str(toml_src).unwrap();
    assert!(matches!(cfg, AuthConfig::OAuth2ClientCredentials { .. }));
}
```

### WR-02: `get_customer_orders` emits a silent empty-address request for C006

**File:** `crates/pmcp-openapi-server/tests/fixtures/contoso-m365.toml:109-117`,
`crates/pmcp-openapi-server/examples/contoso-m365.toml:115-123`

**Issue:** The `get_customer_orders` script handles only C001–C005 in its
`if`/`else if` chain. C006 ("Proseware") is intentionally order-less (documented in
`contoso-m365-workbook.json` as "no contiguous block address"), so `ordersAddr`
remains `""` and the tool issues:

```
GET .../range(address='')?$select=values
```

This is a real HTTP request to the backend with an empty range address — not a
clean "no orders" response. The Graph endpoint would return an error for this malformed
address. The result is confusing for users of the shipped pointable example
(`examples/contoso-m365.toml`) who call `get_customer_orders("C006")`.

The `get_customer` script correctly includes `C006 -> "A7:D7"`, so the asymmetry is
inconsistent.

Note: this does not affect any existing tests (no test calls `get_customer_orders`
with `"C006"`), but it is a silent bad-request path in the shipped example.

**Fix:** Add an explicit `else` guard that returns a descriptive empty result or
an error-friendly message rather than issuing a malformed URL. The
simplest engine-safe approach:

```js
let ordersAddr = "";
if (args.customer_id === "C001") { ordersAddr = "A2:D3"; }
else if (args.customer_id === "C002") { ordersAddr = "A4:D4"; }
else if (args.customer_id === "C003") { ordersAddr = "A5:D5"; }
else if (args.customer_id === "C004") { ordersAddr = "A6:D6"; }
else if (args.customer_id === "C005") { ordersAddr = "A7:D7"; }
if (ordersAddr === "") { return []; }
const r = await api.get(`/drives/CONTOSO_DRIVE/items/ORDERS_ITEM/workbook/worksheets/Orders/range(address='${ordersAddr}')?$select=values`);
return r.values;
```

Apply the same guard in the fixture TOML, the examples TOML, and the inline CONFIG
in `contoso_m365_min.rs` (the min example only has `get_customer`, so only the two
TOML files and the fixture need the fix).

### WR-03: Course chapter section numbers are out of order (Step 4 before Step 3)

**File:** `pmcp-course/src/openapi-graph-m365.md:42,79`

**Issue:** The course chapter presents headers `## Step 4: Authenticate to the
Backend — The Double Lock` at line 42 and then `## Step 3: Two Kinds of Tools` at
line 79. A chapter that introduces authentication before defining what the tools are
reads coherently, but the explicit step numbers 4 and 3 in that order are confusing
and will mislead readers who use step numbers as a sequence. The book chapter
(`pmcp-book/src/openapi-graph-m365.md`) avoids numbered steps entirely and reads
fine; the course chapter introduced the numbers inconsistently.

**Fix:** Either renumber so the auth step follows the tools step (change `Step 4`
to `Step 2` and `Step 3` to `Step 3`, or use `Step 1`/`Step 2` for tools/auth
respectively), or drop the numeric prefix entirely and use descriptive headers
matching the book's style. The latter is lower-risk because no cross-reference links
depend on these numbers.

```markdown
## The Backend (Microsoft Graph)
...

## Outgoing Authentication — The Double Lock
...

## Two Kinds of Tools (`get_customer` / `get_customer_orders`)
```

### IN-01: `test_auth_config_deserializes_snake_case_tag` does not test the new canonical form

**File:** `crates/pmcp-server-toolkit/src/http/auth.rs:946`

**Issue:** The existing deserialization test covers only the `bearer` variant. With
the new aliases in place, there is no test verifying that the old derived forms
(e.g., `type = "o_auth_passthrough"`) still deserialize (they will, because the
alias supplements the derived name, but this is implicit). This is lower-priority
than WR-01 but worth noting: a test named `_snake_case_tag` that exercises only
`bearer` is a misleading test name.

**Suggestion:** Rename the test to `test_auth_config_deserializes_bearer_snake_case`
or expand it to exercise all six variants. The WR-01 fix addresses the more urgent
gap; this is a naming/coverage quality concern only.

### IN-02: Rollover-proof test does not exercise an actual year-rollover scenario

**File:** `crates/pmcp-openapi-server/tests/contoso_m365_code_mode.rs:327`

**Issue:** `contoso_m365_code_mode_rollover_math_equals_pinned_window_start` proves
the month-subtraction math equals the pinned `window_start` for the
canonical `ref_date = "2026-05-28"`. Since May minus 3 = February, no year-rollover
branch (`m - 3 < 1`) is exercised by this test. The rollover ternary
(`m2 = m - 3 < 1 ? m - 3 + 12 : m - 3`) is logically correct and the test verifies
the no-rollover path, but a fixture date near January (e.g., `REF = "2026-02-15"`)
would trigger the year-rollover branch and increase confidence. The pinned fixture
cannot be changed without cascading updates to the expected set, so this is a
future-improvement note only — the current test is not wrong.

**Suggestion:** Add a standalone rollover unit test using a synthetic `REF` such as
`"2026-01-15"` (expected `window_start = "2025-10-15"`) that exercises the
`y -= 1, m += 12` path without touching the canonical fixture.

### IN-03: Doc comment at auth.rs:56 lists `oauth_passthrough` as if it is the primary tag

**File:** `crates/pmcp-server-toolkit/src/http/auth.rs:56`

**Issue:** The top-level `AuthConfig` doc comment lists the config values as
`oauth_passthrough` (the alias/documented form) rather than the canonical
`o_auth_passthrough` (the `rename_all = "snake_case"` derived form). Since `alias`
does not affect serialization, the true canonical tag for round-trips is still
`o_auth_passthrough`, but the doc comment implies `oauth_passthrough` is the primary
name. This is intentional (the docs document the user-facing contract), but it means
the doc comment and the serialization output describe different strings. A round-trip
test that serializes an `OAuthPassthrough` variant to TOML and asserts `type =
"o_auth_passthrough"` would make this explicit.

**Suggestion:** Add a short note to the variant doc comment clarifying that
serialization emits the snake_case derived form (`o_auth_passthrough`) while
deserialization accepts both. Example addition to the existing doc comment at
line 130–133:

```
/// Note: serialization emits `o_auth_passthrough` (the `rename_all = "snake_case"`
/// derived form); deserialization accepts both `o_auth_passthrough` and the alias
/// `oauth_passthrough`. Configs written by humans should use `oauth_passthrough`.
```
