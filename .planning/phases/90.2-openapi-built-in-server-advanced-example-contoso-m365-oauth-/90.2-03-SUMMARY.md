# Phase 90.2 Plan 03: Contoso M365 Code Mode Headline Query Summary

**An offline Rust Code Mode test proves the demo's hero query ("customers who bought more than 100 in the last 3 months") end-to-end: it drives `JsCodeExecutor` over the same token-threaded `HttpCodeExecutor` the `execute_code` handler builds per request, the script reads the SAME Graph range-read API the `get_customer`/`get_customer_orders` tools expose (all-rows blocks), joins orders to customers, filters to the trailing 3 months of a PINNED reference date with engine-accurate `YYYYMMDD`-integer date math (no `Date` builtin), sums per customer, and returns the deterministic set — asserted against the EXACT `expected_headline_set` LOADED from the canonical workbook json, with both backend reads required to carry the forwarded user bearer and to be invoked exactly once.**

## (c) Boundary-customer classification result

- **C003 (boundary, IN):** its single order is dated EXACTLY `window_start` (2026-02-28, amount 150); the filter is inclusive of the lower bound (`>= startNum`), so C003 totals 150 > 100 and is IN.
- **C004 (just-outside, OUT):** its only order (2026-02-27, amount 500) is one day before `window_start`, so its in-window total is 0 and it is OUT — despite the large amount, proving the date filter (not the amount) drives exclusion.
- Both are restated as explicit `assert!`s in addition to the set-equality proof, so a fixture edit that breaks the boundary fails loudly.

## (d) Month-rollover-safe trailing-3-month approach used

- NO `Date` builtin. The script parses `REF` into `y`/`m`/`d` via `parseInt` on fixed `YYYY-MM-DD` substrings.
- Month subtraction with explicit year rollover, expressed without reassignment (the subset forbids `Expr::Assign`/`++`/`while`/`throw`): `const m2 = m - 3 < 1 ? m - 3 + 12 : m - 3;` and `const y2 = m - 3 < 1 ? y - 1 : y;`.
- `computedStart` is reassembled as a zero-padded `YYYY-MM-DD` string (`toFixed(0)` integer formatting + `REF.slice(8,10)` for the unchanged day) and proven `=== START` by the dedicated `contoso_m365_code_mode_rollover_math_equals_pinned_window_start` test.
- **Date comparison reality (Genchi Genbutsu):** the engine's relational operators coerce operands to numbers, so ISO-string lexicographic comparison fails (date strings -> NaN). The window filter therefore uses `YYYYMMDD` integer encodings: `refNum = y*10000 + m*100 + d`, `startNum = y2*10000 + m2*100 + d`, and each order date is encoded the same way inline. The trailing-3-month logic is genuinely demonstrated (not hardcoded), and the result is stable across calendar time.

## (e) "Via the tools" reconciliation (P902-CODEMODE)

Documented in the test header comment AND here. The `get_customer`/`get_customer_orders` connector tools are thin literal `customer_id -> A{row}:D{row}` mappers over a SINGLE Microsoft Graph range-read op (`/drives/.../workbook/worksheets/{sheet}/range(address='...')?$select=values`) with no aggregation. The headline aggregation reads the **SAME Graph range-read API the tools expose** — identical endpoint shape, identical `$select=values` projection, identical forwarded user bearer — but over the all-rows block addresses (`all_customers_address`/`all_orders_address` = `A2:D7`) so the cross-row join + filter + sum is computable in a single pass. Reading the whole-table block instead of per-id ranges is REQUIRED for the aggregation and stays within the single existing range-read op: **no new Graph op, no scope expansion, the locked tool interface is unchanged.**

## Deviations from Plan

The plan instructed the in-window date filter to use "ISO `YYYY-MM-DD` lexicographic string comparison". During execution this FAILED at runtime: the code-mode engine's relational operators (`>=`/`<=`) coerce both operands to numbers (`comparison_op` -> `to_number`), so a date STRING parses to `NaN` and every comparison is false, producing an empty set. This is a Rule 1 (auto-fix bug) / Rule 3 (blocking) correction: the filter was reimplemented with `YYYYMMDD` integer encoding (`parseInt` on date substrings -> `y*10000 + m*100 + d`), which the engine compares correctly. The trailing-3-month semantics, inclusivity of the lower bound, and the asserted set are unchanged — only the comparison representation differs from the plan's wording. The same engine reality also forced the rollover math away from `throw`-based validation (the subset has no `throw`) to a dedicated rollover-proof test that returns `computedStart` and asserts it equals the pinned `window_start`.

(Two engine-accuracy adjustments, both forced by the JS subset and proven by the green tests: (1) `api.get` paths are inlined string literals — binding the URL to a `const` first fails compilation; (2) integer string formatting uses `toFixed(0)` because the engine stores numbers as f64 and `"" + n` yields `"2.0"`.)
