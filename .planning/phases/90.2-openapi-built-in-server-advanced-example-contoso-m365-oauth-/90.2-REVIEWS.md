# Cross-AI Plan Review — Phase 90.2

## Codex Review
_Overall risk: MEDIUM_

**Summary**

The plan set is directionally strong and matches the phase intent: config/test/example/docs only, no `src/` changes, script tools for id-to-range arithmetic, offline passthrough assertion, and a Rust Code Mode test for the deterministic business query. I spot-checked the key technical claims against the repo: `ServerTester::new(..., Some("tok"), Some("http"), None)` does inject `Authorization: Bearer tok`; `oauth_passthrough` forwards captured tokens; `execute_code` does require an approval token; and script tools are the right choice because single-call HTTP tools do literal path substitution, not arithmetic.

The main risks are cross-plan drift and ordering: fixture data is recorded in a summary rather than a machine-readable fixture, Plan 04 docs can run before the actual fixture/query summaries exist, and Plan 03’s Code Mode test appears to use raw `api.get` range reads rather than the two MCP tools even though P902-CODEMODE says “via the tools.”

## 90.2-01-PLAN.md

**Strengths**

- Correctly scopes Plan 01 to fixture/config/spec/scenario only, with no `src/` changes.
- Chooses script tools for `get_customer` and `get_customer_orders`, which is technically correct given the connector cannot compute `customer_id` to Excel range math.
- Calls out strict TOML field names and prompt/resource URI exact-match risk.
- Correctly treats `Date`/wall-clock usage as forbidden and requires a pinned fixture model.
- Includes an OData path fallback plan for `OpenApiSchema::parse`, which is prudent.

**Concerns**

- **HIGH:** P902-FIXTURE is not actually captured as a machine-readable fixture. Plan 01 says workbook rows are “fixed here,” but the durable source is only `90.2-01-SUMMARY.md`. Plan 02 and Plan 03 then manually recreate rows in Rust mocks. That invites drift between config docs, parity mocks, Code Mode mocks, and docs.
- **MEDIUM:** The fallback OpenAPI path `/range` with `address` as a query parameter may no longer represent the real Microsoft Graph range-read shape. If used, it risks weakening P902-SPEC unless explicitly documented as a parser-compatible approximation while runtime scripts still call the real Graph path.
- **MEDIUM:** Validation is deferred too far. Plan 01’s automated verify only checks files exist; syntax, script compilation, and strict TOML parsing all fail later in Plan 02. That makes Plan 01 capable of producing broken foundation artifacts.
- **LOW:** The secret grep wording is confused. It says no `${TENANT` literal secret, but `${VAR}` placeholders are explicitly allowed by the spec.

**Suggestions**

- Add `tests/fixtures/contoso-m365-workbook.json` or equivalent in Plan 01 as the canonical Customers/Orders dataset, then have Plan 02 and Plan 03 load it or generate mocks from it.
- Add a lightweight Plan 01 validation command, even if it is a small temporary or existing parser test invocation, to check `from_toml_strict_validated` and `OpenApiSchema::parse` before downstream plans.
- Define the orders block formula concretely in Plan 01: fixed block size, row offset, and empty-row handling.

**Risk Assessment:** **MEDIUM**. The technical direction is right, but the fixture is too summary-driven for a phase whose success depends on deterministic data consistency.

## 90.2-02-PLAN.md

**Strengths**

- Correctly uses the `ServerTester::new` 4th argument to inject an inbound bearer.
- Strong passthrough proof: wiremock requires `Authorization: Bearer contoso-user-tok`, and recorded backend requests are checked afterward.
- Good drift guard for fixture and pointable example via shared resource/prompt surface assertions.
- Correctly places pointable config under `examples/` because `tests/` is excluded from the crate package.
- Mirrors the 90.1 parity harness, which reduces implementation risk.

**Concerns**

- **MEDIUM:** The “two-sided passthrough proof” acceptance criterion requires temporarily editing the test from `Some("contoso-user-tok")` to `None`. That is manual, easy to forget, and risky in an autonomous execution flow.
- **MEDIUM:** Example config copying from the test fixture can still drift after Plan 02. The helper catches resource/prompt drift, but not necessarily tool script, workbook ids, or auth details unless the test explicitly asserts them.
- **MEDIUM:** The secret grep acceptance is internally inconsistent: it says grep for `${TENANT` / `${CLIENT_SECRET` should return nothing “beyond placeholders,” but a grep command cannot distinguish allowed placeholders from disallowed secrets.
- **LOW:** `ServerTester` also sends `X-API-Key`; harmless here, but the plan should ensure no test accidentally treats that as the delegated credential.

**Suggestions**

- Replace the manual mutation proof with an automated negative test or a second scenario run without the bearer that asserts failure.
- Add explicit pointable-example assertions for `oauth_passthrough`, both tool names, both `customer_id` params, and `range(address=`.
- Use secret checks that reject JWT-shaped values, obvious Azure GUID-like tenant/client ids only if they are not placeholders, and real `client_secret` values. Do not reject `${VAR}` placeholders.

**Risk Assessment:** **LOW-MEDIUM**. The passthrough strategy is sound; the main issues are test ergonomics and drift coverage.

## 90.2-03-PLAN.md

**Strengths**

- Correctly makes Code Mode a Rust test rather than a YAML scenario because `execute_code` requires an HMAC approval token.
- Correctly uses `request_executor_from_extra` / `JsCodeExecutor` to test token-threaded outbound Graph reads.
- Requires a pinned reference date and forbids wall-clock APIs.
- Requires both Customers and Orders mock reads to receive the forwarded bearer.
- Good requirement that the final customer-id set is exact and deterministic.

**Concerns**

- **HIGH:** The plan may not satisfy the literal P902-CODEMODE wording that the workflow loads data “via the tools.” Task 1 uses direct `api.get` calls to Graph ranges, not `get_customer` / `get_customer_orders` MCP tool calls. This may be acceptable architecture, but the plan must reconcile the mismatch or change the implementation.
- **MEDIUM:** The date-window math is under-specified. Computing “REF minus 3 months” manually can produce invalid dates like `2026-02-30` depending on the pinned `REF`. Since determinism matters more than calendar generality, the lower bound should probably be a pinned literal too.
- **MEDIUM:** Direct `JsCodeExecutor` testing bypasses the actual `validate_code` → `execute_code` handler/HMAC path. Research justifies this, but if the acceptance says “execute_code workflow,” a Rust test could thread the approval token and exercise the handlers more faithfully.
- **LOW:** The test should assert request counts, not just final result and headers, so a hard-coded result or missing backend read cannot pass accidentally.

**Suggestions**

- Decide explicitly: either amend P902-CODEMODE language to “via the same Graph range-read API exposed by the tools,” or implement a test that actually invokes the MCP tools before aggregating.
- Use `const REF = "2026-05-30"; const START = "2026-02-28";` or choose a reference date where subtracting 3 months is unambiguous.
- Add wiremock `.expect(1)` for both Customers and Orders range reads and assert the recorded request count is exactly two.
- Consider one additional test that exercises `validate_code` + `execute_code` with a threaded approval token if handler-level coverage is important.

**Risk Assessment:** **MEDIUM-HIGH**. The deterministic-query mechanics are good, but the “via tools” ambiguity is a real acceptance risk.

## Gemini Review
_Overall risk: LOW — APPROVED_

This review evaluates the implementation plans for **Phase 90.2: Contoso M365 OpenAPI Showcase**. The plans are highly surgical, adhering strictly to the "config + fixture + docs only" constraint while leveraging advanced patterns like delegated passthrough and script-tool arithmetic.

### 1. Summary
The plan set is exemplary in its technical depth and alignment with previous phase patterns (90.1 London Tube). It successfully bridges the gap between high-level business narratives ("Excel-over-AI") and low-level protocol assertions (delegated token forwarding). By opting for Rust-based Code Mode tests to bypass HMAC threading limitations in YAML and utilizing JS script tools to handle row-address arithmetic that the connector lacks, the plans exhibit a sophisticated understanding of the SDK's internal mechanics. The wave-based dependency structure is sound, ensuring that fixtures are validated before integration tests or examples are finalized.

### 2. Strengths
*   **Mechanistic Precision:** The plans correctly identify that `execute_code` requires an HMAC approval token that standard YAML scenarios cannot easily thread (**Plan 03**), correctly mandating a Rust test to verify the deterministic headline query.
*   **Proactive Risk Mitigation:** **Plan 01 Task 2** includes a documented fallback strategy for the non-standard OData parenthesized path segment (`range(address='...')`), ensuring the phase can proceed even if the `OpenApiSchema` parser is stricter than anticipated.
*   **Strict Security Posture:** Every task involving configuration (**Plan 01 Task 1, Plan 02 Task 2**) enforces the "no real secrets" rule, utilizing `${VAR}` placeholders and dev-guarded inline secrets that match established repo patterns.
*   **Verification Rigor:** **Plan 02 Task 1** implements a "two-sided passthrough proof" (asserting failure when the token is missing and success when present), which is the gold standard for verifying security-sensitive forwarding logic.
*   **Drift Prevention:** The use of an `assert_contoso_code_mode_surface` helper (**Plan 02 Task 1**) shared between the fixture test and the pointable example ensures that the shipped artifacts never drift from the tested state.

### 3. Concerns
*   **JS Engine `Date` Limitation (LOW):** While **Plan 03** correctly identifies the lack of a `Date` builtin, the complexity of implementing "trailing-3-month string comparison" in pure JS math/string logic without standard date libraries is high. There is a risk of subtle off-by-one errors in the month/year rollover logic within the `HEADLINE` script.
*   **OData Path Substitution (MEDIUM):** **Plan 01 Task 2** assumes `reqwest` will handle the unencoded parentheses/quotes in the path. While `london-tube` proved templated paths work, the OData syntax is significantly more "symbol-heavy." If `reqwest` or the server-side router chokes on the literal parens, the fallback to query params needs to be applied immediately to avoid Wave 1 stalls.
*   **Resource URI Typos (LOW):** **Plan 01 Task 1** correctly flags the risk of typo'd URIs in `include_resources`. However, since these URIs are also used in the docs (**Plan 04**), a typo in Wave 1 would rot the documentation text.

### 4. Suggestions
*   **Stabilize JS Date Logic:** In **Plan 03 Task 1**, suggest providing a concrete JS helper snippet for the month-subtraction logic (e.g., `(month - 3 < 1 ? year - 1 : year)`) to ensure the "last 3 months" calculation handles January/February rollovers correctly without a `Date` object.
*   **Refine Path Fallback:** In **Plan 01 Task 2**, if the primary OData path fails, prefer a script-based `api.get` call with an explicitly constructed URL string over changing the `[[tools]]` definition to query params. This keeps the tool interface cleaner for the user.
*   **Expand Governance Narrative:** In **Plan 04 Task 1**, ensure the "admin consent" explanation explicitly mentions that the server's scope is *capped* by the consent, but the *actual* access is further restricted by the individual user's permissions (double-lock pattern).

### 5. Risk Assessment: LOW
The risk is exceptionally low because the underlying machinery (`OAuthPassthroughAuth`, `TokenCaptureAuthProvider`) is already built and tested. The plans focus entirely on orchestration and documentation. The primary technical hurdle—OData path parsing—is mitigated by a documented fallback. The reliance on a pinned reference date eliminates "test rot" from time-sensitive queries.

**Decision:** The plans are **APPROVED** for execution.

## Consensus Summary

Both reviewers verified the core technical claims against the repo and agree the architecture is sound and the phase is config/fixture/docs/test-only with no `src/` changes. They diverge on overall risk: **Codex = MEDIUM** (execution-discipline risks around fixture drift + docs ordering), **Gemini = LOW / APPROVED** (machinery already built; fallback documented). Net: the plans are executable; two structural fixes would de-risk it.

### Agreed Strengths
- Correct mechanism choices, independently verified: script tools for the `customer_id`→Excel-range arithmetic (connector does literal string-replace only); a Rust Code Mode test because `execute_code` needs an HMAC approval token YAML can't thread; `ServerTester` 4th-arg injects the inbound bearer and `oauth_passthrough` forwards it.
- Documented OData-path fallback in Plan 01 Task 2 (prudent risk mitigation).
- Strict no-secrets posture (`${VAR}` placeholders + dev-guarded inline secret).
- Two-sided passthrough proof (assert success-with-token AND failure-without) + the `assert_contoso_code_mode_surface` drift-prevention helper (both called "gold standard"/strong).
- Pinned reference date eliminates time-based test rot.

### Agreed Concerns
- **OData path substitution (MEDIUM, both):** the parenthesized `range(address='...')` segment is symbol-heavy; if `reqwest`/`OpenApiSchema::parse` chokes on the literal parens/quotes, the documented fallback must be applied immediately to avoid a Wave-1 stall. Gemini suggests preferring a script-built `api.get` URL over switching the tool to query params.
- **Resource-URI typos rot the docs (LOW, both):** `include_resources` URIs are also referenced in Plan 04's chapters, so a Wave-1 typo propagates into documentation.

### Divergent Views (worth resolving before execute)
- **Plan 04 (docs) wave placement — Codex HIGH, Gemini LOW.** Codex: Plan 04 is Wave 1 but its content needs the final fixture rows + deterministic Code Mode result from Plans 01/03 (and literally says "if 90.2-03-SUMMARY exists," which it won't in Wave 1) → move Plan 04 to depend on [01, 02, 03] (Wave 3), or split into draft-then-reconcile. Gemini judged the wave structure "sound." **Recommendation: heed Codex** — the docs describe shipped behavior (the headline query's exact result set), so they should run after the tests that establish it. Cheapest fix: make Plan 04 `depends_on: [01]` at minimum (so resource URIs/config exist) and have it quote the deterministic result from 03's SUMMARY, or move it to a third wave.
- **Fixture durability — Codex HIGH, Gemini not flagged.** Codex: P902-FIXTURE data lives only in `90.2-01-SUMMARY.md` prose, and Plans 02/03 re-create rows in Rust mocks → drift risk across config/parity/code-mode/docs. Suggests a canonical `tests/fixtures/contoso-m365-workbook.json` that 02/03 load. Gemini instead credited the `assert_contoso_code_mode_surface` helper as sufficient drift control. **Recommendation: partially heed Codex** — a single canonical dataset (or a shared `const` the Rust tests import) removes the multi-copy drift surface; weigh against the added Wave-1 artifact.
- **Code Mode "via the tools" (Codex MEDIUM, Gemini silent).** Codex: Plan 03 may use raw `api.get` range reads rather than invoking `get_customer`/`get_customer_orders`, even though P902-CODEMODE frames the query as composed *over the tools*. Worth confirming the Code Mode script calls the two tools (or documenting that raw range reads are acceptable for the deterministic-set assertion).
- **JS date math (Gemini LOW, Codex silent).** Gemini: the "trailing 3 months" comparison in pure JS (no `Date`) risks month/year-rollover off-by-one; suggests a concrete month-subtraction helper in the Plan 03 script.

### Recommended action
Heed Codex's two HIGH items before execution — they are cheap and prevent real friction:
1. Re-wave Plan 04 so docs run after the fixture (and ideally the deterministic result) exist — at minimum `depends_on: [01]`; ideally after 02/03.
2. Make the workbook dataset canonical (shared fixture/const) so config, parity mock, Code Mode mock, and docs cannot drift.

Then optionally apply the MEDIUM/LOW polish (confirm Code Mode calls the tools; harden the OData fallback trigger; add the JS month-rollover helper; strengthen the governance grep to require the full double-lock narrative).
