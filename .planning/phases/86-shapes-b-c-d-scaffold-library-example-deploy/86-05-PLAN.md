<objective>
Build Shape D: make `cargo pmcp deploy` handle a config-driven server with ZERO `TargetEntry`
enum changes (D-09, D-10, SHAP-D-01), resolving the four review HIGHs concentrated here:

- **H3 (single-crate Lambda build):** the Plan 01 spike confirmed `find_lambda_package_dir`
  (builder.rs:312) only resolves a `<server>-lambda` dir or a workspace `*-lambda` package with a
  `bootstrap` binary — a single-crate config-driven project (where the project ROOT is the Lambda
  package) currently `bail!`s. This plan extends `find_lambda_package_dir` (builder.rs) to also
  resolve "the project root IS the Lambda package" when a config-driven single crate is detected,
  with a unit test. `builder.rs` is therefore in `files_modified` (it was missing before).
- **H1 (asset path):** the emitted `deploy.toml` lists `[assets] include = ["config.toml", "schema.sql"]`
  so `bundle_assets_if_configured` places them under `assets/` in the zip → `/var/task/assets/` on
  Lambda, exactly where the Plan 01 `pmcp::assets::load_string` resolver reads. The SQLite DB
  bootstraps to `/tmp/demo.db` (the emitted `main.rs` already uses `demo_db_path()` — `/var/task` is
  read-only). No `main.rs` change is needed; the entrypoint shipped by Plan 03 deploys unchanged.
- **H4 (secret posture):** bundling the scaffold's local `config.toml` as-is would ship the inline DEV
  `token_secret`. The deploy path MUST source the secret from a ref, not the inline default (D-06).
  Mechanism: the deploy bundling uses a config whose `token_secret` is the env-expanded
  `${CODE_MODE_SECRET}` (the toolkit config parser already supports `${...}` expansion). An
  acceptance criterion + a packaging test PROVE the bundled artifact does NOT contain the inline
  dev-secret literal.
- **M3 (detection testability + pmcp.run selection):** `get_target_id` (deploy/mod.rs:1017) reads only
  `--target-type` > `deploy.toml target_type` > `aws-lambda` default; it does NOT infer pmcp.run from
  project shape. So the emitted `deploy.toml` sets `target_type = "pmcp-run"`. The detection predicate
  `is_config_driven_project` is unit-tested via an in-module `#[cfg(test)]` test (a bin-only `pub(crate)`
  fn cannot be reached from an integration test).

Purpose: config-only server deploys to pmcp.run reusing the existing pipeline, with no breaking
changes, the right asset/DB/secret posture, and single-crate build support.
Output: detection seam + single-crate builder fix + scaffold deploy.toml + detection/regression/
packaging tests.
</objective>
