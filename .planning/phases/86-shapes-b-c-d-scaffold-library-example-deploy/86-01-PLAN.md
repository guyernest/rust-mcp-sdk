## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| schema.sql file → SqliteConnector::execute_batch | DDL/INSERT text read from disk and executed against the demo DB |
| LAMBDA_TASK_ROOT env → demo_db_path resolution | runtime environment decides the writable DB location (/tmp on Lambda) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-86-01-01 | Tampering | execute_batch SQL text | accept | execute_batch runs trusted developer-authored schema.sql (the scaffold's own bundled DDL), NOT untrusted client input — no parameterization needed for a batch bootstrap; client query params still flow through the parameterized `execute` path (T-84-04-01), unchanged here. |
| T-86-01-02 | Information Disclosure | ConnectorError from a bad batch | mitigate | Errors map to `ConnectorError::Query(e.to_string())` whose Display is the bounded rusqlite message; the helper does not echo the full schema.sql text or any file path into the error (consistent with T-84-04-02 path-free errors). |
| T-86-01-03 | Denial of Service | spawn_blocking pool exhaustion on a huge batch | accept | Single bootstrap call at startup, demo-scale schema; not a per-request path. Low risk. |
| T-86-01-04 | Tampering | demo_db_path opening a DB at an attacker-influenced path | accept | The path is decided solely by the `LAMBDA_TASK_ROOT` runtime env (set by the Lambda runtime) — `/tmp/demo.db` or a fixed local `demo.db`; no user/client input flows into it. |
</threat_model>
