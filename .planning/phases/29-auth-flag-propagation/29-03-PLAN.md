    3. Resolve auth and build the Authorization header value before constructing PreviewConfig:
       ```rust
       let auth_method = auth_flags.resolve();
       let auth_header = match &auth_method {
           AuthMethod::None => None,
           AuthMethod::ApiKey(key) => Some(format!("Bearer {}", key)),
           AuthMethod::OAuth { .. } => {
               // For OAuth, acquire token once at startup
               use pmcp::client::oauth::{default_cache_path, OAuthConfig, OAuthHelper};
               let AuthMethod::OAuth { client_id, issuer, scopes, no_cache, redirect_port } = &auth_method else { unreachable!() };
               let cache_file = if *no_cache { None } else { Some(default_cache_path()) };
               let config = OAuthConfig {
                   issuer: issuer.clone(),
                   mcp_server_url: Some(url.clone()),
                   client_id: client_id.clone(),
                   scopes: scopes.clone(),
                   cache_file,
                   redirect_port: *redirect_port,
               };
               let helper = OAuthHelper::new(config)
                   .map_err(|e| anyhow::anyhow!("OAuth setup failed: {e}"))?;
               let token = helper.get_access_token().await
                   .map_err(|e| anyhow::anyhow!("OAuth token acquisition failed: {e}"))?;
               Some(format!("Bearer {}", token))
           },
       };
       ```
       Note: Preview is a long-running server. We acquire the token once at startup. Token refresh during long sessions is a separate concern (documented, not implemented now).

       If OAuthHelper does not have a `get_access_token()` method, fall back to using `resolve_auth_middleware()` and then extracting -- but the simpler approach is to construct the header string directly. Check if OAuthHelper exposes a method to get the raw token. If not, use `resolve_auth_middleware()` and construct a reqwest client with default headers instead of passing a string. The simplest reliable approach: for API key, construct "Bearer {key}" directly. For OAuth, use `resolve_auth_middleware()` and note that McpProxy cannot use the middleware chain directly (it uses raw reqwest). Instead, for OAuth, log a warning that OAuth with preview requires direct token -- or better, use the OAuthHelper to acquire the token and construct the header.

       If `OAuthHelper::get_access_token()` is not available: resolve with `resolve_auth_middleware` which returns an HttpMiddlewareChain. Since McpProxy uses raw reqwest (not the middleware chain), for OAuth with preview, print a warning: "OAuth with preview is not yet supported. Use --api-key for authenticated preview." and set auth_header to None. API key works directly.
