**Part A: Create `src/server/typed_prompt.rs` (mirrors typed_tool.rs, per D-05/D-06/D-07/D-08)**

Create a new file `src/server/typed_prompt.rs` containing `TypedPrompt<T, F>`:

1. Define struct `TypedPrompt<T, F>` where:
   - `T: DeserializeOwned + JsonSchema + Send + Sync + 'static`
   - `F: Fn(T, RequestHandlerExtra) -> Pin<Box<dyn Future<Output = Result<GetPromptResult>> + Send>> + Send + Sync`
   - Fields: `name: String`, `description: Option<String>`, `arguments: Vec<PromptArgument>`, `handler: F`, `_phantom: PhantomData<T>`
   - Add a doc comment on the struct that includes this **string-only limitation note**:
     ```rust
     /// # String-Only Arguments
     ///
     /// MCP prompt arguments are transmitted as `HashMap<String, String>`. TypedPrompt
     /// converts each value to `serde_json::Value::String` before deserializing into `T`.
     /// This means struct fields **must** be `String` or `Option<String>` types. Non-string
     /// field types (e.g., `i32`, `bool`, `f64`) will fail deserialization because
     /// `Value::String("42")` does not coerce to `i32` via `serde_json::from_value`.
     ///
     /// If you need numeric or boolean arguments, accept them as `String` and parse
     /// manually in your handler, or use `#[serde(deserialize_with = "...")]` custom
     /// deserializers.
     ```

2. Implement `TypedPrompt::new(name, handler)` that:
   - Calls `schemars::schema_for!(T)` to get the JSON schema
   - Extracts `PromptArgument` entries from schema properties: iterate `schema["properties"]` object entries, for each `(name, prop)` create `PromptArgument::new(name)`, add `.with_description(desc)` if `prop["description"]` exists, call `.required()` if name is in `schema["required"]` array
   - Stores the extracted arguments in `self.arguments`

3. Implement `with_description(self, desc) -> Self` builder method.

4. Implement `PromptHandler for TypedPrompt<T, F>`:
   - `handle()`: Convert `HashMap<String, String>` to `serde_json::Value::Object` (per D-06: each value becomes `Value::String`), then call `serde_json::from_value::<T>(value)` with error mapped to `Error::invalid_params(format!("Invalid arguments for prompt '{}': {}", self.name, e))`, then call `(self.handler)(typed_args, extra).await`
   - `metadata()`: Build `PromptInfo::new(&self.name)`, add `.with_description()` if present, add `.with_arguments()` if non-empty, return `Some(info)`

5. Implement `fmt::Debug` for `TypedPrompt<T, F>` (same pattern as TypedTool).

6. Add `pub mod typed_prompt;` to `src/server/mod.rs` (near the existing `pub mod typed_tool;` line). Add `pub use typed_prompt::TypedPrompt;` in the re-exports section of mod.rs.

**Part B: Create `pmcp-macros/src/mcp_prompt.rs` (mirrors mcp_tool.rs, per D-09 through D-13)**

Create a new file `pmcp-macros/src/mcp_prompt.rs` containing:

1. Define `McpPromptArgs` darling struct (per D-02/D-04):
   ```rust
   #[derive(Debug, FromMeta)]
   pub struct McpPromptArgs {
       pub(crate) description: String,
       #[darling(default)]
       pub(crate) name: Option<String>,
   }
   ```

   Note: NO annotations, NO ui fields -- prompts are simpler than tools.

2. Implement `pub fn expand_mcp_prompt(args: TokenStream, input: ItemFn) -> syn::Result<TokenStream>` following the exact pattern from `mcp_tool.rs::expand_mcp_tool()` with these CRITICAL differences:

   a. **Struct name suffix**: Use `"{}Prompt"` (not `"{}Tool"`). Example: `fn code_review` -> `CodeReviewPrompt` (per D-13).

   b. **Parameter classification**: Use `mcp_common::classify_param()` for all params. Support `Args(Type)`, `State`, `Extra`, reject `SelfRef` with error "standalone #[mcp_prompt] functions cannot have &self -- use #[mcp_server] for impl block prompts".

   c. **Args deserialization**: Instead of `serde_json::from_value(args)`, generate the HashMap-to-Value-to-struct conversion (per D-06):
   ```rust
   let value = serde_json::Value::Object(
       args.into_iter()
           .map(|(k, v)| (k, serde_json::Value::String(v)))
           .collect()
   );
   let typed_args: #at = serde_json::from_value(value)
       .map_err(|e| pmcp::Error::invalid_params(
           format!("Invalid arguments for prompt '{}': {}", #prompt_name, e)
       ))?;
   ```

   d. **Handle signature**: The PromptHandler::handle takes `args: std::collections::HashMap<String, String>` (not `serde_json::Value`).

   e. **No result serialization**: Prompts return `GetPromptResult` directly (no `serde_json::to_value(result)` wrapper). The handle body should end with the function call result directly.

   f. **Metadata generation**: Generate `PromptInfo` (not `ToolInfo`). NO output schema. NO annotations. Use `schemars::schema_for!` on the args type to extract prompt arguments at runtime:
   ```rust
   fn metadata(&self) -> Option<pmcp::types::PromptInfo> {
       let mut info = pmcp::types::PromptInfo::new(#prompt_name)
           .with_description(#description);
       // Only generate argument extraction if args type exists
       let schema = schemars::schema_for!(#args_type);
       let json_schema = serde_json::to_value(&schema).unwrap_or_default();
       let arguments = {
           let properties = json_schema.get("properties").and_then(|p| p.as_object());
           let required_fields: Vec<String> = json_schema.get("required")
               .and_then(|r| r.as_array())
               .map(|arr| arr.iter().filter_map(|v| v.as_str().map(String::from)).collect())
               .unwrap_or_default();
           let Some(props) = properties else { return Some(info); };
           props.iter().map(|(name, prop)| {
               let mut arg = pmcp::types::PromptArgument::new(name);
               if let Some(desc) = prop.get("description").and_then(|d| d.as_str()) {
                   arg = arg.with_description(desc);
               }
               if required_fields.contains(name) {
                   arg = arg.required();
               }
               arg
           }).collect::<Vec<_>>()
       };
       if !arguments.is_empty() {
           info = info.with_arguments(arguments);
       }
       Some(info)
   }
   ```
   For no-args prompts, skip the schema extraction entirely and return `Some(PromptInfo::new(name).with_description(desc))`.

   Also add the same string-only limitation note as a doc comment on the generated struct:
   ```rust
   /// **Note:** MCP prompt arguments are string-only. Struct fields should use
   /// `String` or `Option<String>`. Non-string types (`i32`, `bool`, etc.) will
   /// fail deserialization. Use `#[serde(deserialize_with)]` for custom parsing.
   ```

   g. **Return type**: Do NOT attempt to extract output schema. The return type is always `Result<GetPromptResult>`. Do NOT call `mcp_common::output_schema_tokens`. Do NOT call `mcp_common::is_value_type`.

   h. **State injection**: Same pattern as mcp_tool -- `with_state()` method, `state: Option<Arc<T>>` field, resolution in handle body (per D-11).

   i. **Sync/async**: Support both sync `fn` and `async fn` (per discretion -- mirror tool pattern). Sync prompts are wrapped in the async PromptHandler::handle.

   j. **Constructor**: Generate `pub fn prompt_name() -> PromptNamePrompt { PromptNamePrompt {} }` (per D-13). Rename original function to `__prompt_name_impl` (per Phase 58 pattern).
