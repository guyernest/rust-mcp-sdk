- Wave 1 lays the test seam + config types so 79-02 and 79-03 implement against contracts (interface-first ordering). REVISION 3: `OnFailure::Rollback` variant DELETED + custom Deserialize hard-rejects "rollback" (HIGH-G2). `InfraErrorKind::AuthMissing` DELETED (HIGH-C2). `WidgetConfig.build/.install` are `Option<Vec<String>>` argv arrays (Codex MEDIUM).
- Wave 2 ships the build half end-to-end (failure modes A + B closed at deploy time). REVISION 3: `PMCP_WIDGET_DIRS` colon-list replaces single-dir env (HIGH-C1). Yarn-PnP detection skips install heuristic (Codex MEDIUM).
- Wave 3 ships the verify half end-to-end (failure mode C closed). REVISION 3: subprocess argv adds `--format=json`; consumes `serde_json::from_str::<PostDeployReport>(stdout)`; ALL regex parsers DELETED (`parse_conformance_summary`, `parse_apps_summary`, `build_failure_recipes`, `outcome_summary`, `format_failure_banner`'s noun/metric/recipe fallback); `resolve_auth_token` DELETED (subprocess inherits env per HIGH-C2); exit code becomes 3 (broken-but-live, HIGH-2 distinct from infra exit 2 and pre-cutover exit 1); `emit_ci_annotation` fires when `CI=true` (HIGH-2).
- Wave 4 closes the loop: doctor distinguishes WidgetDir vs include_str! crates (Codex MEDIUM); `cargo pmcp app new --embed-widgets` opts into `include_str!` + writes build.rs with PMCP_WIDGET_DIRS list + local-discovery fallback (HIGH-C1 + HIGH-G1); fuzz target now also exercises OnFailure rollback-rejection path (HIGH-G2); example demonstrates PMCP_WIDGET_DIRS list contract; help-text + rustdoc + README updated; version bumps include mcp-tester 0.6.0 (Plan 79-05's new public module).

**Why not 4 waves (rolling Wave 0 into Wave 3):** Wave 0's contract change touches BOTH `mcp-tester` (new public module) AND `cargo-pmcp` (3 subcommand changes) — separate enough from Wave 3's deploy-orchestrator work that folding would push Wave 3 over the ~50% context target. Also: Wave 0 lands BEFORE Wave 1 because Wave 1's `TestOutcome::Passed/TestFailed` types reference the upstream `mcp_tester::TestSummary` (no schema breakage from late binding).

<locked_planner_decisions>

1. **Subprocess argv shape (REVISION 3 HIGH-1 supersession):**
   - Pre-revision-3: argv was `<exe> test {check,conformance,apps} <URL> [--api-key <TOKEN>] [--quiet]`.
   - REVISION 3: argv is `<exe> test {check,conformance,apps} <URL> --format=json [--mode <mode> for apps]`. NO `--api-key` (REVISION 3 HIGH-C2: subprocess inherits parent env). NO `--quiet`. `--format=json` ships in Plan 79-05 (Wave 0).
   - Stdout: ONE valid `mcp_tester::post_deploy_report::PostDeployReport` JSON document. Parsed via `serde_json::from_str`.
   - Binary path: `std::env::current_exe()?` (unchanged).

2. **Cache invalidation env var contract (REVISION 3 HIGH-C1 supersession):**
   - Pre-revision-3: single `PMCP_WIDGET_DIR` (last-widget-wins broken for multi-widget projects).
   - REVISION 3: `PMCP_WIDGET_DIRS` colon-separated list (Unix `PATH` convention). 79-02's orchestrator joins ALL `[[widgets]]` entries' `absolute_output_dir` paths into a single env-var value before `target.build(&config).await?`. 79-04's generated build.rs splits on `:` and emits per-dir `cargo:rerun-if-changed`.
   - Empty-list case: do NOT set the env var so 79-04's local-discovery fallback (HIGH-G1) takes over for direct `cargo run` scenarios.

3. **Hybrid in-process vs uniform subprocess:** (unchanged) Lock: uniform subprocess for all three.

4. **`--no-post-deploy-test` skips warmup grace:** (unchanged) Lock: yes.

5. **Runnable example scope:** (unchanged) Lock: `cargo-pmcp/examples/widget_prebuild_demo.rs` schema-direct demo. REVISION 3: example uses `PMCP_WIDGET_DIRS` (list form) per HIGH-C1.

6. **`on_failure="rollback"` rejection UX (REVISION 3 HIGH-G2 supersession):**
   - Pre-revision-3: WARN at deploy-START + treat as Fail at runtime.
   - REVISION 3: HARD-REJECT at config-validation time (custom `Deserialize` impl on `OnFailure` per 79-01) AND at clap parse time (custom `value_parser` on `--on-test-failure` per 79-03+79-04). NO runtime fallback. NO deploy-START WARN. Deploy refuses to start until user changes the value.
   - Verbatim error: `on_failure='rollback' is not yet implemented in this version of cargo-pmcp. Change to 'fail' (default) or 'warn'. Auto-rollback support will land in a future phase that verifies the existing DeployTarget::rollback() trait implementations.` (constant `ROLLBACK_REJECT_MESSAGE` in 79-01).

7. **Crate version bumps (REVISION 3 added mcp-tester bump per Plan 79-05 dependency):**
   - `cargo-pmcp 0.11.0 → 0.12.0` — MINOR bump (additive).
   - `mcp-tester 0.5.3 → 0.6.0` — MINOR bump (REVISION 3: NEW public module `post_deploy_report` per Plan 79-05; additive).
   - `pmcp 2.6.0` — UNCHANGED.
   - `mcp-preview 0.3.0` — UNCHANGED.

   - `cargo-pmcp/Cargo.toml` `mcp-tester` dep version bumped to `0.6.0`.
   - **Action:** version bump is a Wave 4 step (79-04) co-located with CHANGELOG entries. Do NOT bump in earlier waves.

8. **(NEW REVISION 3) Subprocess auth resolution (HIGH-C2 supersession):**
   - Pre-revision-3: parent-side `resolve_auth_token` helper read MCP_API_KEY env → Phase 74 cache → emitted WARN; injected via `Command::env("MCP_API_KEY", token)` to subprocess.
   - REVISION 3: helper REMOVED entirely. Subprocess inherits parent's full env via Tokio Command default (no `env_clear()`). Child resolves auth via existing `AuthMethod::None` path at `cargo-pmcp/src/commands/auth.rs:106-109` which already calls `try_cache_token` → Phase 74 cache lookup → automatic refresh near-expiry. Verified by reading auth.rs lines 28-47 + 102-109.
