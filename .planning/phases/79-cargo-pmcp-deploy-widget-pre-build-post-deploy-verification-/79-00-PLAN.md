<locked_planner_decisions>

Areas the researcher flagged as planner decisions, locked here for executors:

1. **Subprocess argv shape (verified 2026-05-03 via `cargo-pmcp test {check, conformance, apps} --help`):**
   - `cargo-pmcp test check <URL> [--timeout <SECS>] [--api-key <TOKEN>] [--quiet]`
   - `cargo-pmcp test conformance <URL> [--timeout <SECS>] [--api-key <TOKEN>] [--quiet]`
   - `cargo-pmcp test apps <URL> --mode claude-desktop [--timeout <SECS>] [--api-key <TOKEN>] [--quiet]`
   - Auth flag is `--api-key` (NOT `--auth-token`); env-var alias `MCP_API_KEY` is also supported by clap's `env=` attribute. Pass via flag for direct subprocess invocation; for security (avoid `ps` listing leak — see threat model T-79-04 in 79-03), prefer setting `MCP_API_KEY` in the spawned `Command::env()` and omit the flag.
   - Binary path: `std::env::current_exe()?` (NOT `cargo` or `cargo-pmcp`) — guarantees version match. The cargo-pmcp main.rs dispatches identically whether invoked as `cargo pmcp ...` (via cargo subcommand mechanism) or `cargo-pmcp ...` (direct binary).

2. **`PMCP_WIDGET_DIR` propagation through CDK Node.js spawn (researcher A4):**
   - Empirical precedent: Phase 76 IAM passes `PMCP_ORGANIZATION_ID`/`PMCP_SERVER_ID`/`MCP_SERVERS_TABLE` from Rust → CDK → Lambda env via `process.env.PMCP_*` reads at `commands/deploy/init.rs:571,592-594`. Same mechanism applies to `PMCP_WIDGET_DIR` for the cargo-build child of CDK.
   - For the `cargo lambda build` direct path (`deployment/builder.rs:134` `Command::new("cargo").args(["lambda", "build", ...]).envs(...)`), env-var inheritance is the default (`std::process::Command` inherits parent env unless `.env_clear()` is called — verified at line 134-148 there is no `env_clear`).
   - **Lock:** set `std::env::set_var("PMCP_WIDGET_DIR", absolute_output_dir)` ONCE in the orchestrator (`commands/deploy/mod.rs::execute_async`) before line 720 `target.build(&config).await?`. Do NOT set per-target. 79-02 includes a Wave 2 task that adds an integration test confirming the env var reaches a child `cargo build` invocation.

3. **Hybrid in-process vs uniform subprocess (researcher Open Question 3):**
   - **Lock: uniform subprocess for all three (`check`, `conformance`, `apps`).** The ~100ms saved by in-process `check` does not justify maintaining two abstractions. All three go through `tokio::process::Command::new(current_exe())` with a single `spawn_test_subprocess` helper.

4. **`--no-post-deploy-test` skips warmup grace (researcher Open Question 4):**
   - **Lock: yes.** When `--no-post-deploy-test` is set OR `[post_deploy_tests].enabled = false`, the orchestrator skips both the `tokio::time::sleep(warmup_grace_ms)` AND the test invocations. The 2s warmup is part of the verify lifecycle and serves no purpose otherwise; this also keeps the no-test-deploy path 2s faster.

5. **Runnable example scope (researcher Open Question 5):**
   - **Lock:** `cargo-pmcp/examples/widget_prebuild_demo.rs` (NOT `deploy_with_widgets.rs` — the latter implies a full deploy demo, which would need a live Lambda).
   - Scope: schema-direct demo. Sets up a tempdir with a synthetic `widget/` dir + a no-op `package.json` + a `package-lock.json` (npm fallback). Calls the public `cargo_pmcp::deployment::widgets::run_widget_build` function against it. Asserts that `PMCP_WIDGET_DIR` is exposed and that `target/` of a fake Rust crate gets a `cargo:rerun-if-changed` invalidation signal (verified by reading `cargo build` rerun-if output via stderr capture). Pattern matches Phase 77's `multi_target_monorepo.rs` (schema-layer simulation, no real cloud deps). Lives in 79-04.

6. **`on_failure="rollback"` rejection UX timing (researcher recommendation):**
   - **Lock:** WARN at deploy-START (BEFORE any deploy work begins), not at test-failure-time. Rationale: operator should see the gotcha early so they can `Ctrl-C` and reconfigure before the deploy commits, rather than after the deploy is live and the test has failed.
   - Exact message: `WARNING: on_failure='rollback' is reserved for a future phase — see Phase 79 deferred items. Treating as 'fail' for this deploy.`
   - Lifecycle moment: `commands/deploy/mod.rs::execute_async`, after `DeployConfig::load(&project_root)?` (line 701) but before `emit_target_banner_if_resolved` (line 719). Implementation in 79-03.

7. **Crate version bumps (per CLAUDE.md release rules):**
   - `cargo-pmcp 0.11.0 → 0.12.0` — MINOR bump (additive: new behavior, new flags, new opt-out config sections).
   - `pmcp 2.6.0` — UNCHANGED. No core SDK changes.
   - `mcp-tester 0.5.3` — UNCHANGED. Subprocess-spawn decision avoids any new public API requirement.
   - `mcp-preview 0.3.0` — UNCHANGED.
   - `pmcp-widget-utils` — UNCHANGED.
   - **Action:** version bump is a Wave 4 step (79-04) co-located with CHANGELOG entry. Do NOT bump in earlier waves.
