    Implement `cargo-pmcp/src/pentest/attacks/session_security.rs` with the full session attack suite per D-04.

    **Main entry:**
    ```rust
    pub async fn run(
        tester: &mut ServerTester,
        surface: &AttackSurface,
        limiter: &PentestRateLimiter,
        config: &PentestConfig,
    ) -> Vec<SecurityFinding>
    ```
    Early return empty vec if `!surface.has_session_support` (no sessions = skip suite, add Info finding noting sessions not supported).

    **Important design decision per research pitfall 3:** Session security tests MUST create fresh ServerTester instances for each test to avoid session state leakage. The `tester` param from the engine is used only to extract connection parameters (URL, timeout, transport). Each SS test creates its own tester(s).

    **Helper: `create_fresh_tester(url: &str, config: &PentestConfig) -> Result<ServerTester>`**
    Creates a new ServerTester with the same URL, config.timeout, no auth (session tests probe the session layer, not auth), HTTP transport forced.

    **Helper: `shannon_entropy(strings: &[String]) -> f64`**
    Compute Shannon entropy over the concatenated character frequencies of all session ID strings. Formula: H = -sum(p_i * log2(p_i)) where p_i is frequency of each unique character divided by total characters. Good entropy for UUIDs is typically > 3.5 bits.

    **Helper: `extract_session_id_from_initialize(url: &str, config: &PentestConfig) -> Result<Option<String>>`**
    Send a raw HTTP POST to the URL with a JSON-RPC initialize request using reqwest directly. Parse the response Mcp-Session-Id header. This is needed because ServerTester may not expose the raw response header. Use reqwest (already in cargo-pmcp deps) to:
    1. POST to url with Content-Type: application/json
    2. Body: `{"jsonrpc":"2.0","method":"initialize","params":{"protocolVersion":"2025-11-25","capabilities":{},"clientInfo":{"name":"pmcp-pentest","version":"0.1.0"}},"id":1}`
    3. Read response header `Mcp-Session-Id`
    4. Return it if present, None if absent

    **Attack implementations:**

    - **SS-01: Session ID Entropy** (severity: High, destructive: false)
      Collect 10 session IDs by calling extract_session_id_from_initialize 10 times (rate-limited: `limiter.wait().await` between each). Compute Shannon entropy. Check all are unique (put into HashSet, compare lengths). If entropy < 3.0 or not all unique, report High finding. If entropy >= 3.0 and all unique, report Info finding (entropy acceptable). Evidence: "Shannon entropy: {:.2}, unique: {}/{}, sample: {first_3_ids}".
      Remediation: "Use cryptographically random session IDs (e.g., UUID v4 or 32+ bytes of crypto random hex)."

    - **SS-02: Session Replay** (severity: High, destructive: false)
      1. Create fresh tester, initialize, capture the session ID from a tool call or initialize.
      2. Use raw reqwest to send a tools/list request with the captured Mcp-Session-Id header.
      3. Verify it works (expected: 200 OK with tools result).
      4. Send a notification or let the session expire (wait 2 seconds).
      5. Send the same tools/list request again with the same session ID.
      6. If the server still accepts it with valid response, report High finding: "Server accepts replayed session ID after disconnect."
      7. If server rejects (4xx or error), report Info finding: "Server correctly rejects replayed sessions."
      Rate-limit each request. Evidence: the HTTP status code received on replay attempt.
      Remediation: "Invalidate session IDs on disconnect. Implement session expiration."

    - **SS-03: Session Fixation** (severity: High, destructive: false)
      1. Use raw reqwest to send an initialize request with a pre-set Mcp-Session-Id header: "attacker-controlled-session-id-12345".
      2. Check if the server accepts the client-provided session ID (returns it back in the response header) or generates its own.
      3. If server echoes back the attacker-provided session ID, report High finding: "Server accepts client-provided session IDs (session fixation vulnerable)."
      4. If server generates a different ID or rejects, report Info finding: "Server generates its own session IDs."
      Rate-limit. Evidence: the session ID in the response vs what was sent.
      Remediation: "Always generate session IDs server-side. Reject client-provided Mcp-Session-Id on initialize."

    - **SS-04: Concurrent Sessions** (severity: Medium, destructive: false)
      1. Open 5 sessions simultaneously using tokio::join! on 5 extract_session_id_from_initialize calls (burst -- do NOT rate-limit these, per research pitfall 2 about concurrent session tests needing burst traffic).
      2. Check all 5 returned valid, distinct session IDs.
      3. If all succeed with unique IDs, report Info finding: "Server supports concurrent sessions."
      4. If some fail or return duplicate IDs, report Medium finding.
      5. This test is informational -- concurrent session support is not inherently a vulnerability, but duplicate session IDs under concurrency indicates a PRNG issue.
      Evidence: number of successful sessions, number of unique IDs.
      Remediation: "Ensure session ID generation is thread-safe and uses a cryptographic PRNG."

    - **SS-05: Session Timeout** (severity: Medium, destructive: false)
      1. Create a session via initialize.
      2. Wait 5 seconds (short timeout -- we can't wait for real timeout in a pen test).
      3. Send a tools/list request with the session ID.
      4. If server still accepts it, report Info finding: "Session still valid after 5s (timeout not testable in short scan)."
      5. Note in the finding description that real timeout testing requires longer scans.
      This is intentionally a weak test -- just verifies the session mechanism works over time. Evidence: response status after wait period.
      Remediation: "Implement session timeouts (recommended: 30 minutes of inactivity)."

    - **SS-06: Cross-Session Data Leak** (severity: Critical, destructive: false)
      1. Create session A. Initialize. Call a tool (pick first tool from surface.tools) with a distinctive argument containing "PENTEST_SESSION_A_DATA_MARKER".
      2. Create session B. Initialize. Call the SAME tool with different arguments.
      3. Check if session B's response contains "PENTEST_SESSION_A_DATA_MARKER".
      4. If marker appears in session B, report Critical finding: "Cross-session data leakage detected."
      5. If marker not found, report Info finding: "No cross-session data leakage detected."
      Rate-limit each request. Evidence: whether session A's data appeared in session B's response.
      Remediation: "Ensure all tool state is scoped to the session. Never share mutable state between sessions."
