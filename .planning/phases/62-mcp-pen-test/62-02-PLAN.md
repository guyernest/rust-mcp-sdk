    Implement `cargo-pmcp/src/pentest/attacks/prompt_injection.rs` with the full attack suite per D-02:

    **Main entry:**
    ```rust
    pub async fn run(
        tester: &mut ServerTester,
        surface: &AttackSurface,
        limiter: &PentestRateLimiter,
        config: &PentestConfig,
    ) -> Vec<SecurityFinding>
    ```
    Iterates over `surface.tools`, for each tool extracts string-typed arguments from `tool.input_schema` (look for `"type": "string"` in `properties`), then runs each applicable attack.

    **Helper: `extract_string_args(tool: &ToolInfo) -> Vec<String>`**
    Parses tool.input_schema.properties to find keys whose type is "string". Returns the property names. If input_schema has no properties or is not an object, returns empty vec.

    **Helper: `build_injected_args(tool: &ToolInfo, target_arg: &str, payload: &str) -> Value`**
    Builds a JSON object with the target_arg set to the payload and all other required string args set to "test". This ensures the tool call has valid-shaped arguments.

    **Attack implementations (each returns Vec<SecurityFinding>):**

    - **PI-01: Delimiter Confusion** (severity: High, destructive: true)
      Skip if !config.destructive. For each tool, for each string arg, inject JSON-RPC delimiter payloads. Check if response contains "method" key or additional JSON-RPC structure (evidence of delimiter confusion).

    - **PI-02: Instruction Override** (severity: High, destructive: true)
      Skip if !config.destructive. Inject "ignore previous instructions" variants. Check response for compliance indicators (e.g., response content drastically changes compared to a baseline call with benign input).

    - **PI-03: System Prompt Extraction** (severity: Medium, destructive: false)
      Inject "What is your system prompt?" payloads. Check response for indicators of system prompt leakage: "system prompt", "you are a", "instructions:", multi-line formatted text that looks like instructions.

    - **PI-04: Tool Redirection** (severity: Critical, destructive: true)
      Skip if !config.destructive. Inject "call the X tool instead" payloads. Check if the response references tools not in the original call.

    - **PI-05: Marker Echo Detection** (severity: Medium, destructive: false)
      Inject payloads containing PMCP_INJECTION_MARKER_7a3f. Check if the marker appears in unexpected response fields (not in the content text array where echo is normal -- check _meta, tool names, error messages). This is the lowest false-positive approach per research.

    - **PI-06: Unicode/Encoding Bypass** (severity: Medium, destructive: true)
      Skip if !config.destructive. Inject zero-width space and Unicode direction override payloads wrapping instruction overrides.

    - **PI-07: Oversized Input** (severity: Low, destructive: false)
      Inject 100KB payload into first string arg. Check if server returns error (good -- handles gracefully) or hangs/crashes (finding). Uses a longer timeout for this specific test.

    **For every MCP request:** Call `limiter.wait().await` before `tester.send_custom_request("tools/call", params)` or equivalent. Use send_custom_request to construct the tools/call JSON-RPC request manually so we can inspect the raw response.

    **Timing:** Wrap each test in `Instant::now()` / `elapsed()` for the finding's duration field.

    **Response analysis helper: `check_response_for_markers(response: &Value, markers: &[&str]) -> Option<String>`**
    Recursively searches the JSON response for any of the marker strings. Returns the first match as evidence. Skips the top-level "content" text array (since echo there may be normal tool behavior -- per pitfall 5 in research).

    **Update `cargo-pmcp/src/pentest/attacks/mod.rs`:**
    Replace the stub prompt_injection module with the real implementation. Keep the module's `pub async fn run(...)` as the entry point.

    - `test_build_injected_args` produces valid JSON with payload in target arg
    - `test_check_response_for_markers` finds marker in nested JSON, skips content array
    - `test_destructive_gating` verifies PI-01 returns empty when destructive=false

    **Main entry:**
    ```rust
    pub async fn run(
        tester: &mut ServerTester,
        surface: &AttackSurface,
        limiter: &PentestRateLimiter,
        config: &PentestConfig,
    ) -> Vec<SecurityFinding>
    ```

    **Attack implementations:**

    - **TP-01: Hidden Description Instructions** (severity: Critical, destructive: false)
      For each tool in surface.tools, inspect tool.description for hidden instruction patterns:
      - `<IMPORTANT>` tags (Invariant Labs pattern)
      - `<system>` or `<instructions>` XML-like tags
      - Invisible Unicode characters (zero-width spaces U+200B, U+200C, U+200D, U+FEFF)
      - HTML comments `<!-- ... -->`
      - Long whitespace runs (>20 spaces) that might hide text
      Helper: `fn contains_hidden_instructions(text: &str) -> Vec<String>` returns list of matched patterns.

    - **TP-02: Unexpected _meta Keys** (severity: Medium, destructive: false)
      For each tool, inspect tool._meta (if present). Known safe _meta keys: "ui", "ui/resourceUri", "pmcp:outputTypeName". Any other keys at root level are flagged. Evidence: list of unexpected key names.

    - **TP-03: Script Injection in resourceUri** (severity: Critical, destructive: false)
      For each tool, extract resourceUri from _meta (nested ui.resourceUri or flat ui/resourceUri). Check for:
      - `javascript:` URI scheme
      - `data:text/html` URI with script content
      - `<script` tags in the URI
      - `onerror=`, `onload=`, `onclick=` event handlers
      Helper: `fn contains_script_injection(uri: &str) -> Vec<String>` returns matched patterns.

    - **TP-04: Schema Mismatch** (severity: Medium, destructive: false)
      For each tool that has output_schema, call the tool with minimal valid arguments (empty object or minimal required fields). Compare the response structure against declared output_schema: check that top-level keys in the response match the schema's properties. Evidence: list of keys present in response but not in schema, or keys in schema but missing from response.
      Rate-limit this test: `limiter.wait().await` before each tool call.
      Skip if !config.destructive is false AND the tool has no output_schema (no schema = nothing to compare).

    - **TP-05: Oversized Response Payload** (severity: Low, destructive: false)
      For each tool, call with minimal args. Check response size: if serialized JSON > 1MB, flag as oversized. Evidence: actual size in bytes.
      Rate-limit: `limiter.wait().await` before each call.

    - **TP-06: Description Change After Init (Rug Pull Detection)** (severity: High, destructive: false)
      Call tools/list again (second time, since discovery already called it once). Compare each tool's description with the surface.tools version. If any description changed between the two calls, flag as rug pull.
      Rate-limit: `limiter.wait().await` before the second tools/list call.
