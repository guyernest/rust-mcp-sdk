<bug_a>
**Bug (a): `strip_js_comments` is not string-literal aware.**

Current implementation (lines 209-233 of `app_validator.rs`):

```rust
fn strip_js_comments(src: &str) -> String {
    let no_html = html_comment_re().replace_all(src, "");
    let no_block = js_block_comment_re().replace_all(&no_html, "");
    let no_line = js_line_comment_re().replace_all(&no_block, "");
    no_line.into_owned()
}
```

Where `js_block_comment_re` is `(?s)/\*.*?\*/` and `js_line_comment_re` is `//[^\r\n]*`.

Both regexes match comment delimiters anywhere — including inside JS string literals. The block-comment failure mode (cost-over-time / savings-summary) is the load-bearing one; the line-comment failure mode is the smaller one but has the same root cause.

**Required fix:** rewrite `strip_js_comments` to track string-literal context character-by-character. Inside a string literal (`'...'`, `"..."`, or `` `...` ``), `/*` and `//` are NOT comment delimiters and must be preserved. Outside strings, they ARE comment delimiters and consume to the matching close (`*/` or end-of-line, respectively).

State machine:

```
Initial state: out (outside any string)
  '/'-then-'*' in `out`: enter `block_comment` state (consume both, output nothing)
  '/'-then-'/' in `out`: enter `line_comment` state (consume both, output nothing)
  ''' in `out`: enter `single_string` state (output ''')
  '"' in `out`: enter `double_string` state (output '"')
  '`' in `out`: enter `template_string` state (output '`')
  any other char in `out`: output as-is

In `block_comment`:
  '*'-then-'/': exit to `out` (consume both, output nothing)
  any other char: consume, output nothing

In `line_comment`:
  '\n' or '\r': exit to `out` (output the newline so subsequent lines aren't merged)
  any other char: consume, output nothing

In `single_string`:
  '\\'-followed-by-any: consume both, output both (escape sequence)
  ''' (unescaped): exit to `out` (output ''')
  any other char: output as-is

In `double_string`:
  '\\'-followed-by-any: consume both, output both
  '"' (unescaped): exit to `out` (output '"')
  any other char: output as-is

In `template_string`:
  '\\'-followed-by-any: consume both, output both
  '`' (unescaped): exit to `out` (output '`')
  any other char: output as-is
  (Accepted simplification: do NOT recurse into ${...} interpolations. Real-world prod
  bundles rarely embed signal literals inside template interpolations, and full
  recursive parsing would require tracking brace depth — adds complexity for marginal
  gain. Property test exercises arbitrary input.)
```

Total LOC budget: ~80 lines for the state machine. PMAT cog estimated ~15-20 (single function, mostly flat match arms).

<bug_b>
**Bug (b): G2 constructor regex requires literal `name`/`version` values.**

Current regex (line 91-95 of `app_validator.rs`):

```regex
new [a-zA-Z_$][a-zA-Z0-9_$]{0,5}\(\s*\{\s*name\s*:\s*"[^"]+"\s*,\s*version\s*:\s*"[^"]+"\s*\}
```

Real prod (every cost-coach widget):

```js
function Rw(t,e){var i=new yl({name:"cost-coach-"+t,version:"1.0.0"})...
```

The `name:` value is `"cost-coach-"+t` — a string concatenation with the function parameter `t`. The cycle-1 regex requires `name\s*:\s*"[^"]+"\s*,\s*` — i.e. a quoted string literal followed by a comma. After the closing `"` of `"cost-coach-"`, the next char is `+` (not `,`), so the regex fails.

**Required fix:** widen the regex to accept any expression value for `name` and `version`. The load-bearing positive signal is the presence of both `name` and `version` keys in the constructor's first-arg object. Mangled-id length cap also needs widening (some prod widgets use 1-char ids like `c`, others use 2-3 like `Bh`, `Aa`, `Cc`).

Replacement regex (Option A — preferred):

```regex
new [a-zA-Z_$][a-zA-Z0-9_$]{0,20}\(\s*\{[^}]{0,200}\bname\s*:[^,}]{0,100}\bversion\s*:
```

Rationale:
- `[a-zA-Z_$][a-zA-Z0-9_$]{0,20}` — mangled id widening from `{0,5}` to `{0,20}` (still bounded; minified ids virtually never exceed 6-8 chars but cap is generous)
- `\(\s*\{` — opening of constructor call's first-arg object (whitespace tolerant)
- `[^}]{0,200}` — anything inside the object UP TO `}`, capped at 200 chars (prevents catastrophic backtracking; covers any realistic prefix between `{` and the `name` key)
- `\bname\s*:` — `name` key (word-boundary ensures we don't match e.g. `firstName`)
- `[^,}]{0,100}` — any value-expression for `name`, capped at 100 chars (covers `"literal"`, `"prefix"+t`, `getName()`, `template ${var}`)
- `\bversion\s*:` — `version` key

This regex matches:
- ✓ Cycle-1 unit-test positives: `new yl({name:"cost-coach-cost-summary",version:"1.0.0"})` (literal values)
- ✓ Cycle-1 unit-test positives: `new App({name: "tool", version: "1.0.0"})` (whitespace tolerance)
- ✓ Real prod: `new yl({name:"cost-coach-"+t,version:"1.0.0"})` (concatenation)
- ✓ Real prod: `new c({name:"cost-coach-"+t,version:"1.0.0"})` (1-char mangled id)
- ✗ Cycle-1 unit-test negatives: `new Date(2026,1,1)` (no `name`/`version` keys) — `[^}]{0,200}` would match `2026,1,1` but `\bname\s*:` won't fire
- ✗ Cycle-1 unit-test negatives: `new URL("http://x")` (no object payload) — `\(\s*\{` requires opening brace

**G2 false-positive risk:** the widened regex could match unrelated `new Foo({name: ..., version: ...})` calls on benign code. The new property test `prop_g2_cycle2_no_false_positive_on_unrelated_keys` proves the regex does NOT match arbitrary `new <Class>({<random keys>})` shapes — it requires both `name` and `version` to appear, which is far less likely to occur on benign code than just `name`.
</bug_b>

    ```rust
    /// Best-effort comment stripper. Strips HTML, JS block, and JS line comments
    /// from `src` while preserving comment delimiters that appear INSIDE JS
    /// string literals.
    ///
    /// Cycle-2 fix: the cycle-1 implementation used unconditional regex
    /// replacement, which destroyed thousands of bytes of code in cost-coach
    /// prod bundles where a JS string literal contained `"/*..."` (a CSP
    /// frame-src directive value) — the regex saw that `/*` as a block-comment
    /// opener and matched the next `*/` it found, far away. See
    /// `tests/fixtures/widgets/bundled/real-prod/CAPTURE.md` for evidence.
    ///
    /// State machine: outside-strings (`out`) → block_comment / line_comment /
    /// single_string / double_string / template_string. Inside any string state,
    /// `/*` and `//` are NOT comment delimiters and are preserved in the output.
    /// Escape sequences (`\<any>`) are consumed as a unit inside string states.
    ///
    /// Accepted simplification: template-literal interpolations `${...}` are NOT
    /// recursively parsed — comment delimiters inside an interpolation expression
    /// are still treated as part of the template string. Real-world prod bundles
    /// rarely embed signal literals inside template interpolations.
    ///
    /// HTML comments are stripped by the existing `html_comment_re` regex BEFORE
    /// this function runs (in `extract_inline_scripts`); this function operates
    /// on JS bodies post-HTML-strip.
    #[allow(dead_code)]
    fn strip_js_comments(src: &str) -> String {
        // First strip HTML comments (they don't appear inside JS string literals
        // in any realistic bundle — keeping the regex pass for them keeps the
        // cycle-1 invariant that <!--...--> is removed).
        let after_html = html_comment_re().replace_all(src, "");

        let bytes = after_html.as_bytes();
        let mut out = String::with_capacity(bytes.len());
        let mut i = 0;
        let n = bytes.len();

        // States:
        //   0 = outside strings/comments
        //   1 = inside /* ... */ block comment
        //   2 = inside // line comment
        //   3 = inside '...' single-quoted string
        //   4 = inside "..." double-quoted string
        //   5 = inside `...` template-literal string
        let mut state: u8 = 0;

        while i < n {
            let c = bytes[i];
            match state {
                0 => {
                    if c == b'/' && i + 1 < n && bytes[i + 1] == b'*' {
                        state = 1;
                        i += 2;
                        continue;
                    }
                    if c == b'/' && i + 1 < n && bytes[i + 1] == b'/' {
                        state = 2;
                        i += 2;
                        continue;
                    }
                    if c == b'\'' {
                        state = 3;
                        out.push(c as char);
                        i += 1;
                        continue;
                    }
                    if c == b'"' {
                        state = 4;
                        out.push(c as char);
                        i += 1;
                        continue;
                    }
                    if c == b'`' {
                        state = 5;
                        out.push(c as char);
                        i += 1;
                        continue;
                    }
                    out.push(c as char);
                    i += 1;
                }
                1 => {
                    if c == b'*' && i + 1 < n && bytes[i + 1] == b'/' {
                        state = 0;
                        i += 2;
                        continue;
                    }
                    i += 1;
                }
                2 => {
                    if c == b'\n' || c == b'\r' {
                        state = 0;
                        out.push(c as char); // preserve newline so subsequent lines aren't merged
                        i += 1;
                        continue;
                    }
                    i += 1;
                }
                3 | 4 | 5 => {
                    // Escape: consume the backslash + next char as a unit.
                    if c == b'\\' && i + 1 < n {
                        out.push(c as char);
                        out.push(bytes[i + 1] as char);
                        i += 2;
                        continue;
                    }
                    let close = match state {
                        3 => b'\'',
                        4 => b'"',
                        5 => b'`',
                        _ => unreachable!(),
                    };
                    if c == close {
                        state = 0;
                        out.push(c as char);
                        i += 1;
                        continue;
                    }
                    out.push(c as char);
                    i += 1;
                }
                _ => unreachable!(),
            }
        }

        out
    }
    ```
