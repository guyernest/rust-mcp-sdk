<auth_architecture_note>
## Outgoing auth: static providers vs the per-request passthrough provider (H1 / D-05 / Pitfall 1)

`[backend.auth]` has SIX modes — `none` + five authenticated variants (`api_key`,
`bearer`, `basic`, `oauth2_client_credentials`, `oauth_passthrough`). They split into
TWO construction strategies:

- **Static providers** (`none`/`api_key`/`bearer`/`basic`/`oauth2_client_credentials`):
  fully determined by `config.toml` (operator credentials / `${ENV}` secrets). Built
  ONCE at startup via `create_auth_provider(&AuthConfig)` and shared as
  `Arc<dyn HttpAuthProvider>`. They ignore any inbound MCP client token.

- **Per-request passthrough** (`oauth_passthrough`): needs the INCOMING MCP client
  token for EACH request — it cannot be fully built at startup. Per CONTEXT D-05,
  `oauth_passthrough` "bridges via the pmcp `AuthContext` token-capture pattern";
  RESEARCH Pitfall 1 cites the reference bridge `pmcp_server.rs:37-64` — the inbound
  `TokenCaptureAuthProvider` stores the client token in `AuthContext`, and the
  outbound provider forwards that captured token as the outgoing `Authorization`
  (or configured `target_header`).

**Design consequence for `apply` (this plan):** `HttpAuthProvider::apply` accepts an
OPTIONAL inbound token so a SINGLE trait serves both strategies:
`async fn apply(&self, headers, query, inbound_token: Option<&str>)`. Static providers
ignore `inbound_token`; `OAuthPassthroughAuth` forwards it (and, if `incoming_token`
was captured at construction via `create_passthrough_auth_provider`, prefers the
per-request `inbound_token` arg when present). Plan 04 (`HttpCodeExecutor`) carries
the per-request token to `apply`; Plan 06 wires the inbound `TokenCaptureAuthProvider`
so the captured token lands in `AuthContext` and is threaded from
`RequestHandlerExtra` into the outgoing `apply` call. DO NOT defer the
`oauth_passthrough` variant — design the optional-token path in now.
</auth_architecture_note>
