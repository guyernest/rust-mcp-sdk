<objective>
Lift the `HttpCodeExecutor` seam (impl `pmcp_code_mode::HttpExecutor`, OAPI-05) and
perform the one real refactor of the phase: generalize the toolkit's
`code_mode_tools_from_executor` + its two handlers from hardcoded `Arc<SqlCodeExecutor>`
+ `"sql"` flavor to `Arc<dyn CodeExecutor>` + a `ValidationFlavor` enum (OAPI-10 / D-02),
so SQL and OpenAPI share ONE wiring function. The existing SQL path MUST stay green.

Purpose: This single seam feeds BOTH Code Mode (via JsCodeExecutor) and script tools
(Plan 05). The generalization unblocks the OpenAPI code-mode wiring without breaking SQL.
Output: HttpCodeExecutor + a backend-agnostic `code_mode_tools_from_executor`.

NOTE (H2 — feature gate): `HttpCodeExecutor` references `pmcp_code_mode::HttpExecutor`,
which lives behind `pmcp-code-mode/js-runtime`. The bare `code-mode` feature does NOT
forward `js-runtime`, so this plan MUST gate `HttpCodeExecutor` (and its tests) under
`#[cfg(feature = "openapi-code-mode")]` (the Plan 90-01 umbrella) — NOT
`all(feature="http", feature="code-mode")`. The earlier draft's gate is the same bug the
plan-checker fixed for 01/05/06; it is fixed here.

NOTE (H1 — per-request passthrough): `HttpCodeExecutor` must be able to carry the
per-request inbound MCP token so `oauth_passthrough` works (the Code-Mode / script-tool
path is exactly where the passthrough token flows). See Task 1.
</objective>
