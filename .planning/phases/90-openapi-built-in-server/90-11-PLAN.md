<objective>
`${VAR}` / `env:VAR` secret expansion (added in 90-07) is wired ONLY into the `api_key`
arm of `create_auth_provider`. The `bearer` (`token`), `basic` (`password`), and
`oauth2_client_credentials` (`client_secret`) arms pass their credential VERBATIM, so
`token = "${GITHUB_PAT}"` ships the literal `${GITHUB_PAT}` to the backend — a silent
auth failure. The Bearer field doc even advertises `${VAR}` support the code does not
deliver.

Purpose: Close the WR-/altitude finding (cross-variant secret resolution). Generalize
the existing `api_key` env-ref discipline into ONE chokepoint applied to every
credential field as it enters `create_auth_provider`. OAPI-03 (the 5/6-variant auth
surface) is only correct once every variant resolves its secret consistently.

Output: A single `resolve_secret_ref` chokepoint (consolidating the brace/env parsing
currently duplicated between `resolve_api_key_value` in auth.rs and `expand_braced_var`
in code_mode.rs) applied to token/password/client_secret/api_key; unit tests per
variant proving `${VAR}` and `env:VAR` resolve and the literal never reaches the
provider.
</objective>

<resolution_semantics>
The existing `resolve_api_key_value` (the precedent) maps unset/empty refs to `""`
(so an optional api_key is OMITTED). For bearer/basic/oauth2 the existing arms already
treat an EMPTY resolved credential as "no auth" (Bearer empty token -> NoAuth; Basic
empty user+pass -> NoAuth; OAuth2 empty client_id/secret -> NoAuth). So reusing the
same "unset -> empty string" rule is consistent: a `required:true` bearer with an unset
`${VAR}` collapses to NoAuth (no literal placeholder on the wire), which is the correct
failure mode (no auth sent) rather than shipping `${VAR}`. Do NOT change the
empty-credential->NoAuth branches; only resolve the value BEFORE those checks.
</resolution_semantics>
