<objective>
Make the `oauth_passthrough` per-request token path WORK at runtime. Today the seam
is architecturally complete but dead: `ScriptToolHandler::handle` and
`ExecuteCodeHandler::handle` both ignore `RequestHandlerExtra`, executing over a
fixed dispatch-time `HttpCodeExecutor` with `inbound_token: None`. Additionally
`dispatch()` installs `MissingTokenAuth`/`NoAuth` for passthrough backends — neither
forwards the captured token to `target_header`, so even a threaded token would not
reach the backend.

Purpose: Close VERIFICATION truths #3 and #8 (OAPI-03 / OAPI-05) — the single
root-cause blocking gap of Phase 90. Resolve the cross-crate constraint by moving the
per-request derivation INTO the toolkit (the handlers live in the toolkit; the binary
cannot be a dependency of the toolkit).

Output: A toolkit-resident `request_executor_from_extra` seam called by both handlers;
`dispatch()` builds an `OAuthPassthroughAuth` provider for passthrough backends; an
end-to-end wiremock test proving the forwarded `Authorization` header reaches the
backend through the handler path for both `required:true` and `required:false`.
</objective>

<architecture_decision>
CROSS-CRATE CONSTRAINT (mandatory): `request_executor` currently lives in the BINARY
(`assemble.rs`). The handlers that must call it (`ScriptToolHandler`,
`ExecuteCodeHandler`) live in the TOOLKIT. The toolkit cannot depend on the binary.
Therefore the per-request derivation MUST live in the toolkit.

Chosen design (resolves the JsCodeExecutor private-field problem WITHOUT breaking the
SQL path):

1. Toolkit free fn `request_executor_from_extra(base: &HttpCodeExecutor, extra: &RequestHandlerExtra) -> HttpCodeExecutor`
   in `code_mode.rs` (feature `openapi-code-mode`): reads `extra.auth_context().and_then(|c| c.token.clone())`
   and returns `base.clone().with_inbound_token(token)`. This is the toolkit-resident
   replacement for the binary's dead `request_executor`.

2. `ScriptToolHandler` already holds `http_exec: HttpCodeExecutor` directly. In `handle`,
   replace `self.http_exec.clone()` with `request_executor_from_extra(&self.http_exec, &extra)`.
   Rename the param `_extra` -> `extra`. The pre-compiled `self.plan` reuse stays unchanged.

3. `ExecuteCodeHandler` (the OpenAPI path) currently holds a type-erased
   `Arc<dyn CodeExecutor>` (a `JsCodeExecutor` whose inner `http` is private), so it
   cannot rederive per-request. Fix WITHOUT breaking SQL: add an OPTIONAL per-request
   derivation source to `ExecuteCodeHandler`:
     enum ExecSource {
       Static(Arc<dyn CodeExecutor>),                       // SQL path (SqlCodeExecutor) — unchanged behavior
       PerRequestHttp { base: HttpCodeExecutor, exec_config: ExecutionConfig },  // OpenAPI path
     }
   In `handle`, after token/hash verification:
     - Static(e)  => e.execute(code, variables)
     - PerRequestHttp { base, exec_config } =>
         let http_exec = request_executor_from_extra(base, &extra);   // rename _extra -> extra
         JsCodeExecutor::new(http_exec, exec_config.clone()).execute(code, variables)
   The `PerRequestHttp` variant is feature-gated `openapi-code-mode` (HttpCodeExecutor/
   JsCodeExecutor are only in scope there); the `Static` variant stays under bare
   `code-mode` so the SQL build is unaffected.

4. Add a toolkit entry point for the OpenAPI per-request path:
   `code_mode_http_tools_from_executor(builder, config, base: HttpCodeExecutor, exec_config: ExecutionConfig, flavor)`
   (feature `openapi-code-mode`) that constructs `ExecuteCodeHandler` with the
   `PerRequestHttp` source. The existing `code_mode_tools_from_executor` keeps the
   `Static` source (SQL path) unchanged.

5. `dispatch()` MUST install an `OAuthPassthroughAuth` provider for passthrough backends
   so the threaded token is actually forwarded to `target_header`. Replace
   `create_auth_provider(&backend.auth)` with `create_passthrough_auth_provider(&backend.auth, None)`
   (None = no construction-time token; the per-request token arrives via `apply`'s
   `inbound_token`). For all non-passthrough configs `create_passthrough_auth_provider`
   delegates to `create_auth_provider`, so static-auth behavior is unchanged.

6. `assemble.rs::build_server` switches the OpenAPI code-mode wiring from
   `code_mode_tools_from_executor(.., JsCodeExecutor wrapped Arc, ..)` to
   `code_mode_http_tools_from_executor(.., http_exec, exec_config, ValidationFlavor::OpenApi)`.
   Remove the now-dead binary `request_executor` (and its unit test) OR have it delegate
   to the toolkit `request_executor_from_extra` re-export — prefer REMOVAL (dead code is
   the WR-01 root cause). Update `assemble.rs` imports accordingly.

The `required:false` rule: `OAuthPassthroughAuth::apply` already forwards the token when
present and returns `Ok(())` (no header) when absent. Installing `OAuthPassthroughAuth`
(step 5) for `required:false` — instead of the current `NoAuth` — is what makes the
"forward when present" behavior real. Do NOT install `NoAuth` for passthrough.
</architecture_decision>
</context>
