<objective>
Wire the shared `task_dispatch` unit (built and proven in Plan 01) into the
HTTP-facing `Server` and `ServerBuilder`: add the `task_store`/`task_router`
backend fields + setters (HTASK-01), run the shared capability rule at `build()`
so a store-backed `Server` advertises `tasks` (HTASK-01), replace the `tasks/*`
hard-reject at `mod.rs:1166-1172` with delegation to `route_tasks_endpoint`
(HTASK-02), and bolt the create-path onto `Server::handle_call_tool` as a
post-execution branch keyed on `req.task` (HTASK-02). Resolve the two
planning-spike seams the research flagged: where the create-path branch lives,
and the `Result<Value>` ↔ `JSONRPCResponse` response-shape adapter.

The newly-reachable `tasks/*` path is the highest-risk surface in the phase. This
plan makes auth reachability explicit: the tasks/* dispatch is inserted at the
post-auth assembly layer (adapter (a)) so it cannot bypass the auth validation the
rest of the dispatcher performs, and the TaskSupport matrix + cross-owner isolation
are tested rather than assumed.

Purpose: `Server` is what `StreamableHttpServer` calls; until it serves the
lifecycle, pmcp.run must shim through `ServerCore::handle_request`. This plan
removes the rejection and makes `Server::builder().tool(t).task_store(store).build()`
serve tasks — using the SAME shared unit, never a second copy.

Output: `src/server/mod.rs` with the backend fields/setters, the build-time
capability rule, the tasks/* delegation, the create-path branch, plus unit +
property + non-task regression + TaskSupport-matrix + cross-owner-IDOR tests
proving HTASK-01/02/04.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → Server::handle_request | Untrusted JSON-RPC over real HTTP now reaches the task lifecycle for the FIRST time (previously rejected). The new path must enforce the same auth + owner-scoping the ServerCore path already does, and must be inserted DOWNSTREAM of auth resolution. |
