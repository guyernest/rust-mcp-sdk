# Phase 102 PRD — Lift the task lifecycle onto the high-level `Server` / HTTP path

**Milestone:** v2.10 SDK DX — follow-up to Phase 101
**Source:** Phase 101 execution discovery (TASKDX acceptance test had to pair a `pmcp::Client` with `ServerCore` over an in-process duplex transport because the HTTP path can't serve `tasks/*`). Verified against pmcp 2.10.0.
**Goal:** Make the SDK's `tasks/*` lifecycle available over the **high-level `Server` and `StreamableHttpServer`**, so an HTTP-hosted server (e.g. pmcp.run's Lambda servers) can serve task-based tools **without driving `ServerCore::handle_request` directly via a shim**. The eventual end state for consumers is: `Server::builder().tool(t).task_store(store).build()` → serve over `StreamableHttpServer` → tasks just work.

---

## Background — what Phase 101 delivered, and the gap it surfaced

Phase 101 made tools-as-Tasks correct-by-construction, but the entire task lifecycle (the `task_store` plumbing, the endpoint-backed capability rule, the task-augmented create-path, and the `tasks/get | result | list | cancel` dispatch) lives **only on `ServerCore` / `ServerCoreBuilder`**. The SDK has **two parallel request dispatchers**:

- `ServerCore::handle_request` (`src/server/core.rs`, `impl ProtocolHandler for ServerCore` at ~`:884`) — **has** the task dispatch.
- `Server::handle_request` (`src/server/mod.rs`) — the **HTTP-facing** dispatcher. It **hard-rejects** `tasks/*`: `src/server/mod.rs:1166-1169`
  ```
  ClientRequest::TasksGet(_) | TasksResult(_) | TasksList(_) | TasksCancel(_)
      => Err(crate::Error::protocol(...))
  ```

`StreamableHttpServer` wraps `Arc<Mutex<Server>>` and dispatches via `Server::handle_request` (`src/server/streamable_http_server.rs:1172`, `:1391`). So **HTTP servers cannot serve `tasks/*`** today; the high-level `Server` has no `TaskStore` field and rejects the methods.

**Consumer impact (pmcp.run):** their servers are HTTP-hosted Lambdas. To adopt the SDK task path in 2.10.0 they must call `ServerCore::handle_request` directly from their Lambda handler (a working but non-obvious shim). This phase removes that shim.

---

## Verified current architecture (read against pmcp 2.10.0 — do NOT re-derive blind)

- `Server` struct: `src/server/mod.rs:315`; `ServerBuilder`: `:1878`. `Server` builds a `tool_infos` cache at construction (`:3568`) mirroring `ServerCore`.
- `Server` and `ServerCore` are **separate** implementations with **duplicated dispatch bodies** — this duplication is exactly why the task work landed on one and not the other. Phase 102 must **share** the task lifecycle, not add a third copy.
- The Phase 101 task machinery to lift / share lives in:
  - `src/server/builder.rs` — `task_store(Arc<dyn TaskStore>)`, `with_task_store` (legacy `TaskRouter`), and `apply_tasks_capability_rule()` (endpoint-backed capability + `Required`-without-backend → `build()` Err).
  - `src/server/core.rs` — the `ToolCallOutcome::TaskCreated { task_id, task_value, result }` create-path (store-minted id written to wire `task.taskId` + `_meta`), and the `tasks/*` routing block (store-first, `TaskRouter` fall-through, `-32002` pending error).
  - `src/server/task_store.rs` — the `TaskStore` trait (`create`/`get`/`update_status`/`list`/`cancel` + `set_result`/`get_result`/`supports_results`), `InMemoryTaskStore`.
  - `resolve_task_owner(auth_context)` (owner-scoping) is consumed by the `tasks/*` routing.
- `ServerCore: ProtocolHandler`; `Server::handle_request` is its own inherent dispatch (the HTTP loopback test harness lives at `tests/workflow_prompt_e2e_test.rs:54-97` — `StreamableHttpServer` + `StreamableHttpTransport` on `127.0.0.1:0`).

## Requirements

### HTASK-01 — High-level `ServerBuilder`/`Server` gains the task backend + capability rule
The high-level `ServerBuilder` accepts a `TaskStore` (and honors `with_task_support` tools) with the SAME endpoint-backed capability behavior Phase 101 added to `ServerCoreBuilder`: registering a store auto-advertises the top-level `tasks` capability; a `TaskSupport::Required` tool with no backing store/router makes `build()` return `Err`; an explicitly-configured capability set is never clobbered. The capability rule is the SAME implementation Phase 101 already wrote (shared, not re-derived).

### HTASK-02 — `Server::handle_request` serves the full task lifecycle, sharing logic with `ServerCore`
Replace the `tasks/*` hard-reject at `mod.rs:1166-1169` with the real lifecycle: the task-augmented `tools/call` create-path (store-minted id, `_meta` related-task, terminal-result persistence) AND `tasks/get | result | list | cancel` (store-first typed serialization, `TaskRouter` fall-through, `-32002` specified pending error, owner-scoping via `resolve_task_owner`). **The task-lifecycle logic MUST be SHARED with `ServerCore`** — factored into one reusable unit both dispatchers call — NOT a divergent second copy. (The two-dispatcher drift is the root cause this phase exists to retire.)

### HTASK-03 — End-to-end over `StreamableHttpServer`, proven by a real HTTP round-trip
A server built with `Server::builder().tool(t).task_store(store).build()` and served via `StreamableHttpServer` serves the complete lifecycle over real HTTP. Acceptance is a **live HTTP loopback round-trip** (real `StreamableHttpServer` + `StreamableHttpTransport` on `127.0.0.1:0`, the `workflow_prompt_e2e_test` harness) driving `initialize → call(task) → tasks/get (poll) → tasks/result` and asserting the same invariants as Phase 101: non-empty typed `CallToolResult.content`; `CreateTaskResult.task.taskId == tasks/get id == _meta.relatedTask.taskId`; `initialize` advertises `tasks`; pending `tasks/result` returns the specified error. This **replaces** the in-process duplex harness Phase 101 was forced to use.

### HTASK-04 — No regressions, additive API, consumer shim removed
No regression to the `ServerCore` task path (all Phase 101 tests green) or existing `Server`/HTTP behavior. No breaking change to public wire shapes (`Task`/`GetTaskResult`/`CallToolResult`/`CreateTaskResult`/`ServerCapabilities`) or to the `Server`/`ServerBuilder` public API (additive only). `#[cfg(not(target_arch="wasm32"))]` boundary preserved. ALWAYS coverage (unit/property/integration/example/doctest) + a worked HTTP example (e.g. `s46_http_tool_as_task`). `make quality-gate` AND `make doc-check` green. Net result: an HTTP-hosted server serves `tasks/*` through the high-level `Server` with **no `ServerCore::handle_request` shim**.

## Out of scope / scope fences (LOCKED)

- **Do NOT change the `tasks/*` wire contract** — Phase 101 froze it; this phase only changes *which dispatcher* serves it.
- **Do NOT duplicate the task logic** into a divergent third copy — the deliverable is a SINGLE shared implementation used by both `Server` and `ServerCore`. (A planner proposing copy-paste fails the goal.)
- **Do NOT break the existing `ServerCore` path or the `TaskRouter` fallback** — both stay green.
- **Out of scope:** durable/persistent `TaskStore` backends (DynamoDB etc. — that is the consumer's `TaskStore` impl, not the SDK's); task streaming over SSE/WebSocket; the legacy `pmcp-tasks` crate; any change to the high-level vs low-level builder split beyond adding the task backend.
- Keep WASM out (task path is non-wasm, as in Phase 101).

## Key design question for the planner (recommend, then confirm against the code)

How to **share** the task lifecycle between `Server` and `ServerCore` with minimal churn. Options:
- **(A, recommended)** Extract the task create-path + `tasks/*` endpoint dispatch into a free-standing `server::task_dispatch` unit parameterized by `(task_store, task_router, tool_infos, resolve_owner, …)`, called by both `handle_request` bodies. Cleanest; directly retires the drift.
- **(B)** Have `Server` own/delegate task methods to an internal `ServerCore` it constructs. Lower code movement but couples the two types and may double-handle `tools/call`.
- **(C)** Duplicate into `Server` — **rejected** (reintroduces the exact drift).
The hard part is the create-path: it is entangled with `tools/call` result handling and the `tool_infos`/`task_support` lookup. The planner must map the shared seam against `core.rs` (the `TaskCreated` branch + tasks routing) and `mod.rs` (the `Server` `tools/call` + dispatch) before committing.

## Success criteria (what must be TRUE)

1. `Server::builder().tool(t, task_tool).task_store(store).build()?` served via `StreamableHttpServer` serves `initialize → call(task) → tasks/get → tasks/result` over real HTTP — a live HTTP loopback test drives it green.
2. The high-level `initialize` advertises `tasks` when a store/router backs it; an unbacked `Required` tool errors at `build()`.
3. The task lifecycle is a SINGLE shared implementation used by both `Server` and `ServerCore` (no duplicated dispatch body); both paths' tests pass.
4. All Phase 101 `ServerCore` tests (tasks/result-from-store, store-minted id, TaskRouter fallback, pending error) still green; no wire-shape change (`src/types/tasks.rs` Task/GetTaskResult/CreateTaskResult diff = none).
5. Worked HTTP example + ALWAYS coverage; `make quality-gate` and `make doc-check` green; a new minor `pmcp` version is publishable.
6. A pmcp.run-shaped HTTP server serves tasks with NO `ServerCore::handle_request` shim (demonstrated by the example/test using only `Server` + `StreamableHttpServer`).

## Acceptance gate (carried from Phase 101)
Protocol-shape requirements are "resolved" ONLY via a live round-trip — here specifically over the **real HTTP transport**, not the in-process duplex shim and not a hand-built fixture.

## Downstream
Once shipped, the pmcp.run migration note's "drive `ServerCore` directly" shim section is removed — they build a high-level `Server` with a `task_store` and serve over HTTP unchanged.
