Extract Phase 101's `ServerCore`-only task machinery into ONE shared
`src/server/task_dispatch.rs` unit (Option A from the research — the only sharing
strategy that retires the two-dispatcher drift), then refactor `ServerCore` and
`ServerCoreBuilder` to call it with ZERO behavior change. This is the foundational
"share" half of the phase: nothing wires into `Server` yet, but the shared seam
exists and is proven against the Phase 101 test suite.

Purpose: HTASK-02 and HTASK-01 both mandate that the task lifecycle be a SINGLE
shared implementation, never a divergent second copy. Building and proving the
shared unit FIRST (against the already-green `ServerCore` path) means Plan 02 wires
`Server` into a unit that is known-correct. Re-implementing (drift) is the failure
mode this plan structurally prevents.

The create-path gate (`maybe_build_task_created`) is the seam Plan 02 leans on most,
so this plan makes it a TRULY shared, self-enforcing helper: it takes an explicit
`task_requested: bool` and enforces the COMPLETE gate internally (no reliance on the
caller having already checked a precondition), and it is proven HERE with `#[cfg(test)]`
truth-table unit tests rather than waiting for Plan 02.

Output: `src/server/task_dispatch.rs` (new shared unit, wasm-gated, with a `#[cfg(test)]`
gate truth-table module), `ServerCore` and `ServerCoreBuilder` refactored to delegate to
it, module registered in `mod.rs`. All Phase 101 tests still green.

The capability rule to lift (src/server/builder.rs:881-904) is currently a
ServerCoreBuilder method reading self.task_store / self.task_router /
self.tool_infos / self.capabilities. The two builders hold tool_infos at
DIFFERENT lifecycle points (ServerCoreBuilder: field filled at .tool(); ServerBuilder:
local built inside build()), so the rule MUST become a free fn over explicit params:

    pub(crate) fn apply_tasks_capability_rule(
        capabilities: &mut ServerCapabilities,
        tool_infos: &HashMap<String, ToolInfo>,
        has_backend: bool, // task_store.is_some() || task_router.is_some()
    ) -> Result<()>

default_tasks_capability() (builder.rs:849-859) is the FROZEN ServerTasksCapability
shape the client assert_capability expects — call it from the free fn or move it
alongside; do not re-derive its JSON.

resolve_task_owner (core.rs:972-990) is an inherent ServerCore method (NOT a free fn,
despite CONTEXT phrasing) reading self.task_router then self.task_store. It depends
ONLY on (&task_router, &task_store, Option<&AuthContext>).

ToolCallOutcome::TaskCreated { task_id, task_value, result } enum is at core.rs:299-314;
its dispatch branch is core.rs:1222-1235; the create-path gate is inside
handle_call_tool at core.rs:589-640 (return ToolCallOutcome::TaskCreated at :626).

build_task_created_response (core.rs:1007-1064) mints the wire envelope; the tail
(core.rs:1053-1063) sets task.taskId AND _meta.relatedTask.taskId to the SAME
store-minted id (D-STORE-MINTS-ID three-way invariant).

handle_tasks_result (core.rs:1076-1126): store-first → router → -32002 pending →
-32601 no-backend (FROZEN error codes). extract_terminal_result (core.rs:458-468).

tasks/* arms (core.rs:1312-1421): TasksGet/TasksList/TasksCancel each a pure fn of
(task_store, task_router, resolve_owner, params, auth); -32601 "Tasks not enabled"
when neither backend; -32603 on store error.

success_response / error_response: ServerCore's JSON-RPC envelope builders (the fns that
wrap a Value or an (code, message) into a `JSONRPCResponse`). These MUST become single-source
in task_dispatch (Concern #3 / Codex MEDIUM "envelope drift"): move the bodies ONCE into
task_dispatch and have ServerCore call into them — do NOT duplicate the bodies. Locate the
current associated fns on ServerCore (search `fn success_response` / `fn error_response` in
core.rs) and re-home them.

THE SHARED GATE CONTRACT (Concern #1 — "truly shared, not shared-ish"):

    pub(crate) async fn maybe_build_task_created(
        &self,
        id: RequestId,
        value: &Value,
        task_support: Option<TaskSupport>,
        task_requested: bool,            // explicit — NOT a caller precondition
        auth_context: Option<&AuthContext>,
    ) -> Option<JSONRPCResponse>

The helper itself enforces the FULL gate; the caller passes raw facts, never a pre-filtered
precondition. Returns Some(envelope) IFF ALL hold:
  task_requested == true
  && self.task_store.is_some()                         // backend present
  && matches!(task_support, Some(TaskSupport::Required) | Some(TaskSupport::Optional))
  && value carries BOTH a taskId and a status               // task-shaped value
Otherwise returns None ("fall through to a normal CallToolResult"). TaskSupport::Forbidden
and None, an absent task field, an absent backend, or a non-task-shaped value ALL yield None
with no error leak.
