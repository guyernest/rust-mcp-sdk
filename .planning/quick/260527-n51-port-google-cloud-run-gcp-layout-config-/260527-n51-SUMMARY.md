# Quick Task 260527-n51: Port Google Cloud Run config + multi-crate Dockerfile + unified --target Summary

Ported the 0.6.x Google Cloud Run `[gcp]`+`[layout]`+`[runtime]` config schema, the surgical multi-crate-isolated Dockerfile generator (byte-matching the reference golden), and the local `docker buildx` deploy path into 0.14.0; made `DeployConfig.aws` an `Option` behind a panic-on-None `aws()` accessor; and unified `--target` to accept a known backend type OR a configured named target (hard-erroring otherwise). All pre-existing aws-lambda / pmcp-run / cloudflare-workers shapes still parse and behave unchanged.

## Deviations from Plan

### Reference-divergence resolution (planned, documented)
**`aws()` panic-vs-static-default:** The locked-decision text said "stored/static default"; the actual 0.6.x reference uses `.expect()` (panic-on-None). Per "Port 0.6.x semantics verbatim" the **reference won** — `aws()` panics with "deploy.toml is missing the required [aws] block …" and the always-runs resolver reads the `Option` directly (never via `aws()`), so gcp-only configs never panic.

### `.aws` migration count (vs the "18" in the original issue text)
The actual migrated sites are **7 AWS reads + 2 resolver reads + 1 resolver test write = 10** — NOT 18. The 7 AWS reads moved to `config.aws()` (deploy/deploy.rs ×4, aws_lambda deploy.rs/init.rs/mod.rs). The 2 resolver reads + 1 test write read the `Option` directly (the resolver runs for gcp-only configs and must not panic). The `secrets/registry.rs providers.aws.*` and `secrets/config.rs .join(".aws")` sites are DIFFERENT structs / a filesystem path literal and were correctly left untouched.

### RuntimeConfig ported (planned, required)
`RuntimeConfig` (`base` + `apt_packages`) was ported as part of the same #258/#259/#260 feature set — it is not separable from the dockerfile port; the distroless-default/debian-opt-in golden tests depend on it.

### Rule 3 — lib-surface mount for the integration test
The bin-only `targets::*` tree (which owns the Dockerfile generator) cross-depends on `commands::*` and is NOT in the lib surface that external integration tests link against. To let `tests/cloud_run_local_build.rs` drive the **real** `generate_dockerfile` (not a hand-copied Dockerfile), `src/lib.rs` now mounts `deployment::google_cloud_run::{env,dockerfile}` (and re-exports `DeployConfig`/`LayoutConfig`) via `#[path]`, mirroring the established `deployment::{config,iam,widgets}` lib-mount pattern. The bin `deployment::mod` re-export was narrowed to `{DeployConfig, LayoutConfig}` to keep it warning-clean.

### Rule 1 — `doc_lazy_continuation` fix
The ported `BUILDER_APT_LAYER` doc comment had a line starting with `+ libssl-dev` that rust-1.95 clippy flagged as a lazy list-item continuation. Reworded to ``` `pkg-config` and `libssl-dev` ``` (the only NEW default-clippy hit my port introduced; fixed so the default-clippy error count stayed at the pre-existing baseline of 14).
